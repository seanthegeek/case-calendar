# Scripts (`scripts/`)

Maintainer-facing reference for the helper scripts in this directory — the
caveats and the reasoning behind each one. For operator-facing, runnable how-to
(flags, examples, the dev → prod workflow), see [README.md](README.md).

## One-shot maintenance scripts (`scripts/`)

- **`scripts/reprocess_entries.py <entry_id>...`** — re-run the LLM pipeline against specific stored entries (by CourtListener `entry_id`, not docket position). Builds a synthetic entry from the local store, clears the entry's fingerprint to bypass dedup, and calls `CaseSyncer.process_entry`. Useful after prompt changes when you want to see the new behavior on existing rows. CAUTION: the LLM can allocate fresh `hearing_key`s if it interprets a previously-vague row as multiple specific events — duplicates may need manual cleanup.
- **`scripts/classify_significance.py`** — classifies hearing significance using a focused single-question LLM prompt (no full action extraction, so no risk of the duplicate-key issue above). Default is dry-run on NULL-significance rows; `--all` widens to every row, `--apply` writes verdicts back to the DB.
- **`scripts/heal_proceeding_notes.py`** — deterministic backfill (no LLM, no CourtListener) for hearings whose `notes` regressed to a pre-hearing administrative notice (clerk's notice of Zoom access / courtroom change / scheduling) instead of the record of what happened. Calls `sync.heal_proceeding_notes`, which rebuilds each affected row's notes from its own record source entries (minute entry / transcript / clerk's notes of proceedings held), touching only rows whose notes are empty or an administrative notice. The sync-time MARK_HELD-supersede + dedupe-aware-notes rules only prevent NEW regressions; rows already collapsed in the store (their rich-notes sibling deleted by a dedupe merge) need this sweep because re-running sync can't recover them. Dry-run by default; `--apply` writes back. Only rewrites the `notes` column.
- **`scripts/heal_drifted_keys.py`** — deterministic canonicalization (no LLM, no CourtListener) for hearings AND deadlines left with a `base-<digits>` key-drift suffix (the "Sentencing Lytvynenko 2" class — the cross-record drift the dedupe sweeps collapse, where the survivor historically kept the `-N` key because it carried more `source_entry_ids`). Calls `sync.heal_drifted_keys` (hearings) and `sync.heal_drifted_deadline_keys` (deadlines), which repair ONLY provable-drift rows by two signals: (1) the survivor's `audit_notes` record it absorbed its own suffix-free base AND that base no longer exists → rename `base-N` → `base` (the new row starts with a NULL `m365_event_id` so the next emit re-creates the Graph event cleanly — the old key's gcal/Graph events orphan, the one-time cost of a rename; for hearings the recomputed key-derived title is name-aware per `_fallback_hearing_title`, for deadlines it's the plain humanization); (2) the suffix-free `base` still exists at the same slot (`starts_at_utc` for hearings, `due_at_utc` for deadlines) in the same logical PACER group → delete the `-N` row after folding its `source_entry_ids` onto `base`. A meaningful trailing number (sequential status conferences, trial days, dates, distinct same-day deadlines, `-docket-<id>` records) is NEVER touched — it has neither signal (its base, if any, sits at a different slot and carries no absorption note). The deadline half is NOT the forbidden deterministic same-slot deadline sweep — see the "Deadlines deliberately have NO same-slot dedupe sweep" carve-out in Key Design Decisions. The code fix in `llm.py`/`sync.py` (multi-record group canonicalization for the extractor + the dedupe-merge canonical-survivor preference) prevents NEW drift; this sweep repairs rows already collapsed in the store, which re-running sync can't re-cluster. Dry-run by default; `--apply` writes back. Renaming a key changes its ICS UID + gcal/M365 event id, so subscribers' clients re-create those events once.
- **`scripts/heal_key_derived_titles.py`** — deterministic cleanup (no LLM, no CourtListener) for hearings AND deadlines whose title is still a fallback humanization of the key: a hearing carrying a redundant defendant name on a single-defendant docket ("Sentencing Lytvynenko" → "Sentencing"), and either kind carrying a key abbreviation ("Status Conf" → "Status Conference", "Govt Status Report" → "Government Status Report"). Calls `sync.heal_key_derived_titles`, which rewrites ONLY rows whose CURRENT title is a fallback-derived form (`_title_is_key_derived`, which recognizes the plain humanization plus the name-aware/expanded fallback forms) — an explicit LLM title is never touched — judging single-vs-multi-defendant from each case's own hearing keys (`_case_defendant_names`) so genuine co-defendant dockets keep their names rendered `"<Type> - <Name>"`, and expanding abbreviations via `_KEY_ABBREVIATIONS`. The write-time fallbacks (`CaseSyncer._hearing_fallback_title`, `_fallback_deadline_title`) now produce these clean titles for NEW events; this script repairs the rows stored before that landed. Touches only the `title` column of either table (no key change, no schema change). Dry-run by default; `--apply` writes back. Deadlines get abbreviation expansion only — NOT name-stripping (their keys are freeform, not "type + defendant").

### Deployment scripts (`scripts/`, bash)

These are the operator wrappers for running the production deployment over ssh. They are checked in (no host-specific values are hardcoded) and read every deployment-identifying value from `.env` through a shared loader, so the same scripts work for any operator who fills in their own `.env`.

- **`scripts/_prod-env.sh`** — sourced (not executed) by every deployment script. Resolves the repo-root `.env` from its own location (so the scripts work from any CWD), reads ONLY the five `CC_PROD_*` keys it needs (rather than sourcing the whole secrets file, which would pull every key into the shell and could choke on a value bash can't evaluate), validates that all five are present (exits with a clear error naming the missing one otherwise), and exports `CC_PROD_SSH` = `<user>@<host>` for convenience. The five keys — `CC_PROD_HOST`, `CC_PROD_SSH_USER`, `CC_PROD_APP_DIR`, `CC_PROD_SERVICE` (systemd unit AND unix service account, assumed identical), `CC_PROD_STAGE_DIR` (a dir on prod writable by the ssh user, for staging scp drops) — are deployment-IDENTIFYING, so they live in the gitignored `.env` and are DELIBERATELY NOT mirrored into `.env.example` (the loader's header documents them instead, since `.env.example` is the fresh-checkout template and these are one operator's host details). This is the pattern to follow for any future host-specific value: externalize to `.env`, document in the script, leave out of `.env.example`.
- **`scripts/sync-prod`** — run `case-calendar sync` on prod over ssh, streaming output back to the terminal (`PYTHONUNBUFFERED=1` + `env` wrapper, since `sudo` strips env and python block-buffers a non-TTY stdout). Forwards extra args. Doesn't stop the service — sync and serve coexist under WAL journaling + `busy_timeout`.
- **`scripts/reconcile-prod`** — run `case-calendar reconcile` on prod over ssh, same shape as `sync-prod` (same `_prod-env.sh` loader, ssh-agent check, `PYTHONUNBUFFERED=1` + `sudo -u … env` wrapper). Runs the cheap placeholder re-check instead of a full sync — re-fetches only entries matching `sync.is_pending_enrichment` by id to pick up the upstream enrichment the webhook path can't see (see the placeholder-reconcile Key Design Decision and [CourtListener issue #7423](https://github.com/freelawproject/courtlistener/issues/7423)). Forwards extra args (`--case`, `--days`). Doesn't stop the service — reconcile and serve coexist under WAL + `busy_timeout`.
- **`scripts/sync-via-prod`** — push local `config.yaml` to prod, sync there (prod's CourtListener token does the work), restart the service so the running serve process picks up the new config, then pull the resulting store back to dev via SQLite's online-backup API. Config flows dev → prod, DB flows prod → dev (intentional). `--prune` runs `prune --apply` between code-pull and sync; all other args forward to `sync`. Backs up the local store before the swap.
- **`scripts/pull-prod-db`** — copy prod's `case-calendar.sqlite` down to dev (DB only, online-backup snapshot, no service downtime). Backs up the local store first.
- **`scripts/push-db-to-prod`** — the inverse: upload local store + `.env` + `config.yaml` to prod, pull code, re-emit, restart. Prompts for confirmation (it overwrites prod state). Stages in `CC_PROD_STAGE_DIR` then `cp`s into place (preserving destination ownership/perms a `mv` would clobber) and removes the staged `.env` afterward so secrets don't linger.
- **`scripts/upgrade-prod`** — apt upgrades, push local `.env`, pull code, `uv sync`, re-emit, restart, and flag if a reboot is required. Leaves the prod DB and `config.yaml` alone.
