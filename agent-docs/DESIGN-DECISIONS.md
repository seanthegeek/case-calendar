# Key Design Decisions

> Part of the Case Calendar project instructions, split out of [`AGENTS.md`](../AGENTS.md) so the always-loaded instruction file stays within agentic tools' context-loading limits. This is canonical project guidance — the same status as AGENTS.md, not a private scratch note.

## Court/clerk variation is a primary concern; accuracy beats clean abstractions.

Every district and clerk handles CM/ECF differently — how hearings get noticed (electronic clerk's notes vs scheduling orders vs minute entries), what surfaces in PACER vs how RECAP / CourtListener mirror it, whether minute entries inline the figures or attach a judgment PDF, how reschedules and continuances are signaled, whether paperless orders exist for a given event class, what vocabulary the order uses for the same proceeding the stipulation proposed. These idiosyncrasies propagate up to every layer — extractor, verify pass, dedup sweep, summary pipeline, description renderer — and each layer should assume the next docket will do it differently and prefer the extra branch over forcing the pattern. When a fix has to choose between simpler code and accurately reflecting what a specific clerk actually does, accuracy wins; one wrong event on a public calendar erodes subscriber trust far more than a small extra case in the code. The corollary: when you see a new clerk behavior in the wild, add a test that pins the correct handling so the next refactor doesn't quietly regress on it.

## LLM-driven extraction over per-court regexes.

Courts describe hearings inconsistently (`Set/Reset Hearings` minute entries, `ELECTRONIC NOTICE OF RESCHEDULING`, embedded in PDF text, plain prose in clerk's notes). Maintaining regexes per court is a treadmill; the LLM sees the entry plus the case's known-hearing list and decides ADD_HEARING vs RESCHEDULE_HEARING vs UPDATE_DETAILS vs CANCEL_HEARING in one call.

## Default to the small/fast LLM tier for extraction, not flagship.

The per-entry extractor is structured-output work (date + key + significance + a short title), not deep reasoning. The defaults are Haiku, Flash Lite, and gpt-5.4-nano — cents per case per day even on busy dockets. Resist proposing flagship-tier defaults; the cheap models are well-fit for this task and a 100× cost hike for ambiguously-better extraction is a bad trade. Case summaries are the separate, opt-in track that justifies a higher tier — see the per-track defaults in `llm.py` (`_DEFAULT_MODELS` vs `_DEFAULT_SUMMARY_MODELS`). **The default is now a per-track split: Gemini (`gemini-3.1-flash-lite`) for extraction, Anthropic (`claude-sonnet-4-6`) for summaries.** Auto-detect key priority when neither `LLM_PROVIDER` nor a track override is set is therefore per-track: the extraction track is **gemini > anthropic > openai** (`_detect_extraction_provider`, passing `_EXTRACTION_KEY_PRIORITY`) and the summary/base track is **anthropic > gemini > openai** (`_detect_provider`, default `_SUMMARY_KEY_PRIORITY`), so a zero-config operator with all three keys lands on that split. The flip earned its place: through 0.11.0 the extraction default was Anthropic because Gemini, relying on its training priors alone, silently classified a long tail of substantive federal-procedure deadline classes — PSR, Speedy Trial Act exclusions, surrender for service of sentence, civil-forfeiture claim/answer, substantive sealing motion practice, exhibit-filing deadlines, certified administrative record — as `procedural-minor` and dropped them at the render-time significance gate. 0.13.0 added the structured `DEADLINE_SIGNIFICANCE_RULES` block to the unified `SYSTEM_PROMPT`: it enumerates those substantive classes explicitly and biases the default to `major`. Because the classes are now named IN THE PROMPT for every provider (apples-to-apples) rather than left to a model's intrinsic priors, Gemini classifies them as major too — note the framing is "the prompt carries the priors for all providers now," NOT "Gemini's training improved." The measured result (see `model-comparison/SCORECARD.md`): Gemini's `D met/pass` is the best among the hosted extractors (129 vs Anthropic's 143) and its spurious-cancellation rate is well below Anthropic's (`D canc` 4 vs 43), its per-entry (636 vs 784) and aggregate (376 vs 476) deviations are the best overall, and it stays \~3.7× cheaper on a full-caseload backfill (docs/cost.md: $1.87 vs $6.99) and \~2× faster per call (median 1.5 s vs 3.1 s) than Anthropic — so Gemini is the right default for the high-volume, cost-dominant extraction track. Anthropic stays the summary default on the SCORECARD's Phase 3 blind grading: Sonnet graded A (accurate, most detailed, clean — the reference) vs Gemini 2.5 Pro's A− (a touch less detail, and it omitted the China nexus on us-v-ding — a case-distinguishing fact this caseload turns on). Honest caveat: the ruleset enumerates the classes the project currently knows about, so an operator whose caseload carries substantive classes the ruleset doesn't name should still verify against their own dockets — but the silent-drop risk is materially reduced, not merely traded, because the enumeration is in-prompt for both providers. The two tracks can still be pinned independently: `LLM_EXTRACTION_PROVIDER` overrides the extractor track and `LLM_SUMMARY_PROVIDER` overrides the summary track, both with-or-without an `LLM_PROVIDER` global default (which forces one provider for both). **Local inference via Ollama defaults to `gpt-oss:20b` (\~13 GB resident — measured 12.9 GB on an A5000 — MXFP4 4-bit, 20B params, LEVEL-based thinking) for BOTH tracks (`_DEFAULT_MODELS["ollama"]` and `_DEFAULT_SUMMARY_MODELS["ollama"]`).** It earned the default on two axes: it was the best-scoring LOCAL extractor in the benchmark (lowest aggregate deviation of the local models tested — see `model-comparison/SCORECARD.md`), AND it fits a mainstream 16 GB card with room for a working context window. Its level-only thinking is also a stability win under the bounded budget — it can't run away the way the boolean-thinkers (`qwen3.5:9b`, `glm-4.7-flash`) did when their reasoning was unbounded. gpt-oss:20b is the ONLY local model the benchmark recommends for extraction: the next-best local, `gemma4:e4b` (9.6 GB), scored 1241 per-entry deviation vs gpt-oss's 710 — nearly double, over-extracting harder than any hosted model — and the rest trailed further or couldn't complete a run (`granite3.3:8b` 1461, `granite4.1:8b` 1869, `llama3.2:3b` 2367, `qwen3.5:9b` unstable, `deepseek-r1:8b`/`:14b` output-volume degenerate in both thinking modes; see `model-comparison/SCORECARD.md`). On a card too small for gpt-oss:20b, the recommendation is HOSTED extraction (Gemini, the cheap default), not a smaller local model — `gemma4:e4b` / `gemma4:e2b` run on 12 GB / 8 GB but are a measured accuracy downgrade, documented as last resorts, not recommendations. NO local model is good enough for PRODUCTION summaries: Phase 3's blind grading topped every local summary out at a C (accurate figures but too thin or too clunky to publish) while the hosted models graded A / A−, so a public calendar keeps summaries on the hosted tier (the hybrid setup in `docs/local-llms.md`) — the all-local default exists for testing and private use, not production. Larger local models are NOT the quality fix: the dense 24B/30B models ran 3-6× slower than gpt-oss on a 24 GB card while scoring no better, and `gemma4:31b` (\~20 GB of weights — alongside a summary-sized context window there's no room for the KV cache, so it OOMs or spills and crawls on a 24 GB card and wants a 32 GB GPU) has never been benchmarked on either track, so don't present it as an upgrade path. Don't "fix" the Ollama default UP to `gemma4:31b` on the assumption gpt-oss:20b is an oversight; the default is intentional (best-local-extractor + hardware fit) and documented in `docs/local-llms.md`. (Earlier defaults were `gemma4:31b` then `gemma4:e4b`; gpt-oss:20b superseded both once the benchmark showed it both fits common cards AND extracts better locally.)

## Ollama sampling and seed: greedy (`temperature=0`) is the DEFAULT (scorecard-valid); in-spec sampling is an OPT-IN escape hatch, not the default.

The domain layer pins `temperature=0` (greedy) on every call for determinism and the LLM-cache's byte-identical replay, and `_call_ollama_native` / `_call_ollama_openai_compat` FORWARD it like the hosted paths. Greedy IS off-spec for local thinking models — the [Qwen3 card](https://huggingface.co/Qwen/Qwen3-8B) states verbatim "DO NOT use greedy decoding, as it can lead to performance degradation and endless repetitions" (recommending temperature 0.6 / top-p 0.95 / top-k 20 for thinking mode) and the [DeepSeek-R1 card](https://huggingface.co/deepseek-ai/DeepSeek-R1) recommends temperature 0.5–0.7 "to prevent endless repetitions" — and it is the documented trigger for the `qwen3.5:9b` / `deepseek-r1:8b` reasoning runaways (first mis-attributed in `model-comparison/SCORECARD.md` as inherent instability). A June 2026 probe (benchmark snapshot, RX 7900 XTX) confirmed the mechanism: under greedy these models ran their reasoning to the full bounded budget with an EMPTY answer (qwen 64–67K reasoning chars / 0 answer on stress entries; deepseek 15,958 chars / 0 answer on a TRIVIAL order — the greedy × schema-grammar combination specifically, since dropping EITHER stopped it), and switching that entry to in-spec sampling let it complete. **So why keep greedy as the default?** Because a full re-benchmark of the RECOMMENDED local models under in-spec sampling (gemma4:e4b at its Modelfile sampling; gpt-oss:20b ×2 for its non-determinism, seed 42, `--no-llm-cache`) came back NEUTRAL-to-slightly-WORSE — gemma per-entry deviation 1241→1258 / aggregate 985→998 (deterministic, so a real if tiny regression); gpt-oss per-entry 710→690/701 and aggregate 396→373/418, i.e. greedy sits INSIDE gpt-oss's own run-to-run band (statistically unchanged). Switching buys no accuracy on the models we actually recommend, while it WOULD invalidate the SCORECARD (every local row was measured greedy) and the cache's replay soundness. So greedy stays the default and the sampling knobs are opt-in: **`OLLAMA_TEMPERATURE`** (`_ollama_temperature_override`) overrides the temperature — set it to a card value, e.g. 0.6, to escape a qwen/deepseek runaway — and **`OLLAMA_SEED`** (`_ollama_seed`) pins the seed. Both apply to the native AND OpenAI-compatible paths; an unset/malformed value falls back (override → caller's greedy `temperature`; seed → Ollama's own random seed). **Determinism, where available, comes from the SEED, not a low temperature** — any `temperature > 0` samples, so a fixed `OLLAMA_SEED` is the PREREQUISITE for reproducibility but is NECESSARY, not SUFFICIENT, on GPU: the same probe (seed 42, run twice) reproduced byte-identical output ONLY for schema-constrained extraction on a NON-reasoning model (gemma; qwen's short output likewise), while a reasoning model's extraction (gpt-oss — different md5, even a different `deadline_key`) and free-text summaries on BOTH models diverged run-to-run despite the seed (inferred cause: GPU floating-point nondeterminism in the unconstrained reasoning/summary generation, which a grammar-pinned extraction is largely immune to). The `_LLMCache` key in `build_provider_stores.py` folds in `(_ollama_seed, _ollama_temperature_override)` for ollama ONLY when one is set, so the default greedy run keeps its original key (no cache orphaning) while an opt-in sampled run re-keys. Even fully in-spec, qwen still runs away intermittently and deepseek still over-emits, so gpt-oss:20b (level-based, inherently bounded reasoning) remains the only recommended local extractor. See the corrected `qwen3.5:9b` / `deepseek-r1:8b` sections and the "in-spec sampling attempt" note in `model-comparison/SCORECARD.md`, and the model-card convention near the top of this file. **If you ever DO move the default to sampling, you must re-benchmark every local row and republish the SCORECARD under the new policy — don't mix greedy and sampled rows in one table.**

## Models that reject the `temperature` parameter get one retry without it, remembered per model — no per-model allow-list.

The domain layer pins `temperature=0` on every call so re-runs are repeatable, but newer hosted models are dropping sampling controls entirely: Anthropic's model-migration guide documents that `temperature` / `top_p` / `top_k` are removed on the Claude 5 family and on Opus 4.7 and later — any value returns a 400 — and OpenAI's reasoning tiers restrict `temperature` to its default. Without handling, pointing `LLM_MODEL` at such a model would turn every LLM call into a hard failure. The hosted call paths in `llmkit/providers.py` (`_call_anthropic` / `_call_openai` / `_call_gemini`) handle it dynamically: when a call fails with an error that both names `temperature` AND reads as a not-supported rejection (`_is_temperature_unsupported_error` — a bad-VALUE error like an out-of-range number still propagates, since retrying without the knob would paper over a real config mistake), the call is retried once without the parameter and the `(provider, model)` pair is recorded in `_TEMPERATURE_UNSUPPORTED` so every later call in the process skips the doomed first attempt. This was chosen over a maintained list of model names because the list would go stale on every model release — the dynamic detection costs one rejected request per process per affected model and then self-corrects. Two consequences to keep in mind: (1) on such a model the calls run at the provider's default sampling, so the repeatability the `temperature=0` pin bought is unavailable there — the one-time warning log says so; (2) the memory is per `(provider, model)`, so an older model on the same provider keeps its deterministic pin. The Ollama path is deliberately excluded — it has its own temperature policy and operator overrides (see the Ollama sampling note above), and the rejection shape is a hosted-API behavior.

## Structured output (schema-enforced JSON extraction) is always on.

The per-entry extractor's output is hard-constrained to a JSON Schema (`llm.ACTIONS_SCHEMA`) by each provider's structured-output mechanism, threaded as the `schema=` argument through `llmkit.providers._dispatch_llm_call`. Always ON with no opt-out knob — none is needed even for generic OpenAI-compatible servers: verified June 2026 from current source that vLLM (`JsonSchemaResponseFormat` → xgrammar) and LM Studio's llama.cpp engine (`json_schema_to_grammar` → GBNF) BOTH accept and enforce our exact closed schema — `additionalProperties:false`, all-required, enums, nullable `[type, null]` unions; both accept-and-ignore the OpenAI `strict` flag and enforce via their grammar backend regardless. **The schema is CLOSED and MINIMAL-required:** every object sets `additionalProperties: false` and DECLARES every field, but only `type` (the action discriminator, a non-null enum) and the top-level `actions` array are `required`; every other field is a nullable union. Two hard-won constraints sit behind that exact shape, each pinned by a live benchmark, not a guess:
  - **Closed + every-field-declared** because Gemini's `response_schema` is a strict generative grammar that emits ONLY the fields the schema declares — an under-declared / open schema made Gemini emit typeless / empty objects and run away (2026-06 live check: 5/8 entries degenerate + 3 runaways, one a 206K-char partial before `OutputTruncatedError`). The other three providers (Anthropic forced tool-use, OpenAI `json_schema`, Ollama native `format`) treat an open schema loosely and were fine on it, but the closed schema is REQUIRED for Gemini and works for all four — so it's the single canonical schema.
  - **Minimal-required (only `type`), NOT all-required.** Forcing every field `required` (the first closed draft) sharply depressed Gemini's RECALL — it emitted actions on roughly half as many entries (clean benchmark: action-bearing entries \~365 → 184, misses +99) because each action then had to carry all 18 fields. Only OpenAI's strict mode actually needs all-required, so `providers._to_openai_strict_schema` fills that in for the OpenAI path ALONE, keeping the canonical schema minimal-required for the providers (Gemini especially) that don't need it.

  The MECHANISM is domain-free and lives in `llmkit` (the `schema=` param + the four per-provider enforcers + the `_to_gemini_schema` dialect adapter — union `["t","null"]` → `{type, nullable}`, strip `additionalProperties` — and `_to_openai_strict_schema`); the DOMAIN (`case_calendar.llm`) supplies `ACTIONS_SCHEMA` (passed unconditionally — no gate). **Why always-on (the benchmark verdict — neutral-or-positive across the board):** accuracy-NEUTRAL on Gemini (per-entry deviation 645→636, aggregate 381→376 — within run-to-run noise; recall preserved 373→365) WHILE cutting tokens (Gemini extraction input −4%, output −23% — the grammar stops spurious/verbose generation, and fewer spurious rows feed a leaner accumulated known-events context into later calls); a measurable WIN on the local `gpt-oss:20b` (per-entry 731→694, −5%, aggregate 423→402, by suppressing its spurious over-emission with the hard `format` grammar — the same re-emission failure mode the bounded-thinking note describes); and Anthropic / OpenAI validation-clean (0 degenerate). The Gemini token win is small precisely BECAUSE Gemini already emits clean JSON; the gpt-oss accuracy win is where the hard grammar earns its keep, which is the point of keeping it on for everyone. **Benchmark methodology learned the hard way:** compare OFF vs ON only with BOTH runs `--no-llm-cache` — a warm-cache OFF carried cached extractions for the \~24 frozen-snapshot PDF entries that a fresh ON couldn't reproduce, which manufactured an apparent recall "tradeoff" that vanished once both ran fresh. **A prompt-slim experiment was tried and REVERTED:** dropping the JSON-scaffold from `SYSTEM_PROMPT` (redundant once the schema enforces structure — a DRY win on its face) saved \~167 prompt tokens/call but came back token-NEGATIVE overall (+2.5% input) — without the scaffold the model emitted slightly more actions, and the same context-accumulation loop turned those into more input tokens than the slimmer prompt saved. So the scaffold STAYS: the schema is the structural guarantee, the prompt keeps the human-readable shape. Don't re-add a slim prompt variant without re-measuring NET tokens end-to-end (not just prompt size).

## Token usage is logged from real counts; dollar cost is an opt-in ESTIMATE layered on top — both log-only, not persisted.

Cost docs can only ever give a rough range, and an operator running this unattended needs to verify their real spend. So every LLM call records its actual token counts (`case_calendar/llmkit/usage.py`): one `llm-tokens call …` line at INFO per call carrying `purpose` / `provider` / `model` / `docket` / `in` / `out` / `cached` / `cache_write`, plus per-docket subtotals, per-model subtotals, and a run `TOTAL` logged at the end of `sync` / `summarize`. The per-model subtotal lines are how an operator reads the extractor track's spend separately from the summary track's — they run on different model tiers, so the by-model split is the by-track split without `llmkit` having to know the domain track names (it buckets by the opaque model string, as it does for docket). The counts are NORMALIZED across providers so `in` always means total prompt tokens (cached included) and `cached` / `cache_write` are broken out — Anthropic reports cache reads/writes separately from `input_tokens` while OpenAI/Gemini fold them in, and papering over that would make the providers' numbers non-comparable. **Cost estimation is a separate, optional layer that keeps `llmkit` price-free:** the ledger exposes a `set_price_estimator(fn)` hook (`fn(model, TokenUsage, provider) -> float | None`); when set, each `llm-tokens` line also carries a `cost_est=` field and the run TOTAL accumulates it. The estimator itself lives in the domain layer (`case_calendar/costs.py`) as a small static per-model rate table hand-copied from each provider's pricing page and dated with `PRICES_VERIFIED` — it prices each token slice (uncached input / cache read / cache write / output) at its own rate (the cache split matters because the system prompt is cached on nearly every call). It is explicitly an estimate: a model absent from the table (a legacy model or an unrecognized operator `LLM_MODEL` override) logs `cost_est=?` and the TOTAL notes how many calls were unpriced, rather than emitting a wrong number. The library stays pricing-free on purpose — baking a price table into `llmkit` would reintroduce the maintenance treadmill the tokens-only design avoided AND bloat the reusable package; the consumer supplies prices. Deliberately log-only (tokens AND cost): persisting to SQLite would mean a schema change (and the back-up-before-migration rule) for telemetry an operator can recover by summing the `TOTAL` log lines, and the value is verification-of-cost, not an operational input the code reads back. The recorder defends against the telemetry itself becoming a failure mode — every usage field is read through `_as_int`, so a missing field or an SDK shape change degrades to a 0 count rather than raising inside a working LLM call. `serve` intentionally skips the run-total path (per-call lines only, which DO include `cost_est`): its webhook worker threads and the debounce Timer thread would race on the ledger `reset`, and a "total" isn't a meaningful unit for a long-running daemon — a `sync` backfill is where cost measurement matters. `reset` clears the estimator too (full reset to fresh state), so `cli._log_llm_setup` re-attaches it at the start of each command.

## Anthropic prompt caching has a per-model minimum-prompt-size threshold; smaller prompts pay full input rate.

The Haiku 4.5 floor is 4096 tokens — NOT 2048, which is the retired Haiku 3.5's floor. An earlier version of this note misread the docs' per-model table and wrongly claimed the documented number was inaccurate; in fact Anthropic documents 4096 for Haiku 4.5 and our empirical measurements agree with it. Still confirm with `count_tokens` + a real call's `cached` / `cache_write` counters after any prompt edit, because the chars/4 estimate is too coarse to trust near a boundary. `case_calendar/llmkit/providers.py::_call_anthropic` attaches `cache_control: ephemeral` to the system block on every call regardless of purpose, so the marker is applied uniformly to all four extractor-track / verify-track / dedupe-track prompts. But Anthropic only ACTUALLY caches a system prompt once it clears a per-model token floor. The documented floors are **1024 tokens on Sonnet 4.x / Opus 4.8, 4096 tokens on Haiku 4.5 (and on Opus 4.5 / 4.6 / 4.7); 2048 is the floor for the retired Haiku 3.5**. The 0.11.0 verify-pass-determinism work confirmed the 4096 figure empirically: first the merged `VERIFY_SYSTEM_PROMPT` at \~2000 measured tokens didn't cache (under 4096), then a deliberate bump to a `count_tokens`-measured **2941 tokens** ALSO didn't cache (`cached=0`, `cache_write=0` across all 130 verify calls in the validation build) — consistent with the 4096 floor, since 2941 < 4096. `SYSTEM_PROMPT` at 9302 measured tokens DOES cache, pinning the floor from the other side (somewhere above 2941 and at or below 9302 — and the docs say 4096). So the floor is not a mystery or a doc error; the verify prompt is simply under it. The 2941-token bump was reverted after that finding (no observable behavior win, real cost increase, and no cache offset because it was still below 4096) — the current prompt is back at \~2000 tokens. Below the floor, the marker is a no-op: nothing is written to cache, nothing is read from cache, and every call pays the full uncached input rate. Token counts for the current prompts (`count_tokens`-measured where noted): `SYSTEM_PROMPT` **9302 tok** (caches on Haiku ✓), `VERIFY_SYSTEM_PROMPT` \~2000 tok (well under the 4096 Haiku floor, so does NOT cache on Haiku ✗; bringing it into cache would require growing it past 4096 tokens with substantive content, a stretch given the rule space is already enumerated — the practical alternative is routing the verify track to Sonnet, which caches at 1024 tokens per the docs), `DEDUPE_HEARING_SYSTEM_PROMPT` \~480 (doesn't cache anywhere ✗), `SUMMARY_SYSTEM_PROMPT` \~8000 (caches on Sonnet ✓). The cost impact today is small because verify / dedupe calls are infrequent compared to per-entry extraction — roughly $0.55 per full sync on Haiku 4.5 for the still-uncached verify track. But it has two consequences worth keeping in mind: (1) **prompt-slim passes that shrink a prompt PAST a threshold can quietly hurt cost**, by dropping it from cache-eligible to full-rate. Before shrinking any prompt that's currently caching, check its size against the floor for the model it runs on. (2) **prompt-merge passes that consolidate small sibling prompts can quietly help cost** when the combined prompt clears the threshold — but trust `count_tokens` + an empirical caching check, since chars/4 can land you on the wrong side of the floor. **The chars/4 token estimate is too coarse to trust within \~5%** — the 0.11.0 verify consolidation estimated 2104 tokens using chars/4 but actually landed at \~2000 tokens by Anthropic's tokenizer (either number is far under the 4096 Haiku floor, so the prompt doesn't cache there regardless). Cross-calibration against a known-caching prompt's measured cached count is closer: `SYSTEM_PROMPT` is 35569 chars and reports 9296 cached tokens (3.83 chars/token); but even that gives you the SIZE without telling you the THRESHOLD. The recorder in `usage.py` is what makes this visible in production: `cached` and `cache_write` are broken out as their own counters, so a sweep over the `llm-tokens call` lines for a sync immediately shows which prompts are caching and which aren't — sweep verify-track calls after any prompt edit to confirm the threshold is still cleared.

## Stable hearing keys, not stable timestamps.

A "logical" hearing (e.g. "sentencing for Wang") may be rescheduled multiple times. We let the LLM assign a kebab-case `hearing_key` on first observation and re-use it on follow-ups, so the same calendar event is updated rather than duplicated. Google event IDs are derived from this key.

## Regex pre-filter before LLM.

Only entries whose text matches hearing-related vocabulary are sent to the LLM. Briefs, attorney withdrawals, sealed-document placeholders, etc. are dropped for free.

## Three-tier short-circuit

to keep API + LLM costs near zero on quiet days: (1) per-docket — skip the entire docket if its `date_modified` hasn't advanced since the last sync (no entries API call, no LLM); (2) per-entry — `iter_entries(modified_after=cutoff)` filters server-side to entries newer than our store's last-modified cutoff; (3) per-fingerprint — even if an entry comes back, dedup against `(docket_id, entry_id, content_fingerprint)` skips re-LLM-ing on entries whose substantive content didn't change. The docket last-modified cutoff is only advanced on a clean run, so a mid-sync error retries the whole docket on the next run. The cutoff bump sits AFTER the iteration loop in linear control flow (no try/finally) — any exception that escapes the loop, including `BaseException` subclasses like `KeyboardInterrupt` (Ctrl+C) and `SystemExit`, must propagate past the bump without advancing it. The previous try/except/finally form used `except Exception:` which silently let those slip through with `iterated_ok=True`, advancing the cutoff despite the interrupted iteration and making the next sync's docket-level short-circuit skip the unprocessed entries entirely. Future refactors must preserve the linear shape; an `except Exception:` around the loop is a regression and pytest pins three cases (KeyboardInterrupt / SystemExit / generic Exception) to catch it. **The same short-circuit also gates the end-of-case sweeps (added 0.13.1).** `sync_case` tracks whether ANY docket in the case advanced past the per-docket short-circuit; when none did, the LLM-backed verify / dedupe sweeps (`_verify_scheduled_hearings`, `_verify_pending_deadlines`, `_dedupe_concurrent_hearings`, `_dedupe_concurrent_held_hearings`, `_dedupe_nearslot_hearings`) are skipped, because they read their candidate rows and docket-entry context entirely from the store — unchanged when nothing landed — and every domain LLM call pins `temperature=0`, so re-running would re-bill identical verdicts. Before this, a fully-quiet sync still spent one verify call per non-terminal row (≈$0.14 / 107 calls on the current caseload) re-confirming rows that couldn't have changed. The gate is safe with respect to `serve`: the webhook path calls `process_entry` only and NEVER advances a docket's stored `date_modified`, so a webhook-touched docket always fails the short-circuit on the next poll and runs its sweeps then — the gate cannot strand a webhook-added row. The time-driven `_auto_mark_passed_stale` is deliberately OUTSIDE the gate (it flips `pending`→`passed` as due dates elapse, makes no LLM call). `sync --reverify` (CLI flag) forces all sweeps regardless — for after a verify-prompt / model change or an out-of-band store edit (`scripts/reprocess_entries.py`, `scripts/classify_significance.py`) that mutated rows without advancing any docket. A future refactor that wants to tighten the gate (e.g. to `entries_processed > 0`) must keep the conservative direction — it can over-run the sweeps but must never skip them on a case whose rows could have changed.

## Summary pipeline reads the local entries cache, not CourtListener, when sync just wrote the data.

`sync.process_entry` persists description + recap_documents (with `plain_text`) for any entry that matched the hearing/deadline pre-filter OR `summary.is_primary_document` / `summary.is_disposition`. `find_primary_documents` then reads those rows via `Store.get_entries_with_body` — same shape as a CourtListener response, so the rest of the summary pipeline doesn't branch. Before this, `refresh_stale` ran right after sync and burned three duplicate `/docket-entries/` pages per stale docket (two oldest-first + one newest-first), all to find entries CourtListener had already returned to sync minutes earlier. CourtListener is now only hit when the local cache is genuinely cold (first-ever sync of a docket, or pre-fix data lacking op/disp body text).

## No separate PDF text cache.

Once a hearing is extracted from a PDF, the structured fields are in the `hearings` table and we never re-consult that PDF — follow-up reschedules / dial-in updates arrive on new docket entries with their own PDFs. Description rendering reads the entry's stored `description` + `recap_documents` JSON (already on the `entries` row), not a separate PDF-text excerpt column. The rare case of "same entry re-processed because its fingerprint changed" pays for one IA round-trip, which is cheap compared to the operational complexity of an immutable-blob cache.

## PDF retries on availability flips.

RECAP coverage is incomplete — many entries' PDFs get uploaded later, and the upstream `plain_text` for an image-only PDF can stay empty indefinitely (CourtListener does not OCR documents — that's our job, run locally via poppler + tesseract). The entry fingerprint includes each recap_doc's `is_available` / `is_sealed` / `plain_text`-presence flags, so a previously-missing PDF re-triggers processing automatically once it appears, and a previously-empty `plain_text` re-triggers processing once we (or any other downstream actor whose output makes it back into CourtListener's mirror) populate it.

## The summary LLM is instructed to refuse rather than fabricate.

Even with the garbled-text detector in [Garbled `plain_text` is not trusted; we OCR ourselves when it is](#garbled-plain_text-is-not-trusted-we-ocr-ourselves-when-it-is) catching upstream PDF-encoding noise, other failure modes can still leave the summary LLM with too little signal — a PDF that won't OCR cleanly, an indictment that's still sealed and only the docket-text stub is available, or an entry whose recap_documents are all unavailable on RECAP. The `SUMMARY_SYSTEM_PROMPT` ends with a CRITICAL rule that on those inputs the model MUST output the exact sentence "Documents available for this docket are insufficient to generate a reliable summary." and nothing else; the matching constant `llm.SUMMARY_INSUFFICIENT_DOCUMENTS` is what `summary.summarize_docket` greps for to log a warning so the operator can investigate (typically: install poppler/tesseract so our own OCR fallback can run — CourtListener does not OCR documents, so a perpetually-empty `plain_text` means nothing upstream is going to fix it for us). The fallback prose is stored and rendered like any other summary — subscribers see the explicit refusal on the index page rather than silence or a confabulated narrative. This rule overrides the prompt's 2-4-sentence length guidance whenever it applies; it must not be weakened in future prompt revisions because it's the last-mile guard against the failure mode that produced the us-v-dubranova "CSRERI / Roskomnadzor" hallucination.

## Garbled `plain_text` is not trusted; we OCR ourselves when it is.

Some PDFs use custom font subsets with no `/ToUnicode` map, so pypdf can't decode them — it emits glyph-index tokens (`/i255 /1 /2 ...`) or maps the raw bytes 1:1 into Latin-1 codepoints (`ÿ`/`ÿ`/punctuation noise). CourtListener's upstream pipeline runs pypdf the same way, so the `plain_text` field on a recap_document can be a multi-KB string that's non-empty and has the page header/footer in real ASCII while the body is gibberish. The summary LLM cannot tell garbage text from real text — fed gibberish, it confabulates plausible-sounding facts to fill the gap. The canonical instance was the us-v-dubranova first superseding indictment, where CourtListener's `plain_text` was 27KB of `ÿ`-noise and the summary asserted a fictitious "Center for the Study and Regulation of Internet Relations (CSRERI), also called 'Roskomnadzor'" — neither name appears in the actual indictment (the real-world group is named "CISM" and operates within the "NoName057(16)" hacktivist collective, both of which OCR reads cleanly). `pdf.is_usable_text` therefore inspects every extracted string before returning it. It's a single positive predicate that bundles every check a caller needs to decide whether to keep an extraction result or fall through to the next stage of the chain: (a) **length** — at least 100 chars, since a 2-page+ document extracting to less is almost certainly a failure not worth feeding to the LLM; (b) **font-encoding gibberish** — flagged when the ASCII-alpha ratio of non-whitespace characters falls below 0.4; real English prose runs 70-80% alpha even header-heavy and garbled extracts reliably score under 10%, so the threshold has wide margin on both sides; (c) **PACER-stamps-only extraction** — image-only scans with a thin OCR overlay covering just the page-header band let pypdf read several KB of clean ASCII off the headers but the document body (which lives only in the page images, not in any text layer) never reaches the caller. The text passes the alpha-ratio gate trivially because page stamps are mostly letters and digits, so the detector also strips the standard PACER stamp pattern (`Case <docket> Document <n> [Filed <date>] [Page <i> of <n>] [PageID:/Page ID #: <id>]`) and treats the result as useless if less than 100 chars of body survive. The us-v-schmitz indictment was the canonical case: pypdf returned 1538 chars of pure header stamps from an 18-page scan with no body text, sailed past the alpha-ratio gate, and the OCR fallback never ran until this detector caught the shape; the document itself was a real 18-page indictment that OCRs cleanly to 20 KB. The predicate runs at three points along the extraction chain — CourtListener's `plain_text`, our local pypdf output, and OCR output — and the first usable stage wins; if every stage fails the predicate, the last non-empty string is returned as a degraded-but-non-null result so callers can see the entry was attempted. The function was historically named `looks_garbled` and returned True for unusable text; it was inverted to its current positive form so callers read `if is_usable_text(text): return text` instead of the verbose `len(text) >= _MIN_USEFUL_CHARS and not looks_garbled(text)` pairing, and so its name reflects every check it does, not just the original garbled-detection one. The summary pipeline and the operator-document path (`extract_text_from_url` for `extra_documents`) share the same chain.

## `is_sealed` is the only hard gate; once we download, our pipeline owns the result.

Two related rules govern `pdf.extract_text`. (1) **`is_available=False` is NOT a gate on the fetch.** The pipeline always attempts `fetch_pdf_bytes` when CourtListener's `plain_text` is empty or unusable, regardless of the cached `is_available` flag — only `is_sealed=True` short-circuits the fetch (sealed PDFs aren't on RECAP; a fetch would just 404). The us-v-lytvynenko regression that drove this rule had a cached recap_doc with `is_available=False, plain_text=''` while upstream CourtListener had since flipped to `is_available=True` with full text and populated `filepath_local` — gating on the cached flag meant we never tried to fetch, so pypdf and OCR never ran on a document we could perfectly well have read. `fetch_pdf_bytes` itself decides whether the document is fetchable by checking the URL fields (`filepath_ia` / `filepath_local`); when both are empty it returns `None` after just looking at the dict, so the cost of removing the gate is bounded — when storage truly has nothing, we behave the same as before, just via the URL check inside `fetch_pdf_bytes` instead of an upstream `is_available=False` bail. When storage has drifted ahead of the flag (which CourtListener can exhibit between an upstream RECAP upload and the next `is_available` flip), we successfully recover instead of bailing. (2) **Once we download a PDF, the result comes from OUR pipeline (pypdf → OCR) — never from CourtListener's `plain_text`.** The plain_text fallback at the END of `extract_text` (after pypdf and OCR both ran on the downloaded bytes) used to be `return text or plain or None`, which would re-inject CourtListener's plain_text after we'd already chosen to ignore it at the top of the function. That re-injected exactly the upstream pypdf garbage (font-encoding gibberish, PACER-stamps-only extraction) we'd just rejected, since CourtListener's upstream extraction ran the same pypdf pass we run locally. The new contract: after `fetch_pdf_bytes` succeeds, the return value is `text or None` — pypdf's output if neither pypdf nor OCR cleared `is_usable_text`, otherwise None. CourtListener's `plain_text` is only ever returned in two places, both before the download attempt or as a "we never got to run our pipeline" fallback: the top-of-function short-circuit when `is_usable_text(plain)` passes, and the `if not pdf_bytes: return plain or None` path when the fetch itself failed (no URLs or every URL 404'd) and we never got bytes to feed into pypdf. Paperless orders and minute entries that genuinely have no fetchable document still produce `None` (no URLs → `fetch_pdf_bytes` returns None → `plain or None` is None when plain is also empty); entries that exist purely as docket-text (no `recap_documents` at all) are processed by the LLM from their description without any PDF stage at all.

## No redundant `recap-documents` calls.

The `/docket-entries/` endpoint already returns each recap_document inline with `is_available`, `is_sealed`, `plain_text`, `filepath_local`, `filepath_ia`. We use that inline copy directly rather than refetching `/recap-documents/{id}/`, which would be one wasted call per document on first sync.

## Court-local timezones, computed once.

Hearings are stored in UTC but ICS / Google events emit the courthouse's local time tagged with the court's IANA `TZID`. Calendar viewers display in their own tz while the event preserves "this is a 3 PM Pacific hearing" semantics across DST transitions and travel. We rely on the receiver's bundled IANA tz database for DST resolution and ship the Python `tzdata` package to keep our own `zoneinfo` lookups consistent across hosts. The court → tz table in `courts.py` is hand-maintained; CourtListener doesn't surface this.

## Sticky timezone on hearing updates.

`timezone` is set on first insert and preserved across reschedules / detail updates. A hearing happens in one courthouse, even when sibling dockets in different timezones reference it — without this, a cand entry referencing a cadc oral argument would clobber the tz to PT and shift the displayed wall-clock time.

## Multi-docket cases collapsed into one logical case.

Some cases have multiple docket numbers (e.g. district + appellate, or related defendants). The `dockets:` list in `config.yaml` aggregates them under one `case_id`, so hearings from any docket land in the same calendar.

## Case summaries are keyed by logical PACER docket, not by CourtListener docket_id.

CourtListener's reconciler can store the same logical PACER docket as multiple `docket_id` rows when the upstream `pacer_case_id` changed mid-life — see [CourtListener issue #7345](https://github.com/freelawproject/courtlistener/issues/7345). The canonical example is the Akhter twins case (`1:25-cr-00307`, E.D. Va.): three CourtListener docket_ids (71989485, 73333500, 73320754) with `pacer_case_id` values 585302, 585303, 585304 each carry a partial, non-overlapping slice of the PACER entries — entries 51-66 on one, 117-136 on another, 78-128 on the third. The `case_summaries` table primary key is therefore `(case_id, docket_number, court_id)` rather than `(case_id, docket_id)`, and `summary.summarize_docket` pools entries across every CourtListener docket_id in the group via `summary.find_primary_documents_for_group` (which deduplicates entries by PACER `entry_number` with a `(date_filed, description)` fallback for null-numbered paperless entries — see [Entry dedup across a docket group](#entry-dedup-across-a-docket-group)). The renderer collapses same-group summaries to one paragraph in the index, no docket label, and renders the docket NUMBER once as the primary clickable link. The primary link's URL is the `absolute_url` of whichever sibling CourtListener docket_id appears FIRST in `case.dockets` (config order, not freshest CourtListener docket — different from the summary pipeline's `Store.get_docket_group_ids` which picks the freshest). The choice is intentional: it gives the operator a stable way to pick the canonical CourtListener page subscribers see by reordering the `dockets:` array in `config.yaml`. Beneath the primary link, when the group has more than one CourtListener record, `index._render_cl_records` renders a muted "CourtListener records (same docket): 1 · 2 · 3" line — every record in config order, each individually clickable (its own `absolute_url`, carried in the model's per-docket `cl_records` list). This is full transparency + one-click access to each record (each holds a different slice of the entries) WITHOUT misleading a lay reader: the docket number is shown only ONCE, the records are explicitly labeled "(same docket)" with a tooltip, and the word "record" — never "docket" — is used for the siblings. The vocabulary is the disambiguator: a separate CourtListener record is NOT a separate docket or case. Genuinely distinct proceedings (a district case + its appeal) remain separate `dockets_meta` entries with their own numbers, each with its own (possibly multi-record) line. Document this in `docs/configuration.md` so the operator knows the convention. A future contributor who "simplifies" the schema back to `docket_id` keying will re-introduce the three-near-duplicate-summaries-per-case bug. The `_migrate_case_summaries_to_group_key` migration runs automatically on Store init and renames the pre-migration table to `case_summaries_pre_group_migration` as a rollback escape hatch.

## Entry dedup across a docket group.

When `find_primary_documents_for_group` pools entries from N CourtListener docket_ids in the same `(docket_number, court_id)` group, the same logical PACER entry appears on each CourtListener sibling with a DIFFERENT CourtListener `id` but the SAME PACER `entry_number`. The dedup key is therefore `("num", entry_number)` when `entry_number` is non-null; for paperless minute orders that have no entry_number we fall back to `("desc", date_filed, description[:200])`. First-seen wins, and the freshest CourtListener docket_id is walked first (the order from `Store.get_docket_group_ids`, which sorts by `date_modified DESC`) so a stale cached copy on an older CourtListener sibling never beats a fresh one — EXCEPT when the later sibling's copy has populated `plain_text` on its main recap_document and the first-seen copy doesn't, in which case the populated copy replaces the empty one. CourtListener can leave one CourtListener row's recap_document with an empty `plain_text` while another CourtListener row in the same logical PACER group has the extracted body; without the upgrade the summary LLM is fed metadata-only and refuses with the `SUMMARY_INSUFFICIENT_DOCUMENTS` sentence. The canonical case is us-v-schmitz (`1:24-cr-00234`, D.N.J.) — the freshest CourtListener sibling 73292090's indictment recap_document had `plain_text=""` while older sibling 73353898's same logical entry carried 20 KB of text. This dedup runs BEFORE `_attach_text`, so each logical PACER entry's PDF is extracted exactly once regardless of how many CourtListener siblings carry it — no redundant PDF reads, no redundant OCR cost. The upgrade rule doesn't add reads (it picks between copies already in hand).

## Group pooling does NOT short-circuit per-sibling CourtListener fetches.

`find_primary_documents_for_group` calls `find_primary_documents(cl, docket_id, store)` per CourtListener sibling. The per-docket function already short-circuits on its own warm cache (it returns from `get_entries_with_body` without a CourtListener fetch when its cache has a primary that isn't stale), so in steady state — after sync has populated each sibling's cache — the group function makes ZERO `/docket-entries/` calls regardless of how many CourtListener siblings are in the group. The only CourtListener fetches happen during sync itself (necessary — that's how sync discovers entries to cache) or during a stale-cache rebuild (the us-v-moucka path). A "skip the cold sibling's CourtListener fetch when a warm sibling's cache has a primary" optimization is UNSAFE and must NOT be added: it sacrifices the cold sibling's unique entries, which is the entire reason for pooling — the Akhter case proves that CourtListener siblings carry NON-overlapping slices of the PACER docket, so skipping a cold one drops real data. The corollary: an "all warm" cache short-circuit at the pooling layer would save zero fetches over the existing per-docket short-circuit. Leave the per-sibling calls in place.

## ICS first, third-party calendars opt-in.

ICS works with anything that subscribes to a URL (Proton, Apple, etc.), needs no OAuth, and is always written. Google Calendar push requires `google_credentials_path` in the config and a staged token cache; Microsoft 365 / Outlook push requires an Entra app registration's `m365_client_id` (config or env) and a staged auth record. Both auto-enable in `emit_calendars` whenever the per-backend token cache file is present — no CLI flag — so `sync` / `serve` / `emit` push uniformly. Both can be on simultaneously; `emit_calendars` initializes each backend at most once per call and reuses it across calendars.

## Webhook URL host is independent of `public_base_url`.

`public_base_url` is for the public ICS hosting (Caddy → `out/` directory); the webhook receiver typically lives on a different host (the example Caddy config in the docs uses `calendars.example.com` for feeds and `webhook.example.com` for the receiver). `case-calendar webhook-url --host <host>` takes the webhook host as a flag rather than reading `public_base_url`, because a legitimate deployment is "run the webhook receiver privately, don't expose any public ICS feeds" — and that user has no `public_base_url` to read. Don't fold the two surfaces into a shared config key; either add a separate optional key with `--host` as a fallback, or leave the flag as the source of truth.

## All input data is treated as untrusted.

Docket entries and PDF text can include arbitrary user-submitted content; the system prompt explicitly tells the model not to follow any instructions found inside.

## Notifications use attendees, not email reminders.

Google Calendar's email reminder field only fires for the calendar owner; we add `notify_emails` as event attendees instead, so arbitrary addresses get the invitation + their own reminders. The `reminders` config emits per-event overrides (popup for the owner, plus an email VALARM rendered into the ICS for clients that support it).

## Privacy-aware notification config.

`notify_emails` is exposed in the public ICS/gcal feed via `ATTENDEE` lines, so we document this trap in `config.example.yaml`: don't list addresses you don't want public. For public calendars, `reminders: [{method: popup, ...}]` works without leaking — popup fires in the subscriber's own calendar app from a `VALARM:DISPLAY` block.

## Polling and webhook share one path.

Both `case-calendar sync` (polling) and `case-calendar serve` (webhook) call `CaseSyncer.process_entry`, so a hearing extracted via webhook is byte-identical to one extracted via polling. Webhook payloads include the docket entry with `recap_documents` inline (per the CourtListener docs), so the only CourtListener API call a webhook triggers is one `/dockets/{id}/` lookup the first time we see a new docket — to learn its `court_id` (timezone) and citation. After that, zero CourtListener calls per webhook delivery; the LLM is the only outbound dependency.

## Docket alerts are reconciled automatically; opt out, not in.

Configured dockets only receive webhook pushes when the authenticated CourtListener account has a `docket-alerts` subscription on each one. `case_calendar.alerts.ensure_docket_alerts`, called from `cmd_sync` and `cmd_serve` startup, lists existing subscriptions and POSTs new ones for any docket in `config.yaml` that isn't already covered. Adding a case to the config wires up its docket alert on the next sync without manual UI clicks; this is the default precisely because forgetting that step is the most common operator-facing mistake (you add a case, no webhooks ever fire for it, the polling sync silently masks the gap). The flag is opt-out (`ensure_docket_alerts: false`) for operators who maintain subscriptions through a different surface — bulk CSV upload, an admin tool, etc. — and don't want the project's reconciler touching the list. The reconcile is deliberately one-way: it ADDS missing subscriptions but never deletes stale ones (removing a case from `config.yaml` doesn't unsubscribe — the operator can do that in the CourtListener UI if desired). Per-docket failures are logged at WARNING and don't abort sync/serve; a complete list-call failure marks every docket `'failed'` and skips creates entirely (avoids spamming duplicates or skipping blindly against an unknown baseline).

## Sync and serve are separate processes sharing one SQLite file, made safe with WAL + busy_timeout.

Polling (`case-calendar sync`, typically run via cron) and the webhook receiver (`case-calendar serve`, typically a systemd unit) usually run as two distinct OS processes pointed at the same `data/case-calendar.sqlite`. SQLite's default rollback-journal mode + `busy_timeout=0` would make the second writer raise `sqlite3.OperationalError: database is locked` *immediately* on any commit overlap — a webhook landing mid-sync would bubble up to a server-side 500 (CourtListener retries via `Idempotency-Key`, so no data loss but transient errors in the log), and a sync that loses a race would abort the whole invocation. `Store.__init__` therefore opens every connection with `PRAGMA journal_mode=WAL` (lets multiple readers and one writer coexist; main writes go append-only to a `<db>-wal` sidecar and get checkpointed back into the main file later) and `PRAGMA busy_timeout=5000` (when a writer does hit a contending lock, block-and-retry for up to 5 seconds before raising — long enough to cover any of our actual transactions, all of which are single-row UPSERTs). The webhook server's in-process `_lock` (`WebhookServer.process_locked`) still serializes worker threads *within* the serve process; WAL + `busy_timeout` close the gap *between* processes. Hot backups must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main file, or run `PRAGMA wal_checkpoint(TRUNCATE)` first to consolidate everything back into the main file before copying.

## Placeholder reconcile catches webhook-delivered stubs that enrich later, at O(filing activity) cost not O(dockets).

A CourtListener `DOCKET_ALERT` webhook fires once, when an entry is first docketed, and the payload reflects the entry's state at that instant — frequently a stub: empty `description`, a `recap_document` with `is_available=False` and no `plain_text`. CourtListener fills in the entry text and makes the document available afterward but does NOT fire a second webhook for that enrichment ([CourtListener issue #7423](https://github.com/freelawproject/courtlistener/issues/7423)), so `serve` never re-delivers the filled-in entry and the per-docket short-circuit can leave it for the next full `sync`. The canonical instance is us-v-kejia-wang entry #42: an endorsed order extending a self-surrender deadline (June 10 → July 10, 2026) was delivered as an empty stub, enriched \~90 minutes later with the operative date, and the deadline stayed stale until a poll re-read it. `case-calendar reconcile` closes this without reverting to the per-docket cost of a full poll: `CaseSyncer.reconcile_placeholders` re-fetches ONLY the entries still pending enrichment (`sync.is_pending_enrichment` — empty body plus an unsealed, not-yet-available document — surfaced cheaply by `Store.get_empty_body_entries_since` with a `--days` age cutoff), one `CourtListener.get_docket_entry` request each, and re-runs the existing `process_entry`. If the entry enriched, its fingerprint flips and the normal pipeline reschedules and (via the `upsert_*` stale chokepoint) regenerates the summary; if not, the fingerprint matches and it's a no-op (the `entry_seen` dedup short-circuits before any LLM call). Because the candidate set is the handful of recent placeholders rather than every docket, cost scales with filing activity like the webhook path, not with caseload size like polling — adding a quiet case adds zero reconcile cost. Two properties keep it honest: the predicate is deliberately TIGHT (the empty-body guard excludes paperless orders whose full text IS the `description` and that carry a permanently-`is_available=False` doc — without it the predicate matches every paperless order ever stored, \~908 vs \~131 in one snapshot), and the reconcile is an OPTIMIZATION, not a correctness boundary — whatever it misses (e.g. an entry with a present description but a date only in a not-yet-available PDF) is still caught by the infrequent full `sync` catch-all, so the predicate can stay tight without risking permanent loss. The intended deployment is `serve` (real-time first-touch) + a frequent cheap `reconcile` cron (enrichment) + an infrequent full `sync` (catch-all). A cutoff-capping fix to the short-circuit itself was rejected: in a real store 23/40 dockets have `docket.date_modified > max(entry.date_modified)` as their normal steady state (docket-level metadata re-saves), so capping the stored cutoff at the newest entry would turn the cheap short-circuit into a full re-walk on the majority of dockets every poll.

## Significance gate at render, not at extraction.

Every hearing is stored in the DB regardless of importance — that preserves the audit trail and lets the LLM use procedural events as context (`known_hearings` plus the new RELATED DOCKET ENTRIES block). Calendar visibility is filtered at render: the LLM tags each action with `significance='major' | 'minor'`, and the ICS/gcal renderers skip minor rows. Classification is by the proceeding's PURPOSE, not its EFFECT: final and CIPA pretrials and any pretrial/status conference whose agenda is a substantive motion are major; initial / Rule 16 / scheduling-only pretrials and phone calls whose only purpose is to rule on a Motion to Continue / Extend are minor — any rescheduling that comes out of those calls shows up on the trial / hearing row itself.

## Past-date alone is not evidence a hearing occurred.

The old `_auto_mark_held_stale` sweep flipped any past 'scheduled' row to 'held' on the assumption that "date passed → it happened". Wrong for trials — they get continued or vacated by plea agreement without an explicit cancellation entry, the calendar date passes, and the row was falsely flipped to 'held' (see the us-v-moucka regression where the 4/13 trial was marked 'held' even though the docket had no minute entry, verdict form, transcript, or judgment-after-trial). Replaced by `_verify_scheduled_hearings` extended to past-dated rows: the verify-pass LLM only returns MARK_HELD when it can cite a minute entry / verdict / transcript / judgment-after as evidence. Past rows the docket doesn't confirm stay 'scheduled' — accurately communicating "outcome not confirmed" rather than asserting 'held' on guesswork. The summary LLM's existing trial-vs-plea invariant ("a trial DATE in a scheduling order is NOT proof a trial OCCURRED") matches this signal: status='scheduled' on a past trial row is the calendar's way of saying the same thing. Next sync re-verifies; once more entries land confirming the outcome, the row transitions to its terminal status.

## Same-slot dedup sweep, group-aware.

The per-entry extractor sees `known_hearings` and is told not to allocate a new `hearing_key` for a slot another hearing already occupies on the same docket — but a court order often describes a hearing using different vocabulary than the stipulation that proposed it ("Motion Hearing" in the signed order vs "Hearing on Motion for Summary Judgment" in the joint stipulation), and the LLM can fail to recognize the same event under the new label, allocating a fresh key with a `-N` suffix. The same failure mode also surfaces across CourtListener siblings of one logical PACER docket: the per-entry extractor processing a freshly-synced sibling can allocate a fresh key for an event the OTHER sibling already has under a different key (the didenko `sentencing-didenko` vs `sentencing-didenko-2` shape). The verify pass operates on one hearing at a time and has no view of siblings, so it can't catch either failure mode on its own (it'll happily CONFIRM both rows independently). Two end-of-sync sweeps close the gap, both clustering by the LOGICAL PACER slot `(docket_number, court_id, starts_at_utc)` rather than by CourtListener `docket_id`, so cross-CourtListener-sibling drift collapses into one cluster the same way intra-docket vocabulary drift does:
  - `_dedupe_concurrent_hearings` for FUTURE `scheduled` clusters: sends each cluster to `llm.resolve_duplicate_hearings`. MERGE_INTO picks a target_key; the syncer folds the siblings' `source_entry_ids` onto the target, appends a `[dedupe]` audit line on the target naming the absorbed sibling keys + the LLM's reason, and **DELETEs the sibling rows outright** via `Store.delete_hearing`. KEEP_BOTH and UNCLEAR are no-ops — the LLM is the safety net because two distinct proceedings can legitimately be stacked back-to-back at the same time on a single docket. Zero LLM cost on quiet syncs (no clusters → no LLM call); one extra call per duplicate cluster when one exists.
  - `_dedupe_concurrent_held_hearings` for `held` clusters (past OR future): no LLM call needed, deterministic merge. A court cannot have *held* two hearings simultaneously on the same logical PACER docket — same-slot held is unambiguously a key-drift duplicate. Canonical row is picked by `(-len(source_entry_ids), last_updated, hearing_key)` (most sources, oldest, alphabetically first); siblings get their `source_entry_ids` merged onto the canonical, the canonical's `audit_notes` gets a `[dedupe-held]` line naming the absorbed sibling keys, and the **sibling rows are DELETEd outright** via `Store.delete_hearing`. Earlier behavior flipped siblings to `status='cancelled'` and kept the row, which inflated H_canc deviation in the provider scorer — each key-drift artifact counted as a spurious cancellation in the totals even though it was never a real court-ordered cancellation. The audit trail of which keys were absorbed now lives on the canonical row instead. The held sweep runs AFTER the scheduled sweep so any RESCHEDULE_HEARING / CANCEL_HEARING outcome from the LLM-driven scheduled pass has settled first. `cmd_sync` adds the case's calendar to `affected_calendars` when any of `stats["deduped"]`, `stats["deduped_held"]`, or `stats["deduped_nearslot"]` is non-zero, so a sweep-only sync still re-renders the ICS — otherwise an absorbed sibling staying in the cached ICS would survive until the next sync touched an entry.
  - `_dedupe_nearslot_hearings` for NEAR-slot clusters the two exact-slot sweeps above miss — the class that surfaced when Gemini became the extraction default: it proliferates a fresh `hearing_key` for a proceeding it already has at a *near* (not identical) slot, which then renders TWICE. `Store.find_nearslot_hearing_clusters` surfaces two tiers of candidate, both keyed by the LOGICAL PACER slot `(docket_number, court_id)`: (1) **same court-local date, different exact slot, within one status** — a date-only midnight row plus a timed row of one proceeding, or two same-day vocab-drift keys; (2) **same SINGULAR proceeding type + defendant, `held`, across dates** — a once-only proceeding (sentencing / arraignment / change-of-plea / initial-appearance, per `store._singular_defendant_base`, which collapses `sentencing-mcgonigal` and `sentencing-mcgonigal-2` to one base) recorded at both its scheduled date and the date it was actually held (the `sentencing-mcgonigal` @12-18 vs `sentencing-mcgonigal-2` @12-14 shape). Clusters are DISJOINT (the singular tier skips rows the same-date tier already claimed). Each goes to `llm.resolve_duplicate_hearings` — this sweep is LLM-gated, NOT deterministic, precisely because a court CAN hold two DIFFERENT hearings on one day at different times (a morning motion hearing + an afternoon status conference), so only the model, seeing the titles + docket text, should decide MERGE_INTO vs KEEP_BOTH. The shared `DEDUPE_HEARING_SYSTEM_PROMPT` was generalized from "EXACT same slot" framing to cover near slots and tells the model, for a once-only proceeding recorded at two dates, to MERGE_INTO the row whose date a minute entry / verdict / judgment confirms (the HELD date over a stale scheduled date). MERGE_INTO DELETEs the absorbed rows with a `[dedupe-nearslot]` audit line; runs AFTER the two exact-slot sweeps (so they've already merged identical-slot dups, leaving this only the near-slot residue). The model-comparison harness (`build_provider_stores.py::_replay_case`) MUST call this alongside the other sweeps — its sweep list mirrors `sync_case`, and a sweep added to one but not the other silently doesn't run in the comparison build.
  - **Remaining gap (narrowed):** the near-slot sweep clusters WITHIN a single status (tier 1) and the singular tier is `held`-only, so a logical hearing that is `held` on one CourtListener record of a group but `scheduled` at a different slot on a sibling record — CROSS-status near-dup — is still collapsed by none of the three sweeps. Closing it has to reach across status without re-introducing the over-merge risk the LLM-gated tiers guard against; left for a focused follow-up (render-time group collapsing, like `calendars/index.py` already does for summaries, remains the likely shape).

## Multi-record groups are unified for the extractor so it stops MINTING `-N` drift keys (prevention), and the dedupe merge prefers the suffix-free survivor (cleanup).

Two coordinated pieces fix the "Sentencing Lytvynenko 2" class — a drift suffix that leaked into the subscriber-facing title because the title falls back to `key.replace("-", " ").title()` when the LLM gives no explicit title.
  - **Prevention (root cause).** When CourtListener splits one logical PACER docket across several `docket_id` rows (the `pacer_case_id` reconciler, bug #7345 — [Case summaries are keyed by logical PACER docket, not by CourtListener docket_id](#case-summaries-are-keyed-by-logical-pacer-docket-not-by-courtlistener-docket_id)), the extractor's Cross-docket rule (`llm.py` SYSTEM_PROMPT, "NEVER apply an update/terminal action to a known row whose docket_id differs from the entry's") MISFIRES: the two records ARE the same docket, but processing the second record's entry the model sees the existing key on the other `docket_id`, is forbidden to reuse it, and mints `sentencing-lytvynenko-2`. `CaseSyncer._handle_entry` now resolves the current docket's group via `Store.get_docket_group_ids(docket_number, court_id)` and, when `len(group) > 1`, passes `group_docket_ids` + `canonical_docket_id = min(group)` (a STABLE pick, unlike the method's `date_modified DESC` ordering) into `llm.extract_actions`. `build_user_message` renders every in-group `docket_id` AS the canonical id — the entry's own line AND each known-hearing / known-deadline line — so the cross-docket rule reads "same docket" and the model reuses the existing key. Only the PROMPT VIEW is normalized; the STORED `docket_id` and the telemetry `docket=` stay the real CourtListener id (provenance / per-docket token accounting unaffected). Scoping is exact: genuinely-distinct dockets differ in `docket_number` or `court_id`, so they're not in the group, not normalized, and the rule still fires for them (district + appellate, co-defendants on separate dockets). No system-prompt prose change was needed — normalization alone makes the existing rule do the right thing — so `SYSTEM_PROMPT` stays at its ~9302-token cache size.
  - **Cleanup (safety net).** When a dedupe cluster holds both `base` and `base-<digits>`, `sync._canonical_drift_key` returns `base` and the merge re-points the survivor to it regardless of `source_entry_ids` count — so residual drift the prevention misses still collapses onto the clean key (and clean key-derived title / ICS UID / event ids). Applied in the EXACT-slot sweeps ONLY: the LLM-gated `_dedupe_concurrent_hearings` (via `_apply_dedupe_action`, default `prefer_canonical=True`) and the deterministic `_dedupe_concurrent_held_hearings` (via the `_rank` canonical tier). The near-slot sweep passes `prefer_canonical=False` because there the rows sit at DIFFERENT dates and which survives is SEMANTIC (a singular cross-date merge must keep the HELD date, which may be the `-N` row) — the LLM's `target_key` encodes that and must not be overridden by a cosmetic key preference. A `base-N` survivor whose own title is key-derived also adopts an absorbed sibling's EXPLICIT title via `sync._best_dedupe_title` (the `_best_proceeding_notes` precedent for titles); when every title is key-derived the survivor's now-canonical key already yields a clean one. Together these mean a meaningful trailing `-N` (sequential `status-conf-X-2..9`, `trial-X-day-2`, date suffixes, `-docket-<id>` records) is NEVER stripped — it only collapses when it duplicates a same-slot logical event. Rows already collapsed in the store before this landed (the suffix-free sibling already deleted, so re-sync can't re-cluster them) are repaired by `scripts/heal_drifted_keys.py` (see the maintenance-scripts section).
  - **Name-aware fallback title (the redundant-name half).** Collapsing the key to `base` still left the subscriber-facing title as the humanized key — `sentencing-lytvynenko` reads "Sentencing Lytvynenko", since the key carries the defendant slug for disambiguation. The prompt itself says to append the defendant ONLY on a multi-defendant docket (`llm.py` SYSTEM_PROMPT "--- Titles ---"), so the fallback now applies that rule deterministically: `sync._fallback_hearing_title(key, defendant_names)` drops the defendant token on a single-defendant docket ("Sentencing") and renders it `"<Type> - <Name>"` on a co-defendant one ("Sentencing - Prince"). `defendant_names` comes from `sync._case_defendant_names`, which reads the non-vocabulary tokens of the case's own hearing keys against the conservative `sync._HEARING_TYPE_WORDS` proceeding-word set — a case is single-defendant iff that set has ≤1 distinct name. The set is conservative on purpose: a proceeding word MISSING from it only biases a case toward "multi-defendant" and KEEPS the name (the safe direction, no wrong strip), so only unambiguous court-vocabulary words belong in it, never plausible surnames. The fallback is wired into both hearing write-time sites (`CaseSyncer._hearing_fallback_title`) so NEW untitled hearings come out clean, and the `heal_drifted_keys` Signal-1 rename uses it too. DEADLINES deliberately do NOT get this name-stripping — their keys are freeform ("govt-response-to-forfeiture-motion"), not "type + defendant", so a name heuristic would misfire; deadline fallbacks keep the plain humanization. Rows stored before this landed are repaired by `scripts/heal_key_derived_titles.py`, which recomputes ONLY currently-key-derived titles (an explicit LLM title is never rewritten) and judges single-vs-multi-defendant from each case's own keys (so genuine co-defendant dockets — Akhter twins, Ashtor + Prince, Zheng + English — keep their names).
  - **Abbreviation expansion in fallback titles.** Keys use abbreviations (`conf`, `govt`), so an untitled event's humanized fallback read "Status Conf" / "Govt Status Report" — jarring next to the model-titled "Status Conference" / "Government …" everywhere else (the only reason a row shows the fallback at all is the LLM happened not to title that one event; the abbreviation was always latent and the name-strip above merely made "Status Conf" stand out). `sync._KEY_ABBREVIATIONS` (`conf`→"Conference", `govt`→"Government") is applied by BOTH `_fallback_hearing_title` and the deadline-only `_fallback_deadline_title` (deadlines now route their write-time fallback through the latter instead of the bare `key.replace("-"," ").title()`). Expansion only ever runs on an UNTITLED row's key — an explicit LLM title is never passed through a humanizer — so a wrong expansion can only mildly mis-title a fallback, never corrupt a real title; keep the map to unambiguous court-vocabulary abbreviations that are single title-cased words (an acronym like `psr`→"PSR" would need separate casing handling — `.title()` would re-lowercase it — and is deliberately NOT in the map). `_title_is_key_derived` recognizes the expanded AND unexpanded fallback forms (the `expand=False` variant exists for exactly this) so `heal_key_derived_titles` — which now sweeps deadlines too — can recompute a row left "Status Conf" by an earlier name-only heal up to "Status Conference". The grounding (verified against a June 2026 prod store): every `status-conf-*` hearing across all cases read "Status Conference" from an explicit LLM title EXCEPT `us-v-schmitz/status-conf-schmitz`, which the LLM never titled.

## Deadlines deliberately have NO same-slot dedupe sweep — and must not get a deterministic one.

The three hearing sweeps above have no deadline analogue, on purpose. The deterministic `_dedupe_concurrent_held_hearings` is only safe because it rests on a PHYSICAL IMPOSSIBILITY: a single court cannot hold two hearings at once on one docket, so a same-slot `held` cluster is unambiguously key-drift. That premise does NOT transfer to deadlines — a single calendar date legitimately carries many genuinely distinct filing deadlines, so "two deadline rows share an exact `due_at_utc`" is normal, not evidence of drift. The first line of defense still covers deadlines (the per-entry extractor sees `known_deadlines` and is told to reuse an existing key rather than allocate a new one, which prevents most true duplicates at extraction time); deadlines simply lack the end-of-sync backstop sweeps that catch the residual key-drift for hearings. Grounding (verified against `us-v-ding`, docket `3:24-cr-00141` N.D. Cal., docket_id 68317014, in a June 2026 store snapshot): of 77 deadline rows, 11 slots had 2+ deadlines sharing an exact `due_at_utc`, and EVERY cluster was distinct deadlines coinciding on a date — multiple trial-day transcripts whose public-release or redaction windows land on the same day (e.g. `2026-05-01` carries the public-release deadlines for the 01-06, 01-07, and 01-09 transcripts, three different documents under well-formed date-suffixed keys, NOT `-N` drift), plus pairs of separate obligations set by one order (`govt-reply-pflaum-exclusion` + `parties-sealing-justification-submission` on `2026-01-07`). Zero were same-logical-deadline key-drift. A deterministic same-slot deadline merge would therefore have DELETED real deadlines — collapsing the three distinct transcript public-release rows on `2026-05-01` into one drops two genuine items from the feed, the exact agenda-clarity regression the design avoids. If a deadline duplicate problem is ever observed, the fix MUST be LLM-gated (the near-slot hearing sweep's shape, where the model decides MERGE_INTO vs KEEP_BOTH from the titles), never deterministic — but on current evidence the deadline duplicate rate is effectively zero, so even an LLM sweep would be cost with no benefit. The separate, lesser issue of a subscriber seeing several near-identical same-day same-type deadline rows is a PRESENTATION choice (optional cosmetic grouping in the agenda preview), not a store-level merge, and never involves deleting rows. (This note exists because a prior session misread these distinct-transcript clusters as "transcript-release key-drift" and proposed a deterministic deadline analogue of `_dedupe_concurrent_held_hearings`; the grounding above is why that's wrong.) **Carve-out — the one-time `heal_drifted_deadline_keys` heal is NOT this forbidden sweep.** The maintenance function `sync.heal_drifted_deadline_keys` (run via `scripts/heal_drifted_keys.py`, which covers both tables) DOES collapse deadline duplicates, but ONLY a literal `base` / `base-<digits>` drift pair coexisting at the SAME `due_at_utc` in the SAME logical PACER group — the exact cross-record drift the split-docket bug #7345 produces (`transcript-release-lytvynenko-12-17` on one CourtListener record + `transcript-release-lytvynenko-12-17-2` on its sibling). That is a different signal from "two deadline rows share a slot": the us-v-ding distinct transcripts are keyed `...-0106` / `...-0107` / `...-0109` (descriptive, NOT base/base-N of one another), so the drift heal can never cluster them. It is also a one-time heal, not an end-of-sync sweep, and the 0.18.x extractor's multi-record group canonicalization prevents NEW deadline drift at the source. The prohibition stands for any DETERMINISTIC SAME-SLOT deadline SWEEP (merging rows merely because they coincide on a date); the provable base/base-N drift heal is allowed precisely because it CANNOT touch coincidentally-same-day distinct deadlines.

## Per-row confidence pass (deterministic at temperature=0, source-entry-aware as of 0.11.0).

Every `scheduled` hearing — past OR future — and every `pending` deadline gets audited by a focused LLM call (the merged `VERIFY_SYSTEM_PROMPT`, called via `verify_hearing` / `verify_deadline` in `llm.py`); every PAST `cancelled` hearing does too. The model receives the candidate row + a window of relevant docket entries assembled by `CaseSyncer._verify_context_entries`, with THREE sources merged + de-duplicated: (1) the most recent hearing-relevant entries on the docket, (2) hearing-relevant entries filed AROUND the row's own date (backed by `Store.get_relevant_entries_in_date_range`, anchored on `date_filed` so an OCR / metadata re-sync that bumps `date_modified` can't hide them), and (3) the row's SOURCE entries — the docket entries that originally allocated the row, looked up via `Store.get_entries_by_ids`. The source-entry inclusion is the 0.11.0 fix for the McGonigal-trial-class regression: at temperature=0 the model would emit DELETE_HALLUCINATION on past rows whose scheduling order was older than both the recent and around-date windows, because the rule "you've seen the original source entry and concluded it does NOT actually schedule this row" was unsatisfiable. Source-entry context makes the rule satisfiable; the matching deterministic guard `CaseSyncer._delete_hallucination_allowed` enforces it the other way — DELETE_HALLUCINATION verdicts are downgraded to UNCLEAR with a WARN log if any source entry wasn't in the recent_entries the LLM actually received (a defense-in-depth backstop against the row being deleted from the store, source_entry_ids being malformed, etc.). The around-date-window is the older fix for the parallel past-hearing miss class: the entry that proves a hearing occurred (a minute entry / verdict / transcript / judgment) is filed on or shortly after the hearing date, and later docket activity can push it out of the most-recent window — feeding only the recent window left such rows stuck at 'scheduled' because the LLM never saw the evidence (the us-v-mcgonigal 12/14/2023 sentencing miss). All three context sources widen what the LLM SEES without loosening the bar for MARK_HELD; the prompt still requires a cited record. The verify call's `max_tokens` budget is 1500 (bumped from 512 in 0.11.0 — the us-v-knoot `trial-knoot` truncation observed at 512 was the canonical case). The pass returns one of: CONFIRM (no-op), RESCHEDULE_HEARING, CANCEL_HEARING, MARK_HELD (hearings), MARK_FILED (deadlines), REINSTATE (cancelled hearings only), DELETE_HALLUCINATION, or UNCLEAR. State coverage is asymmetric on purpose — future `cancelled` rows are NOT verified because a deliberately cancelled future row should stay cancelled until something actively un-cancels it. For past-dated rows the prompt is explicit on both sides: MARK_HELD requires a minute entry / verdict / transcript / judgment-after as evidence, not just an elapsed date; REINSTATE requires explicit positive signal that the cancellation is wrong, not just absence of evidence. Cost is one LLM call per non-terminal row per sync **on cases where a docket advanced** — as of 0.13.1 the whole verify / dedupe sweep set is skipped for a case whose dockets all hit the per-docket short-circuit (the verdicts can't change without new entries at `temperature=0`; see the gate note under the Three-tier short-circuit decision), so a quiet case costs zero verify calls. At the small/fast tier the per-active-case cost stays well under a cent per day even on dockets with many past rows (though the verify track is currently uncached on Haiku — see the prompt-cache threshold note above).

## CANCEL_HEARING/MARK_HELD on unknown keys with a date insert directly.

When the LLM emits a CANCEL_HEARING or MARK_HELD targeting a `hearing_key` that isn't in the store — common when a memo endorsement adjourns a hearing whose original scheduling entry was filtered out — and the action carries a `local_date`, the syncer creates a new row directly in the terminal status (`cancelled` or `held`). Without this fallback, those entries silently no-oped and the audit trail lost the proceeding. Without `local_date` we drop the action and log a warning so the operator can investigate. The LLM prompt now requires `local_date` on both CANCEL_HEARING and MARK_HELD specifically to support this path.

## MARK_HELD date-proximity validation.

When the LLM emits a MARK_HELD action carrying a `local_date`, the syncer compares it against the existing hearing's `starts_at_utc`. If they differ by more than 2 days the action is rejected and logged — the LLM probably matched the minute entry onto the wrong logical hearing (e.g. shoehorning a 3/6 CIPA hearing onto a 3/8 status conference). The real hearing's row gets caught by a properly-keyed MARK_HELD on a subsequent entry, or stays 'scheduled' if no such evidence ever materializes.

## Cross-entry context for the LLM.

Each LLM call gets two bonus context streams alongside the entry being processed: explicit cross-references parsed from the entry text (`granting 65 Motion ...` → entry 65) and the last 5 hearing-relevant entries on the same docket (temporal proximity). This is how `[Telephonic Pretrial Conference]` orders that don't say "CIPA" get correctly titled `Telephonic Pretrial Conference (CIPA)` — the originating CIPA motion lives a few entries earlier and the LLM picks up the framing. To make this work without burning storage on irrelevant entries, the `entries.description` column is populated only for hearing-relevant entries; filter-failed entries (notices, attorney appearances, etc.) keep a fingerprint stub but no text body.

## Per-entry known-events context is scoped to the current docket's court, not the case.

Aggregated multi-docket cases come in two shapes: (a) co-defendants on separate dockets in the *same* court — really one logical proceeding, contexts should aggregate; (b) parallel actions in *different* courts (e.g. district + circuit appeal + sibling district under a different statute) — separate proceedings, contexts must NOT bleed. Without this scope, a "stay appellate proceedings" order on docket B in court Y would surface court X's hearings and deadlines in the LLM context and trigger CANCEL_HEARING actions against them. `Store.get_hearings_in_court` / `get_deadlines_in_court` do the LEFT JOIN against `dockets.court_id` so the per-entry extractor only sees same-court siblings. Rows with NULL docket_id or NULL court_id are included so a warm-up sync doesn't silently drop context. The end-of-sync verify pass is naturally docket-scoped (it reads `get_recent_relevant_entries(docket_id, …)`) and didn't need a fix.

## Iterate oldest-first within a sync.

The CourtListener API returns entries newest-first (which gives us a cheap `modified_after` early-stop), but cross-references resolve cleanly only when an entry's referenced motion is already in the local store. So `iter_entries` buffers a docket's worth of new entries and yields them sorted ascending by `date_modified`. Memory cost is bounded by `max_pages * page_size`; even a first-time sync of a busy docket is well under 10 MB.

## No artificial cap on Retry-After.

CourtListener's daily request bucket is small — 125/day on the free tier, rising with a paid Free Law Project membership (Tier 1 at $10/mo is 300/day, up to Tier 4 at 1,400/day; see [free.law/membership](https://free.law/membership/)) — and can return Retry-After near 24h once exhausted. We honor it rather than abort, so a long-running sync resumes itself across the daily reset without manual restarts. We still log the URL, body, and rate-limit headers on every 429 so it's obvious which bucket fired (per-minute, per-hour, per-day) and whether requests are landing as anonymous (a CourtListener-flagged misconfiguration symptom).

## Filing deadlines tracked in parallel to hearings, in the same store.

The `deadlines` table mirrors `hearings` (stable kebab-case keys, `significance` major/minor, `gcal_event_id` for idempotent push, `source_entry_ids` for audit), and `cli._deadline_to_hearing` maps a deadline row into a hearing-shaped dict so ICS and gcal renderers stay deadline-agnostic. The hearing_key namespace is shared at the LLM layer (the addendum prompt explicitly tells the model to use a separate `deadline_key` field), and the renderers prefix the gcal/ICS UID with `deadline:` so a hearing and a deadline can never collide.

## Deadlines without an explicit time fire late in the court-local business day.

Most scheduling orders set a date but no clock time ("response due by 5/24"). The syncer fills in 4pm court-local time at write time (`sync.DEADLINE_DEFAULT_LOCAL_TIME = "16:00"`, applied in `_deadline_local_to_utc`) so the calendar pops a useful end-of-day reminder; the rendered duration is short (15 min). When the order DOES specify a time ("must be filed by 9 AM"), the LLM passes it through verbatim and it's used as-is.

## Deadlines tracked uniformly on every docket.

All cases (criminal, civil, appellate, magistrate, specialty) extract filing deadlines on every entry — no per-case opt-in, no docket-number auto-detect. The single `llm.SYSTEM_PROMPT` covers both hearings and deadlines; the renderer's significance gate (`major` vs `minor`) decides what reaches subscribers, which keeps procedural noise off calendars without losing the audit trail. The previous criminal-off / civil-on auto-detect was a cost optimization that produced inconsistent treatment of serious criminal cases (sentencing-memo deadlines, PSR objections, in-limine briefing) and an unprincipled split between siblings of multi-docket cases (e.g. district criminal + appellate). Uniform tracking removes a configuration knob and keeps prompts apples-to-apples across all dockets — a hearing on a `-cr-` docket sees the exact same prompt as a deadline on a `-cv-` docket, so model behavior is comparable.

## Transcript filings — three distinct treatments.

Three rules in the deadline portion of `SYSTEM_PROMPT` distinguish transcript-related entries that look similar at the regex layer but warrant different action: (1) "ORDER for Transcript" / "Transcript Order" / "Order Form" entries are PRIVATE REQUESTS to purchase a copy of a transcript — they are NOT court orders or deadlines, emit IGNORE; (2) transcript-redaction-request deadlines (e.g. "Notice of Intent to Request Redaction due ...") are deadlines but procedural — extract with `significance="minor"` so the audit trail keeps them but subscriber calendars don't; (3) transcript public-release deadlines (the date a filed transcript becomes publicly viewable on the docket) are substantive — extract with `significance="major"`. The three are pinned by `tests/test_llm.py::TestSystemPromptTranscriptRules` so a future prompt revision can't silently drop them.

## Auto-emit closes the loop.

Both `cmd_sync` (after each `sync_case`) and the webhook handler (after each `_handle_docket_alert`) call `cli.emit_calendars(cfg, store, only_calendars=...)` scoped to the calendars whose rows actually changed. ICS write is microseconds and idempotent — there's no separate flush step. gcal and M365 push auto-run inside `emit_calendars` whenever the per-backend token cache file exists; there's no CLI flag for it on `sync` / `serve` / `emit`, so the operator's workflow is just "run `setup gcal` / `setup m365` once, then everything pushes". The one-time `setup` subcommand is its own command precisely because its OAuth browser flow can't happen headless. Emit failures inside the webhook handler are logged but never fail the ack — the store is already updated, and a CourtListener retry would just re-process.

## Case summaries are documents-only — never inference.

The case-summary LLM (`llm.generate_docket_summary`) must state ONLY what the source documents say. Do not add prompt rules that ask the model to infer status from the *absence* of records — e.g., do not nudge it to label defendants "fugitives" merely because no appearance / arraignment is in the docket. Conservative refusal to assert facts the documents don't support is the correct behavior, not a defect. This applies to every future revision of `SUMMARY_SYSTEM_PROMPT`. **Custody / fugitive status is the canonical instance:** state "remains a fugitive" / "remains at large" / "in custody" ONLY when a source document (the indictment, a disposition, an operator `extra_documents` NOTE) says so; when the record doesn't establish it, OMIT custody entirely — say NOTHING about it. Do NOT announce that the status is "unknown" or "cannot be determined" ("X's custody status cannot be determined from the available record") either: stating what the record does NOT show is pointless noise, not a useful fact, and it leaks the same absence-of-record failure mode the rule below forbids. This *reverses* the earlier "the honest output is UNKNOWN" framing — silence is the honest output, because a lay subscriber gains nothing from "we don't know whether X was arrested." (The deterministic guard now flags "cannot be determined from the" / "it is unknown whether" / "<custody|arrest|appearance> status … unknown" alongside the fugitive-by-absence inference.) Custody is never inferred from missing arrest/appearance entries in either direction. The prompt formerly *licensed* the fugitive inference (`"remains a fugitive" (when ... no apparent arrest is reflected in the docket)`) — that license was removed because it directly contradicted this rule. The us-v-jin / us-v-berezhnoy regressions ("remain at large with no public docket entries reflecting arrests") were the result, and us-v-zewei (charged, then arrested and extradited) is the case where the inference would have been flatly false. See the post-generation truthfulness guard decision below for the deterministic backstop.

## Case summaries must not assert the ABSENCE of scheduling, activity, or disposition.

The complement of the documents-only rule above. The structured-events scaffold and the disposition set passed to the summary LLM reflect only what is visible in the public docket entries fetched from CourtListener; a docket that has been sealed at any point (initial sealing is routine in criminal cases pre-arrest, partial re-sealing can recur, and some dockets get re-sealed after a RECAP capture — the Dubranova `2:25-cr-00578 (C.D. Cal.)` regression that drove this rule is the canonical example) can have ongoing scheduling and filings that are simply not visible to us. Treating "the scaffold is empty" as "no hearings have been recorded" silently turns missing-from-RECAP into a positive claim about case posture. The matching `SUMMARY_SYSTEM_PROMPT` block forbids phrasings of the form "no hearings have been recorded", "no deadlines are set", "the case remains pending" (as a closing positive claim), "no disposition has been entered", "the docket shows no recent activity", and equivalents. The legal-terminology list at the top of the prompt no longer recommends "remains pending" / "case remains pending" because that phrasing was the primary vehicle for the failure mode. Future prompt revisions must preserve the forbidden-phrasing list as essential. **The rule is about the CLAIM, not the wording**, and **hedging does not exempt it**: the model reworded around the literal forbidden strings (us-v-berezhnoy: "no disposition documents have been **filed**", "the docket **does not reflect** any scheduled hearings", "no judgment is reflected **in the available record**"). For hearings / deadlines / dispositions the required output is SILENCE — describe the charges and any documented disposition, then stop. **There is no longer a custody-status exception:** custody, too, is omitted when undocumented (the earlier "→ unknown" carve-out was removed — see the documents-only rule above — because "X's custody status cannot be determined" is itself pointless absence-of-record noise, and the operator decided a lay subscriber gains nothing from being told what we don't know). A related class is **speculative / conditional future outcomes and routine sentencing boilerplate** — these are also forbidden: a consequence hanging on an event that hasn't happened ("X will be remanded to the Bureau of Prisons if a term of imprisonment is imposed", "if convicted, X faces …", "should the court impose …") is an unknown dressed up as a fact, and the mechanics of sentencing are not news. Keep the scheduled EVENT and its date ("sentencing is scheduled for June 3, 2026") but drop the hypothetical-consequence clause. The deterministic guard flags `if (convicted|found guilty)`, `if a (term of )?(imprisonment|…) is imposed`, and `should the court impose`. (us-v-martino.) A related figure-fabrication rule lives in the same prompt: **state a specific dollar amount (restitution / forfeiture / fine) only when it appears LEGIBLY in the provided document text — never reconstruct one from garbled OCR.** Court forms with hand-filled amounts OCR into noise (the us-v-chapman restitution "Total" line came through as "Total AD2, O52. 1S"), and the model decoded that garble into a confident-looking dollar amount that differed from one run to the next — neither value is trustworthy, and we have no clean source for the real figure. When the amount isn't cleanly legible, state that the obligation exists without a number; do NOT decode it (and neither should a contributor — recording a specific figure read off that garble in docs or comments is the same error). **And when restitution specifically is unreadable, drop the OTHER monetary figures too — even legible ones.** `summary._restitution_amount_unreadable` detects the case (a *granted* restitution order is among the dispositions — keyed off the strict `_is_disposition_document` classifier so a typed *proposed* order on a motion never counts — but no clean `$N,NNN` extracts from it) and `summarize_docket` passes a `DOCKET FINANCIAL ADVISORY` into the prompt; the matching `SUMMARY_SYSTEM_PROMPT` rule then suppresses all specific dollar amounts for restitution AND forfeiture (money judgments / dollar-sum forfeiture), collapsing to "was ordered to pay restitution" (and "and forfeiture" generically). Otherwise a separately-printed forfeiture order's legible figures (us-v-chapman: a clean $284,666.92 + $176,850 from order #43) read to a subscriber as the defendant's total liability while the larger, unknown restitution (handwritten in entered order #49) is invisible. The fixed special assessment and identified-property (non-dollar) forfeiture are the exceptions — neither is a dollar total a reader would sum. This deliberately overrides the "imposed sentence figures must appear" and "identified-property forfeiture stays" guidance for dollar amounts, and only while the advisory is present.

## Post-generation truthfulness guard on case summaries (detect + prevent).

Five coordinated layers stop the summary LLM from publishing claims the documents don't support. (1) **Prompt rules** (the prevention) — the documents-only custody-OMIT rule (undocumented custody is omitted, NOT framed as "unknown"), the silence-on-procedural-absence rule (phrasing-invariant), the no-speculative/conditional-outcome rule (drop "if convicted …" / "… if a term of imprisonment is imposed" consequence clauses; keep the scheduled event + date), and the no-OCR-decode figure rule, all in `SUMMARY_SYSTEM_PROMPT` (above). (2) **A deterministic guard over the generated text** (the backstop, in `summary.py`), because prompt rules are soft and for a brand-new case there is no prior good summary to fall back to. `summary._audit_summary_text` flags absence-of-record claims (matched by CONSTRUCTION — negation + a procedural-record noun — not literal strings, so rewordings like "filed" vs "entered" are caught), custody "we don't know" noise ("cannot be determined from the" / "it is unknown whether" / "<custody|arrest|appearance> status … unknown" — also construction-matched), speculative/conditional outcomes ("if convicted" / "if a term of imprisonment is imposed" / "should the court impose"), and ungrounded custody/flight claims (flagged only when the claim's keyword is absent from the source-document corpus, so a document-stated fugitive status passes). `_generate_guarded_summary` runs it, and on a hit **retries generation once** with the violation fed back via `generate_docket_summary(correction=...)`, then keeps whichever attempt has fewer violations and logs a WARNING if any persist — the summary is never blocked ("retry-then-keep+WARN"). (3) **A grounding check for fabricated dates/amounts** — `summary._audit_summary_grounding` flags a "Month D, YYYY" date or "$N" figure in the prose that is traceable to NONE of: the structured-events scaffold (hearing/deadline dates), the source documents, or the operator-supplied notes the model is also given (the `aggregation_note` and any `extra_documents` notes). The match is whitespace-normalized, so a PDF line-break artifact like "June 26,\n\n2024" still matches. The operator-notes part of the corpus is essential — without it the guard false-positives on facts an operator deliberately supplied: the us-v-gholinejad regression was an appeal docket whose summary correctly cited a sentencing date conveyed from the sibling district docket via the `aggregation_note`, which the guard then flagged as fabricated because its corpus was documents-only. This class is **retry → strip → WARN**: because summaries auto-generate as a service, a WARN alone can sail unread into a subscriber's calendar, so a hit ACTS rather than just logging. On a hit `_generate_guarded_summary` retries generation once with the unsupported figures fed back via `generate_docket_summary(correction=...)` and an instruction to drop them or restate only what the documents support; if the retry comes back clean (and introduces no tier-1/2 regression) it is adopted. If the retry STILL carries an ungrounded figure, `summary._strip_ungrounded_sentences` deterministically removes whichever whole sentence(s) carry one (per-sentence detection reuses `_audit_summary_grounding`, so only grounded sentences survive; if every sentence is stripped the canonical refusal sentence is returned), and a WARNING is always logged so an operator can review the removal. The retry-before-strip ordering is what makes this safe for the FP-prone class: dates appear in nearly every summary and formatting variance ("5/6/26" vs "May 6, 2026", "$16M" vs "16,000,000") gives the check a real false-positive rate — re-shown the documents, the model can keep and reformat a figure it can actually support (recovering the false positive), and only a figure it genuinely can't support reaches the deterministic strip. The cost asymmetry vs the absence/custody class is now in the STRIP, not the WARN: a false positive that survives the retry costs a real sentence rather than only a log line, which is why the matching deliberately BIASES toward silence (month-only ranges unmatched, "X million" approximations skipped, amounts matched comma-insensitively as substrings). This caught the live us-v-chapman $492,052.15 fabrication and only false-positived on a line-broken date that whitespace normalization then fixed. `summary._split_sentences` does the segmentation, with an abbreviation guard (`U.S.`, middle initials, `Mr.`/`No.`/`Inc.`, case-caption `v.`) so a period mid-sentence isn't mistaken for a boundary; the backstop runs only after the model defied an explicit correction, so "good enough" segmentation plus the WARN is the right cost. (4) **A deterministic redundant-forfeiture guard** (tier 4) — `summary._audit_summary_forfeiture` flags a forfeiture MONEY JUDGMENT whose dollar amount equals a restitution amount also stated in the prose (the us-v-knoot "$15,100 in restitution" + "forfeiture money judgment of $15,100" → reads as $30,200 misread), and `_generate_guarded_summary` runs the same **retry → strip → WARN** shape as the grounding guard: retry once asking the model to OMIT the forfeiture per the redundant-forfeiture omission rule, and if it STILL states the doubled figure, `summary._strip_redundant_forfeiture` deterministically excises the forfeiture clause — surgically when it is joined to other content (keeping that content, e.g. a trailing "is appealing his conviction" clause on an appellate summary), dropping a whole sentence only when the sentence is solely the forfeiture statement. The detector tolerates the inline-link markup that may still wrap the phrase (`[forfeiture money judgment](doc:Dn)`) because it runs on the token-bearing prose BEFORE link resolution; it is scoped to the literal "forfeiture money judgment" wording so identified-property forfeiture (a named house / vehicle / wallet) is never touched, and amounts compare comma-insensitively. CAVEAT: detection is amount-equality only, so two CO-DEFENDANTS who each happen to draw the same-dollar restitution and forfeiture could trip it — rare, the retry gives the model a chance to keep them separate first, and a WARNING always logs for review (matching the grounding guard's risk posture). This became necessary when the appellate-summary disposition borrow (see the appellate-scope note above) began feeding the trial-court judgment — which carries the forfeiture order — into appellate summaries, widening where the soft omission rule had to hold and surfacing the violation on us-v-knoot's 26-5455 appellate summary. (5) **A deterministic stale-scheduled-date guard** (tier 5) — `summary._audit_summary_stale_event_dates` flags a "`<event>` is scheduled/set/continued for `<date>`" claim whose date matches NO scheduled scaffold row for that event type (sentencing / trial / arraignment / oral argument / change of plea; the court-local and UTC dates of the row both count as agreement), and `_generate_guarded_summary` runs the same **retry → strip → WARN** shape: retry once quoting the scaffold row's current date back to the model, then `_strip_stale_event_date_sentences` removes any sentence still carrying the stale date — the calendar feed carries the event's current date, so dropping the sentence never loses the schedule. This exists because the scaffold-outranks-documents prompt rule (the CRITICAL current-date rule in `SUMMARY_SYSTEM_PROMPT`) is soft and the grounding guard is structurally blind to this failure: the stale date genuinely appears in the source corpus. The us-v-ding case is the canonical instance — the 6/16/2026 minute entry ("Sentencing continued to 8/4/2026") rode into the disposition set, but the 7/22/2026 clerk's notice that moved sentencing to 9/1/2026 did not (a plain scheduling notice is neither a primary document nor a disposition, so it can never reach the document set; the hearing reschedule it drives DOES flip the summary stale and refresh the scaffold), and the regenerated summary said August 4 while the calendar feed rendered from the same store said September 1. The matching BIASES toward silence like the grounding guard: historical phrasing between the event and the verb ("was scheduled for", "originally set for") never flags, an event type with no scheduled scaffold row never flags (the claim may narrate history the scaffold no longer carries, or describe a deadline), and a date matching ANY scheduled row of the type passes (co-defendants each with their own sentencing). All five layers are hermetically tested (string in → expected violations/warnings/removals out); the prompt rules can't be unit-tested for model obedience, which is exactly why the deterministic guard exists. A RELATED deterministic cleanup (not a truthfulness layer — the fact here IS supported) handles operator-note PROVENANCE leaks: the `aggregation_note` and any `extra_documents` NOTE are trusted CONTEXT for the model, never material to attribute to the reader, but the model occasionally writes "the operator notes that he has appealed his conviction" (the us-v-knoot district summary). `summary._audit_summary_attribution` detects the operator-AS-SOURCE lead-in ("the operator notes that …", "according to the operator", "per the operator's note") — scoped so an unrelated "operator" in the prose (a "laptop-farm operator") is untouched — and `_strip_operator_attribution` removes ONLY the lead-in, keeping the clause it introduced (capitalized if it opened a sentence) since the note is trusted and the fact stands on its own. It is applied unconditionally at the end of `_generate_guarded_summary` (always-safe, no retry); the matching `SUMMARY_SYSTEM_PROMPT` rule forbids referencing the operator / aggregation note / "provided context" as a source at all, so the words "operator" / "aggregation note" / "the note" / "provided context" never appear in output.

## Programmatic sealing detection surfaces a DOCKET VISIBILITY ADVISORY to the summary LLM.

Phase 2 sibling of the absence-of-activity rule. `summary.detect_sealing` walks a small slice of the docket (one page oldest-first + one page newest-first) and flags the docket as currently sealed-in-effect when ALL of the following hold: (1) at least one visible entry matches "ORDER granting … to seal" (the `_SEAL_ORDER_RE` pattern), (2) no subsequent "ORDER granting … to unseal" is visible, (3) no disposition document is on the docket (passed in from `find_primary_documents`'s output, so an unavailable disposition wouldn't be classified as one and wouldn't kill the signal), and (4) post-sealing publicly-available entries number at most three. When all four align, the resulting advisory rides into the summary LLM's user message as a `DOCKET VISIBILITY ADVISORY` block (sibling to `AGGREGATION NOTE` — both are trusted operator-supplied metadata, distinct from document text), carrying the sealing order's entry number, filing date, verbatim description, and the observed available-post-seal entry count. The matching `SUMMARY_SYSTEM_PROMPT` rule tells the model to quote those facts and append a "subsequent docket activity may not be publicly visible" hedge, without speculating about what is happening behind the seal. The seal-then-unseal false-positive — common in criminal cases where the indictment is sealed at filing and unsealed at arrest — is suppressed by the three kill signals (unsealing order, disposition presence, substantial post-seal public availability). The walk costs roughly one extra `/docket-entries/` page per cold-cache summary; the cost is bypassed entirely when a disposition is already in hand (the `if dispositions: return None` short-circuit at the top of `detect_sealing`). Tuning the post-seal threshold downward favors more advisories at the cost of more FPs; tuning it upward favors more silence (Phase 1 behavior). The default of three is the empirical sweet spot for the Dubranova class of regressions — one or two post-seal available entries (e.g., a case summary plus a press release reference) is consistent with the seal still being in effect; four or more usually means the case is proceeding publicly.

## Document-quality issues never reach subscriber-facing prose; they get logged for operator review.

Phase 3 sibling of the absence-of-activity and sealing-advisory rules. When a primary document's extracted text is sparse or low-quality, the summary LLM has a SUMMARY_SYSTEM_PROMPT rule ("work around partial or low-quality source documents SILENTLY") that forbids meta-commentary like "The primary document text consists only of page-header citations" or "Based on the available minute entries, [defendant] is charged with…". The model has exactly two options on a partial input: produce a normal subscriber-facing summary from whatever signals it has (primary text fragments + structured-events scaffold + dispositions), or emit the canonical `SUMMARY_INSUFFICIENT_DOCUMENTS` refusal verbatim when even those are too thin. The narration path is explicitly forbidden because subscribers don't need to read about extractor failures — they need a case summary. To keep the failure mode VISIBLE to operators, `summary.summarize_docket` logs a WARNING whenever the extracted primary document text falls below 1500 chars (the empirical floor below which the project has only seen parser failures, not legitimately short charging documents). A sweep of the warning stream is how operators spot diluted summaries before subscribers do. The us-v-moucka regression that drove this rule was a full multi-count indictment (39K of clean `plain_text` on CourtListener itself) that extracted to \~4K of user_chars on our side because the locally-stored copy of the recap_document was missing its `plain_text` field — a parsing issue distinct from "the document is genuinely short". The warning treats both root causes the same way: log it, let the summary LLM produce the cleanest possible subscriber-facing prose from whatever signal exists, and let operators investigate the root cause separately.

## Automatically rebuild stale cached recap_documents at summary read-time.

The compact subset of each recap_document we store in `entries.recap_documents` is built by `store.compact_recap_documents` and includes `plain_text`. Rows synced before `plain_text` was a stored field have everything EXCEPT `plain_text`, and the entry fingerprint only encodes `bool(plain_text)` (not its content), so a fresh sync against a docket whose CourtListener `plain_text` flipped from empty to non-empty BETWEEN the original sync and now sees a matching fingerprint and skips re-processing. The us-v-moucka entry #1 stale-cache regression that drove this rule had that exact shape — locally-stored `plain_text` was empty even though CourtListener carried 39 KB of clean text. `summary._entry_looks_stale` detects the signature (any available, non-sealed main recap_document with empty `plain_text` in the stored copy), `summary._cached_entries_look_stale` runs it over an iterable, and `summary.find_primary_documents` falls through to CourtListener when it fires, AND uses `Store.refresh_entry_recap_documents` to rewrite the local store's recap_documents from the fresh CourtListener data — leaving the fingerprint alone so a real future content change still triggers the normal sync re-process path. The fingerprint-untouched discipline matters: re-fingerprinting here would silently mask "an actual content change just landed" on the next sync. Detection is conservative — sealed and unavailable recap_documents have legitimately empty `plain_text` and don't trigger the fallback. The staleness sweep covers BOTH primary documents (indictments, complaints) AND disposition documents (judgments, plea agreements, verdicts) because the same empty-`plain_text`-on-an-available-main-doc shape can hit either; when stale rows are detected, only those specific stale entries are dropped from the cache view (fresh ones stay) and the CourtListener fallback refreshes the recap_documents for whatever it surfaces in either classification. The whole arrangement means a once-broken cache row repairs itself on the next summary call, without operator intervention and without forcing a full re-sync.

## Case-summary prompt invariants — trial-vs-plea, sentence specifics, dated references, past-but-unconfirmed rows, redundant-forfeiture omission.

Five prompt rules in `SUMMARY_SYSTEM_PROMPT` are essential and must survive future edits:
  - A trial DATE in a scheduling order (or the `starts_at_utc` of a "Jury Trial" row in the structured-events scaffold) is NOT proof a trial occurred. A guilty plea entered before the trial date moots the trial — Fed. R. Crim. P. 11 doesn't itself use "vacate" and courts vary on whether they enter a formal vacatur order or just take the date off-calendar. The model may say "tried before a jury" / "a jury trial was held" only when a verdict form or judgment-after-trial document is in the disposition set. Without that, it must state the plea/dismissal/etc. and stop — without claiming a trial occurred.
  - When a judgment / sentencing-judgment document is provided, the actual imposed sentence (term of imprisonment, supervised release, fine, restitution) is the most important fact about the case and MUST appear in the summary. Vague phrasing like "was sentenced" is wrong when the figures are in the document. When the judgment is unavailable and only a notice of sentencing or a plea agreement's recommendation exists, the model must say "was sentenced on [date]" without speculating about terms — the plea recommendation is not the imposed sentence.
  - If you mention a hearing or deadline by referring to its schedule, STATE THE DATE. Vague phrases like "a trial date set", "a hearing is scheduled", "a final pretrial conference scheduled", "with a hearing pending" are forbidden — they let stale or unverified scheduling slip past a subscriber unnoticed. The structured-events scaffold provides `starts_at_utc` / `due_at_utc` on every row; quote them. If the scaffold lacks a date (null), describe the trigger in the court's own language or omit the reference entirely — never substitute the absence of a date with vague wording.
  - Past-dated `status=scheduled` rows must be described as past-but-unconfirmed, NOT as upcoming. A row whose date has elapsed without the verify-pass LLM being able to MARK_HELD it (no minute entry / verdict / transcript / judgment-after on the docket) AND without an explicit vacatur entry means the public docket is silent on the outcome. The model must NOT speculate about the cause (a sealed order, a missed minute entry, an abandoned case) — the docket alone doesn't say. The honest framing states the original date AND the unconfirmed status without inventing a reason: "a trial was originally set for June 12, 2024; no public docket entry confirms either the proceeding or its vacatur, and the case has continued actively since." The McGonigal/Shestakov regression — summary said "a final pretrial conference scheduled and a trial date set" without dates after the trial-mcgonigal and FPC rows were correctly reverted from cancelled/held to scheduled, because the model hedged on stale-but-not-confirmed-wrong dates — is the canonical case.
  - When a forfeiture money judgment against the SAME defendant equals the TOTAL restitution amount, OMIT the forfeiture money judgment from the summary. The forfeiture order stays on the docket — we choose not to surface it to lay subscribers when it's redundant with restitution, because the forfeiture-money-judgment terminology adds noise without adding information once the dollar figure is already stated as restitution. The Knoot us-v-knoot (3:24-cr-00151, M.D. Tenn.) regression — summary said "$15,100 in restitution ... with a forfeiture money judgment of $15,100 also entered against him" which reads to subscribers as $30,200 total — is the canonical case. Acceptable shape: "$15,100 in restitution, and a $100 special assessment" — full stop, no mention of the forfeiture money judgment. The earlier rule iteration that made the relationship explicit ("in the same amount" / "for the same $15,100") is also NOT acceptable; the operator's preference is to drop it entirely for lay readers. The rule is narrowly scoped by three guardrails the prompt spells out explicitly: (1) the orders must run against the SAME defendant — two co-defendants each ordered $15,100 in restitution are two independent obligations and must be described separately by defendant; (2) the forfeiture must be a MONEY JUDGMENT (in personam order to disgorge a proceeds amount), NOT forfeiture of identified property (houses, cars, bank accounts, cryptocurrency wallets) — identified-property forfeiture takes things, not money, and stays in the summary on its own; a mixed judgment that includes both drops only the money-judgment portion under this rule; (3) the forfeiture money judgment must equal the TOTAL restitution across all payees — a $15,100 forfeiture against $15,100-each-to-two-victims ($30,200 total) does NOT match and the forfeiture stays. Outside of all three, restitution and forfeiture are their own line items. The omission is a DELIBERATE, prompted suppression of redundant detail; the model must not generalize it into silent drops of OTHER financial obligations. Paired with this rule is a positive directive for equal-amount multiple-payee restitution (or forfeiture): when a single defendant is ordered to pay the same amount to N victims (e.g. "$15,100 each to Acme Corp. and Beta Inc."), every payee is its own order — the summary must state the TOTAL across all payees AND either name the payees individually or state how many there are ("$30,200 in restitution, $15,100 each to Acme Corp. and Beta Inc." or "a total of $30,200 in restitution, distributed equally among two victims at $15,100 each"). The vocabulary courts use varies — line-by-line in the judgment, an attached schedule, "restitution to victims as set forth in the attached schedule" — but a per-victim itemization in any form is N orders, not one. The two rules together close opposite failure directions: the omission rule prevents stacking redundant amounts that subscribers would sum to fictional larger totals; the multi-payee rule prevents silently dropping real distinct obligations. Because the omission rule is a soft prompt rule the model applies inconsistently, a deterministic tier-4 backstop (`summary._audit_summary_forfeiture` + `_strip_redundant_forfeiture`, see the post-generation truthfulness-guard note) detects the equal-amount forfeiture-money-judgment-vs-restitution pair in the generated prose and strips the forfeiture clause after a correction retry — the destination behind this prompt rule's bridge.

## Broad `is_disposition` for stale-flagging; narrow `_is_disposition_document` for the LLM doc set.

Two predicates in `summary.py` look alike but serve different jobs. `is_disposition` is intentionally broad: it matches a "Motion for Preliminary Injunction" because the motion's outcome is a disposition-class signal worth refreshing the case summary on. `_is_disposition_document` (used inside `find_primary_documents` only) is strict: it rejects motions, briefs, status reports, case-schedule orders (`ORDER RE 149 STIPULATION ... Motion for Summary Judgment due ...`), continuances, scheduling orders (`ORDER SETTING SENTENCING HEARING`), and minute entries of motion hearings (`Minute Entry ... Motion Hearing re: 6 Motion for Preliminary Injunction held ...`) — entries that mention disposition vocabulary in passing but are not themselves the ruling document. The split exists because feeding a 200-page PI brief or a procedural minute-entry to the summary LLM as "the disposition" produces summaries that describe the wrong posture: the original bug had `case_summaries` describing a granted preliminary injunction as "taken under submission" forever, because the only "disposition" the LLM saw was the motion text. Keep the two predicates separately tested — `is_disposition` should still flag motion practice; `_is_disposition_document` should only accept actual orders / judgments / minute-entries-of-decision.

## Paperless dispositions use the docket text as the document text.

"Electronic Clerk's Notes for proceedings held … Court imposes sentence: 92 months imprisonment, 3 years Supervised Release; $200 Special Assessment" is a real sentencing record — but it's a minute entry with the figures inline in the docket text and no PDF attached. `_attach_text` takes `allow_description_fallback`, enabled for the disposition slot only, so the description body reaches the LLM when no PDF extracts. Primary documents deliberately do NOT get this fallback — an indictment minute-entry stub is not an acceptable substitute for the real charging document, and the asymmetry is locked in by tests.

## Case summaries auto-refresh on docket changes.

`case_summaries.enabled: true` in the top-level config is the gate. The `case_summaries` table carries `stale` and `stale_since` columns per row; `sync.process_entry` flips `stale=1` and bumps `stale_since` to NOW whenever the entry it just saw matches `summary.is_primary_document` or `summary.is_disposition` (a superseding indictment, amended complaint, judgment, plea agreement, verdict, dismissal, dispositive memo). Two refresh paths:
  - `cmd_sync` (polling) calls `summary.refresh_stale` immediately at the end of the sync — polling is already batched into a single pass, so there's nothing to debounce, and the operator running `sync` wants results now.
  - `cmd_serve` (webhook) debounces: each delivery's `emit_fn` calls `_arm_debounce(only_calendars)`, which checks whether any docket on the touched calendars has `stale=1` and, if so, (re)starts a `threading.Timer` for `case_summaries.debounce_seconds` (default 300). Every subsequent delivery within the window cancels the existing timer and starts a new one, so a PACER batch upload that fires 5 webhooks across 90 seconds produces ONE Sonnet call, not 5. When the timer fires, it runs `refresh_stale` against the pending-calendar set and re-emits those calendars to update the index with the new prose. The immediate `emit_calendars` call still fires on each delivery for fast ICS / gcal / M365 push — only the expensive Sonnet step is debounced.
  Missing rows count as stale, so adding a case to the config seeds its summary on the next sync without a separate command. `case-calendar summarize --force` remains as an explicit per-case regeneration path; `case-calendar sync --force-summaries` is the equivalent for the polling flow — both bypass the stale check and regenerate every summary on the configured cases, useful after a prompt or model upgrade. Neither is required for the automatic flow.

  **Calendar-posture changes flip stale too, not just document landings.** `process_entry`'s `is_primary_document` / `is_disposition` check above only fires on document ENTRIES. But the end-of-sync verify and dedupe sweeps change a hearing's or deadline's *posture* (scheduled → held / cancelled / reinstated, or a reschedule) WITHOUT any new document entry — so the document-only trigger missed them, and a summary could go stale silently. The canonical regression is anthropic-v-dow 26-1049 (D.C. Cir.): the oral argument was correctly flipped to `held` by the verify pass, but the summary stayed frozen at "oral argument is scheduled for May 19" because the post-argument entries were supplemental-briefing orders matching neither predicate. The fix is a chokepoint in the store: `Store.upsert_hearing` / `upsert_deadline` flag the docket's summary stale (via `_flag_summary_stale_for_docket`, joining the caller's transaction, no commit of its own) whenever the upsert is a NEW row or changes `status` / `starts_at_utc` (hearing) / `due_at_utc` (deadline). Metadata-only re-saves (notes, dial-in, a dedupe source-id merge on an unchanged row) do NOT flip stale, so there's no churn. This is the one chokepoint every posture-changing mutation funnels through (the per-entry extractor actions, the verify passes, both dedupe sweeps), so it can't be missed; the `_auto_mark_passed_stale` deadline sweep uses a direct UPDATE rather than `upsert_deadline` and is deliberately excluded (a deadline's date merely elapsing is not a prose-changing event).

## Summaries use a higher model tier than the extractor.

Extraction is high-volume, short-context, classification-heavy — Haiku/Flash Lite/gpt-5.4-nano fit. Summaries are low-volume (one call per docket, rarely re-run), long-context (30-100K tokens of legal prose per docket), and synthesis-heavy with legal-terminology nuance — Sonnet/GPT-5.4/Gemini Pro fit. The two tracks are decoupled via separate env vars (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL`) and separate config keys (`case_summaries.provider` / `.model`) so neither track contaminates the other when defaults move.

## Per-docket summaries, not per-case.

Multi-docket aggregated cases (Anthropic v. Dow) get one summary per constituent docket, rendered as a labeled paragraph block in the index. Primary documents live on individual dockets, so this is the natural unit; a judgment landing in one docket only dirties that docket's summary, not its siblings'.

## Static legal disclaimers, never from the model.

"Case descriptions are generated by AI and may contain mistakes" and "Criminal defendants are presumed innocent unless and until convicted in a court of law" are rendered as a static `<footer>` block by the page template, NOT by the LLM. Keeping the legally-loaded text outside the synthesis step guarantees the language is stable regardless of model output or prompt revision. The prompt explicitly tells the model NOT to include these — they're added by the renderer.

## Calendar event previews are rendered into the index from the store, with ICS-identical filtering.

Subscribers can see what a calendar contains — the next hearings and deadlines — without subscribing to the ICS feed or opening it in a calendar client, because each case row on `index.html` carries a compact agenda preview (`calendars/index.py::_render_events`, fed by `_visible_events_for_case` + `_window_events` + `_decorate_event` from `build_calendar_models`). It is BUILD-TIME static, NOT a server backend: the same `emit_calendars` pass that writes the ICS files reads the `hearings` / `deadlines` tables and bakes the events into the page, so Caddy still serves one self-contained file (inline CSS/JS, no external requests). The load-bearing invariant is FEED PARITY — the preview must show exactly the events the ICS feed carries, no more and no fewer. `_visible_events_for_case` guarantees this by running the feed's OWN code rather than re-deriving the filter: it reuses `description.is_calendar_visible` (drops date-less rows and `significance='minor'`) plus the `status=='cancelled'` skip, and maps deadlines through the canonical `cli._deadline_to_hearing` (which folds `_DEADLINE_STATUS_MAP`, so a `met` / `cancelled` deadline is hidden just as the feed hides it — filtering on the RAW deadline status would wrongly surface already-filed deadlines, since `met`→`cancelled` in that map). The `cli` import is function-local because `cli` imports `index` at module top; pulling the helpers up into `index` would be an import cycle, so the cycle is broken at the call site instead. The preview is a WINDOW, not the whole feed: a small recent-past grace (`_EVENT_GRACE_DAYS`, default 14, for "what just happened") plus upcoming, capped at `_EVENT_MAX_RECENT_PAST` (2) past and `_EVENT_MAX_UPCOMING` (6) upcoming with an expandable `+N more` disclosure (native `<details>`, no JS, revealing the remaining upcoming events on click); times render in the courthouse timezone PER EVENT, so a multi-docket case spanning venues labels each row with its own court citation. `build_calendar_models` takes an injectable `now` so the window is deterministic in tests. The FULL visible set (not just the windowed rows) folds into the client-side search haystack via `event_search_titles`, so a proceeding is findable even when it's past the `+N more` cutoff. A render-time month-grid widget and any external / iframe ICS viewer were both rejected: the grid conveys less for far more bytes and is hostile on mobile / a 50–100-case page, and an embedded viewer breaks the self-contained, no-external-requests guarantee (the entire point of the static page is that one file is served). `tests/test_index_events.py` pins the parity (preview visible-set count == ICS `VEVENT` count over the full filter spectrum), the windowing / caps / grace, the court-local decoration (incl. date-only → `time TBD`/`time unknown` and bogus-tz fallback), and XSS escaping of event titles.

## Inline document links in case summaries (LLM picks the words, deterministic resolution validates).

The per-docket summaries hyperlink the words themselves, newspaper-style — the defendants "**were charged**" links to the indictment, "**pled guilty**" to the plea agreement, "**sentenced him**" to the judgment. The linked span is a SHORT action/term phrase (two or three words: the verb plus a word or two, or the short name of the order — "forfeiture money judgment", "preliminary injunction"), NOT a single bare word and NOT the trailing detail. The prompt is explicit that the link covers only the action, not the specifics that follow it — link "were charged", not "charged with wire fraud and five counts of money laundering"; link "was sentenced", not "sentenced to 60 months imprisonment and $2M restitution" — so the charge names, sentence terms, dollar amounts, and dates stay as plain text after the linked phrase. It is never a footnote marker / "[1]" / "(see Doc 1)" — readers see and tap the phrase. **The model is unreliable about exactly where the span starts and ends, so `summary._tidy_link_spans` deterministically normalizes the boundaries after resolution: it pulls an immediately-preceding auxiliary / linking verb (was/were/is/are/be/been/has/have/had) INTO the link ("was [charged]" → "[was charged]", looped so "has been [charged]" fully absorbs) and pushes a DANGLING trailing preposition (of/to/with/for/on/in/against/upon/into) OUT ("[convicted at trial of]" → "[convicted at trial] of", "[charged with]" → "[charged] with"). It only fires on a verb/preposition sitting flush against the link, so the prose's meaning is untouched — just the link boundary. This is the reliable guarantee; the prompt rule is the supporting nudge. The link target is the supporting document's PDF, resolved via `pdf.recap_document_url` (CourtListener storage → Internet Archive mirror fallback — CourtListener is the authoritative current source, IA a downstream mirror), the SAME preference the calendar event-body renderer uses (the two now share that helper — `description._document_url` delegates to it). The approach is LLM-driven, not a per-phrase regex map, for the reason every other extraction layer is: the model is the one that read the documents and knows which phrase each supports, and a deterministic phrase→doc-type table is both a treadmill and can't satisfy "any document referenced in any generated summary." The mechanism keeps the model inside the truthfulness discipline the rest of the pipeline enforces:
  - Each document fed to the summary LLM is shown with a **prompt-only reference token** in its header (`[D1]`, `[D2]`, … assigned by `summary._assign_document_refs` across primary documents, then dispositions, then `extra_documents`, in render order). The token NEVER reaches the page. The model links a phrase by writing a markdown-link-shaped marker whose target is the token: `[were charged with wire fraud](doc:D1)`.
  - `summary._resolve_document_links` rewrites those markers AFTER generation and BEFORE the prose is stored: a known token with a URL becomes `[phrase](real-url)`; a known token whose document has no reachable URL (paperless minute orders, not-yet-uploaded or sealed PDFs — `_entry_document_url` returns `None`, skipping sealed docs) collapses back to the bare phrase; a token the model **invented** (not in the set we fed it) also collapses to the bare phrase AND logs a WARNING (the link analogue of the grounding guard). So a summary can never link to a document that wasn't in its input set — the worst the model can do is link the wrong in-set document or leave a phrase unlinked, both of which are the same class of error as the prose itself and covered by the periodic prose-vs-tables audit.
  - **Ordering matters for the guards:** the post-generation truthfulness + grounding guards (`_audit_summary_text` / `_audit_summary_grounding`) run inside `_generate_guarded_summary` on the TOKEN-bearing prose, and resolution happens after. The `doc:Dn` token carries no date / dollar amount / URL, so it doesn't perturb the guards; resolving before them would inject real URLs the grounding date/amount regexes might trip on. Future refactors must keep resolution strictly after the guard pass.
  - **Storage:** the resolved `[phrase](https://…)` markdown lives in the existing `case_summaries.summary` column — no schema change. A document-URL availability flip is handled by the existing stale-regeneration machinery (the entry fingerprint flips, `sync.process_entry` flags the summary stale, `refresh_stale` re-resolves), so frozen-at-generation URLs self-correct on the next refresh.
  - **Span boundaries + weak targets:** `summary._tidy_link_spans` (above) normalizes verb/preposition boundaries. Two more rules govern WHAT is linkable: a brief direct object may stay in the span ("dismissed count three"), but a jury **verdict form is never a link target** — `_is_verdict_form` maps it to `None` in the link map even when it has a URL, because its extracted text is the checkbox template (identical blank or filled; the jury's marks aren't in the text layer), so linking "returned a verdict" to it is misleading (the us-v-ding regression). The form still rides into the prompt as summary context, but a matching `SUMMARY_SYSTEM_PROMPT` rule governs that context too: a verdict form confirms a verdict was RETURNED (and licenses "a jury trial was held") but NOT the per-count findings, so the model states the actual outcome (convicted/acquitted on which counts) only when the provided text states it, and otherwise says just "a jury trial was held and the jury returned its verdict on [date]" — never a vacuous "covering all N counts" clause (which conveys nothing, since every verdict covers the submitted counts). The specific convictions enter the summary once a judgment, which states them in text, is available. The result-bearing link target, when present, is that judgment. And a prompt rule keeps each docket's summary scoped to its own proceedings — purely-appellate events (appellate counsel appointment, appellate transcripts, the court-of-appeals case number) belong in the appellate docket's summary, not the district docket's, even though the district clerk dockets the notice of appeal and related appellate paperwork on the district docket (the us-v-gholinejad regression where the district summary narrated the appellate public-defender appointment). The flip side: an appellate CRIMINAL docket's summary must LEAD with the appeal and state the underlying conviction + sentence ON APPEAL (the 26-5455 / us-v-knoot regression, where the appellate summary said only that the defendant "was charged" and never the conviction or sentence under review). Two halves fix it: (1) `summary._borrow_primary_from_siblings` now borrows the trial-court sibling's DISPOSITION documents (judgment / plea / verdict / dismissal) alongside the charging document it already borrowed, so the conviction/sentence the appeal is about actually reaches the appellate summary instead of being discarded — an appellate docket re-files neither its charging document nor the trial-court judgment, so without the borrow the summary has only the borrowed indictment and falls back to a charges-only "was charged" framing; (2) the matching `SUMMARY_SYSTEM_PROMPT` rule tells the model to lead with the defendant as active appellant ("John Smith is appealing his conviction for … after being sentenced to …") and report the borrowed disposition (labeled `[from sibling <docket> <court>]`) as the established outcome rather than as still-pending allegations. The borrowed disposition text rides into the grounding corpus, so the sentence figures it states are grounded, not flagged. A separate prompt rule — "one ruling is one statement, even when it is recorded across more than one filing" — stops the model splitting a single decision documented across an opinion + the order implementing it into two parallel clauses just to attach a second inline link (the 3:26-cv-01996 / anthropic-v-dow preliminary-injunction regression, where "granted a preliminary injunction … and the accompanying preliminary injunction order enjoins …" read as two rulings; the fix collapses it to "granted a preliminary injunction …, enjoining …").
  - **Rendering + XSS:** `index._render_summary_body` parses only `[words](https?://…)` markers (URL stops at whitespace or `)`), emits `<a target="_blank" rel="noopener">` on the words, and HTML-escapes the prose, the linked words, AND the href — model output stays untrusted. Anything that isn't a well-formed http(s) marker (a stray bracket, a non-link parenthetical) renders as escaped literal text. `index._summary_plain_text` strips the markers to their words for the client-side search haystack so subscribers search the words they read, not embedded URLs. The `[phrase](doc:Dn)` token markers are the ONLY markdown the summary prompt allows; the "no URLs" rule in `SUMMARY_SYSTEM_PROMPT` still holds (the model writes the token, the pipeline fills in the link).

## `extra_documents` works around CourtListener data gaps without forging RECAP entries.

Some documents the public should be able to see don't surface through CourtListener even when they should, and the gaps come from different upstream layers. The canonical case is the Zewei docket (S.D. Tex. 4:23-cr-00523), which has two unrelated examples on a single docket: (a) entries 1-4 — including the indictment — remain sealed in the public record at the courthouse even though the seal was lifted in connection with extradition; this is an upstream sealing decision on the court's side, NOT a CourtListener data bug, and the workaround is `extra_documents` pointing at the DoJ press-release attachment that carries the indictment. (b) Entry 20's PDF (the agreed motion to designate the case as complex) sits in CourtListener's storage bucket but the v4 API reports `is_available: false` because the file was uploaded under an older `pacer_case_id` (`1940635`) than the docket's current one (`1940634`) and CourtListener's reconciler matches on the current id only ([CourtListener bug #7345](https://github.com/freelawproject/courtlistener/issues/7345)); the workaround is `extra_documents` pointing at the `storage.courtlistener.com/recap/...` URL directly. Three options were considered: (1) upload a second copy to RECAP as a workaround — muddies the upstream bug report and writes under-seal stamps into the public RECAP record; (2) auto-search out-of-band sources (DoJ press releases etc.) — brittle discovery, burns LLM tokens on every summary regen, and "we found a PDF on the internet" is a much weaker provenance claim than "we read this from PACER"; (3) `extra_documents` — a per-case YAML list on `CaseConfig` where each entry has just three fields: `docket` (must be one of the case's tracked docket ids), `url`, and a required natural-language `note` describing what the document is and why it was added. There is deliberately no `role` field: real out-of-band documents don't always slot cleanly into "pleading" / "disposition", the operator's note already carries the meaning a role taxonomy would, and the note is what the summary LLM actually reads. `summary._fetch_extra_documents` pulls the bytes via `pdf.extract_text_from_url` (same pypdf → OCR fallback chain as CourtListener recap_docs) and returns a single flat list that rides into the LLM prompt as its own `EXTRA DOCUMENTS PROVIDED BY OPERATOR` section, separate from the CourtListener-walk's primary-document and disposition slots. Each entry's LLM block carries the `OPERATOR-PROVIDED DOCUMENT (sourced outside CourtListener)` header and a `NOTE FROM OPERATOR` line; the summary system prompt is explicit that the note is trusted metadata describing the document's identity ("treat this as the unsealed indictment despite the SEALED stamps"), while the document TEXT is still untrusted in the same way as CourtListener/PACER text — matching the existing AGGREGATION NOTE convention. The `note:` field on each entry is *data fed to the summary LLM*, not operator scratchpad: keep it to one short sentence that identifies the document by name + case citation, and stop there. Bug numbers, `pacer_case_id` workaround details, "remove this entry once X ships" — all of that belongs in the YAML `#` comment block above the case in `config.yaml` (operator-facing) rather than in `note:`, because the LLM is summarizing a federal court case for public subscribers and any mention of CourtListener internals or tooling state in its output would be both off-topic and a leak of internal context. The content-gate that returns `None` when no primary-document text is available is extras-aware: extras alone satisfy it, so a docket whose only document is an operator-supplied URL still gets summarized (the canonical Zewei flow). Automatic dedup against CourtListener runs at the extracted-text level — `summary._filter_extras_already_in_cl` compares a normalized-text sha256 of each extra against the same fingerprint over every CourtListener-surfaced primary / disposition doc on the same docket group, and drops any extra whose body the CourtListener walk already produced. This catches the failure mode where the upstream gap closes after the operator added the entry: someone re-uploads the same PDF to PACER under the new `pacer_case_id` (the bug-#7345 reconciler shape, since FLP closed [the bug](https://github.com/freelawproject/courtlistener/issues/7345) without committing to a fix), or CourtListener's reconciler catches up on its own. Without this dedup, the same document body would reach the summary LLM twice — once via the primary slot, once via the extras block — wasting tokens and giving that document outsized influence. The dedup uses TEXT fingerprints (not URL match, not raw-bytes hash) deliberately: URL match misses cases where CourtListener surfaces the file under a different `filepath_local` than the operator-pinned URL, and raw-bytes hash breaks when one path runs through OCR and the other doesn't. Each drop is logged at WARN naming the URL, so the operator notices and can remove the now-redundant entry from `config.yaml`; the filter is the bridge, the operator's config cleanup is the destination. Misconfigured entries (missing field, docket id not on the case, note not a non-empty string) fail loud via `SystemExit` at config-load time so operator typos surface at boot rather than as silently-skipped documents.

## M365 idempotency without server-side determinism.

Google lets us pick the event id; Graph doesn't. To stay idempotent across reschedules, DB restores, and manual calendar edits, M365 push uses two cooperating mechanisms: (1) cache the server-assigned id in `m365_event_id` so the common path is one PATCH per push; (2) on cache miss / 404, recover via `$filter` against a stable `CaseCalendarKey` single-value extended property keyed to `sha1(case_id::hearing_key)` that we set on every event we create. Plus `transactionId` on the create body so a retried POST from a webhook storm dedupes within Graph itself. The result is the same end-state as the gcal flow: one logical hearing → one calendar event, surviving reschedules, restores, and at-least-once delivery.
