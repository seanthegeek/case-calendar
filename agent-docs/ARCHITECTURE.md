# Architecture

> Part of the Case Calendar project instructions, split out of [`AGENTS.md`](../AGENTS.md) so the always-loaded instruction file stays within agentic tools' context-loading limits. This is canonical project guidance ŌĆö the same status as AGENTS.md, not a private scratch note.


- **case_calendar/cli.py** ŌĆö CLI entry point. Subcommands: `sync` (pull updates from CourtListener into the store + auto-emit affected calendars; takes `--force-summaries` to bundle the equivalent of `summarize --force` into the same CourtListener session, so a prompt or model change can be propagated in one command instead of two, and `--reverify` to force the end-of-sync verify / dedupe sweeps to run even on cases whose dockets all short-circuited ŌĆö by default those sweeps are skipped for a fully-quiet case, see the gate note under the Three-tier short-circuit decision), `serve` (run the webhook receiver + auto-emit affected calendars per delivery), `reconcile` (cheaply re-check recent placeholder entries for the upstream enrichment the webhook path can't see ŌĆö a CourtListener docket alert fires once at entry creation and CourtListener does NOT re-fire a webhook when it later fills in the entry text / makes the document available, see [CourtListener issue #7423](https://github.com/freelawproject/courtlistener/issues/7423); re-fetches ONLY entries matching `sync.is_pending_enrichment` by id via `CourtListener.get_docket_entry`, so cost scales with filing activity not caseload ŌĆö `--days N` age cutoff, default 7, bounds retries on stubs that never enrich; see the placeholder-reconcile Key Design Decision), `emit` (write ICS + push to any configured gcal / M365 backend ŌĆö used standalone for forced re-renders), `setup gcal|m365` (one-time OAuth bootstrap that stages the token cache), `summarize` (generate per-docket AI case summaries; opt-in, gated on `case_summaries.enabled` in config), `show` (dump current hearings + deadlines), `prune` (delete every store row tied to a `docket_id` no longer referenced by any case in the config; dry-run by default, pass `--apply` to actually delete ŌĆö for the case where a docket was removed from `config.yaml` and its rows would otherwise be ignored forever), `webhook-url` (print the ready-to-paste CourtListener webhook URL, composed from `CASE_CALENDAR_WEBHOOK_SECRET` + `--host`; with `--check`, also probes the receiver's secret-gated health endpoint at `<url>/health` and reports whether the host is reachable, Case Calendar is actually answering, and the local secret matches the running receiver ŌĆö the operator-facing check that catches CF intercepts, stale Caddy configs, and secret mismatches in one shot). Loads `.env` via `python-dotenv` before any module reads env vars. Exposes `emit_calendars(cfg, store, *, only_calendars=None, setup_gcal=False, setup_m365=False)` as the shared rendering primitive ŌĆö both `cmd_sync` and the webhook server's `emit_fn` callback go through it, so a single update flows from CourtListener to ICS subscribers without a manual step. Push readiness for gcal / M365 is auto-detected via `_resolve_gcal` / `_resolve_m365` from config (credentials_path / client_id) + token-cache file presence; the `setup_*` flags only matter on the `setup` subcommand path, where they allow the OAuth browser flow to run when no cache exists yet. Also owns the deadline-to-hearing adapter (`_deadline_to_hearing` maps the deadline row into a hearing-shaped dict the renderers accept), `_compose_title` (the title prefixer that adds `[HEARING]` / `[DEADLINE]` / `[time TBD]` / `[time unknown]` flags), and `_maybe_ensure_docket_alerts` (calls `alerts.ensure_docket_alerts` once on `cmd_sync` / `cmd_serve` startup so each configured docket is subscribed to CourtListener docket alerts; gated on the `ensure_docket_alerts` top-level config flag, default true).
- **case_calendar/courtlistener.py** ŌĆö REST v4 client. Wraps `dockets/`, `docket-entries/`, `recap-documents/`, and `docket-alerts/` endpoints with backoff for 429 (`Retry-After`) and 5xx responses. Honors any `Retry-After` value (the daily 300/day bucket can return a wait near 24h; we sleep through rather than abort) and logs URL + body + rate-limit headers on every 429 so it's obvious which bucket fired. Iterator-style pagination via `iter_entries` (with a `modified_after` cutoff for cheap incremental syncs; pages newest-first to preserve the cheap stop-condition but buffers and yields oldest-first so cross-entry references resolve cleanly within a single sync) and `iter_docket_alerts` (one-shot list for the webhook-subscription reconciler). `create_docket_alert(docket_id)` POSTs a new subscription. `_get` and `_post` are thin wrappers around the shared `_request` retry loop ŌĆö both verbs use the same retry shape because CourtListener applies the same rate-limit + transient-failure characteristics to either. Request-rate telemetry: `_request` records a wall-clock timestamp for every HTTP request that reached the server (via `_record_request` ŌĆö including ones that came back 429/5xx and were retried, since those still spend quota; transport failures that never got a response are not counted), keeping a lifetime `_request_total` plus a rolling buffer pruned to the largest reporting window (`_REQUEST_WINDOW_MAX_SECONDS`, 24h). `log_request_stats` (called from `__exit__`, so it covers `sync` / `summarize` / `serve` shutdown uniformly) logs `courtlistener-requests total=ŌĆ” peak/min=ŌĆ” peak/hour=ŌĆ” peak/day=ŌĆ”`, where the peaks are the busiest rolling windows computed by the module-level `_peak_in_window` two-pointer sweep ŌĆö the number to compare against a Free Law Project / CourtListener API tier's hard per-minute/hour/day ceilings. No-op when no requests were made.
- **case_calendar/alerts.py** ŌĆö Reconciles CourtListener docket-alert subscriptions against the docket ids configured in `config.yaml`. `ensure_docket_alerts(cl, docket_ids)` lists the account's existing alerts once, creates a subscription via POST for any configured docket that isn't already covered, and returns a `{docket_id: 'exists' | 'created' | 'failed'}` status dict. List-call failures degrade gracefully ŌĆö every docket reports `'failed'` and no creates are attempted (avoids the "spam duplicates or skip blindly" failure mode against an unknown baseline). Per-docket create failures are logged at WARNING and don't abort the loop. Wired into `cmd_sync` and `cmd_serve` via `cli._maybe_ensure_docket_alerts`, gated on the top-level `ensure_docket_alerts` config flag (default true).
- **case_calendar/courts.py** ŌĆö Court-id ŌåÆ IANA timezone mapping. CourtListener doesn't expose timezones in its API, so we maintain the table here. Coverage is comprehensive: all 13 federal circuits, all 94 district courts, the Supreme Court, specialty courts (Federal Claims, CIT, FISA, Tax, Armed Forces), and territories (PR, VI, Guam, NMI, American Samoa). Multi-tz states (Tennessee, Florida, Indiana, etc.) map to the principal-office tz of each district. Unknown court IDs log a warning and fall back to ET so the gap is visible rather than silent.
- **case_calendar/extractor.py** ŌĆö Cheap regex pre-filter. `is_hearing_relevant` matches hearing-vocabulary entries; `is_deadline_relevant` matches filing-deadline vocabulary (`due by`, briefing schedule, scheduling order, stipulation, etc.); `is_extractable(entry)` is the combined gate the syncer calls ŌĆö any entry matching either vocabulary passes. Deadline tracking is uniform across all dockets now, so there's no per-case opt-in at the filter layer. Briefs, attorney appearances, and similar paperwork entries are skipped before any LLM tokens are spent. Intentionally over-inclusive ŌĆö false positives just cost an extra LLM call; false negatives lose hearings/deadlines.
- **case_calendar/llm.py** ŌĆö Hearing + filing-deadline extractor and case-summary generator. This module owns the domain PROMPTS and the high-level entry points; the provider-agnostic call layer it sits on ŌĆö dispatch across anthropic / gemini / openai, provider auto-detection from `LLM_EXTRACTION_PROVIDER` (track-specific) > `LLM_PROVIDER` (global default) > `*_API_KEY` (extraction-track key priority gemini > anthropic > openai) for the extraction track, with the parallel `LLM_SUMMARY_PROVIDER` > `LLM_PROVIDER` > `*_API_KEY` (summary-track key priority anthropic > gemini > openai) cascade for the summary track ŌĆö so a zero-config operator with all three keys set lands on the default split (Gemini extraction, Anthropic summaries), the small/fast model defaults, lazy SDK imports, the `cache_control: ephemeral` Anthropic prompt-cache marker, max-token-truncation detection, and token telemetry ŌĆö was extracted to the **`case_calendar.llmkit`** subpackage, which it calls via `providers._dispatch_llm_call` / `providers._detect_extraction_provider` (extractor track) / `providers._detect_provider` (base/summary track). `LLM_MODEL` overrides the per-provider default; the small/fast tier (Haiku, Flash Lite, gpt-5.4-nano) fits structured extraction, not deep reasoning. The summary track keeps its own higher-tier model selection here (`_DEFAULT_SUMMARY_MODELS`, `summary_provider_info`). The Anthropic prompt-cache discount currently covers ONLY the per-entry extract prompt ŌĆö the verify_hearing / verify_deadline / dedupe_hearings prompts all fall under Anthropic's per-model minimum-prompt-size threshold and aren't eligible for caching on the Haiku tier (see the matching [design note](DESIGN-DECISIONS.md#anthropic-prompt-caching-has-a-per-model-minimum-prompt-size-threshold-smaller-prompts-pay-full-input-rate) in Key Design Decisions). Four entry points:
  - `extract_actions` runs against ONE docket entry plus the case's known-hearings list AND known-deadlines list (always ŌĆö deadline tracking is uniform across all dockets), and returns 0+ actions: hearing actions (ADD_HEARING / RESCHEDULE_HEARING / UPDATE_DETAILS / CANCEL_HEARING / MARK_HELD / IGNORE) and deadline actions (ADD_DEADLINE / RESCHEDULE_DEADLINE / CANCEL_DEADLINE / MARK_FILED). A single merged `SYSTEM_PROMPT` covers both ŌĆö no conditional addendum, no per-case opt-in.
  - `verify_hearing` and `verify_deadline` BOTH use the merged `VERIFY_SYSTEM_PROMPT` (consolidated in 0.11.0 from the prior two-prompt design ŌĆö `VERIFY_DEADLINE_SYSTEM_PROMPT` is gone). The user message labels which row type ("CANDIDATE HEARING" or "CANDIDATE DEADLINE") and the system prompt has type-tagged action types: common to both (CONFIRM / RESCHEDULE_HEARING / CANCEL_HEARING / DELETE_HALLUCINATION / UNCLEAR), HEARING-ONLY (MARK_HELD / REINSTATE), and DEADLINE-ONLY (MARK_FILED). Each call receives the candidate row plus a window of relevant entries ŌĆö the most recent, the entries filed around the row's own date, AND the row's source entries (the docket entries that originally allocated it, included since 0.11.0). The source-entry inclusion makes the model's DELETE_HALLUCINATION rule satisfiable on rows whose scheduling order is older than both other windows (the McGonigal-trial regression class); pairs with the deterministic `CaseSyncer._delete_hallucination_allowed` guard that downgrades DELETE_HALLUCINATION to UNCLEAR when any source entry wasn't shown to the model. All domain LLM calls pin `temperature=0.0` (since 0.11.0) so the verdict is deterministic given the inputs.
  - `resolve_duplicate_hearings` is the same-slot resolver ŌĆö receives a CLUSTER of 2+ scheduled hearings on the same docket sharing the EXACT same `starts_at_utc`, plus recent docket entries, and returns one of MERGE_INTO (pick a target_key, the caller cancels the others and folds their source_entry_ids into the target) / KEEP_BOTH (rare ŌĆö stacked back-to-back proceedings) / UNCLEAR. A single court can't hold two hearings on one docket simultaneously, so the cluster almost always means the per-entry extractor split one logical event across keys (vocabulary varied between the entries that scheduled it).
  - `generate_docket_summary` is the **case-summary** entry point ŌĆö runs the primary document + disposition documents + a structured-events scaffold (hearings/deadlines with their statuses) through a higher-tier model and returns 2-4 sentences of prose plus the model identifier. Uses a separate provider/model selection (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL` env vars, with config-level `case_summaries.provider` / `.model` taking precedence) so the extractor pipeline's cheap defaults stay decoupled. Per-provider defaults are Sonnet (Anthropic) / Gemini Pro / gpt-5.4 (OpenAI) ŌĆö synthesis-heavy enough to warrant the upgrade. Returns plain text, not JSON; the per-provider callers expose a `json_mode=False` knob to support this.

  Each ADD_HEARING/RESCHEDULE_HEARING/UPDATE_DETAILS action (hearing or deadline) carries a `significance` field (`major` / `minor`) that controls calendar visibility ŌĆö substantive proceedings (trial, sentencing, oral arg, any pretrial conference, evidentiary hearing, plea, etc.) and dispositive briefing deadlines (MTD/MSJ response/reply, sentencing memos, motion-in-limine, Daubert, suppression briefing), plus recurring joint status reports / case-management statements (major on both the deadline side and the hearing side as of 0.13.0), are major; procedural-only events (a phone call set just to rule on a Motion to Continue, proposed orders following a settled disposition, attorney-appearance papers) are minor and filtered out at render time. Hearing significance is decided by the ordered `HEARING_SIGNIFICANCE_RULES`; deadline significance by the parallel `DEADLINE_SIGNIFICANCE_RULES` (both in `case_calendar/llm.py`, each substituted into `SYSTEM_PROMPT` and the hearing one also reused by `scripts/classify_significance.py`). The user message includes a "RELATED DOCKET ENTRIES" block (explicit cross-refs from the new entry's text + the last few hearing-relevant entries on the same docket) so the LLM can name a hearing correctly even when the entry that schedules it doesn't itself name the subject.
- **case_calendar/llmkit/** ŌĆö Provider-agnostic LLM call layer + token telemetry, kept as a self-contained subpackage with NO case_calendar domain knowledge (no court prompts, no hearing/deadline shapes) so it can be lifted out as a standalone project later. The package name is an internal placeholder pending a real one; nothing is published, so it does not compete for the PyPI/GitHub `llmkit` namespace.
  - `providers.py` ŌĆö the per-provider call wrappers (`_call_anthropic` / `_call_openai` / `_call_gemini` over their SDKs with lazy imports + Anthropic `cache_control: ephemeral` on the system block; **`_call_ollama` for local models**, which routes between Ollama's **native `/api/chat`** endpoint ŌĆö `_call_ollama_native`, the only endpoint exposing per-request **thinking control** via the `think` field ŌĆö and a `_call_ollama_openai_compat` fallback for a non-Ollama OpenAI-compatible server (LM Studio / vLLM / llama.cpp, detected by an `/api/show` miss). On real Ollama, a model reporting the `thinking` capability **thinks on every track** (extract / verify / dedupe / summary) with a **BOUNDED output budget** ŌĆö `num_predict = max_tokens + _ollama_think_budget()` (the answer allowance plus a reasoning headroom, default `_DEFAULT_OLLAMA_THINK_BUDGET = 8192`, overridable via `OLLAMA_THINK_BUDGET`). This REPLACED an earlier `num_predict = -1` (unbounded), which let runaway-prone models keep generating until the request hit the 600s timeout ŌĆö surfacing as a hang or an empty "No content" response rather than the clean truncation the unbounded design assumed (observed: `qwen3.5:9b`, a thinking model, ran its reasoning away on \~47% of entries in the scorecard run; `granite4.1:30b`, which does NOT report the `thinking` capability, ran its *output* away to the timeout and failed 100% ŌĆö the bound caps total generation either way). Local inference has no per-token cost, so the budget isn't a money lever; it's a runaway GUARD sized generously enough that disciplined thinkers are untouched (their reasoning fits well inside it) while a degenerate non-stopping generation truncates cleanly to the ordinary `OutputTruncatedError` (entry skipped) in seconds. This includes the case where a thinking model spends its WHOLE budget on reasoning and returns EMPTY content with `done_reason`/`finish_reason` = `length`: both Ollama paths (`_call_ollama_native` and `_call_ollama_openai_compat`) map that to `OutputTruncatedError` like the OpenAI/Gemini paths, rather than the bare `ValueError("No content")` they used to raise (which buried a routine budget-exhaustion truncation under a scary traceback). A genuinely empty response with a non-`length` stop reason still raises the `ValueError`. SUPPRESSING reasoning entirely is still wrong (an earlier design sent `think = false` on the high-volume tracks) ŌĆö that made weaker models **re-emit the KNOWN hearings/deadlines they were shown back as spurious actions** (the native re-emission: a single dense entry could dump 20-plus `ADD_DEADLINE` actions, mostly date-less, inflating extraction-error scores and slowing generation); letting the model reason fixes that and is measurably more accurate (gemma4:e4b full-benchmark deviation 1241 thinking vs 1945 not ŌĆö see `model-comparison/SCORECARD.md`). The escape hatch for a model that runs away or is simply too slow is `OLLAMA_FORCE_NO_THINK` (explicit `think = false` + bounded output for a boolean-thinking model; a no-op for the level-only gpt-oss family). Note Phase 3 of the benchmark found reasoning HARMS local SUMMARIES ŌĆö gemma / qwen / glm all graded worse thinking-on (qwen ran its reasoning away, glm hung), the inverse of the extraction result above (see `model-comparison/SCORECARD.md`) ŌĆö but the code still thinks on every track (the think decision in `_call_ollama_native` is purpose-independent for boolean-thinkers); `OLLAMA_FORCE_NO_THINK` is the operator setting for a summary-only local run, and no local model is recommended for production summaries regardless (Phase 3's best local grade was a C ŌĆö see the [local-default design note](DESIGN-DECISIONS.md#default-to-the-smallfast-llm-tier-for-extraction-not-flagship)). The trade is wall-clock only: a thinking model is slower per call (gemma \~20s, qwen3.5:9b \~41s vs \~7s not-thinking), which for a free local model is an accepted cost. The **gpt-oss family is the exception** (`_ollama_requires_thinking_level` / `_OLLAMA_LEVEL_THINKING_MODELS`): per the Ollama docs its reasoning CANNOT be disabled ŌĆö a boolean `think` is ignored ŌĆö only tuned by **level** (`low`/`medium`/`high`); its deepest trace is too slow for high-volume work, so the code sends `low` on the extract/verify/dedupe tracks and `high` for summaries, both still bounded by `max_tokens + OLLAMA_THINK_BUDGET`. **`OLLAMA_THINK_LEVEL`** (`low`/`medium`/`high`, resolved by `_ollama_think_level`) overrides that per-track default across every track at once ŌĆö the operator knob for gpt-oss reasoning depth, and the control the level-sweep benchmark uses to compare the levels; a malformed value falls back to the per-track default, and it's a no-op for boolean-thinking models (which use `OLLAMA_FORCE_NO_THINK`). The OpenAI-compatible endpoint ignores `think`, so the compat fallback has no thinking control (a heavy thinker there can still overrun ŌĆö prefer real Ollama, or a non-thinking model on a generic server). **Sampling on the Ollama path: greedy default, opt-in overrides.** The domain layer pins `temperature=0` (greedy) and the Ollama path FORWARDS it like the hosted paths ŌĆö greedy is the default. Greedy is off-spec for local thinking models (the Qwen3 card forbids it; it's below DeepSeek-R1's range) and is the documented trigger for the reasoning runaways, but a re-benchmark found in-spec sampling neutral-to-worse for the recommended local models, so it stays opt-in rather than default (full reasoning in the [Ollama-sampling-and-seed design note](DESIGN-DECISIONS.md#ollama-sampling-and-seed-greedy-temperature0-is-the-default-scorecard-valid-in-spec-sampling-is-an-opt-in-escape-hatch-not-the-default) in Key Design Decisions). `OLLAMA_TEMPERATURE` (`_ollama_temperature_override`) overrides the temperature (the escape hatch for a qwen/deepseek runaway ŌĆö set it to a card value like 0.6) and `OLLAMA_SEED` (`_ollama_seed`) pins the seed for reproducibility (necessary but not sufficient on GPU ŌĆö see [that design note](DESIGN-DECISIONS.md#ollama-sampling-and-seed-greedy-temperature0-is-the-default-scorecard-valid-in-spec-sampling-is-an-opt-in-escape-hatch-not-the-default)). Both apply to the native AND OpenAI-compatible paths; an unset/malformed value falls back (override ŌåÆ the caller's greedy `temperature`; seed ŌåÆ Ollama's own random seed). Native usage is parsed by `usage.from_ollama`. See `_ollama_think_budget` and `docs/local-llms.md`), the `_dispatch_llm_call` dispatch, provider auto-detection (`_detect_provider` for the base/global resolution reading `LLM_PROVIDER` then `*_API_KEY` in a `key_priority` order that defaults to the summary track's `anthropic > gemini > openai`; `_detect_extraction_provider` layers `LLM_EXTRACTION_PROVIDER` on top and passes the extraction track's `gemini > anthropic > openai` key priority so the extractor track defaults to Gemini and can be pinned independently of the summary track ŌĆö the two module-level tuples are `_EXTRACTION_KEY_PRIORITY` / `_SUMMARY_KEY_PRIORITY`), the small/fast `_DEFAULT_MODELS`, an optional `schema=` JSON-Schema parameter threaded to every backend for **structured output** (always on, no opt-out ŌĆö see the [Structured Output design note](DESIGN-DECISIONS.md#structured-output-schema-enforced-json-extraction-is-always-on); enforced per-provider: OpenAI `response_format` json_schema strict via `_to_openai_strict_schema`, Gemini `response_schema` via the `_to_gemini_schema` dialect adapter, Ollama native `format`, Anthropic forced tool-use (best-effort, NOT `strict` ŌĆö Anthropic's native `strict` / `output_config.format` structured output went GA in 2026, but 2026-06-11 live checks returned 400 for `ACTIONS_SCHEMA` under `strict:true` ŌĆö "Schema is too complex", or "Grammar compilation timed out" after \~3 minutes ŌĆö even though the schema sits within every documented strict limit: 16 union-typed params exactly AT the 16 cap, 16 optional params under the 24 cap. Anthropic support confirmed the cause is an internal compiled-grammar complexity ceiling beyond the documented caps, dominated by OPTIONAL parameters (each roughly doubles part of the grammar state space): the all-required variant ŌĆö 16 unions, 0 optional, exactly what `_to_openai_strict_schema` produces ŌĆö compiles and works, while minimal-required fails even at 14 unions (a hearing-only split) and only compiles at 9 (deadline-only). We stay on plain forced tool-use anyway: the compiling all-required shape is the exact change that halved Gemini's recall in the benchmark (see the [Minimal-required constraint](DESIGN-DECISIONS.md#structured-output-schema-enforced-json-extraction-is-always-on) under the Structured Output design note) and inflates output tokens (every action emits all 17 fields, mostly null), and splitting the schema in two doubles per-entry calls and breaks the single hearing-vs-deadline decision (a forced deadline-only probe relabeled a sentencing hearing `ADD_DEADLINE`). Plain forced tool-use handles the schema and was validation-clean ŌĆö do NOT re-add `strict:true` as-is, it 400s every call; revisit only with a re-benchmark of the all-required form on Anthropic, or after Anthropic raises the compilation ceiling), with shared helpers `_openai_json_schema_format` / `_STRUCTURED_OUTPUT_NAME`), `OutputTruncatedError` (raised when a model stops at `max_tokens` so callers skip the entry rather than parse a truncated half-JSON), and `provider_info`.
  - `usage.py` ŌĆö token telemetry (log-only; nothing persisted). `TokenUsage` normalizes the cross-provider difference (`input` = total prompt tokens with the cached portion INCLUDED, `output`, `cached` = served from cache, `cache_write` = written to cache ŌĆö Anthropic reports cache reads/writes SEPARATE from `input_tokens` so they're folded into `input`; OpenAI/Gemini already fold them). `from_anthropic` / `from_openai` / `from_gemini` read each SDK's response through `_as_int`, which coerces any non-numeric field (missing attribute, `None`, a test `MagicMock`) to 0 so telemetry can never turn a working call into a crash. `TokenLedger` (thread-safe ŌĆö the webhook server records from worker threads) accumulates per-docket, per-model, and grand totals for one process run; module-level `record` logs one `llm-tokens call ŌĆ”` line per call and folds it into the subtotals, `log_summary` dumps per-docket subtotals, then per-model subtotals, then a `ŌĆ” TOTAL ŌĆ”` line (which also carries `models=N`) at a run boundary, `reset` clears it. The per-model lines are what separate the extractor track's cost from the summary track's: the two run on different model tiers (cheap small/fast vs higher synthesis), so reading the run total by model tells you which track spent what ŌĆö and the startup `extraction LLM:` / `summary LLM:` log lines name which model is which. Model is the grouping axis (not a domain `extractor`/`summary` label) precisely so `llmkit` stays domain-free ŌĆö it buckets by the opaque model string the same way it buckets by docket. A model the price table doesn't cover shows `cost_est=?` on its per-model line (priced-or-not is per-model deterministic, so a model key present in the cost map ŌĆö even at `$0.0000` ŌĆö means it was priced).
  - `__init__.py` ŌĆö re-exports the stable public API over the underscore-prefixed internals: `dispatch` (= `_dispatch_llm_call`), `detect_provider`, `provider_info`, `OutputTruncatedError`, `DEFAULT_MODELS`, plus `usage`, `TokenUsage`, `TokenLedger`.
  - Wiring: `providers`' call functions invoke `usage.record(purpose=ŌĆ”, provider=ŌĆ”, model=ŌĆ”, tokens=ŌĆ”, docket=ŌĆ”)` after each SDK call (purpose Ōłł `extract` / `verify_hearing` / `verify_deadline` / `dedupe_hearings` / `summary`, threaded through `_dispatch_llm_call` from the callers in `case_calendar.llm`); `cli.cmd_sync` / `cmd_summarize` call `llmkit.usage.log_summary` + `reset` at the end of the run. `serve` is left out of the run-total path on purpose ŌĆö its webhook handlers and the debounce Timer thread would race on `reset`, so it relies on the always-on per-call lines instead (see the [design note](DESIGN-DECISIONS.md#token-usage-is-logged-from-real-counts-dollar-cost-is-an-opt-in-estimate-layered-on-top--both-log-only-not-persisted)).
- **case_calendar/costs.py** ŌĆö Rough USD cost ESTIMATOR plugged into the price-free `llmkit` ledger via `llmkit.usage.set_price_estimator` (wired in `cli._log_llm_setup`). `estimate_cost(model, TokenUsage)` looks the model up in `_RATES_USD_PER_MTOK` (exact id, then longest-matching prefix so a dated id like `claude-haiku-4-5-20251001` still resolves) and prices each token slice at its own rate ŌĆö uncached input (`input ŌłÆ cached ŌłÆ cache_write`), cache read, cache write, output. The table is hand-copied from each provider's published pricing page and dated with `PRICES_VERIFIED`; Anthropic, Gemini, and the current OpenAI 5.4 + 5.5 family (every `-mini`/`-nano`/`-pro` variant, so a variant can't prefix-fall-back to its base) are filled at their standard tier. Returns `None` for any unlisted model (a legacy model or an unrecognized `LLM_MODEL` override), which the ledger surfaces as `cost_est=?` and a partial-estimate note on the run TOTAL. Deliberately a domain-layer module (not in `llmkit`) so the reusable package ships no prices.
- **case_calendar/pdf.py** ŌĆö PDF text extraction with fallbacks. Order: CourtListener `plain_text` ŌåÆ `pypdf` on the downloaded PDF (CourtListener storage preferred, Internet Archive mirror as fallback) ŌåÆ `pdftoppm` + `tesseract` OCR (optional system dependency). CourtListener is the authoritative, current source today; the Internet Archive is a downstream mirror that can lag, so `pdf.recap_document_url` / `pdf.fetch_pdf_bytes` prefer CourtListener storage (`https://storage.courtlistener.com/<filepath_local>`) and keep `filepath_ia` only as a backstop. Sealed PDFs and not-yet-uploaded PDFs return `None` and are NOT cached ŌĆö the next sync re-fingerprints the entry and retries once the PDF arrives or gets OCR'd.
- **case_calendar/store.py** ŌĆö SQLite state. Tables: `dockets` (id + last `date_modified` (the polling short-circuit cutoff) + `date_last_filing` (CourtListener's "most recent document filing date"; what the index's "Last filing" column reads) + cached metadata for description rendering), `entries` (dedup of already-processed entries, keyed by `(docket_id, entry_id)` with content fingerprint, `entry_number` for cross-ref lookup, `description` / `short_description` / `recap_documents` JSON stored for entries that either passed the regex pre-filter OR matched `summary.is_primary_document` / `summary.is_disposition` ŌĆö every other entry gets a fingerprint-only stub so dedup still works without storing dead-weight notice text. `recap_documents` keeps a compact subset of each CourtListener recap_document ŌĆö `id`, `document_number`, `attachment_number`, status flags, `filepath_ia`, `filepath_local`, `plain_text` ŌĆö built by `store.compact_recap_documents`, that the description renderer turns into clickable per-doc URLs at emit time, and that the summary pipeline uses to find primary documents + dispositions locally via `get_entries_with_body` instead of re-fetching docket-entries pages from CourtListener right after sync wrote them down ŌĆö `plain_text` rides along in that stored subset so `pdf.extract_text` short-circuits without a PDF fetch), `hearings` (per-case logical hearings keyed by `(case_id, hearing_key)` with `significance` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list), `deadlines` (per-case filing deadlines keyed by `(case_id, deadline_key)` with `due_at_utc` / `status` Ōłł {pending, met, passed, cancelled} / `significance` / `deadline_type` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list ŌĆö parallel structure to hearings so they share renderers), `case_summaries` (per-docket prose summaries keyed by `(case_id, docket_number, court_id)` ŌĆö the logical PACER docket, not the CourtListener docket_id; see the [group-keying design note](DESIGN-DECISIONS.md#case-summaries-are-keyed-by-logical-pacer-docket-not-by-courtlistener-docket_id) ŌĆö with `summary` text, `model` identifier, `source_entry_ids` JSON listing the CourtListener entries that informed the LLM, and a `stale` boolean that `sync.process_entry` flips when it sees a new primary document or disposition so the automatic refresher knows to regenerate; populated by `summary.refresh_stale` during sync / webhook emit, or explicitly by `case-calendar summarize`), `courts` (CourtListener court metadata, citation_string), `webhook_events` (idempotency-key dedup). The `m365_event_id` columns are populated by `set_m365_id_for_hearing` / `set_m365_id_for_deadline` after a Graph create; gcal ids are deterministic so `gcal_event_id` is currently unused at runtime. Connection is `check_same_thread=False` so the webhook server's worker threads can use the same Store instance under a server-wide lock; the connection is also opened with `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000` so the polling `sync` process and the long-running `serve` process can safely share the same DB file (see the [concurrency design note](DESIGN-DECISIONS.md#sync-and-serve-are-separate-processes-sharing-one-sqlite-file-made-safe-with-wal--busy_timeout)).
- **case_calendar/sync.py** ŌĆö Per-case orchestration. For each docket: fetch entries newer than the store's last-modified cutoff, run the regex filter, optionally pull PDFs, resolve cross-references (`granting 65 Motion ...` ŌåÆ look up entry 65 from the local store) plus the last 5 hearing-relevant entries on the docket as temporal context, hand entry + known-hearings list + known-deadlines list + related entries to the LLM, apply returned actions to the hearings/deadlines tables. The known-hearings / known-deadlines context is sourced via `Store.get_hearings_in_court` / `get_deadlines_in_court` so it's scoped to siblings in the SAME court as the current docket ŌĆö same-court co-defendant dockets still share context, but parallel proceedings in different venues do not (a stay order in court B must not contaminate court A's events). `_needs_pdf` decides whether to pull an entry's PDF text for the LLM. It reads the PDF of EVERY order / endorsement entry that reached the filter (matched by `_IS_ORDER`), even when the description inlines a hearing time or names only the signing judge ŌĆö an order's operative dates routinely live ONLY in the PDF body (a schedule table under backward-looking "WHEREAS" prose, or follow-on deadlines a ruling sets) and aren't echoed in the one-line docket description, and a clerk naming the judge would otherwise trip `_DETAIL_HINTS` and suppress the fetch. The two narrower scheduling-order patterns (`_ORDER_GRANTS_SCHEDULING_MOTION`, `_SETS_SCHEDULE`) still run first to catch grant-a-scheduling-motion / stipulation phrasings that DON'T carry the literal word "order" (e.g. a minute entry granting a continuance). Conversely `_TRANSCRIPT_DOC` is checked FIRST and short-circuits to skip the fetch for transcript documents, transcript-purchase orders, and transcript filing notices ŌĆö a transcript is a verbatim record of testimony (often hundreds of OCR-heavy pages) with no forward-looking scheduling, and the held-date / redaction / public-release deadlines it implies all live in the docket description, never the body; the transcript guard precedes the order rule because "TRANSCRIPT ORDER" contains the word "order". Cost is bounded: only entries that already passed the hearing/deadline pre-filter reach `_needs_pdf`, `_maybe_fetch_pdfs` no-ops when no recap_document is fetchable, storage GETs don't spend the CourtListener API quota, and the LLM call happens regardless ŌĆö so the marginal cost of reading an order PDF is bandwidth plus a few input tokens. Local time ŌåÆ UTC conversion happens here using the court's IANA timezone. End-of-case sweeps run in order: (1) `_verify_scheduled_hearings` ŌĆö per-hearing LLM confidence pass for every scheduled hearing (BOTH future and past), fed the recent-activity window on the docket PLUS the hearing-relevant entries filed around the hearing's own date (`_verify_context_entries` ŌåÆ `Store.get_relevant_entries_in_date_range`) so a past hearing's outcome record (a minute entry / judgment filed days later) stays in view even when later filings pushed it out of the recent window; past-dated rows only flip to 'held' when the LLM cites a minute entry / verdict / transcript / judgment-after as evidence of occurrence ŌĆö date-passed alone is not enough, so trials vacated by plea stay 'scheduled' instead of being falsely marked 'held'; (2) `_dedupe_concurrent_hearings` ŌĆö resolves same-docket same-`starts_at_utc` clusters via `llm.resolve_duplicate_hearings`, runs AFTER verify so any RESCHEDULE_HEARING/CANCEL_HEARING from verify gets a chance to clear concurrency before the dedup LLM is consulted; (3) `_dedupe_concurrent_held_hearings` ŌĆö deterministic merge of same-slot `held` clusters (a court can't have held two hearings at once, so a same-slot `held` pair is unambiguous key-drift); (4) `_dedupe_nearslot_hearings` ŌĆö LLM-gated merge of the near-slot (not exact-slot) clusters the two exact-slot sweeps miss; (5) `_verify_pending_deadlines` does the same for pending deadlines, and `_auto_mark_passed_stale` flips past-due pending deadlines to `passed` (the one sweep that runs unconditionally). See the [same-slot dedup sweep design note](DESIGN-DECISIONS.md#same-slot-dedup-sweep-group-aware) for how the three dedupe sweeps divide the work. The verify and dedupe sweeps (verify-hearings, the three dedupe sweeps, verify-deadlines) are SKIPPED for a case when no docket in it advanced past the per-docket short-circuit this sync ŌĆö their inputs come entirely from the store, which is unchanged when nothing landed, so at `temperature=0` the verdicts can't differ from the prior sync (see the gate note under the Three-tier short-circuit decision). `_auto_mark_passed_stale` runs UNCONDITIONALLY ŌĆö it's the one sweep driven by the wall clock rather than docket changes, and makes no LLM call, so gating it would strand an elapsed deadline at `pending` on a quiet docket. `sync --reverify` forces all sweeps regardless of the gate. Exposes `CaseSyncer.process_entry` so the webhook server can reuse the exact same per-entry pipeline.
- **case_calendar/summary.py** ŌĆö Per-docket case-summary pipeline (opt-in, off by default). For each docket: hits CourtListener directly to find primary documents (`is_primary_document` matches `INDICTMENT` / `SUPERSEDING INDICTMENT` / `INFORMATION` / `COMPLAINT` / amended variants at the head of the entry description) and dispositions (`is_disposition` matches `JUDGMENT` / `VERDICT` / `ORDER OF DISMISSAL` / `PLEA AGREEMENT` / `MEMORANDUM OPINION` etc.), pulls PDF text via the existing `pdf.extract_text` fallback chain, gathers the docket's hearings + deadlines from the store as a structured scaffold, and hands the whole package to `llm.generate_docket_summary`. Dispositions get an extra fallback: when no PDF text extracts, the entry description itself is used as the disposition text (`allow_description_fallback=True` on `_attach_text`). This is how a paperless "Electronic Clerk's Notes for proceedings held ŌĆ” Court imposes sentence: 92 months imprisonment ŌĆ”" entry ŌĆö where the imposed sentence lives inline in the docket text with no judgment PDF attached ŌĆö still reaches the LLM. Primary documents deliberately do NOT get this fallback (a minute-entry stub is not a substitute for an indictment). When `store` is passed in (the normal path from `summarize_docket`), `find_primary_documents` reads from the local `entries` table first via `Store.get_entries_with_body` ŌĆö `sync.process_entry` now persists description + recap_documents for op/disp matches as well as hearing-relevant ones, so right after a sync the local cache already has everything the summary pipeline needs. Falls back to CourtListener only on a cold docket (first ever sync, or pre-fix data where op/disp entries are still stub-only); the CourtListener fallback walks two pages oldest-first for primary documents and three pages newest-first for dispositions (busy civil cases with heavy post-disposition briefing can push the dispositive order well past page 1). The picker for the LLM document set uses the stricter `_is_disposition_document` ŌĆö see the [broad-vs-narrow design note](DESIGN-DECISIONS.md#broad-is_disposition-for-stale-flagging-narrow-_is_disposition_document-for-the-llm-doc-set). Three entry points: `summarize_docket` runs the full pipeline for one docket and persists the row; `summarize_case` loops the dockets on one case (used by `case-calendar summarize`); `refresh_stale` walks all configured cases and regenerates any row marked `stale=1` (or missing entirely) ŌĆö this is the automatic path called from `cmd_sync` and the webhook `emit_fn` so a freshly-landed judgment regenerates the prose in the same emit cycle. No-document-text outcomes ALWAYS produce a row ŌĆö `summarize_docket` never silently drops a docket. Subscribers see a description block under every docket link on the index regardless of why a real summary couldn't be produced; the row carries one of FOUR distinct fallback messages so operators (and careful readers) can tell the failure modes apart at a glance, each operator-actionable in a different direction. (1) `llm.SUMMARY_PRIMARY_DOCUMENT_SEALED` ("The primary document(s) are currently sealed.") ŌĆö written when each identified primary entry's main recap_document has `is_sealed=True`. Operator-actionable: wait for the unseal; the entry fingerprint includes `is_sealed`, so an upstream unseal flips the fingerprint and the next sync's `process_entry` refreshes the cache automatically. (2) `llm.SUMMARY_PRIMARY_DOCUMENT_NOT_AVAILABLE` ("The primary document(s) are not yet available on RECAP.") ŌĆö written when each identified primary entry's main recap_document has no source for text (no `filepath_ia`, no `filepath_local`, empty `plain_text`, not sealed); literally nothing for the extraction chain to fetch. Operator-actionable: wait for the RECAP upload, or add an `extra_documents` workaround. (3) `llm.SUMMARY_PRIMARY_DOCUMENT_UNREADABLE` ("The primary document(s) could not be read.") ŌĆö written when the extraction chain had something to work with (a URL or upstream plain_text) but couldn't produce usable text. Operator-actionable: install poppler/tesseract if not already, investigate the PDF itself (image-only without text layer? corrupted? unusual font subsetting?). Also serves as the catch-all when multiple primaries are in different states ŌĆö the most general framing that stays accurate. (4) `llm.SUMMARY_INSUFFICIENT_DOCUMENTS` ("Documents available for this docket are insufficient to generate a reliable summary.") ŌĆö written when no entry on the docket matched `is_primary_document` at all (truly cold: appellate filings whose opener is a clerical "case opened" entry, dockets with only procedural notices, paperless-only filings). Operator-actionable: check whether the docket is sealed at the docket level or whether an `extra_documents` entry pointing at a DoJ press-release attachment would surface the indictment. The same `SUMMARY_INSUFFICIENT_DOCUMENTS` string ALSO comes back from `llm.generate_docket_summary` when the LLM exercises its own prompt-level refusal on garbled text ŌĆö the renderer doesn't distinguish the two sources, both produce the same subscriber-facing line. Classification is done by `_primary_failure_state(entry)` on each identified primary; the WARN log line emitted alongside the row carries the per-state breakdown (e.g., "primary=2 (sealed=1, not-available=0, unreadable=1) disposition=0") so operators can read the log without re-running the summary. The canonical regression that drove this split is us-v-lytvynenko, whose indictment recap_document carried `description='Indictment'` but `is_available=False` (PDF not yet uploaded to RECAP) ŌĆö under the original silent-drop behavior the docket vanished from the index entirely, and under the next iteration ("could not be read" as a catch-all) the message didn't tell subscribers whether to wait for unsealing, wait for the upload, or assume the document was permanently unreadable. All fallback messages are written directly without an LLM round-trip ŌĆö there's nothing for the LLM to summarize from. `source_entry_ids` on the fallback row points at any primary/disposition entries we tried to summarize from, so the audit trail is preserved even when the body itself is the fallback message. Stale-flag refresh (see the matching [Case summaries auto-refresh on docket changes](DESIGN-DECISIONS.md#case-summaries-auto-refresh-on-docket-changes) design note) still applies ŌĆö when the PDF later becomes available or a new primary lands, `sync.process_entry` flips the row stale and `refresh_stale` regenerates from real text on the next emit cycle, replacing the fallback message with a real summary.
- **case_calendar/serve.py** ŌĆö Webhook receiver for CourtListener `DOCKET_ALERT` events (real-time alternative to polling). Stdlib `ThreadingHTTPServer`; serializes processing with a server-wide lock so SQLite writes don't race. Auth model is the URL secret (CourtListener has no signing mechanism), and `Idempotency-Key` headers are stored to make CourtListener retries no-ops. The same store + entry-fingerprint dedup makes double-delivery safe even without the idempotency check. `WebhookServer` accepts an optional `emit_fn(only_calendars: set[str]) -> None` callback that the handler invokes after each successful docket-alert delivery, scoped to the calendars whose rows actually changed ŌĆö `cmd_serve` builds this callback to call `cli.emit_calendars` with the configured ICS paths; gcal / M365 push auto-runs inside `emit_calendars` whenever a token cache is present. Emit failures are logged but never fail the webhook ack: the store is already updated, and a CourtListener retry would just re-process the same entries. Exposes two GET endpoints alongside the POST receiver: `/health` is an unauthenticated liveness probe (returns `{"status":"ok"}` ŌĆö for Caddy / uptime checks), and `/webhooks/case-calendar/<secret>/health` is the secret-gated identity probe that returns `{"service":"case-calendar","tracking":{...}}` so `webhook-url --check` can confirm the operator's secret matches the running receiver in one shot.
- **case_calendar/calendars/description.py** ŌĆö Shared event-body builder used by both ICS and gcal renderers. Composes notes ŌåÆ judge / panel ŌåÆ dial-in ŌåÆ case citation ŌåÆ docket URL ŌåÆ per-document URLs (`Documents:` block, one line per attached doc ŌĆö `65: <url>` for the main doc, `65-1: <url>` for attachment 1; `(sealed)` / `(not yet uploaded to RECAP)` replace the URL when no PDF is reachable) ŌåÆ PACER docket entry numbers (`Docket entries: 65, 82`) ŌåÆ CourtListener entry IDs. The docket-numbers line is what subscribers see in the CourtListener UI; it sits directly above the opaque CourtListener entry IDs so the audit trail reads docket-first. Both `Documents:` and `Docket entries:` are populated in `cli.emit_calendars` from `Store.get_entry_documents` / `Store.get_entry_numbers(source_entry_ids)`; source entries that lack a docket position (paperless minute orders) are silently dropped, and missing or empty doc lists produce no `Documents:` block at all. URL preference is CourtListener storage (`https://storage.courtlistener.com/<filepath_local>`) ŌåÆ Internet Archive mirror (`filepath_ia`) as fallback ŌĆö CourtListener is the authoritative current source; IA is a downstream mirror that can lag. New documents (e.g. an attachment uploaded later) land in the description automatically: the entry's fingerprint already includes per-doc availability flags, so a doc-add flips the fingerprint, `process_entry` re-runs and overwrites the stored `recap_documents` JSON, and the auto-emit (sync + webhook) re-renders the calendar. Also exports `no_time_title_prefix(starts_at_utc)` which returns `[time TBD]` for future date-only events (we expect the time to be set later) and `[time unknown]` for past ones (we never learned it). Renderers are deadline-agnostic: deadlines are mapped to hearing-shaped dicts in `cli._deadline_to_hearing` (with `hearing_key` namespaced as `deadline:<key>`) before they reach this layer.
- **case_calendar/calendars/ics.py** ŌĆö Hand-rolled RFC 5545 output. UIDs are derived from `case_id + hearing_key` so regenerations are stable across runs (and the `deadline:` prefix on mapped deadline keys keeps their UIDs in a separate namespace from real hearings). Rows with `significance='minor'` and rows with `status='cancelled'` are filtered out entirely ŌĆö the calendar tracks major case moments only, not procedural noise or events that are no longer happening. Each timed event is tagged with `DTSTART;TZID=<IANA tz>:<local time>`; receiving calendar apps resolve DST via their own bundled IANA databases (no VTIMEZONE shipped ŌĆö bare TZIDs work in Google, Apple, Proton, Outlook 2019+, Thunderbird). The title carries `[time TBD]` / `[time unknown]` only when the event is date-only; we don't add `[HELD]` (the date itself shows what's past).
- **case_calendar/calendars/gcal.py** ŌĆö Google Calendar push. Event IDs are deterministic (`sha1(case_id::hearing_key)`), so `patch` is idempotent and reschedules / detail updates land on the same event ŌĆö and the `deadline:` prefix on mapped deadline keys means a hearing and a deadline that happen to share a slug get distinct gcal events. Times are sent as `dateTime: "<local>", timeZone: "<IANA>"` so Google preserves the courthouse tz across DST and viewer-tz displays. First-run uses InstalledAppFlow OAuth; tokens cache to `tokens/google-token.json` (relative to the project directory) ŌĆö the daemon can't run InstalledAppFlow headless, so the operator must run `case-calendar setup gcal` once interactively to stage the refresh token. Optional `notify_emails` and `reminders` decorate events with attendees + reminder overrides; emails are added as Google Calendar attendees (Google's email reminders only fire for the calendar owner ŌĆö attendees rely on their own client's reminder settings). `sendUpdates=externalOnly` so attendee invitations actually fire. Hearings flipped to `cancelled` get a `patch` with `{"status": "cancelled"}` against the existing remote event so subscribers stop seeing it ŌĆö but if the event was never created (404) we noop rather than insert+cancel. `significance='minor'` rows are skipped entirely.
- **case_calendar/calendars/index.py** ŌĆö Static `index.html` renderer for the public calendar directory. `build_calendar_models(cfg, store, public_base_url=...)` assembles the view-model (calendars in config order, each with subscribe links + a case list carrying docket metadata, `date_filed` = min(entries.date_filed) across the case's dockets, `last_filing_date` = max(dockets.date_last_filing) ŌĆö the value CourtListener exposes as "most recent document filing date", distinct from `date_modified` which also bumps on OCR / metadata churn). `write_index(path, calendars=...)` writes the page. Self-contained ŌĆö inline CSS + JS, no external requests, no CDN. CSS declares `:root { color-scheme: light dark; }` and the `<head>` carries `<meta name="color-scheme" content="light dark">`, the pair of signals Darkreader checks when deciding whether a page has built-in dark mode; both must ship together for Darkreader to skip its own filter. System preference is honored via `@media (prefers-color-scheme: dark)`, a header `<button>` lets users override and persist in `localStorage`, and an inline pre-paint `<script>` in `<head>` applies the saved preference before stylesheets load so there's no flash. Sort is client-side: each `<li.case>` carries `data-name` (lowercased for case-insensitive compare), `data-filed`, `data-last-filing` (ISO date prefix), and the JS re-appends siblings in place when the per-section `<select>` or direction checkbox changes; empty values sort last regardless of direction. The default sort is "Last filing" descending. `cli.emit_calendars` writes the index after the per-calendar ICS / gcal / M365 work ŌĆö **globally**, ignoring `only_calendars`, because the page is the cross-calendar view and even an unaffected calendar's row may have moved if a sibling docket's date_last_filing advanced in the same sync. Per-case AI summaries (from the `case_summaries` table) render as a labeled paragraph block inside each case row ŌĆö when a case aggregates multiple dockets the paragraphs are prefixed with `<docket_number> (<court_citation>)` so subscribers can tell which suit each sentence describes; single-docket cases drop the label. Below the summary, each case row also renders a per-case **Upcoming events preview** (`_render_events`, fed by `_visible_events_for_case` / `_window_events` / `_decorate_event` in `build_calendar_models`): a compact agenda of the case's next hearings and deadlines so a visitor can see what a calendar contains without subscribing ŌĆö see the [Calendar event previews design note](DESIGN-DECISIONS.md#calendar-event-previews-are-rendered-into-the-index-from-the-store-with-ics-identical-filtering) for the feed-parity invariant and windowing. A static footer block carries the two legally-sensitive disclaimers (AI-generated-may-contain-mistakes; presumption of innocence) that must NEVER come from the LLM ŌĆö see the [Static legal disclaimers design note](DESIGN-DECISIONS.md#static-legal-disclaimers-never-from-the-model). Opt-in via top-level config keys `index_path`, optional `public_base_url` (when set, the index emits absolute `https://` + `webcal://` subscribe links; when omitted, only relative filenames), and optional `site_title`. Caddy is the intended fronting server. The Caddy config is documented inline in [docs/public-page.md](docs/public-page.md) (feeds) and [docs/webhooks.md](docs/webhooks.md) (receiver) rather than as a checked-in `Caddyfile` ŌĆö operators paste the block into `/etc/caddy/Caddyfile`, edit the domain, and reload, so there's a single source of truth in the docs instead of a repo template to keep in sync. The documented config carries two site blocks ŌĆö a `calendars.example.com` site that serves the `out/` directory with `text/calendar` content-type on `*.ics` and a sensible `Cache-Control`, plus a `webhook.example.com` site that reverse-proxies the `case-calendar serve` receiver ŌĆö so a single Caddy process handles both the public ICS / index hosting and the private webhook endpoint. The systemd unit for `serve` is likewise documented inline in [docs/webhooks.md](docs/webhooks.md) (no checked-in `case-calendar.service`).
- **case_calendar/calendars/m365.py** ŌĆö Microsoft 365 / Outlook push via Microsoft Graph. Uses the **official** `msgraph-sdk` + `azure-identity` libraries (not the third-party `O365` package). Auth is `InteractiveBrowserCredential` with `TokenCachePersistenceOptions` and a serialized `AuthenticationRecord`: first run opens a browser to grant `Calendars.ReadWrite`, the record JSON is cached at `tokens/m365-token.json` (relative to the project directory), and the daemon thereafter constructs the credential with `disable_automatic_authentication=True` to refresh silently ŌĆö failing fast instead of trying to open a browser on a server. Graph generates event ids server-side (unlike Google's deterministic ids), so we persist the returned id on the hearings/deadlines row as `m365_event_id`, recover a stale cache via a `CaseCalendarKey` `$filter` lookup, and set `transactionId` on every create for retry dedup ŌĆö see the [M365 idempotency design note](DESIGN-DECISIONS.md#m365-idempotency-without-server-side-determinism). The SDK is async; `sync()` wraps `_sync_async` in `asyncio.run` so callers stay synchronous. Per-calendar config: `m365_calendar_id` routes to a specific Outlook calendar (`/me/calendars/{id}/events`); omit it (or set `m365_use_default_calendar: true`) to push to the user's default calendar (`/me/events`). Cancellation = `DELETE` (Graph already mails attendee cancellations on organizer DELETE, no separate verb). `significance='minor'` rows are skipped. Email reminders aren't representable in Graph's single `reminderMinutesBeforeStart`, so only popup reminders are honored ŌĆö the shortest popup minutes wins.
