# Agents

## Project Overview

Case Calendar — a CLI / webhook service that pulls federal court hearing dates AND filing deadlines from CourtListener / RECAP and emits them to ICS feeds (subscribable from Proton, Apple, etc.) and optionally pushes them to Google Calendar and/or Microsoft 365 / Outlook. Built for tracking cases where docket-watching by hand is too much: cybercrime prosecutions (e.g. DPRK IT-worker fraud), multi-docket tech litigation (e.g. Anthropic v. DOW), and similar. Designed to run unattended — `case-calendar serve` accepts CourtListener webhooks, processes each delivery, and re-renders the affected ICS file on the spot. The end-to-end "when does a subscriber actually see this?" latency is bounded by signals outside our control on either side — see the [Limitations](docs/index.md#limitations) page; the project's own internal stage (webhook arrival → ICS file written) is the only step that runs in seconds.

## Conventions

These rules apply to anyone — human or agent — making changes to this repo. They are intentionally checked in (rather than living in any one agent's private scratch memory) so that every collaborator picks them up the same way.

- **Wait for explicit commit AND push permission on the default branch — these are separate grants.** Finish the implementation, run the tests, summarize the diff, then **stop and ask**. The author decides when a change is ready to land; auto-committing makes review noisier and harder to reverse. "Commit this" mid-session counts as permission for that one commit, not a standing grant — and crucially, permission to commit is NOT permission to push. Pushing publishes the change to the remote where collaborators / CI / production deploys can pick it up, and is much harder to walk back than a local commit. Wait for an explicit "push it" before `git push`. If the prior commit was itself unauthorized, do NOT push it to "tidy up" — surface the situation and let the author decide whether to keep, amend, or reset.
- **Self-test before every `git commit`:** has the author typed "commit" (or an unambiguous equivalent — "ok to commit", "commit this", "commit and push") in a present-tense imperative since your last commit? If no, **ask**. Conditional phrasings like "if everything works we can push" or "we could commit this" or "if it looks good ..." are NOT authorizations — they are plans you must confirm before acting on. Treat the literal text of the user's last message as the source of truth, not your own interpretation of where the conversation is going.
  - **Self-test before every `git push`:** has the author typed "push" since your last push? Same rule. Permission to commit is NEVER permission to push.
  - **Exception — branches you created in-session** When you have explicitly created a feature branch yourself (e.g. `git checkout -b feat/something`) in that session, commit and push to THAT branch freely without per-step permission. The entire branch is reviewed at PR-open, so the per-commit gate adds review noise without adding safety. The exception is scoped to branches Claude created in the current session; it does NOT extend to `main`, to other long-lived branches, or to branches the author created.
- **Back up the SQLite store before any schema or migration change.** Before running any schema-changing SQL (ALTER TABLE, CREATE/DROP, hand-rolled column rewrites, anything that mutates table shape) against a real store on disk, copy the DB file aside first (e.g. `cp case-calendar.sqlite case-calendar.sqlite.bak-YYYY-MM-DD-<reason>`). The store holds the full operational history (entries, hearings, deadlines, case summaries, webhook idempotency keys, source-entry provenance) — none of which is reconstructible without a multi-day CourtListener re-sync that burns the daily quota, costs LLM dollars, and still won't recover anything CourtListener has since rolled past. Migrations that look reversible on paper aren't always — a column rename + backfill can lose precision, an aborted migration mid-write can leave the DB in a shape no code path expects. The backup is the escape hatch. This applies in production AND in any dev environment whose DB the author cares about; ask if uncertain. (Tests use ephemeral tmp-path stores via the `store` fixture — those don't need backups.) **The store runs in WAL journaling mode** (see the concurrency design note below in Key Design Decisions), so a hot copy must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main `.sqlite` file, OR run `PRAGMA wal_checkpoint(TRUNCATE)` first to fold the WAL contents back into the main file before copying just that one file. A bare `cp` of only the main file taken while writes are in flight will silently miss any committed-but-not-yet-checkpointed rows.
- **Project-specific rules belong in AGENTS.md, not in any agent's private memory store.** If you (Claude Code, Cursor, Codex, Aider, anything that has a "save this preference for next time" surface) catch yourself about to write down a rule that's actually about the codebase rather than about working with this particular user, write it here instead. Memory is fine for user-profile facts and tool-use preferences; project rules should be portable across agents.
- **Every behavior change ships with its test** — see [Testing philosophy](#testing-philosophy) for the full statement and the verification command.
- **Spell "CourtListener" in full everywhere.** No "CL" abbreviation, anywhere: not in user-facing documentation, not in AGENTS.md, not in source comments or docstrings, not in commit messages, not in PR descriptions, not in tests, not in identifier names. The full name reads only marginally longer than the abbreviation and removes a layer of jargon contributors and subscribers shouldn't have to translate. The only acceptable abbreviation is the lowercase `cl` parameter identifier for a `CourtListener` client object — an obvious local-binding name from context. Everything else is a defect: standalone "CourtListener" in prose, but also CamelCase suffixes like `FakeCL` (→ `FakeCourtListener`) or `_CLClient` (→ `_CourtListenerClient`). The CamelCase form slips through naive `\bCL\b` regex sweeps because lowercase→uppercase isn't a word boundary, so any bulk rename pass must also grep specifically for `CourtListener[^a-zA-Z0-9_]` and `CourtListener$` to catch identifier suffixes.
- **Do NOT claim CourtListener OCRs documents.** It doesn't. CourtListener's `plain_text` field on a `recap_document` is whatever extraction the Free Law Project's ingestion pipeline produced upstream — if that pipeline can't extract text (image-only PDFs without an embedded text layer), the field stays empty or garbled. Documents only become text-searchable when a downstream consumer (this project, via local poppler + tesseract, is one such consumer) runs its own OCR, OR when the originating PACER filing already shipped with embedded text. Saying "CourtListener hasn't OCR'd it yet" is wrong; the accurate framing is "CourtListener's `plain_text` was empty / unreliable, so we OCR the PDF ourselves." Applies in user docs, code comments, commit messages, and PR descriptions equally. (This also matters for the latency framing — a PDF that needs OCR is NOT a "wait for CourtListener to catch up" condition; it's a "wait for someone to run OCR on it, possibly us on next sync" condition.)
- **Plain language over jargon.** Comments, docstrings, AGENTS.md, commit messages, PR descriptions, and user-facing docs should describe what the code does in words a non-specialist would understand. Avoid terminology imported from neighboring fields that only loosely applies — e.g., "projection" from relational algebra to describe "the subset of recap_document fields we keep in the local store", or "compaction" / "denormalization" / similar when a plain description works. When a domain term IS the right word (because the code really is implementing that concept, or the reader needs to look it up to understand a library), use it AND a brief in-place gloss the first time it appears. When a term is borrowed loosely, replace it with the literal description. Past offenders that were swept out and should not return: "projection" of recap_documents (now "the subset of each recap_document we store locally" / "cached recap_documents"); "high-water mark" / "watermark" for the polling cutoff (now "last-modified cutoff"); "load-bearing" for rules-that-must-not-be-dropped (now "essential"); "happy path" for the success case (now "success path"); "self-heal" / "self-healing" for cache rebuild (now "automatically rebuild" / "rewrite from fresh data" / "repair"); "agentic" / "agentically" for unattended automation (now "automatic" / "unattended"); "first-class concern" for important matters (now "primary concern"); "DDL" for schema changes (now "schema-changing SQL"); "fan-out" for orphan child rows (now described literally). The test is whether a contributor coming into the codebase from a different background would have to stop and search to understand what a term refers to here; when in doubt, prefer the plainer rewrite even if it's a few extra words.
- **Don't make unsupportable claims in comments, docstrings, or PR descriptions.** Phrasings like "the most-missed appellate deadline", "the foundational civil deadline", "the most common scheduling-order requirement", or generalizations from one observation ("appellate filings carry the party name between 'MOTION' and the verb" — true of the one 4th Cir docket we looked at, not necessarily true of appellate filings as a class) are claims about empirical frequency or community practice the author can't usually back. Replace them with verifiable substance: cite the rule / statute / case that establishes the fact ("FRAP 40", "FRCP 12(a)(1)"), name the specific docket or source where the pattern was observed ("Verbatim from 4th Cir docket 25-4607 (Gholinejad)"), or describe the invariant the code protects (the regex constraint a negative test pins). The reader gets a citation they can follow; nobody is asked to take an unattributable claim on faith. Past offenders that were swept out and should not return: "currently the most-missed appellate deadline" (now: "FRAP 40 — petition for panel rehearing / rehearing en banc"); "the foundational civil deadline" (now: "FRCP 12(a)(1)"); "the bare district-court phrasing" applied to court-neutral text; "Common scheduling-order requirement"; "`presentencing` also a common informal form"; "minute entries" applied to appellate docket entries (the term is district-court vocabulary); "entries like 'CIPA Section 5 notice' don't otherwise trip the regex" (often inaccurate after a sibling change in the same diff). **Priorities yes, empirical claims no — including inside LLM prompts.** A prompt can legitimately tell the model how to weight or order its output ("the imposed sentence is the most important fact about the case — include it"; "lead with the operative court and the most important charges or claims"). These are directives about emphasis applied to material the model already sees in the source documents — the model isn't being asked to take a worldly fact on faith, just to prioritize within what's in front of it. What a prompt CANNOT do is assert empirical claims the model couldn't verify from the source documents — don't write "FRAP 40 is the most-missed appellate deadline" or "this is the foundational civil deadline" or "most federal cybercrime cases involve X" into a prompt any more than into a comment, because the model has no way to fact-check the claim and will repeat it to subscribers as documented fact. The test: a *priority directive* ("lead with X if present", "X matters more than Y in the output", "if Z is mentioned, quote the date verbatim") shapes how the model orders / emphasizes what's already in front of it; an *empirical assertion* ("X is the most common", "in practice most courts...", "this is the foundational Y") adds a worldly claim the model can't verify and shouldn't be taught.
- **Fix underlying bugs, never just patch the data.** A manual SQL update or shell command that corrects ONE row of bad state in the local DB doesn't help other users running the same code, doesn't help future data hitting the same bug, and doesn't survive a fresh checkout or re-sync. Every observed bug must result in a code change that prevents the bad state from recurring, even when an immediate manual patch is also applied to unblock the operator. The manual patch is the bridge; the code fix is the destination — both happen, never just the bridge. Example: the us-v-moucka entry #1 stale cached-recap_documents bug surfaced as one row in the local DB whose stored `plain_text` was empty even though CourtListener had it. The bridge was an inline script that rebuilt that one row's stored recap_documents from a fresh CourtListener fetch; the destination was the new staleness detector + automatic cache rebuild in `summary.find_primary_documents` plus `Store.refresh_entry_recap_documents`, so the same bug repairs itself on any other affected docket without operator intervention.

## Architecture

- **case_calendar/cli.py** — CLI entry point. Subcommands: `sync` (pull updates from CourtListener into the store + auto-emit affected calendars; takes `--force-summaries` to bundle the equivalent of `summarize --force` into the same CourtListener session, so a prompt or model change can be propagated in one command instead of two), `serve` (run the webhook receiver + auto-emit affected calendars per delivery), `emit` (write ICS + push to any configured gcal / M365 backend — used standalone for forced re-renders), `setup gcal|m365` (one-time OAuth bootstrap that stages the token cache), `summarize` (generate per-docket AI case summaries; opt-in, gated on `case_summaries.enabled` in config), `show` (dump current hearings + deadlines), `prune` (delete every store row tied to a `docket_id` no longer referenced by any case in the config; dry-run by default, pass `--apply` to actually delete — for the case where a docket was removed from `config.yaml` and its rows would otherwise be ignored forever), `webhook-url` (print the ready-to-paste CourtListener webhook URL, composed from `CASE_CALENDAR_WEBHOOK_SECRET` + `--host`; with `--check`, also probes the receiver's secret-gated health endpoint at `<url>/health` and reports whether the host is reachable, Case Calendar is actually answering, and the local secret matches the running receiver — the operator-facing check that catches CF intercepts, stale Caddy configs, and secret mismatches in one shot). Loads `.env` via `python-dotenv` before any module reads env vars. Exposes `emit_calendars(cfg, store, *, only_calendars=None, setup_gcal=False, setup_m365=False)` as the shared rendering primitive — both `cmd_sync` and the webhook server's `emit_fn` callback go through it, so a single update flows from CourtListener to ICS subscribers without a manual step. Push readiness for gcal / M365 is auto-detected via `_resolve_gcal` / `_resolve_m365` from config (credentials_path / client_id) + token-cache file presence; the `setup_*` flags only matter on the `setup` subcommand path, where they allow the OAuth browser flow to run when no cache exists yet. Also owns the deadline-to-hearing adapter (`_deadline_to_hearing` maps the deadline row into a hearing-shaped dict the renderers accept), `_compose_title` (the title prefixer that adds `[HEARING]` / `[DEADLINE]` / `[time TBD]` / `[time unknown]` flags), and `_maybe_ensure_docket_alerts` (calls `alerts.ensure_docket_alerts` once on `cmd_sync` / `cmd_serve` startup so each configured docket is subscribed to CourtListener docket alerts; gated on the `ensure_docket_alerts` top-level config flag, default true).
- **case_calendar/courtlistener.py** — REST v4 client. Wraps `dockets/`, `docket-entries/`, `recap-documents/`, and `docket-alerts/` endpoints with backoff for 429 (`Retry-After`) and 5xx responses. Honors any `Retry-After` value (the daily 300/day bucket can return a wait near 24h; we sleep through rather than abort) and logs URL + body + rate-limit headers on every 429 so it's obvious which bucket fired. Iterator-style pagination via `iter_entries` (with a `modified_after` cutoff for cheap incremental syncs; pages newest-first to preserve the cheap stop-condition but buffers and yields oldest-first so cross-entry references resolve cleanly within a single sync) and `iter_docket_alerts` (one-shot list for the webhook-subscription reconciler). `create_docket_alert(docket_id)` POSTs a new subscription. `_get` and `_post` are thin wrappers around the shared `_request` retry loop — both verbs use the same retry shape because CourtListener applies the same rate-limit + transient-failure characteristics to either. Request-rate telemetry: `_request` records a wall-clock timestamp for every HTTP request that reached the server (via `_record_request` — including ones that came back 429/5xx and were retried, since those still spend quota; transport failures that never got a response are not counted), keeping a lifetime `_request_total` plus a rolling buffer pruned to the largest reporting window (`_REQUEST_WINDOW_MAX_SECONDS`, 24h). `log_request_stats` (called from `__exit__`, so it covers `sync` / `summarize` / `serve` shutdown uniformly) logs `courtlistener-requests total=… peak/min=… peak/hour=… peak/day=…`, where the peaks are the busiest rolling windows computed by the module-level `_peak_in_window` two-pointer sweep — the number to compare against a Free Law Project / CourtListener API tier's hard per-minute/hour/day ceilings. No-op when no requests were made.
- **case_calendar/alerts.py** — Reconciles CourtListener docket-alert subscriptions against the docket ids configured in `config.yaml`. `ensure_docket_alerts(cl, docket_ids)` lists the account's existing alerts once, creates a subscription via POST for any configured docket that isn't already covered, and returns a `{docket_id: 'exists' | 'created' | 'failed'}` status dict. List-call failures degrade gracefully — every docket reports `'failed'` and no creates are attempted (avoids the "spam duplicates or skip blindly" failure mode against an unknown baseline). Per-docket create failures are logged at WARNING and don't abort the loop. Wired into `cmd_sync` and `cmd_serve` via `cli._maybe_ensure_docket_alerts`, gated on the top-level `ensure_docket_alerts` config flag (default true).
- **case_calendar/courts.py** — Court-id → IANA timezone mapping. CourtListener doesn't expose timezones in its API, so we maintain the table here. Coverage is comprehensive: all 13 federal circuits, all 94 district courts, the Supreme Court, specialty courts (Federal Claims, CIT, FISA, Tax, Armed Forces), and territories (PR, VI, Guam, NMI, American Samoa). Multi-tz states (Tennessee, Florida, Indiana, etc.) map to the principal-office tz of each district. Unknown court IDs log a warning and fall back to ET so the gap is visible rather than silent.
- **case_calendar/extractor.py** — Cheap regex pre-filter. `is_hearing_relevant` matches hearing-vocabulary entries; `is_deadline_relevant` matches filing-deadline vocabulary (`due by`, briefing schedule, scheduling order, stipulation, etc.); `is_extractable(entry)` is the combined gate the syncer calls — any entry matching either vocabulary passes. Deadline tracking is uniform across all dockets now, so there's no per-case opt-in at the filter layer. Briefs, attorney appearances, and similar paperwork entries are skipped before any LLM tokens are spent. Intentionally over-inclusive — false positives just cost an extra LLM call; false negatives lose hearings/deadlines.
- **case_calendar/llm.py** — Hearing + filing-deadline extractor and case-summary generator. This module owns the domain PROMPTS and the high-level entry points; the provider-agnostic call layer it sits on — dispatch across anthropic / openai / gemini, provider auto-detection from `LLM_PROVIDER` / `*_API_KEY`, the small/fast model defaults, lazy SDK imports, the `cache_control: ephemeral` Anthropic prompt-cache marker, max-token-truncation detection, and token telemetry — was extracted to the **`case_calendar.llmkit`** subpackage, which it calls via `providers._dispatch_llm_call` / `providers._detect_provider`. `LLM_MODEL` overrides the per-provider default; the small/fast tier (Haiku, gpt-5.4-nano, Flash Lite) fits structured extraction, not deep reasoning. The summary track keeps its own higher-tier model selection here (`_DEFAULT_SUMMARY_MODELS`, `summary_provider_info`). The Anthropic prompt-cache discount covers all three extractor-track prompts (extract / verify_hearing / verify_deadline) within a sync. Four entry points:
  - `extract_actions` runs against ONE docket entry plus the case's known-hearings list AND known-deadlines list (always — deadline tracking is uniform across all dockets), and returns 0+ actions: hearing actions (ADD / RESCHEDULE / UPDATE_DETAILS / CANCEL / MARK_HELD / IGNORE) and deadline actions (ADD_DEADLINE / RESCHEDULE_DEADLINE / CANCEL_DEADLINE / MARK_FILED). A single merged `SYSTEM_PROMPT` covers both — no conditional addendum, no per-case opt-in.
  - `verify_hearing` runs against ONE candidate hearing plus a window of hearing-relevant docket entries (recent docket activity AND the entries filed around the hearing's own date — selected by the caller; see the verify-pass design note) and returns a single audit decision (CONFIRM / RESCHEDULE / CANCEL / MARK_HELD / DELETE_HALLUCINATION / UNCLEAR) — used by the end-of-sync confidence pass to catch reschedules/cancellations the per-entry extractor missed and to reject rows the LLM hallucinated.
  - `verify_deadline` is the deadline analogue — same shape, returns CONFIRM / RESCHEDULE / CANCEL / MARK_FILED / DELETE_HALLUCINATION / UNCLEAR for one pending deadline.
  - `resolve_duplicate_hearings` is the same-slot resolver — receives a CLUSTER of 2+ scheduled hearings on the same docket sharing the EXACT same `starts_at_utc`, plus recent docket entries, and returns one of MERGE_INTO (pick a target_key, the caller cancels the others and folds their source_entry_ids into the target) / KEEP_BOTH (rare — stacked back-to-back proceedings) / UNCLEAR. A single court can't hold two hearings on one docket simultaneously, so the cluster almost always means the per-entry extractor split one logical event across keys (vocabulary varied between the entries that scheduled it).
  - `generate_docket_summary` is the **case-summary** entry point — runs the primary document + disposition documents + a structured-events scaffold (hearings/deadlines with their statuses) through a higher-tier model and returns 2-4 sentences of prose plus the model identifier. Uses a separate provider/model selection (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL` env vars, with config-level `case_summaries.provider` / `.model` taking precedence) so the extractor pipeline's cheap defaults stay decoupled. Per-provider defaults are Sonnet (Anthropic) / gpt-5.4 (OpenAI) / Gemini Pro — synthesis-heavy enough to warrant the upgrade. Returns plain text, not JSON; the per-provider callers expose a `json_mode=False` knob to support this.

  Each ADD/RESCHEDULE/UPDATE_DETAILS action (hearing or deadline) carries a `significance` field (`major` / `minor`) that controls calendar visibility — substantive proceedings (trial, sentencing, oral arg, any pretrial conference, evidentiary hearing, plea, etc.) and dispositive briefing deadlines (MTD/MSJ response/reply, sentencing memos, motion-in-limine, Daubert, suppression briefing) are major; procedural-only events (a phone call set just to rule on a Motion to Continue, routine joint status reports, proposed orders) are minor and filtered out at render time. The user message includes a "RELATED DOCKET ENTRIES" block (explicit cross-refs from the new entry's text + the last few hearing-relevant entries on the same docket) so the LLM can name a hearing correctly even when the entry that schedules it doesn't itself name the subject.
- **case_calendar/llmkit/** — Provider-agnostic LLM call layer + token telemetry, kept as a self-contained subpackage with NO case_calendar domain knowledge (no court prompts, no hearing/deadline shapes) so it can be lifted out as a standalone project later. The package name is an internal placeholder pending a real one; nothing is published, so it does not compete for the PyPI/GitHub `llmkit` namespace.
  - `providers.py` — the three per-provider SDK wrappers (`_call_anthropic` / `_call_openai` / `_call_gemini`; lazy SDK imports; Anthropic `cache_control: ephemeral` on the system block), the three-way `_dispatch_llm_call`, provider auto-detection (`_detect_provider`, reads `LLM_PROVIDER` / `*_API_KEY`), the small/fast `_DEFAULT_MODELS`, `OutputTruncatedError` (raised when a model stops at `max_tokens` so callers skip the entry rather than parse a truncated half-JSON), and `provider_info`.
  - `usage.py` — token telemetry (log-only; nothing persisted). `TokenUsage` normalizes the cross-provider difference (`input` = total prompt tokens with the cached portion INCLUDED, `output`, `cached` = served from cache, `cache_write` = written to cache — Anthropic reports cache reads/writes SEPARATE from `input_tokens` so they're folded into `input`; OpenAI/Gemini already fold them). `from_anthropic` / `from_openai` / `from_gemini` read each SDK's response through `_as_int`, which coerces any non-numeric field (missing attribute, `None`, a test `MagicMock`) to 0 so telemetry can never turn a working call into a crash. `TokenLedger` (thread-safe — the webhook server records from worker threads) accumulates per-docket, per-model, and grand totals for one process run; module-level `record` logs one `llm-tokens call …` line per call and folds it into the subtotals, `log_summary` dumps per-docket subtotals, then per-model subtotals, then a `… TOTAL …` line (which also carries `models=N`) at a run boundary, `reset` clears it. The per-model lines are what separate the extractor track's cost from the summary track's: the two run on different model tiers (cheap small/fast vs higher synthesis), so reading the run total by model tells you which track spent what — and the startup `extraction LLM:` / `summary LLM:` log lines name which model is which. Model is the grouping axis (not a domain `extractor`/`summary` label) precisely so `llmkit` stays domain-free — it buckets by the opaque model string the same way it buckets by docket. A model the price table doesn't cover shows `cost_est=?` on its per-model line (priced-or-not is per-model deterministic, so a model key present in the cost map — even at `$0.0000` — means it was priced).
  - `__init__.py` — re-exports the stable public API over the underscore-prefixed internals: `dispatch` (= `_dispatch_llm_call`), `detect_provider`, `provider_info`, `OutputTruncatedError`, `DEFAULT_MODELS`, plus `usage`, `TokenUsage`, `TokenLedger`.
  - Wiring: `providers`' call functions invoke `usage.record(purpose=…, provider=…, model=…, tokens=…, docket=…)` after each SDK call (purpose ∈ `extract` / `verify_hearing` / `verify_deadline` / `dedupe_hearings` / `summary`, threaded through `_dispatch_llm_call` from the callers in `case_calendar.llm`); `cli.cmd_sync` / `cmd_summarize` call `llmkit.usage.log_summary` + `reset` at the end of the run. `serve` is left out of the run-total path on purpose — its webhook handlers and the debounce Timer thread would race on `reset`, so it relies on the always-on per-call lines instead (see the design note below).
- **case_calendar/costs.py** — Rough USD cost ESTIMATOR plugged into the price-free `llmkit` ledger via `llmkit.usage.set_price_estimator` (wired in `cli._log_llm_setup`). `estimate_cost(model, TokenUsage)` looks the model up in `_RATES_USD_PER_MTOK` (exact id, then longest-matching prefix so a dated id like `claude-haiku-4-5-20251001` still resolves) and prices each token slice at its own rate — uncached input (`input − cached − cache_write`), cache read, cache write, output. The table is hand-copied from each provider's published pricing page and dated with `PRICES_VERIFIED`; Anthropic, Gemini, and the current OpenAI 5.4 + 5.5 family (every `-mini`/`-nano`/`-pro` variant, so a variant can't prefix-fall-back to its base) are filled at their standard tier. Returns `None` for any unlisted model (a legacy model or an unrecognized `LLM_MODEL` override), which the ledger surfaces as `cost_est=?` and a partial-estimate note on the run TOTAL. Deliberately a domain-layer module (not in `llmkit`) so the reusable package ships no prices.
- **case_calendar/pdf.py** — PDF text extraction with fallbacks. Order: CourtListener `plain_text` → `pypdf` on the downloaded PDF (CourtListener storage preferred, Internet Archive mirror as fallback) → `pdftoppm` + `tesseract` OCR (optional system dependency). CourtListener is the authoritative, current source today; the Internet Archive is a downstream mirror that can lag, so `pdf.recap_document_url` / `pdf.fetch_pdf_bytes` prefer CourtListener storage (`https://storage.courtlistener.com/<filepath_local>`) and keep `filepath_ia` only as a backstop. Sealed PDFs and not-yet-uploaded PDFs return `None` and are NOT cached — the next sync re-fingerprints the entry and retries once the PDF arrives or gets OCR'd.
- **case_calendar/store.py** — SQLite state. Tables: `dockets` (id + last `date_modified` (the polling short-circuit cutoff) + `date_last_filing` (CourtListener's "most recent document filing date"; what the index's "Last filing" column reads) + cached metadata for description rendering), `entries` (dedup of already-processed entries, keyed by `(docket_id, entry_id)` with content fingerprint, `entry_number` for cross-ref lookup, `description` / `short_description` / `recap_documents` JSON stored for entries that either passed the regex pre-filter OR matched `summary.is_primary_document` / `summary.is_disposition` — every other entry gets a fingerprint-only stub so dedup still works without storing dead-weight notice text. `recap_documents` keeps a compact subset of each CourtListener recap_document — `id`, `document_number`, `attachment_number`, status flags, `filepath_ia`, `filepath_local`, `plain_text` — built by `store.compact_recap_documents`, that the description renderer turns into clickable per-doc URLs at emit time, and that the summary pipeline uses to find primary documents + dispositions locally via `get_entries_with_body` instead of re-fetching docket-entries pages from CourtListener right after sync wrote them down — `plain_text` rides along in that stored subset so `pdf.extract_text` short-circuits without a PDF fetch), `hearings` (per-case logical hearings keyed by `(case_id, hearing_key)` with `significance` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list), `deadlines` (per-case filing deadlines keyed by `(case_id, deadline_key)` with `due_at_utc` / `status` ∈ {pending, met, passed, cancelled} / `significance` / `deadline_type` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list — parallel structure to hearings so they share renderers), `case_summaries` (per-docket prose summaries keyed by `(case_id, docket_id)` with `summary` text, `model` identifier, `source_entry_ids` JSON listing the CourtListener entries that informed the LLM, and a `stale` boolean that `sync.process_entry` flips when it sees a new primary document or disposition so the automatic refresher knows to regenerate; populated by `summary.refresh_stale` during sync / webhook emit, or explicitly by `case-calendar summarize`), `courts` (CourtListener court metadata, citation_string), `webhook_events` (idempotency-key dedup). The `m365_event_id` columns are populated by `set_m365_id_for_hearing` / `set_m365_id_for_deadline` after a Graph create; gcal ids are deterministic so `gcal_event_id` is currently unused at runtime. Connection is `check_same_thread=False` so the webhook server's worker threads can use the same Store instance under a server-wide lock; the connection is also opened with `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000` so the polling `sync` process and the long-running `serve` process can safely share the same DB file (see the concurrency design note below).
- **case_calendar/sync.py** — Per-case orchestration. For each docket: fetch entries newer than the store's last-modified cutoff, run the regex filter, optionally pull PDFs, resolve cross-references (`granting 65 Motion ...` → look up entry 65 from the local store) plus the last 5 hearing-relevant entries on the docket as temporal context, hand entry + known-hearings list + known-deadlines list + related entries to the LLM, apply returned actions to the hearings/deadlines tables. The known-hearings / known-deadlines context is sourced via `Store.get_hearings_in_court` / `get_deadlines_in_court` so it's scoped to siblings in the SAME court as the current docket — same-court co-defendant dockets still share context, but parallel proceedings in different venues do not (a stay order in court B must not contaminate court A's events). `_needs_pdf` forces a PDF fetch on orders that grant scheduling-affecting motions (`Motion for Hearing`, `to Continue`, `for Continuance`, etc.) even when the order text inlines a date+time, since the brief description references the underlying motion only by docket position and the PDF often holds dates the description doesn't echo. Local time → UTC conversion happens here using the court's IANA timezone. End-of-case sweeps run in order: (1) `_verify_scheduled_hearings` — per-hearing LLM confidence pass for every scheduled hearing (BOTH future and past), fed the recent-activity window on the docket PLUS the hearing-relevant entries filed around the hearing's own date (`_verify_context_entries` → `Store.get_relevant_entries_in_date_range`) so a past hearing's outcome record (a minute entry / judgment filed days later) stays in view even when later filings pushed it out of the recent window; past-dated rows only flip to 'held' when the LLM cites a minute entry / verdict / transcript / judgment-after as evidence of occurrence — date-passed alone is not enough, so trials vacated by plea stay 'scheduled' instead of being falsely marked 'held'; (2) `_dedupe_concurrent_hearings` — resolves same-docket same-`starts_at_utc` clusters via `llm.resolve_duplicate_hearings`, runs AFTER verify so any RESCHEDULE/CANCEL from verify gets a chance to clear concurrency before the dedup LLM is consulted; (3) `_verify_pending_deadlines` and `_auto_mark_passed_stale` do the same for pending deadlines — both run on every case unconditionally since deadline tracking is uniform. Exposes `CaseSyncer.process_entry` so the webhook server can reuse the exact same per-entry pipeline.
- **case_calendar/summary.py** — Per-docket case-summary pipeline (opt-in, off by default). For each docket: hits CourtListener directly to find primary documents (`is_primary_document` matches `INDICTMENT` / `SUPERSEDING INDICTMENT` / `INFORMATION` / `COMPLAINT` / amended variants at the head of the entry description) and dispositions (`is_disposition` matches `JUDGMENT` / `VERDICT` / `ORDER OF DISMISSAL` / `PLEA AGREEMENT` / `MEMORANDUM OPINION` etc.), pulls PDF text via the existing `pdf.extract_text` fallback chain, gathers the docket's hearings + deadlines from the store as a structured scaffold, and hands the whole package to `llm.generate_docket_summary`. Dispositions get an extra fallback: when no PDF text extracts, the entry description itself is used as the disposition text (`allow_description_fallback=True` on `_attach_text`). This is how a paperless "Electronic Clerk's Notes for proceedings held … Court imposes sentence: 92 months imprisonment …" entry — where the imposed sentence lives inline in the docket text with no judgment PDF attached — still reaches the LLM. Primary documents deliberately do NOT get this fallback (a minute-entry stub is not a substitute for an indictment). When `store` is passed in (the normal path from `summarize_docket`), `find_primary_documents` reads from the local `entries` table first via `Store.get_entries_with_body` — `sync.process_entry` now persists description + recap_documents for op/disp matches as well as hearing-relevant ones, so right after a sync the local cache already has everything the summary pipeline needs. Falls back to CourtListener only on a cold docket (first ever sync, or pre-fix data where op/disp entries are still stub-only); the CourtListener fallback walks two pages oldest-first for primary documents and three pages newest-first for dispositions (busy civil cases with heavy post-disposition briefing can push the dispositive order well past page 1). The picker for the LLM document set uses the stricter `_is_disposition_document` — see the broad-vs-narrow design note below. Three entry points: `summarize_docket` runs the full pipeline for one docket and persists the row; `summarize_case` loops the dockets on one case (used by `case-calendar summarize`); `refresh_stale` walks all configured cases and regenerates any row marked `stale=1` (or missing entirely) — this is the automatic path called from `cmd_sync` and the webhook `emit_fn` so a freshly-landed judgment regenerates the prose in the same emit cycle. No-document-text outcomes ALWAYS produce a row — `summarize_docket` never silently drops a docket. Subscribers see a description block under every docket link on the index regardless of why a real summary couldn't be produced; the row carries one of FOUR distinct fallback messages so operators (and careful readers) can tell the failure modes apart at a glance, each operator-actionable in a different direction. (1) `llm.SUMMARY_PRIMARY_DOCUMENT_SEALED` ("The primary document(s) are currently sealed.") — written when each identified primary entry's main recap_document has `is_sealed=True`. Operator-actionable: wait for the unseal; the entry fingerprint includes `is_sealed`, so an upstream unseal flips the fingerprint and the next sync's `process_entry` refreshes the cache automatically. (2) `llm.SUMMARY_PRIMARY_DOCUMENT_NOT_AVAILABLE` ("The primary document(s) are not yet available on RECAP.") — written when each identified primary entry's main recap_document has no source for text (no `filepath_ia`, no `filepath_local`, empty `plain_text`, not sealed); literally nothing for the extraction chain to fetch. Operator-actionable: wait for the RECAP upload, or add an `extra_documents` workaround. (3) `llm.SUMMARY_PRIMARY_DOCUMENT_UNREADABLE` ("The primary document(s) could not be read.") — written when the extraction chain had something to work with (a URL or upstream plain_text) but couldn't produce usable text. Operator-actionable: install poppler/tesseract if not already, investigate the PDF itself (image-only without text layer? corrupted? unusual font subsetting?). Also serves as the catch-all when multiple primaries are in different states — the most general framing that stays accurate. (4) `llm.SUMMARY_INSUFFICIENT_DOCUMENTS` ("Documents available for this docket are insufficient to generate a reliable summary.") — written when no entry on the docket matched `is_primary_document` at all (truly cold: appellate filings whose opener is a clerical "case opened" entry, dockets with only procedural notices, paperless-only filings). Operator-actionable: check whether the docket is sealed at the docket level or whether an `extra_documents` entry pointing at a DoJ press-release attachment would surface the indictment. The same `SUMMARY_INSUFFICIENT_DOCUMENTS` string ALSO comes back from `llm.generate_docket_summary` when the LLM exercises its own prompt-level refusal on garbled text — the renderer doesn't distinguish the two sources, both produce the same subscriber-facing line. Classification is done by `_primary_failure_state(entry)` on each identified primary; the WARN log line emitted alongside the row carries the per-state breakdown (e.g., "primary=2 (sealed=1, not-available=0, unreadable=1) disposition=0") so operators can read the log without re-running the summary. The canonical regression that drove this split is us-v-lytvynenko, whose indictment recap_document carried `description='Indictment'` but `is_available=False` (PDF not yet uploaded to RECAP) — under the original silent-drop behavior the docket vanished from the index entirely, and under the next iteration ("could not be read" as a catch-all) the message didn't tell subscribers whether to wait for unsealing, wait for the upload, or assume the document was permanently unreadable. All fallback messages are written directly without an LLM round-trip — there's nothing for the LLM to summarize from. `source_entry_ids` on the fallback row points at any primary/disposition entries we tried to summarize from, so the audit trail is preserved even when the body itself is the fallback message. Stale-flag refresh (see the matching "Case summaries auto-refresh on docket changes" design note) still applies — when the PDF later becomes available or a new primary lands, `sync.process_entry` flips the row stale and `refresh_stale` regenerates from real text on the next emit cycle, replacing the fallback message with a real summary.
- **case_calendar/serve.py** — Webhook receiver for CourtListener `DOCKET_ALERT` events (real-time alternative to polling). Stdlib `ThreadingHTTPServer`; serializes processing with a server-wide lock so SQLite writes don't race. Auth model is the URL secret (CourtListener has no signing mechanism), and `Idempotency-Key` headers are stored to make CourtListener retries no-ops. The same store + entry-fingerprint dedup makes double-delivery safe even without the idempotency check. `WebhookServer` accepts an optional `emit_fn(only_calendars: set[str]) -> None` callback that the handler invokes after each successful docket-alert delivery, scoped to the calendars whose rows actually changed — `cmd_serve` builds this callback to call `cli.emit_calendars` with the configured ICS paths; gcal / M365 push auto-runs inside `emit_calendars` whenever a token cache is present. Emit failures are logged but never fail the webhook ack: the store is already updated, and a CourtListener retry would just re-process the same entries. Exposes two GET endpoints alongside the POST receiver: `/health` is an unauthenticated liveness probe (returns `{"status":"ok"}` — for Caddy / uptime checks), and `/webhooks/case-calendar/<secret>/health` is the secret-gated identity probe that returns `{"service":"case-calendar","tracking":{...}}` so `webhook-url --check` can confirm the operator's secret matches the running receiver in one shot.
- **case_calendar/calendars/description.py** — Shared event-body builder used by both ICS and gcal renderers. Composes notes → judge / panel → dial-in → case citation → docket URL → per-document URLs (`Documents:` block, one line per attached doc — `65: <url>` for the main doc, `65-1: <url>` for attachment 1; `(sealed)` / `(not yet uploaded to RECAP)` replace the URL when no PDF is reachable) → PACER docket entry numbers (`Docket entries: 65, 82`) → CourtListener entry IDs. The docket-numbers line is what subscribers see in the CourtListener UI; it sits directly above the opaque CourtListener entry IDs so the audit trail reads docket-first. Both `Documents:` and `Docket entries:` are populated in `cli.emit_calendars` from `Store.get_entry_documents` / `Store.get_entry_numbers(source_entry_ids)`; source entries that lack a docket position (paperless minute orders) are silently dropped, and missing or empty doc lists produce no `Documents:` block at all. URL preference is CourtListener storage (`https://storage.courtlistener.com/<filepath_local>`) → Internet Archive mirror (`filepath_ia`) as fallback — CourtListener is the authoritative current source; IA is a downstream mirror that can lag. New documents (e.g. an attachment uploaded later) land in the description automatically: the entry's fingerprint already includes per-doc availability flags, so a doc-add flips the fingerprint, `process_entry` re-runs and overwrites the stored `recap_documents` JSON, and the auto-emit (sync + webhook) re-renders the calendar. Also exports `no_time_title_prefix(starts_at_utc)` which returns `[time TBD]` for future date-only events (we expect the time to be set later) and `[time unknown]` for past ones (we never learned it). Renderers are deadline-agnostic: deadlines are mapped to hearing-shaped dicts in `cli._deadline_to_hearing` (with `hearing_key` namespaced as `deadline:<key>`) before they reach this layer.
- **case_calendar/calendars/ics.py** — Hand-rolled RFC 5545 output. UIDs are derived from `case_id + hearing_key` so regenerations are stable across runs (and the `deadline:` prefix on mapped deadline keys keeps their UIDs in a separate namespace from real hearings). Rows with `significance='minor'` and rows with `status='cancelled'` are filtered out entirely — the calendar tracks major case moments only, not procedural noise or events that are no longer happening. Each timed event is tagged with `DTSTART;TZID=<IANA tz>:<local time>`; receiving calendar apps resolve DST via their own bundled IANA databases (no VTIMEZONE shipped — bare TZIDs work in Google, Apple, Proton, Outlook 2019+, Thunderbird). The title carries `[time TBD]` / `[time unknown]` only when the event is date-only; we don't add `[HELD]` (the date itself shows what's past).
- **case_calendar/calendars/gcal.py** — Google Calendar push. Event IDs are deterministic (`sha1(case_id::hearing_key)`), so `patch` is idempotent and reschedules / detail updates land on the same event — and the `deadline:` prefix on mapped deadline keys means a hearing and a deadline that happen to share a slug get distinct gcal events. Times are sent as `dateTime: "<local>", timeZone: "<IANA>"` so Google preserves the courthouse tz across DST and viewer-tz displays. First-run uses InstalledAppFlow OAuth; tokens cache to `tokens/google-token.json` (relative to the project directory) — the daemon can't run InstalledAppFlow headless, so the operator must run `case-calendar setup gcal` once interactively to stage the refresh token. Optional `notify_emails` and `reminders` decorate events with attendees + reminder overrides; emails are added as Google Calendar attendees (Google's email reminders only fire for the calendar owner — attendees rely on their own client's reminder settings). `sendUpdates=externalOnly` so attendee invitations actually fire. Hearings flipped to `cancelled` get a `patch` with `{"status": "cancelled"}` against the existing remote event so subscribers stop seeing it — but if the event was never created (404) we noop rather than insert+cancel. `significance='minor'` rows are skipped entirely.
- **case_calendar/calendars/index.py** — Static `index.html` renderer for the public calendar directory. `build_calendar_models(cfg, store, public_base_url=...)` assembles the view-model (calendars in config order, each with subscribe links + a case list carrying docket metadata, `date_filed` = min(entries.date_filed) across the case's dockets, `last_filing_date` = max(dockets.date_last_filing) — the value CourtListener exposes as "most recent document filing date", distinct from `date_modified` which also bumps on OCR / metadata churn). `write_index(path, calendars=...)` writes the page. Self-contained — inline CSS + JS, no external requests, no CDN. CSS declares `:root { color-scheme: light dark; }` and the `<head>` carries `<meta name="color-scheme" content="light dark">`, the pair of signals Darkreader checks when deciding whether a page has built-in dark mode; both must ship together for Darkreader to skip its own filter. System preference is honored via `@media (prefers-color-scheme: dark)`, a header `<button>` lets users override and persist in `localStorage`, and an inline pre-paint `<script>` in `<head>` applies the saved preference before stylesheets load so there's no flash. Sort is client-side: each `<li.case>` carries `data-name` (lowercased for case-insensitive compare), `data-filed`, `data-last-filing` (ISO date prefix), and the JS re-appends siblings in place when the per-section `<select>` or direction checkbox changes; empty values sort last regardless of direction. The default sort is "Last filing" descending. `cli.emit_calendars` writes the index after the per-calendar ICS / gcal / M365 work — **globally**, ignoring `only_calendars`, because the page is the cross-calendar view and even an unaffected calendar's row may have moved if a sibling docket's date_last_filing advanced in the same sync. Per-case AI summaries (from the `case_summaries` table) render as a labeled paragraph block inside each case row — when a case aggregates multiple dockets the paragraphs are prefixed with `<docket_number> (<court_citation>)` so subscribers can tell which suit each sentence describes; single-docket cases drop the label. A static footer block carries two legally-sensitive disclaimers that must NEVER come from the LLM: an "AI-generated, may contain mistakes" note and "Criminal defendants are presumed innocent unless and until convicted in a court of law." Keeping these in the page template ensures the language is stable regardless of model output, and isolates the legally-loaded text from the synthesis step. Opt-in via top-level config keys `index_path`, optional `public_base_url` (when set, the index emits absolute `https://` + `webcal://` subscribe links; when omitted, only relative filenames), and optional `site_title`. Caddy is the intended fronting server (see `Caddyfile` in the repo root, checked in as a working template — operators edit the domain in place and either symlink it to `/etc/caddy/Caddyfile` for the distro systemd unit or run `caddy run --config` against it directly). The `Caddyfile` carries two blocks — an enabled `calendars.example.com` site that serves the `out/` directory with `text/calendar` content-type on `*.ics` and a sensible `Cache-Control`, plus a commented-out `webhook.example.com` that reverse-proxies the `case-calendar serve` receiver — so a single Caddy process handles both the public ICS / index hosting and the private webhook endpoint.
- **case_calendar/calendars/m365.py** — Microsoft 365 / Outlook push via Microsoft Graph. Uses the **official** `msgraph-sdk` + `azure-identity` libraries (not the third-party `O365` package). Auth is `InteractiveBrowserCredential` with `TokenCachePersistenceOptions` and a serialized `AuthenticationRecord`: first run opens a browser to grant `Calendars.ReadWrite`, the record JSON is cached at `tokens/m365-token.json` (relative to the project directory), and the daemon thereafter constructs the credential with `disable_automatic_authentication=True` to refresh silently — failing fast instead of trying to open a browser on a server. Graph generates event ids server-side (unlike Google's deterministic ids), so we persist the returned id on the hearings/deadlines row as `m365_event_id`; subsequent pushes patch by cached id, fall back to a `$filter` lookup on the `CaseCalendarKey` single-value extended property when the cache is stale (DB restore, manual delete) so we recover and re-attach to the existing event instead of orphaning it. `transactionId` is set on every create with the same key so retried POSTs from a webhook storm dedupe server-side. The SDK is async; `sync()` wraps `_sync_async` in `asyncio.run` so callers stay synchronous. Per-calendar config: `m365_calendar_id` routes to a specific Outlook calendar (`/me/calendars/{id}/events`); omit it (or set `m365_use_default_calendar: true`) to push to the user's default calendar (`/me/events`). Cancellation = `DELETE` (Graph already mails attendee cancellations on organizer DELETE, no separate verb). `significance='minor'` rows are skipped. Email reminders aren't representable in Graph's single `reminderMinutesBeforeStart`, so only popup reminders are honored — the shortest popup minutes wins.

## Key Design Decisions

- **Court/clerk variation is a primary concern; accuracy beats clean abstractions.** Every district and clerk handles CM/ECF differently — how hearings get noticed (electronic clerk's notes vs scheduling orders vs minute entries), what surfaces in PACER vs how RECAP / CourtListener mirror it, whether minute entries inline the figures or attach a judgment PDF, how reschedules and continuances are signaled, whether paperless orders exist for a given event class, what vocabulary the order uses for the same proceeding the stipulation proposed. These idiosyncrasies propagate up to every layer — extractor, verify pass, dedup sweep, summary pipeline, description renderer — and each layer should assume the next docket will do it differently and prefer the extra branch over forcing the pattern. When a fix has to choose between simpler code and accurately reflecting what a specific clerk actually does, accuracy wins; one wrong event on a public calendar erodes subscriber trust far more than a small extra case in the code. The corollary: when you see a new clerk behavior in the wild, add a test that pins the correct handling so the next refactor doesn't quietly regress on it.
- **LLM-driven extraction over per-court regexes.** Courts describe hearings inconsistently (`Set/Reset Hearings` minute entries, `ELECTRONIC NOTICE OF RESCHEDULING`, embedded in PDF text, plain prose in clerk's notes). Maintaining regexes per court is a treadmill; the LLM sees the entry plus the case's known-hearing list and decides ADD vs RESCHEDULE vs UPDATE vs CANCEL in one call.
- **Default to the small/fast LLM tier for extraction, not flagship.** The per-entry extractor is structured-output work (date + key + significance + a short title), not deep reasoning. The defaults are Haiku, gpt-5.4-nano, and Gemini Flash Lite — cents per case per day even on busy dockets. Resist proposing flagship-tier defaults; the cheap models are well-fit for this task and a 100× cost hike for ambiguously-better extraction is a bad trade. Case summaries are the separate, opt-in track that justifies a higher tier — see the per-track defaults in `llm.py` (`_DEFAULT_MODELS` vs `_DEFAULT_SUMMARY_MODELS`).
- **Token usage is logged from real counts; dollar cost is an opt-in ESTIMATE layered on top — both log-only, not persisted.** Cost docs can only ever give a rough range, and an operator running this unattended needs to verify their real spend. So every LLM call records its actual token counts (`case_calendar/llmkit/usage.py`): one `llm-tokens call …` line at INFO per call carrying `purpose` / `provider` / `model` / `docket` / `in` / `out` / `cached` / `cache_write`, plus per-docket subtotals, per-model subtotals, and a run `TOTAL` logged at the end of `sync` / `summarize`. The per-model subtotal lines are how an operator reads the extractor track's spend separately from the summary track's — they run on different model tiers, so the by-model split is the by-track split without `llmkit` having to know the domain track names (it buckets by the opaque model string, as it does for docket). The counts are NORMALIZED across providers so `in` always means total prompt tokens (cached included) and `cached` / `cache_write` are broken out — Anthropic reports cache reads/writes separately from `input_tokens` while OpenAI/Gemini fold them in, and papering over that would make the providers' numbers non-comparable. **Cost estimation is a separate, optional layer that keeps `llmkit` price-free:** the ledger exposes a `set_price_estimator(fn)` hook (`fn(model, TokenUsage) -> float | None`); when set, each `llm-tokens` line also carries a `cost_est=` field and the run TOTAL accumulates it. The estimator itself lives in the domain layer (`case_calendar/costs.py`) as a small static per-model rate table hand-copied from each provider's pricing page and dated with `PRICES_VERIFIED` — it prices each token slice (uncached input / cache read / cache write / output) at its own rate (the cache split matters because the system prompt is cached on nearly every call). It is explicitly an estimate: a model absent from the table (a legacy model or an unrecognized operator `LLM_MODEL` override) logs `cost_est=?` and the TOTAL notes how many calls were unpriced, rather than emitting a wrong number. The library stays pricing-free on purpose — baking a price table into `llmkit` would reintroduce the maintenance treadmill the tokens-only design avoided AND bloat the reusable package; the consumer supplies prices. Deliberately log-only (tokens AND cost): persisting to SQLite would mean a schema change (and the back-up-before-migration rule) for telemetry an operator can recover by summing the `TOTAL` log lines, and the value is verification-of-cost, not an operational input the code reads back. The recorder defends against the telemetry itself becoming a failure mode — every usage field is read through `_as_int`, so a missing field or an SDK shape change degrades to a 0 count rather than raising inside a working LLM call. `serve` intentionally skips the run-total path (per-call lines only, which DO include `cost_est`): its webhook worker threads and the debounce Timer thread would race on the ledger `reset`, and a "total" isn't a meaningful unit for a long-running daemon — a `sync` backfill is where cost measurement matters. `reset` clears the estimator too (full reset to fresh state), so `cli._log_llm_setup` re-attaches it at the start of each command.
- **Stable hearing keys, not stable timestamps.** A "logical" hearing (e.g. "sentencing for Wang") may be rescheduled multiple times. We let the LLM assign a kebab-case `hearing_key` on first observation and re-use it on follow-ups, so the same calendar event is updated rather than duplicated. Google event IDs are derived from this key.
- **Regex pre-filter before LLM.** Only entries whose text matches hearing-related vocabulary are sent to the LLM. Briefs, attorney withdrawals, sealed-document placeholders, etc. are dropped for free.
- **Three-tier short-circuit** to keep API + LLM costs near zero on quiet days: (1) per-docket — skip the entire docket if its `date_modified` hasn't advanced since the last sync (no entries API call, no LLM); (2) per-entry — `iter_entries(modified_after=cutoff)` filters server-side to entries newer than our store's last-modified cutoff; (3) per-fingerprint — even if an entry comes back, dedup against `(docket_id, entry_id, content_fingerprint)` skips re-LLM-ing on entries whose substantive content didn't change. The docket last-modified cutoff is only advanced on a clean run, so a mid-sync error retries the whole docket on the next run. The cutoff bump sits AFTER the iteration loop in linear control flow (no try/finally) — any exception that escapes the loop, including `BaseException` subclasses like `KeyboardInterrupt` (Ctrl+C) and `SystemExit`, must propagate past the bump without advancing it. The previous try/except/finally form used `except Exception:` which silently let those slip through with `iterated_ok=True`, advancing the cutoff despite the interrupted iteration and making the next sync's docket-level short-circuit skip the unprocessed entries entirely. Future refactors must preserve the linear shape; an `except Exception:` around the loop is a regression and pytest pins three cases (KeyboardInterrupt / SystemExit / generic Exception) to catch it.
- **Summary pipeline reads the local entries cache, not CourtListener, when sync just wrote the data.** `sync.process_entry` persists description + recap_documents (with `plain_text`) for any entry that matched the hearing/deadline pre-filter OR `summary.is_primary_document` / `summary.is_disposition`. `find_primary_documents` then reads those rows via `Store.get_entries_with_body` — same shape as a CourtListener response, so the rest of the summary pipeline doesn't branch. Before this, `refresh_stale` ran right after sync and burned three duplicate `/docket-entries/` pages per stale docket (two oldest-first + one newest-first), all to find entries CourtListener had already returned to sync minutes earlier. CourtListener is now only hit when the local cache is genuinely cold (first-ever sync of a docket, or pre-fix data lacking op/disp body text).
- **No separate PDF text cache.** Once a hearing is extracted from a PDF, the structured fields are in the `hearings` table and we never re-consult that PDF — follow-up reschedules / dial-in updates arrive on new docket entries with their own PDFs. The 2 KB excerpt for description rendering lives on the `entries` row itself. The rare case of "same entry re-processed because its fingerprint changed" pays for one IA round-trip, which is cheap compared to the operational complexity of an immutable-blob cache.
- **PDF retries on availability flips.** RECAP coverage is incomplete — many entries' PDFs get uploaded later, and the upstream `plain_text` for an image-only PDF can stay empty indefinitely (CourtListener does not OCR documents — that's our job, run locally via poppler + tesseract). The entry fingerprint includes each recap_doc's `is_available` / `is_sealed` / `plain_text`-presence flags, so a previously-missing PDF re-triggers processing automatically once it appears, and a previously-empty `plain_text` re-triggers processing once we (or any other downstream actor whose output makes it back into CourtListener's mirror) populate it.
- **The summary LLM is instructed to refuse rather than fabricate.** Even with the garbled-text detector in [[#garbled-plain_text-is-not-trusted-we-ocr-ourselves-when-it-is]] catching upstream PDF-encoding noise, other failure modes can still leave the summary LLM with too little signal — a PDF that won't OCR cleanly, an indictment that's still sealed and only the docket-text stub is available, or an entry whose recap_documents are all unavailable on RECAP. The `SUMMARY_SYSTEM_PROMPT` ends with a CRITICAL rule that on those inputs the model MUST output the exact sentence "Documents available for this docket are insufficient to generate a reliable summary." and nothing else; the matching constant `llm.SUMMARY_INSUFFICIENT_DOCUMENTS` is what `summary.summarize_docket` greps for to log a warning so the operator can investigate (typically: install poppler/tesseract so our own OCR fallback can run — CourtListener does not OCR documents, so a perpetually-empty `plain_text` means nothing upstream is going to fix it for us). The fallback prose is stored and rendered like any other summary — subscribers see the explicit refusal on the index page rather than silence or a confabulated narrative. This rule overrides the prompt's 2-4-sentence length guidance whenever it applies; it must not be weakened in future prompt revisions because it's the last-mile guard against the failure mode that produced the us-v-dubranova "CSRERI / Roskomnadzor" hallucination.
- **Garbled `plain_text` is not trusted; we OCR ourselves when it is.** Some PDFs use custom font subsets with no `/ToUnicode` map, so pypdf can't decode them — it emits glyph-index tokens (`/i255 /1 /2 ...`) or maps the raw bytes 1:1 into Latin-1 codepoints (`ÿ`/`ÿ`/punctuation noise). CourtListener's upstream pipeline runs pypdf the same way, so the `plain_text` field on a recap_document can be a multi-KB string that's non-empty and has the page header/footer in real ASCII while the body is gibberish. The summary LLM cannot tell garbage text from real text — fed gibberish, it confabulates plausible-sounding facts to fill the gap. The canonical instance was the us-v-dubranova first superseding indictment, where CourtListener's `plain_text` was 27KB of `ÿ`-noise and the summary asserted a fictitious "Center for the Study and Regulation of Internet Relations (CSRERI), also called 'Roskomnadzor'" — neither name appears in the actual indictment (the real-world group is named "CISM" and operates within the "NoName057(16)" hacktivist collective, both of which OCR reads cleanly). `pdf.is_usable_text` therefore inspects every extracted string before returning it. It's a single positive predicate that bundles every check a caller needs to decide whether to keep an extraction result or fall through to the next stage of the chain: (a) **length** — at least 100 chars, since a 2-page+ document extracting to less is almost certainly a failure not worth feeding to the LLM; (b) **font-encoding gibberish** — flagged when the ASCII-alpha ratio of non-whitespace characters falls below 0.4; real English prose runs 70-80% alpha even header-heavy and garbled extracts reliably score under 10%, so the threshold has wide margin on both sides; (c) **PACER-stamps-only extraction** — image-only scans with a thin OCR overlay covering just the page-header band let pypdf read several KB of clean ASCII off the headers but the document body (which lives only in the page images, not in any text layer) never reaches the caller. The text passes the alpha-ratio gate trivially because page stamps are mostly letters and digits, so the detector also strips the standard PACER stamp pattern (`Case <docket> Document <n> [Filed <date>] [Page <i> of <n>] [PageID:/Page ID #: <id>]`) and treats the result as useless if less than 100 chars of body survive. The us-v-schmitz indictment was the canonical case: pypdf returned 1538 chars of pure header stamps from an 18-page scan with no body text, sailed past the alpha-ratio gate, and the OCR fallback never ran until this detector caught the shape; the document itself was a real 18-page indictment that OCRs cleanly to 20 KB. The predicate runs at three points along the extraction chain — CourtListener's `plain_text`, our local pypdf output, and OCR output — and the first usable stage wins; if every stage fails the predicate, the last non-empty string is returned as a degraded-but-non-null result so callers can see the entry was attempted. The function was historically named `looks_garbled` and returned True for unusable text; it was inverted to its current positive form so callers read `if is_usable_text(text): return text` instead of the verbose `len(text) >= _MIN_USEFUL_CHARS and not looks_garbled(text)` pairing, and so its name reflects every check it does, not just the original garbled-detection one. The summary pipeline and the operator-document path (`extract_text_from_url` for `extra_documents`) share the same chain.
- **`is_sealed` is the only hard gate; once we download, our pipeline owns the result.** Two related rules govern `pdf.extract_text`. (1) **`is_available=False` is NOT a gate on the fetch.** The pipeline always attempts `fetch_pdf_bytes` when CourtListener's `plain_text` is empty or unusable, regardless of the cached `is_available` flag — only `is_sealed=True` short-circuits the fetch (sealed PDFs aren't on RECAP; a fetch would just 404). The us-v-lytvynenko regression that drove this rule had a cached recap_doc with `is_available=False, plain_text=''` while upstream CourtListener had since flipped to `is_available=True` with full text and populated `filepath_local` — gating on the cached flag meant we never tried to fetch, so pypdf and OCR never ran on a document we could perfectly well have read. `fetch_pdf_bytes` itself decides whether the document is fetchable by checking the URL fields (`filepath_ia` / `filepath_local`); when both are empty it returns `None` after just looking at the dict, so the cost of removing the gate is bounded — when storage truly has nothing, we behave the same as before, just via the URL check inside `fetch_pdf_bytes` instead of an upstream `is_available=False` bail. When storage has drifted ahead of the flag (which CourtListener can exhibit between an upstream RECAP upload and the next `is_available` flip), we successfully recover instead of bailing. (2) **Once we download a PDF, the result comes from OUR pipeline (pypdf → OCR) — never from CourtListener's `plain_text`.** The plain_text fallback at the END of `extract_text` (after pypdf and OCR both ran on the downloaded bytes) used to be `return text or plain or None`, which would re-inject CourtListener's plain_text after we'd already chosen to ignore it at the top of the function. That re-injected exactly the upstream pypdf garbage (font-encoding gibberish, PACER-stamps-only extraction) we'd just rejected, since CourtListener's upstream extraction ran the same pypdf pass we run locally. The new contract: after `fetch_pdf_bytes` succeeds, the return value is `text or None` — pypdf's output if neither pypdf nor OCR cleared `is_usable_text`, otherwise None. CourtListener's `plain_text` is only ever returned in two places, both before the download attempt or as a "we never got to run our pipeline" fallback: the top-of-function short-circuit when `is_usable_text(plain)` passes, and the `if not pdf_bytes: return plain or None` path when the fetch itself failed (no URLs or every URL 404'd) and we never got bytes to feed into pypdf. Paperless orders and minute entries that genuinely have no fetchable document still produce `None` (no URLs → `fetch_pdf_bytes` returns None → `plain or None` is None when plain is also empty); entries that exist purely as docket-text (no `recap_documents` at all) are processed by the LLM from their description without any PDF stage at all.
- **No redundant `recap-documents` calls.** The `/docket-entries/` endpoint already returns each recap_document inline with `is_available`, `is_sealed`, `plain_text`, `filepath_local`, `filepath_ia`. We use that inline copy directly rather than refetching `/recap-documents/{id}/`, which would be one wasted call per document on first sync.
- **Court-local timezones, computed once.** Hearings are stored in UTC but ICS / Google events emit the courthouse's local time tagged with the court's IANA `TZID`. Calendar viewers display in their own tz while the event preserves "this is a 3 PM Pacific hearing" semantics across DST transitions and travel. We rely on the receiver's bundled IANA tz database for DST resolution and ship the Python `tzdata` package to keep our own `zoneinfo` lookups consistent across hosts. The court → tz table in `courts.py` is hand-maintained; CourtListener doesn't surface this.
- **Sticky timezone on hearing updates.** `timezone` is set on first insert and preserved across reschedules / detail updates. A hearing happens in one courthouse, even when sibling dockets in different timezones reference it — without this, a cand entry referencing a cadc oral argument would clobber the tz to PT and shift the displayed wall-clock time.
- **Multi-docket cases collapsed into one logical case.** Some cases have multiple docket numbers (e.g. district + appellate, or related defendants). The `dockets:` list in `config.yaml` aggregates them under one `case_id`, so hearings from any docket land in the same calendar.
- **Case summaries are keyed by logical PACER docket, not by CourtListener docket_id.** CourtListener's reconciler can store the same logical PACER docket as multiple `docket_id` rows when the upstream `pacer_case_id` changed mid-life — see [CourtListener issue #7345](https://github.com/freelawproject/courtlistener/issues/7345). The canonical example is the Akhter twins case (`1:25-cr-00307`, E.D. Va.): three CourtListener docket_ids (71989485, 73333500, 73320754) with `pacer_case_id` values 585302, 585303, 585304 each carry a partial, non-overlapping slice of the PACER entries — entries 51-66 on one, 117-136 on another, 78-128 on the third. The `case_summaries` table primary key is therefore `(case_id, docket_number, court_id)` rather than `(case_id, docket_id)`, and `summary.summarize_docket` pools entries across every CourtListener docket_id in the group via `summary.find_primary_documents_for_group` (which deduplicates entries by PACER `entry_number` with a `(date_filed, description)` fallback for null-numbered paperless entries — see [[#entry-dedup-across-a-docket-group]]). The renderer collapses same-group summaries to one paragraph in the index, no docket label, and renders the docket NUMBER once as the primary clickable link. The primary link's URL is the `absolute_url` of whichever sibling CourtListener docket_id appears FIRST in `case.dockets` (config order, not freshest CourtListener docket — different from the summary pipeline's `Store.get_docket_group_ids` which picks the freshest). The choice is intentional: it gives the operator a stable way to pick the canonical CourtListener page subscribers see by reordering the `dockets:` array in `config.yaml`. Beneath the primary link, when the group has more than one CourtListener record, `index._render_cl_records` renders a muted "CourtListener records (same docket): 1 · 2 · 3" line — every record in config order, each individually clickable (its own `absolute_url`, carried in the model's per-docket `cl_records` list). This is full transparency + one-click access to each record (each holds a different slice of the entries) WITHOUT misleading a lay reader: the docket number is shown only ONCE, the records are explicitly labeled "(same docket)" with a tooltip, and the word "record" — never "docket" — is used for the siblings. The vocabulary is the disambiguator: a separate CourtListener record is NOT a separate docket or case. Genuinely distinct proceedings (a district case + its appeal) remain separate `dockets_meta` entries with their own numbers, each with its own (possibly multi-record) line. Document this in `docs/configuration.md` so the operator knows the convention. A future contributor who "simplifies" the schema back to `docket_id` keying will re-introduce the three-near-duplicate-summaries-per-case bug. The `_migrate_case_summaries_to_group_key` migration runs automatically on Store init and renames the pre-migration table to `case_summaries_pre_group_migration` as a rollback escape hatch.
- **Entry dedup across a docket group.** When `find_primary_documents_for_group` pools entries from N CourtListener docket_ids in the same `(docket_number, court_id)` group, the same logical PACER entry appears on each CourtListener sibling with a DIFFERENT CourtListener `id` but the SAME PACER `entry_number`. The dedup key is therefore `("num", entry_number)` when `entry_number` is non-null; for paperless minute orders that have no entry_number we fall back to `("desc", date_filed, description[:200])`. First-seen wins, and the freshest CourtListener docket_id is walked first (the order from `Store.get_docket_group_ids`, which sorts by `date_modified DESC`) so a stale cached copy on an older CourtListener sibling never beats a fresh one — EXCEPT when the later sibling's copy has populated `plain_text` on its main recap_document and the first-seen copy doesn't, in which case the populated copy replaces the empty one. CourtListener can leave one CourtListener row's recap_document with an empty `plain_text` while another CourtListener row in the same logical PACER group has the extracted body; without the upgrade the summary LLM is fed metadata-only and refuses with the `SUMMARY_INSUFFICIENT_DOCUMENTS` sentence. The canonical case is us-v-schmitz (`1:24-cr-00234`, D.N.J.) — the freshest CourtListener sibling 73292090's indictment recap_document had `plain_text=""` while older sibling 73353898's same logical entry carried 20 KB of text. This dedup runs BEFORE `_attach_text`, so each logical PACER entry's PDF is extracted exactly once regardless of how many CourtListener siblings carry it — no redundant PDF reads, no redundant OCR cost. The upgrade rule doesn't add reads (it picks between copies already in hand).
- **Group pooling does NOT short-circuit per-sibling CourtListener fetches.** `find_primary_documents_for_group` calls `find_primary_documents(cl, docket_id, store)` per CourtListener sibling. The per-docket function already short-circuits on its own warm cache (it returns from `get_entries_with_body` without a CourtListener fetch when its cache has a primary that isn't stale), so in steady state — after sync has populated each sibling's cache — the group function makes ZERO `/docket-entries/` calls regardless of how many CourtListener siblings are in the group. The only CourtListener fetches happen during sync itself (necessary — that's how sync discovers entries to cache) or during a stale-cache rebuild (the us-v-moucka path). A "skip the cold sibling's CourtListener fetch when a warm sibling's cache has a primary" optimization is UNSAFE and must NOT be added: it sacrifices the cold sibling's unique entries, which is the entire reason for pooling — the Akhter case proves that CourtListener siblings carry NON-overlapping slices of the PACER docket, so skipping a cold one drops real data. The corollary: an "all warm" cache short-circuit at the pooling layer would save zero fetches over the existing per-docket short-circuit. Leave the per-sibling calls in place.
- **ICS first, third-party calendars opt-in.** ICS works with anything that subscribes to a URL (Proton, Apple, etc.), needs no OAuth, and is always written. Google Calendar push requires `google_credentials_path` in the config and a staged token cache; Microsoft 365 / Outlook push requires an Entra app registration's `m365_client_id` (config or env) and a staged auth record. Both auto-enable in `emit_calendars` whenever the per-backend token cache file is present — no CLI flag — so `sync` / `serve` / `emit` push uniformly. Both can be on simultaneously; `emit_calendars` initializes each backend at most once per call and reuses it across calendars.
- **Webhook URL host is independent of `public_base_url`.** `public_base_url` is for the public ICS hosting (Caddy → `out/` directory); the webhook receiver typically lives on a different host (the shipped Caddyfile uses `calendars.example.com` for feeds and `webhook.example.com` for the receiver). `case-calendar webhook-url --host <host>` takes the webhook host as a flag rather than reading `public_base_url`, because a legitimate deployment is "run the webhook receiver privately, don't expose any public ICS feeds" — and that user has no `public_base_url` to read. Don't fold the two surfaces into a shared config key; either add a separate optional key with `--host` as a fallback, or leave the flag as the source of truth.
- **All input data is treated as untrusted.** Docket entries and PDF text can include arbitrary user-submitted content; the system prompt explicitly tells the model not to follow any instructions found inside.
- **Notifications use attendees, not email reminders.** Google Calendar's email reminder field only fires for the calendar owner; we add `notify_emails` as event attendees instead, so arbitrary addresses get the invitation + their own reminders. The `reminders` config emits per-event overrides (popup for the owner, plus an email VALARM rendered into the ICS for clients that support it).
- **Privacy-aware notification config.** `notify_emails` is exposed in the public ICS/gcal feed via `ATTENDEE` lines, so we document this trap in `config.example.yaml`: don't list addresses you don't want public. For public calendars, `reminders: [{method: popup, ...}]` works without leaking — popup fires in the subscriber's own calendar app from a `VALARM:DISPLAY` block.
- **Polling and webhook share one path.** Both `case-calendar sync` (polling) and `case-calendar serve` (webhook) call `CaseSyncer.process_entry`, so a hearing extracted via webhook is byte-identical to one extracted via polling. Webhook payloads include the docket entry with `recap_documents` inline (per the CourtListener docs), so the only CourtListener API call a webhook triggers is one `/dockets/{id}/` lookup the first time we see a new docket — to learn its `court_id` (timezone) and citation. After that, zero CourtListener calls per webhook delivery; the LLM is the only outbound dependency.
- **Docket alerts are reconciled automatically; opt out, not in.** Configured dockets only receive webhook pushes when the authenticated CourtListener account has a `docket-alerts` subscription on each one. `case_calendar.alerts.ensure_docket_alerts`, called from `cmd_sync` and `cmd_serve` startup, lists existing subscriptions and POSTs new ones for any docket in `config.yaml` that isn't already covered. Adding a case to the config wires up its docket alert on the next sync without manual UI clicks; this is the default precisely because forgetting that step is the most common operator-facing mistake (you add a case, no webhooks ever fire for it, the polling sync silently masks the gap). The flag is opt-out (`ensure_docket_alerts: false`) for operators who maintain subscriptions through a different surface — bulk CSV upload, an admin tool, etc. — and don't want the project's reconciler touching the list. The reconcile is deliberately one-way: it ADDS missing subscriptions but never deletes stale ones (removing a case from `config.yaml` doesn't unsubscribe — the operator can do that in the CourtListener UI if desired). Per-docket failures are logged at WARNING and don't abort sync/serve; a complete list-call failure marks every docket `'failed'` and skips creates entirely (avoids spamming duplicates or skipping blindly against an unknown baseline).
- **Sync and serve are separate processes sharing one SQLite file, made safe with WAL + busy_timeout.** Polling (`case-calendar sync`, typically run via cron) and the webhook receiver (`case-calendar serve`, typically a systemd unit) usually run as two distinct OS processes pointed at the same `data/case-calendar.sqlite`. SQLite's default rollback-journal mode + `busy_timeout=0` would make the second writer raise `sqlite3.OperationalError: database is locked` *immediately* on any commit overlap — a webhook landing mid-sync would bubble up to a server-side 500 (CourtListener retries via `Idempotency-Key`, so no data loss but transient errors in the log), and a sync that loses a race would abort the whole invocation. `Store.__init__` therefore opens every connection with `PRAGMA journal_mode=WAL` (lets multiple readers and one writer coexist; main writes go append-only to a `<db>-wal` sidecar and get checkpointed back into the main file later) and `PRAGMA busy_timeout=5000` (when a writer does hit a contending lock, block-and-retry for up to 5 seconds before raising — long enough to cover any of our actual transactions, all of which are single-row UPSERTs). The webhook server's in-process `_lock` (`WebhookServer.process_locked`) still serializes worker threads *within* the serve process; WAL + `busy_timeout` close the gap *between* processes. Hot backups must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main file, or run `PRAGMA wal_checkpoint(TRUNCATE)` first to consolidate everything back into the main file before copying.
- **Significance gate at render, not at extraction.** Every hearing is stored in the DB regardless of importance — that preserves the audit trail and lets the LLM use procedural events as context (`known_hearings` plus the new RELATED DOCKET ENTRIES block). Calendar visibility is filtered at render: the LLM tags each action with `significance='major' | 'minor'`, and the ICS/gcal renderers skip minor rows. Classification is by the proceeding's PURPOSE, not its EFFECT: final and CIPA pretrials and any pretrial/status conference whose agenda is a substantive motion are major; initial / Rule 16 / scheduling-only pretrials and phone calls whose only purpose is to rule on a Motion to Continue / Extend are minor — any rescheduling that comes out of those calls shows up on the trial / hearing row itself.
- **Past-date alone is not evidence a hearing occurred.** The old `_auto_mark_held_stale` sweep flipped any past 'scheduled' row to 'held' on the assumption that "date passed → it happened". Wrong for trials — they get continued or vacated by plea agreement without an explicit cancellation entry, the calendar date passes, and the row was falsely flipped to 'held' (see the us-v-moucka regression where the 4/13 trial was marked 'held' even though the docket had no minute entry, verdict form, transcript, or judgment-after-trial). Replaced by `_verify_scheduled_hearings` extended to past-dated rows: the verify-pass LLM only returns MARK_HELD when it can cite a minute entry / verdict / transcript / judgment-after as evidence. Past rows the docket doesn't confirm stay 'scheduled' — accurately communicating "outcome not confirmed" rather than asserting 'held' on guesswork. The summary LLM's existing trial-vs-plea invariant ("a trial DATE in a scheduling order is NOT proof a trial OCCURRED") matches this signal: status='scheduled' on a past trial row is the calendar's way of saying the same thing. Next sync re-verifies; once more entries land confirming the outcome, the row transitions to its terminal status.
- **Same-slot dedup sweep, group-aware.** The per-entry extractor sees `known_hearings` and is told not to allocate a new `hearing_key` for a slot another hearing already occupies on the same docket — but a court order often describes a hearing using different vocabulary than the stipulation that proposed it ("Motion Hearing" in the signed order vs "Hearing on Motion for Summary Judgment" in the joint stipulation), and the LLM can fail to recognize the same event under the new label, allocating a fresh key with a `-N` suffix. The same failure mode also surfaces across CourtListener siblings of one logical PACER docket: the per-entry extractor processing a freshly-synced sibling can allocate a fresh key for an event the OTHER sibling already has under a different key (the didenko `sentencing-didenko` vs `sentencing-didenko-2` shape). The verify pass operates on one hearing at a time and has no view of siblings, so it can't catch either failure mode on its own (it'll happily CONFIRM both rows independently). Two end-of-sync sweeps close the gap, both clustering by the LOGICAL PACER slot `(docket_number, court_id, starts_at_utc)` rather than by CourtListener `docket_id`, so cross-CourtListener-sibling drift collapses into one cluster the same way intra-docket vocabulary drift does:
  - `_dedupe_concurrent_hearings` for FUTURE `scheduled` clusters: sends each cluster to `llm.resolve_duplicate_hearings`. MERGE_INTO picks a target_key; the syncer folds the siblings' `source_entry_ids` onto the target, appends a `[dedupe]` audit line on the target naming the absorbed sibling keys + the LLM's reason, and **DELETEs the sibling rows outright** via `Store.delete_hearing`. KEEP_BOTH and UNCLEAR are no-ops — the LLM is the safety net because two distinct proceedings can legitimately be stacked back-to-back at the same time on a single docket. Zero LLM cost on quiet syncs (no clusters → no LLM call); one extra call per duplicate cluster when one exists.
  - `_dedupe_concurrent_held_hearings` for `held` clusters (past OR future): no LLM call needed, deterministic merge. A court cannot have *held* two hearings simultaneously on the same logical PACER docket — same-slot held is unambiguously a key-drift duplicate. Canonical row is picked by `(-len(source_entry_ids), last_updated, hearing_key)` (most sources, oldest, alphabetically first); siblings get their `source_entry_ids` merged onto the canonical, the canonical's `audit_notes` gets a `[dedupe-held]` line naming the absorbed sibling keys, and the **sibling rows are DELETEd outright** via `Store.delete_hearing`. Earlier behavior flipped siblings to `status='cancelled'` and kept the row, which inflated H_canc deviation in the provider scorer — each key-drift artifact counted as a spurious cancellation in the totals even though it was never a real court-ordered cancellation. The audit trail of which keys were absorbed now lives on the canonical row instead. The held sweep runs AFTER the scheduled sweep so any RESCHEDULE / CANCEL outcome from the LLM-driven scheduled pass has settled first. `cmd_sync` adds the case's calendar to `affected_calendars` when either `stats["deduped"]` or `stats["deduped_held"]` is non-zero, so a sweep-only sync still re-renders the ICS — otherwise an absorbed sibling staying in the cached ICS would survive until the next sync touched an entry.
  - **Known gap (not yet covered):** both sweeps only collapse rows that share the EXACT `(docket_number, court_id, starts_at_utc)` slot, and the held sweep additionally requires all rows to be `held`. A logical hearing that ends up `held` on one CourtListener record of a group but `scheduled` at a DIFFERENT time on a sibling record — cross-status AND non-identical slot — is collapsed by neither sweep, so it can render twice on one calendar (the us-v-wei sentencing in the provider-accuracy adjudication). A fix has to reach across status and tolerate slot differences WITHOUT breaking the bulletproof same-exact-slot argument the held sweep relies on; left for a focused follow-up (likely render-time group collapsing, like `calendars/index.py` already does for summaries, rather than mutating these sweeps).
- **Per-hearing confidence pass (past + future, both states).** Every `scheduled` hearing — past OR future — gets audited by a focused LLM call (`verify_hearing` in `llm.py`), and every PAST `cancelled` hearing does too. The model receives the candidate hearing + a window of hearing-relevant docket entries — the most recent on the docket AND the entries filed around the hearing's OWN date (`CaseSyncer._verify_context_entries`, backed by `Store.get_relevant_entries_in_date_range`, anchored on `date_filed` so an OCR / metadata re-sync that bumps `date_modified` can't hide them). The second set is essential for past hearings on busy dockets: the entry that proves a hearing occurred (a minute entry / verdict / transcript / judgment) is filed on or shortly after the hearing date, and later docket activity can push it out of the most-recent window — feeding only the recent window left such rows stuck at 'scheduled' because the LLM never saw the evidence (the us-v-mcgonigal 12/14/2023 sentencing miss, surfaced by the provider-accuracy adjudication). This widens what the LLM SEES without loosening the bar for MARK_HELD — the prompt still requires a cited record, so the trial-vacated-by-plea false-'held' guard is unaffected, and a FUTURE hearing's window is empty so it behaves exactly as before. The pass returns one of: CONFIRM (no-op), RESCHEDULE, CANCEL, MARK_HELD, REINSTATE (cancelled-row only: revert to scheduled when the cancellation isn't docket-supported), DELETE_HALLUCINATION, or UNCLEAR (no-op, log only). The state coverage is asymmetric on purpose — future `cancelled` rows are NOT verified because a deliberately cancelled future hearing should stay cancelled until something actively un-cancels it. For past-dated rows the prompt is explicit on both sides: MARK_HELD requires a minute entry / verdict / transcript / judgment-after as evidence, not just an elapsed date (UNCLEAR on a past `scheduled` row leaves it as 'scheduled' — the correct "outcome not confirmed" communication); REINSTATE requires explicit positive signal that the cancellation is wrong (no vacatur entry AND docket activity contradicts cancellation), not just absence of evidence (UNCLEAR on a past `cancelled` row leaves it as 'cancelled' — conservative bias). The next sync re-verifies once more entries land. This pass catches missed reschedules / cancellations that the per-entry extractor didn't recognize, kills the hallucination class, prevents the trial-vacated-by-plea false-'held' that the date-only auto-held sweep used to produce, AND catches the inverse-McGonigal failure mode where a live trial got falsely marked cancelled and the case continued past the date. Cost is one LLM call per non-terminal hearing per sync; with the small/fast tier this is well under a cent per case per day even on dockets with many past rows.
- **CANCEL/MARK_HELD on unknown keys with a date insert directly.** When the LLM emits a CANCEL or MARK_HELD targeting a `hearing_key` that isn't in the store — common when a memo endorsement adjourns a hearing whose original scheduling entry was filtered out — and the action carries a `local_date`, the syncer creates a new row directly in the terminal status (`cancelled` or `held`). Without this fallback, those entries silently no-oped and the audit trail lost the proceeding. Without `local_date` we drop the action and log a warning so the operator can investigate. The LLM prompt now requires `local_date` on both CANCEL and MARK_HELD specifically to support this path.
- **MARK_HELD date-proximity validation.** When the LLM emits a MARK_HELD action carrying a `local_date`, the syncer compares it against the existing hearing's `starts_at_utc`. If they differ by more than 2 days the action is rejected and logged — the LLM probably matched the minute entry onto the wrong logical hearing (e.g. shoehorning a 3/6 CIPA hearing onto a 3/8 status conference). The real hearing's row gets caught by a properly-keyed MARK_HELD on a subsequent entry, or stays 'scheduled' if no such evidence ever materializes.
- **Cross-entry context for the LLM.** Each LLM call gets two bonus context streams alongside the entry being processed: explicit cross-references parsed from the entry text (`granting 65 Motion ...` → entry 65) and the last 5 hearing-relevant entries on the same docket (temporal proximity). This is how `[Telephonic Pretrial Conference]` orders that don't say "CIPA" get correctly titled `Telephonic Pretrial Conference (CIPA)` — the originating CIPA motion lives a few entries earlier and the LLM picks up the framing. To make this work without burning storage on irrelevant entries, the `entries.description` column is populated only for hearing-relevant entries; filter-failed entries (notices, attorney appearances, etc.) keep a fingerprint stub but no text body.
- **Per-entry known-events context is scoped to the current docket's court, not the case.** Aggregated multi-docket cases come in two shapes: (a) co-defendants on separate dockets in the *same* court — really one logical proceeding, contexts should aggregate; (b) parallel actions in *different* courts (e.g. district + circuit appeal + sibling district under a different statute) — separate proceedings, contexts must NOT bleed. Without this scope, a "stay appellate proceedings" order on docket B in court Y would surface court X's hearings and deadlines in the LLM context and trigger CANCEL actions against them. `Store.get_hearings_in_court` / `get_deadlines_in_court` do the LEFT JOIN against `dockets.court_id` so the per-entry extractor only sees same-court siblings. Rows with NULL docket_id or NULL court_id are included so a warm-up sync doesn't silently drop context. The end-of-sync verify pass is naturally docket-scoped (it reads `get_recent_relevant_entries(docket_id, …)`) and didn't need a fix.
- **Iterate oldest-first within a sync.** The CourtListener API returns entries newest-first (which gives us a cheap `modified_after` early-stop), but cross-references resolve cleanly only when an entry's referenced motion is already in the local store. So `iter_entries` buffers a docket's worth of new entries and yields them sorted ascending by `date_modified`. Memory cost is bounded by `max_pages * page_size`; even a first-time sync of a busy docket is well under 10 MB.
- **No artificial cap on Retry-After.** CourtListener's daily request bucket is small — 125/day on the free tier, rising with a paid Free Law Project membership (Tier 1 at $10/mo is 300/day, up to Tier 4 at 1,400/day; see [free.law/membership](https://free.law/membership/)) — and can return Retry-After near 24h once exhausted. We honor it rather than abort, so a long-running sync resumes itself across the daily reset without manual restarts. We still log the URL, body, and rate-limit headers on every 429 so it's obvious which bucket fired (per-minute, per-hour, per-day) and whether requests are landing as anonymous (a CourtListener-flagged misconfiguration symptom).
- **Filing deadlines tracked in parallel to hearings, in the same store.** The `deadlines` table mirrors `hearings` (stable kebab-case keys, `significance` major/minor, `gcal_event_id` for idempotent push, `source_entry_ids` for audit), and `cli._deadline_to_hearing` maps a deadline row into a hearing-shaped dict so ICS and gcal renderers stay deadline-agnostic. The hearing_key namespace is shared at the LLM layer (the addendum prompt explicitly tells the model to use a separate `deadline_key` field), and the renderers prefix the gcal/ICS UID with `deadline:` so a hearing and a deadline can never collide.
- **Deadlines without an explicit time fire end-of-business in court tz.** Most scheduling orders set a date but no clock time ("response due by 5/24"). The renderer fills in 5pm court-local time so the calendar pops a useful end-of-day reminder; the rendered duration is short (15 min). When the order DOES specify a time ("must be filed by 9 AM"), the LLM passes it through verbatim and the renderer uses it as-is.
- **Deadlines tracked uniformly on every docket.** All cases (criminal, civil, appellate, magistrate, specialty) extract filing deadlines on every entry — no per-case opt-in, no docket-number auto-detect. The single `llm.SYSTEM_PROMPT` covers both hearings and deadlines; the renderer's significance gate (`major` vs `minor`) decides what reaches subscribers, which keeps procedural noise off calendars without losing the audit trail. The previous criminal-off / civil-on auto-detect was a cost optimization that produced inconsistent treatment of serious criminal cases (sentencing-memo deadlines, PSR objections, in-limine briefing) and an unprincipled split between siblings of multi-docket cases (e.g. district criminal + appellate). Uniform tracking removes a configuration knob and keeps prompts apples-to-apples across all dockets — a hearing on a `-cr-` docket sees the exact same prompt as a deadline on a `-cv-` docket, so model behavior is comparable.
- **Transcript filings — three distinct treatments.** Three rules in the deadline portion of `SYSTEM_PROMPT` distinguish transcript-related entries that look similar at the regex layer but warrant different action: (1) "ORDER for Transcript" / "Transcript Order" / "Order Form" entries are PRIVATE REQUESTS to purchase a copy of a transcript — they are NOT court orders or deadlines, emit IGNORE; (2) transcript-redaction-request deadlines (e.g. "Notice of Intent to Request Redaction due ...") are deadlines but procedural — extract with `significance="minor"` so the audit trail keeps them but subscriber calendars don't; (3) transcript public-release deadlines (the date a filed transcript becomes publicly viewable on the docket) are substantive — extract with `significance="major"`. The three are pinned by `tests/test_llm.py::TestSystemPromptTranscriptRules` so a future prompt revision can't silently drop them.
- **Auto-emit closes the loop.** Both `cmd_sync` (after each `sync_case`) and the webhook handler (after each `_handle_docket_alert`) call `cli.emit_calendars(cfg, store, only_calendars=...)` scoped to the calendars whose rows actually changed. ICS write is microseconds and idempotent — there's no separate flush step. gcal and M365 push auto-run inside `emit_calendars` whenever the per-backend token cache file exists; there's no CLI flag for it on `sync` / `serve` / `emit`, so the operator's workflow is just "run `setup gcal` / `setup m365` once, then everything pushes". The one-time `setup` subcommand is its own command precisely because its OAuth browser flow can't happen headless. Emit failures inside the webhook handler are logged but never fail the ack — the store is already updated, and a CourtListener retry would just re-process.
- **Case summaries are documents-only — never inference.** The case-summary LLM (`llm.generate_docket_summary`) must state ONLY what the source documents say. Do not add prompt rules that ask the model to infer status from the *absence* of records — e.g., do not nudge it to label defendants "fugitives" merely because no appearance / arraignment is in the docket. Conservative refusal to assert facts the documents don't support is the correct behavior, not a defect. This applies to every future revision of `SUMMARY_SYSTEM_PROMPT`. **Custody / fugitive status is the canonical instance:** state "remains a fugitive" / "remains at large" / "in custody" ONLY when a source document (the indictment, a disposition, an operator `extra_documents` NOTE) says so; when the record doesn't establish it, OMIT custody entirely — say NOTHING about it. Do NOT announce that the status is "unknown" or "cannot be determined" ("X's custody status cannot be determined from the available record") either: stating what the record does NOT show is pointless noise, not a useful fact, and it leaks the same absence-of-record failure mode the rule below forbids. This *reverses* the earlier "the honest output is UNKNOWN" framing — silence is the honest output, because a lay subscriber gains nothing from "we don't know whether X was arrested." (The deterministic guard now flags "cannot be determined from the" / "it is unknown whether" / "<custody|arrest|appearance> status … unknown" alongside the fugitive-by-absence inference.) Custody is never inferred from missing arrest/appearance entries in either direction. The prompt formerly *licensed* the fugitive inference (`"remains a fugitive" (when ... no apparent arrest is reflected in the docket)`) — that license was removed because it directly contradicted this rule. The us-v-jin / us-v-berezhnoy regressions ("remain at large with no public docket entries reflecting arrests") were the result, and us-v-zewei (charged, then arrested and extradited) is the case where the inference would have been flatly false. See the post-generation truthfulness guard decision below for the deterministic backstop.
- **Case summaries must not assert the ABSENCE of scheduling, activity, or disposition.** The complement of the documents-only rule above. The structured-events scaffold and the disposition set passed to the summary LLM reflect only what is visible in the public docket entries fetched from CourtListener; a docket that has been sealed at any point (initial sealing is routine in criminal cases pre-arrest, partial re-sealing can recur, and some dockets get re-sealed after a RECAP capture — the Dubranova `2:25-cr-00578 (C.D. Cal.)` regression that drove this rule is the canonical example) can have ongoing scheduling and filings that are simply not visible to us. Treating "the scaffold is empty" as "no hearings have been recorded" silently turns missing-from-RECAP into a positive claim about case posture. The matching `SUMMARY_SYSTEM_PROMPT` block forbids phrasings of the form "no hearings have been recorded", "no deadlines are set", "the case remains pending" (as a closing positive claim), "no disposition has been entered", "the docket shows no recent activity", and equivalents. The legal-terminology list at the top of the prompt no longer recommends "remains pending" / "case remains pending" because that phrasing was the primary vehicle for the failure mode. Future prompt revisions must preserve the forbidden-phrasing list as essential. **The rule is about the CLAIM, not the wording**, and **hedging does not exempt it**: the model reworded around the literal forbidden strings (us-v-berezhnoy: "no disposition documents have been **filed**", "the docket **does not reflect** any scheduled hearings", "no judgment is reflected **in the available record**"). For hearings / deadlines / dispositions the required output is SILENCE — describe the charges and any documented disposition, then stop. **There is no longer a custody-status exception:** custody, too, is omitted when undocumented (the earlier "→ unknown" carve-out was removed — see the documents-only rule above — because "X's custody status cannot be determined" is itself pointless absence-of-record noise, and the operator decided a lay subscriber gains nothing from being told what we don't know). A related class is **speculative / conditional future outcomes and routine sentencing boilerplate** — these are also forbidden: a consequence hanging on an event that hasn't happened ("X will be remanded to the Bureau of Prisons if a term of imprisonment is imposed", "if convicted, X faces …", "should the court impose …") is an unknown dressed up as a fact, and the mechanics of sentencing are not news. Keep the scheduled EVENT and its date ("sentencing is scheduled for June 3, 2026") but drop the hypothetical-consequence clause. The deterministic guard flags `if (convicted|found guilty)`, `if a (term of )?(imprisonment|…) is imposed`, and `should the court impose`. (us-v-martino.) A related figure-fabrication rule lives in the same prompt: **state a specific dollar amount (restitution / forfeiture / fine) only when it appears LEGIBLY in the provided document text — never reconstruct one from garbled OCR.** Court forms with hand-filled amounts OCR into noise (the us-v-chapman restitution "Total" line came through as "Total AD2, O52. 1S"), and the model decoded that garble into a confident-looking dollar amount that differed from one run to the next — neither value is trustworthy, and we have no clean source for the real figure. When the amount isn't cleanly legible, state that the obligation exists without a number; do NOT decode it (and neither should a contributor — recording a specific figure read off that garble in docs or comments is the same error). **And when restitution specifically is unreadable, drop the OTHER monetary figures too — even legible ones.** `summary._restitution_amount_unreadable` detects the case (a *granted* restitution order is among the dispositions — keyed off the strict `_is_disposition_document` classifier so a typed *proposed* order on a motion never counts — but no clean `$N,NNN` extracts from it) and `summarize_docket` passes a `DOCKET FINANCIAL ADVISORY` into the prompt; the matching `SUMMARY_SYSTEM_PROMPT` rule then suppresses all specific dollar amounts for restitution AND forfeiture (money judgments / dollar-sum forfeiture), collapsing to "was ordered to pay restitution" (and "and forfeiture" generically). Otherwise a separately-printed forfeiture order's legible figures (us-v-chapman: a clean $284,666.92 + $176,850 from order #43) read to a subscriber as the defendant's total liability while the larger, unknown restitution (handwritten in entered order #49) is invisible. The fixed special assessment and identified-property (non-dollar) forfeiture are the exceptions — neither is a dollar total a reader would sum. This deliberately overrides the "imposed sentence figures must appear" and "identified-property forfeiture stays" guidance for dollar amounts, and only while the advisory is present.
- **Post-generation truthfulness guard on case summaries (detect + prevent).** Three coordinated layers stop the summary LLM from publishing claims the documents don't support. (1) **Prompt rules** (the prevention) — the documents-only custody-OMIT rule (undocumented custody is omitted, NOT framed as "unknown"), the silence-on-procedural-absence rule (phrasing-invariant), the no-speculative/conditional-outcome rule (drop "if convicted …" / "… if a term of imprisonment is imposed" consequence clauses; keep the scheduled event + date), and the no-OCR-decode figure rule, all in `SUMMARY_SYSTEM_PROMPT` (above). (2) **A deterministic guard over the generated text** (the backstop, in `summary.py`), because prompt rules are soft and for a brand-new case there is no prior good summary to fall back to. `summary._audit_summary_text` flags absence-of-record claims (matched by CONSTRUCTION — negation + a procedural-record noun — not literal strings, so rewordings like "filed" vs "entered" are caught), custody "we don't know" noise ("cannot be determined from the" / "it is unknown whether" / "<custody|arrest|appearance> status … unknown" — also construction-matched), speculative/conditional outcomes ("if convicted" / "if a term of imprisonment is imposed" / "should the court impose"), and ungrounded custody/flight claims (flagged only when the claim's keyword is absent from the source-document corpus, so a document-stated fugitive status passes). `_generate_guarded_summary` runs it, and on a hit **retries generation once** with the violation fed back via `generate_docket_summary(correction=...)`, then keeps whichever attempt has fewer violations and logs a WARNING if any persist — the summary is never blocked ("retry-then-keep+WARN"). (3) **A grounding check for fabricated dates/amounts** — `summary._audit_summary_grounding` flags a "Month D, YYYY" date or "$N" figure in the prose that is traceable to NONE of: the structured-events scaffold (hearing/deadline dates), the source documents, or the operator-supplied notes the model is also given (the `aggregation_note` and any `extra_documents` notes). The match is whitespace-normalized, so a PDF line-break artifact like "June 26,\n\n2024" still matches. The operator-notes part of the corpus is essential — without it the guard false-positives on facts an operator deliberately supplied: the us-v-gholinejad regression was an appeal docket whose summary correctly cited a sentencing date conveyed from the sibling district docket via the `aggregation_note`, which the guard then flagged as fabricated because its corpus was documents-only. This class is **retry → strip → WARN**: because summaries auto-generate as a service, a WARN alone can sail unread into a subscriber's calendar, so a hit ACTS rather than just logging. On a hit `_generate_guarded_summary` retries generation once with the unsupported figures fed back via `generate_docket_summary(correction=...)` and an instruction to drop them or restate only what the documents support; if the retry comes back clean (and introduces no tier-1/2 regression) it is adopted. If the retry STILL carries an ungrounded figure, `summary._strip_ungrounded_sentences` deterministically removes whichever whole sentence(s) carry one (per-sentence detection reuses `_audit_summary_grounding`, so only grounded sentences survive; if every sentence is stripped the canonical refusal sentence is returned), and a WARNING is always logged so an operator can review the removal. The retry-before-strip ordering is what makes this safe for the FP-prone class: dates appear in nearly every summary and formatting variance ("5/6/26" vs "May 6, 2026", "$16M" vs "16,000,000") gives the check a real false-positive rate — re-shown the documents, the model can keep and reformat a figure it can actually support (recovering the false positive), and only a figure it genuinely can't support reaches the deterministic strip. The cost asymmetry vs the absence/custody class is now in the STRIP, not the WARN: a false positive that survives the retry costs a real sentence rather than only a log line, which is why the matching deliberately BIASES toward silence (month-only ranges unmatched, "X million" approximations skipped, amounts matched comma-insensitively as substrings). This caught the live us-v-chapman $492,052.15 fabrication and only false-positived on a line-broken date that whitespace normalization then fixed. `summary._split_sentences` does the segmentation, with an abbreviation guard (`U.S.`, middle initials, `Mr.`/`No.`/`Inc.`, case-caption `v.`) so a period mid-sentence isn't mistaken for a boundary; the backstop runs only after the model defied an explicit correction, so "good enough" segmentation plus the WARN is the right cost. All three layers are hermetically tested (string in → expected violations/warnings/removals out); the prompt rules can't be unit-tested for model obedience, which is exactly why the deterministic guard exists.
- **Programmatic sealing detection surfaces a DOCKET VISIBILITY ADVISORY to the summary LLM.** Phase 2 sibling of the absence-of-activity rule. `summary.detect_sealing` walks a small slice of the docket (one page oldest-first + one page newest-first) and flags the docket as currently sealed-in-effect when ALL of the following hold: (1) at least one visible entry matches "ORDER granting … to seal" (the `_SEAL_ORDER_RE` pattern), (2) no subsequent "ORDER granting … to unseal" is visible, (3) no disposition document is on the docket (passed in from `find_primary_documents`'s output, so an unavailable disposition wouldn't be classified as one and wouldn't kill the signal), and (4) post-sealing publicly-available entries number at most three. When all four align, the resulting advisory rides into the summary LLM's user message as a `DOCKET VISIBILITY ADVISORY` block (sibling to `AGGREGATION NOTE` — both are trusted operator-supplied metadata, distinct from document text), carrying the sealing order's entry number, filing date, verbatim description, and the observed available-post-seal entry count. The matching `SUMMARY_SYSTEM_PROMPT` rule tells the model to quote those facts and append a "subsequent docket activity may not be publicly visible" hedge, without speculating about what is happening behind the seal. The seal-then-unseal false-positive — common in criminal cases where the indictment is sealed at filing and unsealed at arrest — is suppressed by the three kill signals (unsealing order, disposition presence, substantial post-seal public availability). The walk costs roughly one extra `/docket-entries/` page per cold-cache summary; the cost is bypassed entirely when a disposition is already in hand (the `if dispositions: return None` short-circuit at the top of `detect_sealing`). Tuning the post-seal threshold downward favors more advisories at the cost of more FPs; tuning it upward favors more silence (Phase 1 behavior). The default of three is the empirical sweet spot for the Dubranova class of regressions — one or two post-seal available entries (e.g., a case summary plus a press release reference) is consistent with the seal still being in effect; four or more usually means the case is proceeding publicly.
- **Document-quality issues never reach subscriber-facing prose; they get logged for operator review.** Phase 3 sibling of the absence-of-activity and sealing-advisory rules. When a primary document's extracted text is sparse or low-quality, the summary LLM has a SUMMARY_SYSTEM_PROMPT rule ("work around partial or low-quality source documents SILENTLY") that forbids meta-commentary like "The primary document text consists only of page-header citations" or "Based on the available minute entries, [defendant] is charged with…". The model has exactly two options on a partial input: produce a normal subscriber-facing summary from whatever signals it has (primary text fragments + structured-events scaffold + dispositions), or emit the canonical `SUMMARY_INSUFFICIENT_DOCUMENTS` refusal verbatim when even those are too thin. The narration path is explicitly forbidden because subscribers don't need to read about extractor failures — they need a case summary. To keep the failure mode VISIBLE to operators, `summary.summarize_docket` logs a WARNING whenever the extracted primary document text falls below 1500 chars (the empirical floor below which the project has only seen parser failures, not legitimately short charging documents). A sweep of the warning stream is how operators spot diluted summaries before subscribers do. The us-v-moucka regression that drove this rule was a full multi-count indictment (39K of clean `plain_text` on CourtListener itself) that extracted to ~4K of user_chars on our side because the locally-stored copy of the recap_document was missing its `plain_text` field — a parsing issue distinct from "the document is genuinely short". The warning treats both root causes the same way: log it, let the summary LLM produce the cleanest possible subscriber-facing prose from whatever signal exists, and let operators investigate the root cause separately.
- **Automatically rebuild stale cached recap_documents at summary read-time.** The compact subset of each recap_document we store in `entries.recap_documents` is built by `store.compact_recap_documents` and includes `plain_text`. Rows synced before `plain_text` was a stored field have everything EXCEPT `plain_text`, and the entry fingerprint only encodes `bool(plain_text)` (not its content), so a fresh sync against a docket whose CourtListener `plain_text` flipped from empty to non-empty BETWEEN the original sync and now sees a matching fingerprint and skips re-processing. The us-v-moucka entry #1 stale-cache regression that drove this rule had that exact shape — locally-stored `plain_text` was empty even though CourtListener carried 39 KB of clean text. `summary._entry_looks_stale` detects the signature (any available, non-sealed main recap_document with empty `plain_text` in the stored copy), `summary._cached_entries_look_stale` runs it over an iterable, and `summary.find_primary_documents` falls through to CourtListener when it fires, AND uses `Store.refresh_entry_recap_documents` to rewrite the local store's recap_documents from the fresh CourtListener data — leaving the fingerprint alone so a real future content change still triggers the normal sync re-process path. The fingerprint-untouched discipline matters: re-fingerprinting here would silently mask "an actual content change just landed" on the next sync. Detection is conservative — sealed and unavailable recap_documents have legitimately empty `plain_text` and don't trigger the fallback. The staleness sweep covers BOTH primary documents (indictments, complaints) AND disposition documents (judgments, plea agreements, verdicts) because the same empty-`plain_text`-on-an-available-main-doc shape can hit either; when stale rows are detected, only those specific stale entries are dropped from the cache view (fresh ones stay) and the CourtListener fallback refreshes the recap_documents for whatever it surfaces in either classification. The whole arrangement means a once-broken cache row repairs itself on the next summary call, without operator intervention and without forcing a full re-sync.
- **Case-summary prompt invariants — trial-vs-plea, sentence specifics, dated references, past-but-unconfirmed rows, redundant-forfeiture omission.** Five prompt rules in `SUMMARY_SYSTEM_PROMPT` are essential and must survive future edits:
  - A trial DATE in a scheduling order (or the `starts_at_utc` of a "Jury Trial" row in the structured-events scaffold) is NOT proof a trial occurred. A guilty plea entered before the trial date moots the trial — Fed. R. Crim. P. 11 doesn't itself use "vacate" and courts vary on whether they enter a formal vacatur order or just take the date off-calendar. The model may say "tried before a jury" / "a jury trial was held" only when a verdict form or judgment-after-trial document is in the disposition set. Without that, it must state the plea/dismissal/etc. and stop — without claiming a trial occurred.
  - When a judgment / sentencing-judgment document is provided, the actual imposed sentence (term of imprisonment, supervised release, fine, restitution) is the most important fact about the case and MUST appear in the summary. Vague phrasing like "was sentenced" is wrong when the figures are in the document. When the judgment is unavailable and only a notice of sentencing or a plea agreement's recommendation exists, the model must say "was sentenced on [date]" without speculating about terms — the plea recommendation is not the imposed sentence.
  - If you mention a hearing or deadline by referring to its schedule, STATE THE DATE. Vague phrases like "a trial date set", "a hearing is scheduled", "a final pretrial conference scheduled", "with a hearing pending" are forbidden — they let stale or unverified scheduling slip past a subscriber unnoticed. The structured-events scaffold provides `starts_at_utc` / `due_at_utc` on every row; quote them. If the scaffold lacks a date (null), describe the trigger in the court's own language or omit the reference entirely — never substitute the absence of a date with vague wording.
  - Past-dated `status=scheduled` rows must be described as past-but-unconfirmed, NOT as upcoming. A row whose date has elapsed without the verify-pass LLM being able to MARK_HELD it (no minute entry / verdict / transcript / judgment-after on the docket) AND without an explicit vacatur entry means the public docket is silent on the outcome. The model must NOT speculate about the cause (a sealed order, a missed minute entry, an abandoned case) — the docket alone doesn't say. The honest framing states the original date AND the unconfirmed status without inventing a reason: "a trial was originally set for June 12, 2024; no public docket entry confirms either the proceeding or its vacatur, and the case has continued actively since." The McGonigal/Shestakov regression — summary said "a final pretrial conference scheduled and a trial date set" without dates after the trial-mcgonigal and FPC rows were correctly reverted from cancelled/held to scheduled, because the model hedged on stale-but-not-confirmed-wrong dates — is the canonical case.
  - When a forfeiture money judgment against the SAME defendant equals the TOTAL restitution amount, OMIT the forfeiture money judgment from the summary. The forfeiture order stays on the docket — we choose not to surface it to lay subscribers when it's redundant with restitution, because the forfeiture-money-judgment terminology adds noise without adding information once the dollar figure is already stated as restitution. The Knoot us-v-knoot (3:24-cr-00151, M.D. Tenn.) regression — summary said "$15,100 in restitution ... with a forfeiture money judgment of $15,100 also entered against him" which reads to subscribers as $30,200 total — is the canonical case. Acceptable shape: "$15,100 in restitution, and a $100 special assessment" — full stop, no mention of the forfeiture money judgment. The earlier rule iteration that made the relationship explicit ("in the same amount" / "for the same $15,100") is also NOT acceptable; the operator's preference is to drop it entirely for lay readers. The rule is narrowly scoped by three guardrails the prompt spells out explicitly: (1) the orders must run against the SAME defendant — two co-defendants each ordered $15,100 in restitution are two independent obligations and must be described separately by defendant; (2) the forfeiture must be a MONEY JUDGMENT (in personam order to disgorge a proceeds amount), NOT forfeiture of identified property (houses, cars, bank accounts, cryptocurrency wallets) — identified-property forfeiture takes things, not money, and stays in the summary on its own; a mixed judgment that includes both drops only the money-judgment portion under this rule; (3) the forfeiture money judgment must equal the TOTAL restitution across all payees — a $15,100 forfeiture against $15,100-each-to-two-victims ($30,200 total) does NOT match and the forfeiture stays. Outside of all three, restitution and forfeiture are their own line items. The omission is a DELIBERATE, prompted suppression of redundant detail; the model must not generalize it into silent drops of OTHER financial obligations. Paired with this rule is a positive directive for equal-amount multiple-payee restitution (or forfeiture): when a single defendant is ordered to pay the same amount to N victims (e.g. "$15,100 each to Acme Corp. and Beta Inc."), every payee is its own order — the summary must state the TOTAL across all payees AND either name the payees individually or state how many there are ("$30,200 in restitution, $15,100 each to Acme Corp. and Beta Inc." or "a total of $30,200 in restitution, distributed equally among two victims at $15,100 each"). The vocabulary courts use varies — line-by-line in the judgment, an attached schedule, "restitution to victims as set forth in the attached schedule" — but a per-victim itemization in any form is N orders, not one. The two rules together close opposite failure directions: the omission rule prevents stacking redundant amounts that subscribers would sum to fictional larger totals; the multi-payee rule prevents silently dropping real distinct obligations.
- **Broad `is_disposition` for stale-flagging; narrow `_is_disposition_document` for the LLM doc set.** Two predicates in `summary.py` look alike but serve different jobs. `is_disposition` is intentionally broad: it matches a "Motion for Preliminary Injunction" because the motion's outcome is a disposition-class signal worth refreshing the case summary on. `_is_disposition_document` (used inside `find_primary_documents` only) is strict: it rejects motions, briefs, status reports, case-schedule orders (`ORDER RE 149 STIPULATION ... Motion for Summary Judgment due ...`), continuances, scheduling orders (`ORDER SETTING SENTENCING HEARING`), and minute entries of motion hearings (`Minute Entry ... Motion Hearing re: 6 Motion for Preliminary Injunction held ...`) — entries that mention disposition vocabulary in passing but are not themselves the ruling document. The split exists because feeding a 200-page PI brief or a procedural minute-entry to the summary LLM as "the disposition" produces summaries that describe the wrong posture: the original bug had `case_summaries` describing a granted preliminary injunction as "taken under submission" forever, because the only "disposition" the LLM saw was the motion text. Keep the two predicates separately tested — `is_disposition` should still flag motion practice; `_is_disposition_document` should only accept actual orders / judgments / minute-entries-of-decision.
- **Paperless dispositions use the docket text as the document text.** "Electronic Clerk's Notes for proceedings held … Court imposes sentence: 92 months imprisonment, 3 years Supervised Release; $200 Special Assessment" is a real sentencing record — but it's a minute entry with the figures inline in the docket text and no PDF attached. `_attach_text` takes `allow_description_fallback`, enabled for the disposition slot only, so the description body reaches the LLM when no PDF extracts. Primary documents deliberately do NOT get this fallback — an indictment minute-entry stub is not an acceptable substitute for the real charging document, and the asymmetry is locked in by tests.
- **Case summaries auto-refresh on docket changes.** `case_summaries.enabled: true` in the top-level config is the gate. The `case_summaries` table carries `stale` and `stale_since` columns per row; `sync.process_entry` flips `stale=1` and bumps `stale_since` to NOW whenever the entry it just saw matches `summary.is_primary_document` or `summary.is_disposition` (a superseding indictment, amended complaint, judgment, plea agreement, verdict, dismissal, dispositive memo). Two refresh paths:
  - `cmd_sync` (polling) calls `summary.refresh_stale` immediately at the end of the sync — polling is already batched into a single pass, so there's nothing to debounce, and the operator running `sync` wants results now.
  - `cmd_serve` (webhook) debounces: each delivery's `emit_fn` calls `_arm_debounce(only_calendars)`, which checks whether any docket on the touched calendars has `stale=1` and, if so, (re)starts a `threading.Timer` for `case_summaries.debounce_seconds` (default 300). Every subsequent delivery within the window cancels the existing timer and starts a new one, so a PACER batch upload that fires 5 webhooks across 90 seconds produces ONE Sonnet call, not 5. When the timer fires, it runs `refresh_stale` against the pending-calendar set and re-emits those calendars to update the index with the new prose. The immediate `emit_calendars` call still fires on each delivery for fast ICS / gcal / M365 push — only the expensive Sonnet step is debounced.
  Missing rows count as stale, so adding a case to the config seeds its summary on the next sync without a separate command. `case-calendar summarize --force` remains as an explicit per-case regeneration path; `case-calendar sync --force-summaries` is the equivalent for the polling flow — both bypass the stale check and regenerate every summary on the configured cases, useful after a prompt or model upgrade. Neither is required for the automatic flow.

  **Calendar-posture changes flip stale too, not just document landings.** `process_entry`'s `is_primary_document` / `is_disposition` check above only fires on document ENTRIES. But the end-of-sync verify and dedupe sweeps change a hearing's or deadline's *posture* (scheduled → held / cancelled / reinstated, or a reschedule) WITHOUT any new document entry — so the document-only trigger missed them, and a summary could go stale silently. The canonical regression is anthropic-v-dow 26-1049 (D.C. Cir.): the oral argument was correctly flipped to `held` by the verify pass, but the summary stayed frozen at "oral argument is scheduled for May 19" because the post-argument entries were supplemental-briefing orders matching neither predicate. The fix is a chokepoint in the store: `Store.upsert_hearing` / `upsert_deadline` flag the docket's summary stale (via `_flag_summary_stale_for_docket`, joining the caller's transaction, no commit of its own) whenever the upsert is a NEW row or changes `status` / `starts_at_utc` (hearing) / `due_at_utc` (deadline). Metadata-only re-saves (notes, dial-in, a dedupe source-id merge on an unchanged row) do NOT flip stale, so there's no churn. This is the one chokepoint every posture-changing mutation funnels through (the per-entry extractor actions, the verify passes, both dedupe sweeps), so it can't be missed; the `_auto_mark_passed_stale` deadline sweep uses a direct UPDATE rather than `upsert_deadline` and is deliberately excluded (a deadline's date merely elapsing is not a prose-changing event).
- **Summaries use a higher model tier than the extractor.** Extraction is high-volume, short-context, classification-heavy — Haiku/Flash Lite/gpt-5.4-nano fit. Summaries are low-volume (one call per docket, rarely re-run), long-context (30-100K tokens of legal prose per docket), and synthesis-heavy with legal-terminology nuance — Sonnet/GPT-5.4/Gemini Pro fit. The two tracks are decoupled via separate env vars (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL`) and separate config keys (`case_summaries.provider` / `.model`) so neither track contaminates the other when defaults move.
- **Per-docket summaries, not per-case.** Multi-docket aggregated cases (Anthropic v. Dow) get one summary per constituent docket, rendered as a labeled paragraph block in the index. Primary documents live on individual dockets, so this is the natural unit; a judgment landing in one docket only dirties that docket's summary, not its siblings'.
- **Static legal disclaimers, never from the model.** "Case descriptions are generated by AI and may contain mistakes" and "Criminal defendants are presumed innocent unless and until convicted in a court of law" are rendered as a static `<footer>` block by the page template, NOT by the LLM. Keeping the legally-loaded text outside the synthesis step guarantees the language is stable regardless of model output or prompt revision. The prompt explicitly tells the model NOT to include these — they're added by the renderer.
- **Inline document links in case summaries (LLM picks the words, deterministic resolution validates).** The per-docket summaries hyperlink the words themselves, newspaper-style — the defendants "**were charged**" links to the indictment, "**pled guilty**" to the plea agreement, "**sentenced him**" to the judgment. The linked span is a SHORT action/term phrase (two or three words: the verb plus a word or two, or the short name of the order — "forfeiture money judgment", "preliminary injunction"), NOT a single bare word and NOT the trailing detail. The prompt is explicit that the link covers only the action, not the specifics that follow it — link "were charged", not "charged with wire fraud and five counts of money laundering"; link "was sentenced", not "sentenced to 60 months imprisonment and $2M restitution" — so the charge names, sentence terms, dollar amounts, and dates stay as plain text after the linked phrase. It is never a footnote marker / "[1]" / "(see Doc 1)" — readers see and tap the phrase. **The model is unreliable about exactly where the span starts and ends, so `summary._tidy_link_spans` deterministically normalizes the boundaries after resolution: it pulls an immediately-preceding auxiliary / linking verb (was/were/is/are/be/been/has/have/had) INTO the link ("was [charged]" → "[was charged]", looped so "has been [charged]" fully absorbs) and pushes a DANGLING trailing preposition (of/to/with/for/on/in/against/upon/into) OUT ("[convicted at trial of]" → "[convicted at trial] of", "[charged with]" → "[charged] with"). It only fires on a verb/preposition sitting flush against the link, so the prose's meaning is untouched — just the link boundary. This is the reliable guarantee; the prompt rule is the supporting nudge. The link target is the supporting document's PDF, resolved via `pdf.recap_document_url` (CourtListener storage → Internet Archive mirror fallback — CourtListener is the authoritative current source, IA a downstream mirror), the SAME preference the calendar event-body renderer uses (the two now share that helper — `description._document_url` delegates to it). The approach is LLM-driven, not a per-phrase regex map, for the reason every other extraction layer is: the model is the one that read the documents and knows which phrase each supports, and a deterministic phrase→doc-type table is both a treadmill and can't satisfy "any document referenced in any generated summary." The mechanism keeps the model inside the truthfulness discipline the rest of the pipeline enforces:
  - Each document fed to the summary LLM is shown with a **prompt-only reference token** in its header (`[D1]`, `[D2]`, … assigned by `summary._assign_document_refs` across primary documents, then dispositions, then `extra_documents`, in render order). The token NEVER reaches the page. The model links a phrase by writing a markdown-link-shaped marker whose target is the token: `[were charged with wire fraud](doc:D1)`.
  - `summary._resolve_document_links` rewrites those markers AFTER generation and BEFORE the prose is stored: a known token with a URL becomes `[phrase](real-url)`; a known token whose document has no reachable URL (paperless minute orders, not-yet-uploaded or sealed PDFs — `_entry_document_url` returns `None`, skipping sealed docs) collapses back to the bare phrase; a token the model **invented** (not in the set we fed it) also collapses to the bare phrase AND logs a WARNING (the link analogue of the grounding guard). So a summary can never link to a document that wasn't in its input set — the worst the model can do is link the wrong in-set document or leave a phrase unlinked, both of which are the same class of error as the prose itself and covered by the periodic prose-vs-tables audit.
  - **Ordering matters for the guards:** the post-generation truthfulness + grounding guards (`_audit_summary_text` / `_audit_summary_grounding`) run inside `_generate_guarded_summary` on the TOKEN-bearing prose, and resolution happens after. The `doc:Dn` token carries no date / dollar amount / URL, so it doesn't perturb the guards; resolving before them would inject real URLs the grounding date/amount regexes might trip on. Future refactors must keep resolution strictly after the guard pass.
  - **Storage:** the resolved `[phrase](https://…)` markdown lives in the existing `case_summaries.summary` column — no schema change. A document-URL availability flip is handled by the existing stale-regeneration machinery (the entry fingerprint flips, `sync.process_entry` flags the summary stale, `refresh_stale` re-resolves), so frozen-at-generation URLs self-correct on the next refresh.
  - **Span boundaries + weak targets:** `summary._tidy_link_spans` (above) normalizes verb/preposition boundaries. Two more rules govern WHAT is linkable: a brief direct object may stay in the span ("dismissed count three"), but a jury **verdict form is never a link target** — `_is_verdict_form` maps it to `None` in the link map even when it has a URL, because its extracted text is the checkbox template (identical blank or filled; the jury's marks aren't in the text layer), so linking "returned a verdict" to it is misleading (the us-v-ding regression). The form still rides into the prompt as summary context, but a matching `SUMMARY_SYSTEM_PROMPT` rule governs that context too: a verdict form confirms a verdict was RETURNED (and licenses "a jury trial was held") but NOT the per-count findings, so the model states the actual outcome (convicted/acquitted on which counts) only when the provided text states it, and otherwise says just "a jury trial was held and the jury returned its verdict on [date]" — never a vacuous "covering all N counts" clause (which conveys nothing, since every verdict covers the submitted counts). The specific convictions enter the summary once a judgment, which states them in text, is available. The result-bearing link target, when present, is that judgment. And a prompt rule keeps each docket's summary scoped to its own proceedings — purely-appellate events (appellate counsel appointment, appellate transcripts, the court-of-appeals case number) belong in the appellate docket's summary, not the district docket's, even though the district clerk dockets the notice of appeal and related appellate paperwork on the district docket (the us-v-gholinejad regression where the district summary narrated the appellate public-defender appointment).
  - **Rendering + XSS:** `index._render_summary_body` parses only `[words](https?://…)` markers (URL stops at whitespace or `)`), emits `<a target="_blank" rel="noopener">` on the words, and HTML-escapes the prose, the linked words, AND the href — model output stays untrusted. Anything that isn't a well-formed http(s) marker (a stray bracket, a non-link parenthetical) renders as escaped literal text. `index._summary_plain_text` strips the markers to their words for the client-side search haystack so subscribers search the words they read, not embedded URLs. The `[phrase](doc:Dn)` token markers are the ONLY markdown the summary prompt allows; the "no URLs" rule in `SUMMARY_SYSTEM_PROMPT` still holds (the model writes the token, the pipeline fills in the link).
- **`extra_documents` works around CourtListener data gaps without forging RECAP entries.** Some documents the public should be able to see don't surface through CourtListener even when they should, and the gaps come from different upstream layers. The canonical case is the Zewei docket (S.D. Tex. 4:23-cr-00523), which has two unrelated examples on a single docket: (a) entries 1-4 — including the indictment — remain sealed in the public record at the courthouse even though the seal was lifted in connection with extradition; this is an upstream sealing decision on the court's side, NOT a CourtListener data bug, and the workaround is `extra_documents` pointing at the DoJ press-release attachment that carries the indictment. (b) Entry 20's PDF (the agreed motion to designate the case as complex) sits in CourtListener's storage bucket but the v4 API reports `is_available: false` because the file was uploaded under an older `pacer_case_id` (`1940635`) than the docket's current one (`1940634`) and CourtListener's reconciler matches on the current id only ([CourtListener bug #7345](https://github.com/freelawproject/courtlistener/issues/7345)); the workaround is `extra_documents` pointing at the `storage.courtlistener.com/recap/...` URL directly. Three options were considered: (1) upload a second copy to RECAP as a workaround — muddies the upstream bug report and writes under-seal stamps into the public RECAP record; (2) auto-search out-of-band sources (DoJ press releases etc.) — brittle discovery, burns LLM tokens on every summary regen, and "we found a PDF on the internet" is a much weaker provenance claim than "we read this from PACER"; (3) `extra_documents` — a per-case YAML list on `CaseConfig` where each entry has just three fields: `docket` (must be one of the case's tracked docket ids), `url`, and a required natural-language `note` describing what the document is and why it was added. There is deliberately no `role` field: real out-of-band documents don't always slot cleanly into "pleading" / "disposition", the operator's note already carries the meaning a role taxonomy would, and the note is what the summary LLM actually reads. `summary._fetch_extra_documents` pulls the bytes via `pdf.extract_text_from_url` (same pypdf → OCR fallback chain as CourtListener recap_docs) and returns a single flat list that rides into the LLM prompt as its own `EXTRA DOCUMENTS PROVIDED BY OPERATOR` section, separate from the CourtListener-walk's primary-document and disposition slots. Each entry's LLM block carries the `OPERATOR-PROVIDED DOCUMENT (sourced outside CourtListener)` header and a `NOTE FROM OPERATOR` line; the summary system prompt is explicit that the note is trusted metadata describing the document's identity ("treat this as the unsealed indictment despite the SEALED stamps"), while the document TEXT is still untrusted in the same way as CourtListener/PACER text — matching the existing AGGREGATION NOTE convention. The `note:` field on each entry is *data fed to the summary LLM*, not operator scratchpad: keep it to one short sentence that identifies the document by name + case citation, and stop there. Bug numbers, `pacer_case_id` workaround details, "remove this entry once X ships" — all of that belongs in the YAML `#` comment block above the case in `config.yaml` (operator-facing) rather than in `note:`, because the LLM is summarizing a federal court case for public subscribers and any mention of CourtListener internals or tooling state in its output would be both off-topic and a leak of internal context. The content-gate that returns `None` when no primary-document text is available is extras-aware: extras alone satisfy it, so a docket whose only document is an operator-supplied URL still gets summarized (the canonical Zewei flow). Automatic dedup against CourtListener runs at the extracted-text level — `summary._filter_extras_already_in_cl` compares a normalized-text sha256 of each extra against the same fingerprint over every CourtListener-surfaced primary / disposition doc on the same docket group, and drops any extra whose body the CourtListener walk already produced. This catches the failure mode where the upstream gap closes after the operator added the entry: someone re-uploads the same PDF to PACER under the new `pacer_case_id` (the bug-#7345 reconciler shape, since FLP closed [the bug](https://github.com/freelawproject/courtlistener/issues/7345) without committing to a fix), or CourtListener's reconciler catches up on its own. Without this dedup, the same document body would reach the summary LLM twice — once via the primary slot, once via the extras block — wasting tokens and giving that document outsized influence. The dedup uses TEXT fingerprints (not URL match, not raw-bytes hash) deliberately: URL match misses cases where CourtListener surfaces the file under a different `filepath_local` than the operator-pinned URL, and raw-bytes hash breaks when one path runs through OCR and the other doesn't. Each drop is logged at WARN naming the URL, so the operator notices and can remove the now-redundant entry from `config.yaml`; the filter is the bridge, the operator's config cleanup is the destination. Misconfigured entries (missing field, docket id not on the case, note not a non-empty string) fail loud via `SystemExit` at config-load time so operator typos surface at boot rather than as silently-skipped documents.
- **M365 idempotency without server-side determinism.** Google lets us pick the event id; Graph doesn't. To stay idempotent across reschedules, DB restores, and manual calendar edits, M365 push uses two cooperating mechanisms: (1) cache the server-assigned id in `m365_event_id` so the common path is one PATCH per push; (2) on cache miss / 404, recover via `$filter` against a stable `CaseCalendarKey` single-value extended property keyed to `sha1(case_id::hearing_key)` that we set on every event we create. Plus `transactionId` on the create body so a retried POST from a webhook storm dedupes within Graph itself. The result is the same end-state as the gcal flow: one logical hearing → one calendar event, surviving reschedules, restores, and at-least-once delivery.

## One-shot maintenance scripts (`scripts/`)

- **`scripts/reprocess_entries.py <entry_id>...`** — re-run the LLM pipeline against specific stored entries (by CourtListener `entry_id`, not docket position). Builds a synthetic entry from the local store, clears the entry's fingerprint to bypass dedup, and calls `CaseSyncer.process_entry`. Useful after prompt changes when you want to see the new behavior on existing rows. CAUTION: the LLM can allocate fresh `hearing_key`s if it interprets a previously-vague row as multiple specific events — duplicates may need manual cleanup.
- **`scripts/classify_significance.py`** — classifies hearing significance using a focused single-question LLM prompt (no full action extraction, so no risk of the duplicate-key issue above). Default is dry-run on NULL-significance rows; `--all` widens to every row, `--apply` writes verdicts back to the DB.
- **`scripts/build_provider_stores.py`** — builds one complete store per LLM provider from the SAME cached CourtListener data, for a head-to-head comparison (cost AND output) before picking one to push to prod. Copies the warm store per provider into `data/provider-stores/<provider>/`, clears only the LLM-derived tables (hearings / deadlines / case_summaries — the `entries` rows are left untouched so the result is prod-faithful), then replays the real pipeline (`CaseSyncer._handle_entry` per cached entry + the verify/dedupe sweeps + `summary.refresh_stale`) pinned to that provider, and renders each one's ICS + `index.html` into `<provider>/out/` (push-ids stripped, so nothing reaches a real Google / M365 calendar). A shared, thread-safe response cache (CourtListener `_get`/`_post` + `pdf.extract_text` by document id) guarantees each CourtListener fetch happens at most once TOTAL across all providers — never per-provider — and the report prints the one-time CourtListener call count + peak per-minute/hour/day rate alongside per-provider/per-track token cost. Providers build in parallel (thread-local provider selection; `--no-parallel` to serialize), per-provider failures are isolated, and `--validate` diffs each store's row counts against the current prod store. `--fake` exercises the whole flow with synthetic tokens and no spend. Read-only on the live store (operates on copies).

## Testing

```bash
uv sync --extra test
uv run pytest                                    # full suite (~25s, ~600 tests)
uv run pytest tests/test_sync_integration.py    # one file
uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing
```

### Testing philosophy

- **Every change ships with the test that proves it.** Adding code → add the test that exercises the new branch. Fixing a bug → add the test that fails on the old code and passes on the new (so the regression can't sneak back). Changing behavior → update the tests that asserted the old behavior. Deleting code → delete the tests that covered it. The diff that lands a behavior change without a test change is the diff that breaks silently three months from now. CI runs the full suite with `--cov-branch` on every push to `main` and every pull request and fails the build under 90% project coverage (see [`.github/workflows/ci.yml`](.github/workflows/ci.yml)); 90% is a regression floor, not the bar to clear — the local invariant is stronger: no commit should reduce coverage at the module level. Run `uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing` before declaring a change done and confirm the touched modules either held or gained coverage.
- **Hermetic by construction.** No real HTTP, no real LLM, no real Google / Microsoft Graph, no real keyring. Every external dependency has a stub in `tests/conftest.py` or is monkey-patched per test. The `_no_real_token` autouse fixture strips any inherited `*_API_KEY` env vars so a developer with credentials in their shell can't accidentally hit a real provider.
- **Per-provider SDK calls are mocked at the SDK boundary, not the network boundary.** `tests/test_llm.py` injects fake `anthropic` / `openai` / `google.genai` modules into `sys.modules`; `tests/test_m365.py` bypasses `M365CalendarSync.__init__` via `__new__` and feeds a `MagicMock` Graph client with `AsyncMock`-backed `post` / `get` / `patch` / `delete`. This keeps tests fast and lets us assert on the call shape (model kwargs, JSON-mode toggles, extended-property writes) without needing real SDK fixtures.
- **Branch coverage, not just line coverage.** Run with `--cov-branch`. The suite holds at ≥96% branch coverage and every module is at ≥90%. New code should land with the test that drove it; the local invariant is "if a branch can fire, exercise it."
- **Cover error paths, not just success paths.** For each external integration (HTTP 4xx/5xx, 404 on patch, malformed JSON, missing required fields, OAuth token absent), test that the code falls open or fails fast as documented in [Key Design Decisions](#key-design-decisions) rather than crashing. The 5xx-fail-open semantics, the `transaction_id` retry-dedup, and the silent webhook ack on emit failure are all enforced by tests.
- **Webhook integration tests boot a real server.** `tests/test_serve.py` starts an actual `ThreadingHTTPServer` on an ephemeral port and POSTs JSON to it. This catches routing / header / body-parse regressions that an HTTP-stubbed test would miss (411 on missing Content-Length, 413 on oversized payloads, 500 from inside the process-locked critical section).
- **Test files mirror the modules they cover** (`tests/test_store.py` ↔ `case_calendar/store.py`, etc.) so a coverage gap is one file away from its fix. Cross-module integration lives in `tests/test_sync_integration.py` and `tests/test_serve.py`.
- **Test the behavior, not the implementation.** Tests assert on what the renderer writes / what the store persists / what the LLM-shaped call kwargs are, not on which private helper was called. Refactors that preserve behavior shouldn't need test edits; refactors that change behavior should.
- **Unreachable defensive code is a test smell.** If a branch can't be exercised, either delete it or fix the upstream to make it reachable — don't paper over with `# pragma: no cover`. The `extractor._entry_text` filter-empty-blobs change and the `m365._to_local_dt` deletion both came from the coverage report surfacing dead code.
- **No real `sleep` in tests.** Where the production path uses `time.sleep` (CourtListener retry/backoff) or `threading.Timer` (webhook debounce), the tests monkey-patch the sleep / Timer so the test runs at memory speed. Burning wall-clock seconds during pytest is a defect.
- **Cross-check summaries against the calendar tables as an audit pass.** The unit + integration suite catches regressions within each layer, but the LLM-driven layers (extractor, verify pass, summary) can drift in ways no unit test foresees — the model can store a hearing on the wrong key, estimate a date the order didn't fix, or fail to mark a continuance — and each of those bugs is consistent with its layer's contracts in isolation. Periodically dump every case_summaries row alongside its case's hearings + deadlines and read the prose against the structured rows: dates mentioned in the summary should map to existing rows, status flags should agree with how the prose characterizes the proceeding, and every date in the prose should be one the COURT set (not an LLM estimate of an unknown future). This audit surfaced the duplicate same-slot hearing, the conditional deadline that was being stored as a guessed calendar date, and the trial row falsely flipped to 'held' by the date-only auto-held sweep — three different bugs in three different layers, all visible only when you put the prose and the calendar side-by-side. Run this any time the LLM stack is touched (prompt revisions, model upgrades, provider switch).

The suite is hermetic — no real network calls, no real LLM, no real Google API. `tests/conftest.py` provides:

- `store` fixture — fresh tmp-path SQLite store per test
- `make_entry`, `make_recap_doc` factories — synthesize CourtListener-shaped dicts
- `FakeCourtListener` — drop-in for `CourtListener` that records calls and returns canned data
- `_no_real_token` autouse — strips real `*_API_KEY` env vars from inherited environments

Test files mirror the modules they cover (`tests/test_store.py` ↔ `case_calendar/store.py`, etc.). Integration coverage lives in:

- `tests/test_sync_integration.py` — drives `CaseSyncer` end-to-end with `FakeCourtListener` and a stubbed LLM extractor; covers hearing schedule → reschedule → cancel → mark-held flows, the parallel ADD_DEADLINE → RESCHEDULE_DEADLINE → MARK_FILED → auto-passed flow, the docket-level short-circuit, the entry-fingerprint dedup, docket/court metadata caching, the always-on uniform deadline tracking (pinning that `llm.extract_actions` is invoked the same way on both criminal and civil dockets — `known_deadlines` is always a list, no `extract_deadlines` flag), the verify-pending-deadlines end-of-case sweep (CONFIRM / CANCEL / DELETE_HALLUCINATION / MARK_FILED / RESCHEDULE / no-local-date drop / unknown-action drop), the same-docket same-slot dedupe sweep (MERGE_INTO cancels duplicates and folds source_entry_ids into the target with `[dedupe]` provenance note, KEEP_BOTH / UNCLEAR / unknown-target-key are no-ops, past hearings + zero-cluster syncs never invoke the LLM), the past-dated verify pass (MARK_HELD on cited evidence, CANCEL on vacatur, UNCLEAR leaves the row as 'scheduled' rather than guessing 'held', no separate auto-held sweep — `stats["auto_held"]` is gone),
