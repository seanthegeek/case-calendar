# Agents

## Project Overview

case-calendar — a CLI / webhook service that pulls federal court hearing dates AND filing deadlines from CourtListener / RECAP and emits them to ICS feeds (subscribable from Proton, Apple, etc.) and optionally pushes them to Google Calendar and/or Microsoft 365 / Outlook. Built for tracking cases where docket-watching by hand is too much: cybercrime prosecutions (e.g. DPRK IT-worker fraud), multi-docket tech litigation (e.g. Anthropic v. DOW), and similar. Designed to run agentically — `case-calendar serve` accepts CL webhooks, processes each delivery, and re-renders affected calendars on the spot, so subscribers see updates within seconds of a filing without any manual command run.

## Conventions

These rules apply to anyone — human or agent — making changes to this repo. They are intentionally checked in (rather than living in any one agent's private scratch memory) so that every collaborator picks them up the same way.

- **Wait for explicit commit AND push permission — these are separate grants.** Finish the implementation, run the tests, summarize the diff, then stop. The author decides when a change is ready to land; auto-committing makes review noisier and harder to reverse. "Commit this" mid-session counts as permission for that one commit, not a standing grant — and crucially, permission to commit is NOT permission to push. Pushing publishes the change to the remote where collaborators / CI / production deploys can pick it up, and is much harder to walk back than a local commit. Wait for an explicit "push it" before `git push`. If the prior commit was itself unauthorized, do NOT push it to "tidy up" — surface the situation and let the author decide whether to keep, amend, or reset.
- **Back up the SQLite store before any schema or migration change.** Before running any DDL (ALTER TABLE, CREATE/DROP, schema migrations, hand-rolled column rewrites, anything that mutates table shape) against a real store on disk, copy the DB file aside first (e.g. `cp case-calendar.sqlite case-calendar.sqlite.bak-YYYY-MM-DD-<reason>`). The store holds the full operational history (entries, hearings, deadlines, case summaries, webhook idempotency keys, source-entry provenance) — none of which is reconstructible without a multi-day CL re-sync that burns the daily quota, costs LLM dollars, and still won't recover anything CL has since rolled past. Migrations that look reversible on paper aren't always — a column rename + backfill can lose precision, an aborted migration mid-write can leave the DB in a shape no code path expects. The backup is the escape hatch. This applies in production AND in any dev environment whose DB the author cares about; ask if uncertain. (Tests use ephemeral tmp-path stores via the `store` fixture — those don't need backups.) **The store runs in WAL journaling mode** (see the concurrency design note below in Key Design Decisions), so a hot copy must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main `.sqlite` file, OR run `PRAGMA wal_checkpoint(TRUNCATE)` first to fold the WAL contents back into the main file before copying just that one file. A bare `cp` of only the main file taken while writes are in flight will silently miss any committed-but-not-yet-checkpointed rows.
- **Project-specific rules belong in AGENTS.md, not in any agent's private memory store.** If you (Claude Code, Cursor, Codex, Aider, anything that has a "save this preference for next time" surface) catch yourself about to write down a rule that's actually about the codebase rather than about working with this particular user, write it here instead. Memory is fine for user-profile facts and tool-use preferences; project rules should be portable across agents.
- **Every behavior change ships with its test** — see [Testing philosophy](#testing-philosophy) for the full statement and the verification command.

## Architecture

- **case_calendar/cli.py** — CLI entry point. Subcommands: `sync` (pull updates from CL into the store + auto-emit affected calendars; takes `--force-summaries` to bundle the equivalent of `summarize --force` into the same CL session, so a prompt or model change can be propagated in one command instead of two), `serve` (run the webhook receiver + auto-emit affected calendars per delivery), `emit` (write ICS + push to any configured gcal / M365 backend — used standalone for forced re-renders), `setup gcal|m365` (one-time OAuth bootstrap that stages the token cache), `summarize` (generate per-docket AI case summaries; opt-in, gated on `case_summaries.enabled` in config), `show` (dump current hearings + deadlines), `prune` (delete every store row tied to a `docket_id` no longer referenced by any case in the config; dry-run by default, pass `--apply` to actually delete — for the case where a docket was removed from `config.yaml` and its rows would otherwise be ignored forever), `webhook-url` (print the ready-to-paste CourtListener webhook URL, composed from `CASE_CALENDAR_WEBHOOK_SECRET` + `--host`; with `--check`, also probes the receiver's secret-gated health endpoint at `<url>/health` and reports whether the host is reachable, case-calendar is actually answering, and the local secret matches the running receiver — the operator-facing check that catches CF intercepts, stale Caddy configs, and secret mismatches in one shot). Loads `.env` via `python-dotenv` before any module reads env vars. Exposes `emit_calendars(cfg, store, *, only_calendars=None, setup_gcal=False, setup_m365=False)` as the shared rendering primitive — both `cmd_sync` and the webhook server's `emit_fn` callback go through it, so a single update flows from CL to ICS subscribers without a manual step. Push readiness for gcal / M365 is auto-detected via `_resolve_gcal` / `_resolve_m365` from config (credentials_path / client_id) + token-cache file presence; the `setup_*` flags only matter on the `setup` subcommand path, where they allow the OAuth browser flow to run when no cache exists yet. Also owns the deadline-to-hearing adapter (`_deadline_to_hearing` maps the deadline row into a hearing-shaped dict the renderers accept) and `_compose_title` (the title prefixer that adds `[HEARING]` / `[DEADLINE]` / `[time TBD]` / `[time unknown]` flags).
- **case_calendar/courtlistener.py** — REST v4 client. Wraps `dockets/`, `docket-entries/`, `recap-documents/` endpoints with backoff for 429 (`Retry-After`) and 5xx responses. Honors any `Retry-After` value (the daily 300/day bucket can return a wait near 24h; we sleep through rather than abort) and logs URL + body + rate-limit headers on every 429 so it's obvious which bucket fired. Iterator-style pagination via `iter_entries`, with a `modified_after` cutoff for cheap incremental syncs; pages newest-first (preserves the cheap stop-condition) but buffers and yields oldest-first so cross-entry references resolve cleanly within a single sync.
- **case_calendar/courts.py** — Court-id → IANA timezone mapping. CourtListener doesn't expose timezones in its API, so we maintain the table here. Coverage is comprehensive: all 13 federal circuits, all 94 district courts, the Supreme Court, specialty courts (Federal Claims, CIT, FISA, Tax, Armed Forces), and territories (PR, VI, Guam, NMI, American Samoa). Multi-tz states (Tennessee, Florida, Indiana, etc.) map to the principal-office tz of each district. Unknown court IDs log a warning and fall back to ET so the gap is visible rather than silent.
- **case_calendar/extractor.py** — Cheap regex pre-filter. `is_hearing_relevant` matches hearing-vocabulary entries; `is_deadline_relevant` matches filing-deadline vocabulary (`due by`, briefing schedule, scheduling order, stipulation, etc.); `is_extractable(entry, *, want_deadlines)` is the combined gate the syncer calls — hearing-relevant entries always pass, deadline-relevant entries pass only when the case opts into deadline tracking. Briefs, attorney appearances, and similar paperwork entries are skipped before any LLM tokens are spent. Intentionally over-inclusive — false positives just cost an extra LLM call; false negatives lose hearings/deadlines.
- **case_calendar/llm.py** — Provider-agnostic extractor for both hearings and filing deadlines. Selects between anthropic / openai / gemini via `LLM_PROVIDER` env (or auto-detects from whichever `*_API_KEY` is set). `LLM_MODEL` overrides the per-provider default. Defaults are the small/fast tier (Haiku, gpt-5.4-nano, Flash Lite) — this is structured extraction, not deep reasoning. SDKs are imported lazily inside the per-provider call functions. The Anthropic path applies `cache_control: ephemeral` on the system block, so all four prompts (extract / verify_hearing / verify_deadline / extract+deadlines) get prompt-cache discounts within a sync. Four entry points:
  - `extract_actions` runs against ONE docket entry plus the case's known-hearings list (and known-deadlines list when `extract_deadlines=True`) and returns 0+ actions: hearing actions (ADD / RESCHEDULE / UPDATE_DETAILS / CANCEL / MARK_HELD / IGNORE) and, when deadlines are on, deadline actions (ADD_DEADLINE / RESCHEDULE_DEADLINE / CANCEL_DEADLINE / MARK_FILED). The system prompt is `SYSTEM_PROMPT` plus `DEADLINE_PROMPT_ADDENDUM` when deadlines are enabled — keeping the addendum off-by-default keeps the simpler hearings-only prompt cheap on cases that don't need it.
  - `verify_hearing` runs against ONE candidate hearing plus the last 15 hearing-relevant entries on its docket and returns a single audit decision (CONFIRM / RESCHEDULE / CANCEL / MARK_HELD / DELETE_HALLUCINATION / UNCLEAR) — used by the end-of-sync confidence pass to catch reschedules/cancellations the per-entry extractor missed and to reject rows the LLM hallucinated.
  - `verify_deadline` is the deadline analogue — same shape, returns CONFIRM / RESCHEDULE / CANCEL / MARK_FILED / DELETE_HALLUCINATION / UNCLEAR for one pending deadline.
  - `resolve_duplicate_hearings` is the same-slot resolver — receives a CLUSTER of 2+ scheduled hearings on the same docket sharing the EXACT same `starts_at_utc`, plus recent docket entries, and returns one of MERGE_INTO (pick a target_key, the caller cancels the others and folds their source_entry_ids into the target) / KEEP_BOTH (rare — stacked back-to-back proceedings) / UNCLEAR. A single court can't hold two hearings on one docket simultaneously, so the cluster almost always means the per-entry extractor split one logical event across keys (vocabulary varied between the entries that scheduled it).
  - `generate_docket_summary` is the **case-summary** entry point — runs the operative pleading + disposition documents + a structured-events scaffold (hearings/deadlines with their statuses) through a higher-tier model and returns 2-4 sentences of prose plus the model identifier. Uses a separate provider/model selection (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL` env vars, with config-level `case_summaries.provider` / `.model` taking precedence) so the extractor pipeline's cheap defaults stay decoupled. Per-provider defaults are Sonnet (Anthropic) / gpt-5.4 (OpenAI) / Gemini Pro — synthesis-heavy enough to warrant the upgrade. Returns plain text, not JSON; the per-provider callers expose a `json_mode=False` knob to support this.

  Each ADD/RESCHEDULE/UPDATE_DETAILS action (hearing or deadline) carries a `significance` field (`major` / `minor`) that controls calendar visibility — substantive proceedings (trial, sentencing, oral arg, any pretrial conference, evidentiary hearing, plea, etc.) and dispositive briefing deadlines (MTD/MSJ response/reply, sentencing memos, motion-in-limine, Daubert, suppression briefing) are major; procedural-only events (a phone call set just to rule on a Motion to Continue, routine joint status reports, proposed orders) are minor and filtered out at render time. The user message includes a "RELATED DOCKET ENTRIES" block (explicit cross-refs from the new entry's text + the last few hearing-relevant entries on the same docket) so the LLM can name a hearing correctly even when the entry that schedules it doesn't itself name the subject.
- **case_calendar/pdf.py** — PDF text extraction with fallbacks. Order: CL `plain_text` → `pypdf` on the Internet Archive mirror → `pdftoppm` + `tesseract` OCR (optional system dependency). Sealed PDFs and not-yet-uploaded PDFs return `None` and are NOT cached — the next sync re-fingerprints the entry and retries once the PDF arrives or gets OCR'd.
- **case_calendar/store.py** — SQLite state. Tables: `dockets` (id + last `date_modified` (the polling short-circuit watermark) + `date_last_filing` (CL's "most recent document filing date"; what the index's "Last filing" column reads) + cached metadata for description rendering), `entries` (dedup of already-processed entries, keyed by `(docket_id, entry_id)` with content fingerprint, `entry_number` for cross-ref lookup, `description` / `short_description` / `recap_documents` JSON stored for entries that either passed the regex pre-filter OR matched `summary.is_operative_pleading` / `summary.is_disposition` — every other entry gets a fingerprint-only stub so dedup still works without storing dead-weight notice text. `recap_documents` keeps the compact projection (`id`, `document_number`, `attachment_number`, status flags, `filepath_ia`, `filepath_local`, `plain_text`) that the description renderer turns into clickable per-doc URLs at emit time, and that the summary pipeline uses to find operative pleadings + dispositions locally via `get_entries_with_body` instead of re-fetching docket-entries pages from CL right after sync wrote them down — `plain_text` rides along on the compact projection so `pdf.extract_text` short-circuits without a PDF fetch), `hearings` (per-case logical hearings keyed by `(case_id, hearing_key)` with `significance` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list), `deadlines` (per-case filing deadlines keyed by `(case_id, deadline_key)` with `due_at_utc` / `status` ∈ {pending, met, passed, cancelled} / `significance` / `deadline_type` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list — parallel structure to hearings so they share renderers), `case_summaries` (per-docket prose summaries keyed by `(case_id, docket_id)` with `summary` text, `model` identifier, `source_entry_ids` JSON listing the CL entries that informed the LLM, and a `stale` boolean that `sync.process_entry` flips when it sees a new operative pleading or disposition so the agentic refresher knows to regenerate; populated by `summary.refresh_stale` during sync / webhook emit, or explicitly by `case-calendar summarize`), `courts` (CL court metadata, citation_string), `webhook_events` (idempotency-key dedup). The `m365_event_id` columns are populated by `set_m365_id_for_hearing` / `set_m365_id_for_deadline` after a Graph create; gcal ids are deterministic so `gcal_event_id` is currently unused at runtime. Connection is `check_same_thread=False` so the webhook server's worker threads can use the same Store instance under a server-wide lock; the connection is also opened with `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000` so the polling `sync` process and the long-running `serve` process can safely share the same DB file (see the concurrency design note below).
- **case_calendar/sync.py** — Per-case orchestration. For each docket: fetch entries newer than the store's high-water mark, run the regex filter, optionally pull PDFs, resolve cross-references (`granting 65 Motion ...` → look up entry 65 from the local store) plus the last 5 hearing-relevant entries on the docket as temporal context, hand entry + known-hearings list (+ known-deadlines list when deadlines are on) + related entries to the LLM, apply returned actions to the hearings/deadlines tables. The known-hearings / known-deadlines context is sourced via `Store.get_hearings_in_court` / `get_deadlines_in_court` so it's scoped to siblings in the SAME court as the current docket — same-court co-defendant dockets still share context, but parallel proceedings in different venues do not (a stay order in court B must not contaminate court A's events). `_needs_pdf` forces a PDF fetch on orders that grant scheduling-affecting motions (`Motion for Hearing`, `to Continue`, `for Continuance`, etc.) even when the order text inlines a date+time, since the brief description references the underlying motion only by docket position and the PDF often holds dates the description doesn't echo. Local time → UTC conversion happens here using the court's IANA timezone. End-of-case sweeps run in order: (1) `_verify_scheduled_hearings` — per-hearing LLM confidence pass for every scheduled hearing (BOTH future and past); past-dated rows only flip to 'held' when the LLM cites a minute entry / verdict / transcript / judgment-after as evidence of occurrence — date-passed alone is not enough, so trials vacated by plea stay 'scheduled' instead of being falsely marked 'held'; (2) `_dedupe_concurrent_hearings` — resolves same-docket same-`starts_at_utc` clusters via `llm.resolve_duplicate_hearings`, runs AFTER verify so any RESCHEDULE/CANCEL from verify gets a chance to clear concurrency before the dedup LLM is consulted; (3) when deadlines are on, `_verify_pending_deadlines` and `_auto_mark_passed_stale` do the same for pending deadlines. The `resolve_extract_deadlines(case, docket_id=None)` method decides per-call whether to extract deadlines: `case.extract_deadlines=True` is a force-on override; otherwise `_docket_implies_deadlines(docket_number)` classifies by federal docket-number prefix (`-cr-` / `-cm-` / `-po-` / `-mj-cr-` → off, civil/appellate/specialty → on, falls back to on when metadata is missing). Exposes `CaseSyncer.process_entry` so the webhook server can reuse the exact same per-entry pipeline.
- **case_calendar/summary.py** — Per-docket case-summary pipeline (opt-in, off by default). For each docket: hits CL directly to find operative pleadings (`is_operative_pleading` matches `INDICTMENT` / `SUPERSEDING INDICTMENT` / `INFORMATION` / `COMPLAINT` / amended variants at the head of the entry description) and dispositions (`is_disposition` matches `JUDGMENT` / `VERDICT` / `ORDER OF DISMISSAL` / `PLEA AGREEMENT` / `MEMORANDUM OPINION` etc.), pulls PDF text via the existing `pdf.extract_text` fallback chain, gathers the docket's hearings + deadlines from the store as a structured scaffold, and hands the whole package to `llm.generate_docket_summary`. Dispositions get an extra fallback: when no PDF text extracts, the entry description itself is used as the disposition text (`allow_description_fallback=True` on `_attach_text`). This is how a paperless "Electronic Clerk's Notes for proceedings held … Court imposes sentence: 92 months imprisonment …" entry — where the imposed sentence lives inline in the docket text with no judgment PDF attached — still reaches the LLM. Operative pleadings deliberately do NOT get this fallback (a minute-entry stub is not a substitute for an indictment). When `store` is passed in (the normal path from `summarize_docket`), `find_operative_documents` reads from the local `entries` table first via `Store.get_entries_with_body` — `sync.process_entry` now persists description + recap_documents for op/disp matches as well as hearing-relevant ones, so right after a sync the local cache already has everything the summary pipeline needs. Falls back to CL only on a cold docket (first ever sync, or pre-fix data where op/disp entries are still stub-only); the CL fallback walks two pages oldest-first for operative pleadings and three pages newest-first for dispositions (busy civil cases with heavy post-disposition briefing can push the dispositive order well past page 1). The picker for the LLM document set uses the stricter `_is_disposition_document` — see the broad-vs-narrow design note below. Three entry points: `summarize_docket` runs the full pipeline for one docket and persists the row; `summarize_case` loops the dockets on one case (used by `case-calendar summarize`); `refresh_stale` walks all configured cases and regenerates any row marked `stale=1` (or missing entirely) — this is the agentic path called from `cmd_sync` and the webhook `emit_fn` so a freshly-landed judgment regenerates the prose in the same emit cycle. Skipping is silent: dockets without extractable operative-pleading text (sealed indictments, appellate dockets whose petitions are pure cover sheets) are logged and dropped rather than guessed at — the index renderer omits the summary block when the row is missing.
- **case_calendar/serve.py** — Webhook receiver for CourtListener `DOCKET_ALERT` events (real-time alternative to polling). Stdlib `ThreadingHTTPServer`; serializes processing with a server-wide lock so SQLite writes don't race. Auth model is the URL secret (CL has no signing mechanism), and `Idempotency-Key` headers are stored to make CL retries no-ops. The same store + entry-fingerprint dedup makes double-delivery safe even without the idempotency check. `WebhookServer` accepts an optional `emit_fn(only_calendars: set[str]) -> None` callback that the handler invokes after each successful docket-alert delivery, scoped to the calendars whose rows actually changed — `cmd_serve` builds this callback to call `cli.emit_calendars` with the configured ICS paths; gcal / M365 push auto-runs inside `emit_calendars` whenever a token cache is present. Emit failures are logged but never fail the webhook ack: the store is already updated, and a CL retry would just re-process the same entries. Exposes two GET endpoints alongside the POST receiver: `/health` is an unauthenticated liveness probe (returns `{"status":"ok"}` — for Caddy / uptime checks), and `/webhooks/case-calendar/<secret>/health` is the secret-gated identity probe that returns `{"service":"case-calendar","tracking":{...}}` so `webhook-url --check` can confirm the operator's secret matches the running receiver in one shot.
- **case_calendar/calendars/description.py** — Shared event-body builder used by both ICS and gcal renderers. Composes notes → judge / panel → dial-in → case citation → docket URL → per-document URLs (`Documents:` block, one line per attached doc — `65: <url>` for the main doc, `65-1: <url>` for attachment 1; `(sealed)` / `(not yet uploaded to RECAP)` replace the URL when no PDF is reachable) → PACER docket entry numbers (`Docket entries: 65, 82`) → CourtListener entry IDs. The docket-numbers line is what subscribers see in the CL UI; it sits directly above the opaque CL entry IDs so the audit trail reads docket-first. Both `Documents:` and `Docket entries:` are populated in `cli.emit_calendars` from `Store.get_entry_documents` / `Store.get_entry_numbers(source_entry_ids)`; source entries that lack a docket position (paperless minute orders) are silently dropped, and missing or empty doc lists produce no `Documents:` block at all. URL preference is IA mirror (`filepath_ia`) → CL storage (`https://storage.courtlistener.com/<filepath_local>`). New documents (e.g. an attachment uploaded later) land in the description automatically: the entry's fingerprint already includes per-doc availability flags, so a doc-add flips the fingerprint, `process_entry` re-runs and overwrites the stored `recap_documents` JSON, and the auto-emit (sync + webhook) re-renders the calendar. Also exports `no_time_title_prefix(starts_at_utc)` which returns `[time TBD]` for future date-only events (we expect the time to be set later) and `[time unknown]` for past ones (we never learned it). Renderers are deadline-agnostic: deadlines are mapped to hearing-shaped dicts in `cli._deadline_to_hearing` (with `hearing_key` namespaced as `deadline:<key>`) before they reach this layer.
- **case_calendar/calendars/ics.py** — Hand-rolled RFC 5545 output. UIDs are derived from `case_id + hearing_key` so regenerations are stable across runs (and the `deadline:` prefix on mapped deadline keys keeps their UIDs in a separate namespace from real hearings). Rows with `significance='minor'` and rows with `status='cancelled'` are filtered out entirely — the calendar tracks major case moments only, not procedural noise or events that are no longer happening. Each timed event is tagged with `DTSTART;TZID=<IANA tz>:<local time>`; receiving calendar apps resolve DST via their own bundled IANA databases (no VTIMEZONE shipped — bare TZIDs work in Google, Apple, Proton, Outlook 2019+, Thunderbird). The title carries `[time TBD]` / `[time unknown]` only when the event is date-only; we don't add `[HELD]` (the date itself shows what's past).
- **case_calendar/calendars/gcal.py** — Google Calendar push. Event IDs are deterministic (`sha1(case_id::hearing_key)`), so `patch` is idempotent and reschedules / detail updates land on the same event — and the `deadline:` prefix on mapped deadline keys means a hearing and a deadline that happen to share a slug get distinct gcal events. Times are sent as `dateTime: "<local>", timeZone: "<IANA>"` so Google preserves the courthouse tz across DST and viewer-tz displays. First-run uses InstalledAppFlow OAuth; tokens cache to `tokens/google-token.json` (relative to the project directory) — the daemon can't run InstalledAppFlow headless, so the operator must run `case-calendar setup gcal` once interactively to stage the refresh token. Optional `notify_emails` and `reminders` decorate events with attendees + reminder overrides; emails are added as Google Calendar attendees (Google's email reminders only fire for the calendar owner — attendees rely on their own client's reminder settings). `sendUpdates=externalOnly` so attendee invitations actually fire. Hearings flipped to `cancelled` get a `patch` with `{"status": "cancelled"}` against the existing remote event so subscribers stop seeing it — but if the event was never created (404) we noop rather than insert+cancel. `significance='minor'` rows are skipped entirely.
- **case_calendar/calendars/index.py** — Static `index.html` renderer for the public calendar directory. `build_calendar_models(cfg, store, public_base_url=...)` assembles the view-model (calendars in config order, each with subscribe links + a case list carrying docket metadata, `date_filed` = min(entries.date_filed) across the case's dockets, `last_filing_date` = max(dockets.date_last_filing) — the value CL exposes as "most recent document filing date", distinct from `date_modified` which also bumps on OCR / metadata churn). `write_index(path, calendars=...)` writes the page. Self-contained — inline CSS + JS, no external requests, no CDN. CSS declares `:root { color-scheme: light dark; }` and the `<head>` carries `<meta name="color-scheme" content="light dark">`, the pair of signals Darkreader checks when deciding whether a page has built-in dark mode; both must ship together for Darkreader to skip its own filter. System preference is honored via `@media (prefers-color-scheme: dark)`, a header `<button>` lets users override and persist in `localStorage`, and an inline pre-paint `<script>` in `<head>` applies the saved preference before stylesheets load so there's no flash. Sort is client-side: each `<li.case>` carries `data-name` (lowercased for case-insensitive compare), `data-filed`, `data-last-filing` (ISO date prefix), and the JS re-appends siblings in place when the per-section `<select>` or direction checkbox changes; empty values sort last regardless of direction. The default sort is "Last filing" descending. `cli.emit_calendars` writes the index after the per-calendar ICS / gcal / M365 work — **globally**, ignoring `only_calendars`, because the page is the cross-calendar view and even an unaffected calendar's row may have moved if a sibling docket's date_last_filing advanced in the same sync. Per-case AI summaries (from the `case_summaries` table) render as a labeled paragraph block inside each case row — when a case aggregates multiple dockets the paragraphs are prefixed with `<docket_number> (<court_citation>)` so subscribers can tell which suit each sentence describes; single-docket cases drop the label. A static footer block carries two legally-sensitive disclaimers that must NEVER come from the LLM: an "AI-generated, may contain mistakes" note and "Criminal defendants are presumed innocent unless and until convicted in a court of law." Keeping these in the page template ensures the language is stable regardless of model output, and isolates the legally-loaded text from the synthesis step. Opt-in via top-level config keys `index_path`, optional `public_base_url` (when set, the index emits absolute `https://` + `webcal://` subscribe links; when omitted, only relative filenames), and optional `site_title`. Caddy is the intended fronting server (see `Caddyfile` in the repo root, checked in as a working template — operators edit the domain in place and either symlink it to `/etc/caddy/Caddyfile` for the distro systemd unit or run `caddy run --config` against it directly). The `Caddyfile` carries two blocks — an enabled `calendars.example.com` site that serves the `out/` directory with `text/calendar` content-type on `*.ics` and a sensible `Cache-Control`, plus a commented-out `webhook.example.com` that reverse-proxies the `case-calendar serve` receiver — so a single Caddy process handles both the public ICS / index hosting and the private webhook endpoint.
- **case_calendar/calendars/m365.py** — Microsoft 365 / Outlook push via Microsoft Graph. Uses the **official** `msgraph-sdk` + `azure-identity` libraries (not the third-party `O365` package). Auth is `InteractiveBrowserCredential` with `TokenCachePersistenceOptions` and a serialized `AuthenticationRecord`: first run opens a browser to grant `Calendars.ReadWrite`, the record JSON is cached at `tokens/m365-token.json` (relative to the project directory), and the daemon thereafter constructs the credential with `disable_automatic_authentication=True` to refresh silently — failing fast instead of trying to open a browser on a server. Graph generates event ids server-side (unlike Google's deterministic ids), so we persist the returned id on the hearings/deadlines row as `m365_event_id`; subsequent pushes patch by cached id, fall back to a `$filter` lookup on the `CaseCalendarKey` single-value extended property when the cache is stale (DB restore, manual delete) so we recover and re-attach to the existing event instead of orphaning it. `transactionId` is set on every create with the same key so retried POSTs from a webhook storm dedupe server-side. The SDK is async; `sync()` wraps `_sync_async` in `asyncio.run` so callers stay synchronous. Per-calendar config: `m365_calendar_id` routes to a specific Outlook calendar (`/me/calendars/{id}/events`); omit it (or set `m365_use_default_calendar: true`) to push to the user's default calendar (`/me/events`). Cancellation = `DELETE` (Graph already mails attendee cancellations on organizer DELETE, no separate verb). `significance='minor'` rows are skipped. Email reminders aren't representable in Graph's single `reminderMinutesBeforeStart`, so only popup reminders are honored — the shortest popup minutes wins.

## Key Design Decisions

- **Court/clerk variation is a first-class concern; accuracy beats clean abstractions.** Every district and clerk handles CM/ECF differently — how hearings get noticed (electronic clerk's notes vs scheduling orders vs minute entries), what surfaces in PACER vs how RECAP / CourtListener mirror it, whether minute entries inline the figures or attach a judgment PDF, how reschedules and continuances are signaled, whether paperless orders exist for a given event class, what vocabulary the order uses for the same proceeding the stipulation proposed. These idiosyncrasies propagate up to every layer — extractor, verify pass, dedup sweep, summary pipeline, description renderer — and each layer should assume the next docket will do it differently and prefer the extra branch over forcing the pattern. When a fix has to choose between simpler code and accurately reflecting what a specific clerk actually does, accuracy wins; one wrong event on a public calendar erodes subscriber trust far more than a small extra case in the code. The corollary: when you see a new clerk behavior in the wild, add a test that pins the correct handling so the next refactor doesn't quietly regress on it.
- **LLM-driven extraction over per-court regexes.** Courts describe hearings inconsistently (`Set/Reset Hearings` minute entries, `ELECTRONIC NOTICE OF RESCHEDULING`, embedded in PDF text, plain prose in clerk's notes). Maintaining regexes per court is a treadmill; the LLM sees the entry plus the case's known-hearing list and decides ADD vs RESCHEDULE vs UPDATE vs CANCEL in one call.
- **Default to the small/fast LLM tier for extraction, not flagship.** The per-entry extractor is structured-output work (date + key + significance + a short title), not deep reasoning. The defaults are Haiku, gpt-5.4-nano, and Gemini Flash Lite — cents per case per day even on busy dockets. Resist proposing flagship-tier defaults; the cheap models are well-fit for this task and a 100× cost hike for ambiguously-better extraction is a bad trade. Case summaries are the separate, opt-in track that justifies a higher tier — see the per-track defaults in `llm.py` (`_DEFAULT_MODELS` vs `_DEFAULT_SUMMARY_MODELS`).
- **Stable hearing keys, not stable timestamps.** A "logical" hearing (e.g. "sentencing for Wang") may be rescheduled multiple times. We let the LLM assign a kebab-case `hearing_key` on first observation and re-use it on follow-ups, so the same calendar event is updated rather than duplicated. Google event IDs are derived from this key.
- **Regex pre-filter before LLM.** Only entries whose text matches hearing-related vocabulary are sent to the LLM. Briefs, attorney withdrawals, sealed-document placeholders, etc. are dropped for free.
- **Three-tier short-circuit** to keep API + LLM costs near zero on quiet days: (1) per-docket — skip the entire docket if its `date_modified` hasn't advanced since the last sync (no entries API call, no LLM); (2) per-entry — `iter_entries(modified_after=cutoff)` filters server-side to entries newer than our store's high-water mark; (3) per-fingerprint — even if an entry comes back, dedup against `(docket_id, entry_id, content_fingerprint)` skips re-LLM-ing on entries whose substantive content didn't change. The docket high-water mark is only advanced on a clean run, so a mid-sync error retries the whole docket on the next run.
- **Summary pipeline reads the local entries cache, not CL, when sync just wrote the data.** `sync.process_entry` persists description + recap_documents (with `plain_text`) for any entry that matched the hearing/deadline pre-filter OR `summary.is_operative_pleading` / `summary.is_disposition`. `find_operative_documents` then reads those rows via `Store.get_entries_with_body` — same shape as a CL response, so the rest of the summary pipeline doesn't branch. Before this, `refresh_stale` ran right after sync and burned three duplicate `/docket-entries/` pages per stale docket (two oldest-first + one newest-first), all to find entries CL had already returned to sync minutes earlier. CL is now only hit when the local cache is genuinely cold (first-ever sync of a docket, or pre-fix data lacking op/disp body text).
- **No separate PDF text cache.** Once a hearing is extracted from a PDF, the structured fields are in the `hearings` table and we never re-consult that PDF — follow-up reschedules / dial-in updates arrive on new docket entries with their own PDFs. The 2 KB excerpt for description rendering lives on the `entries` row itself. The rare case of "same entry re-processed because its fingerprint changed" pays for one IA round-trip, which is cheap compared to the operational complexity of an immutable-blob cache.
- **PDF retries on availability flips.** RECAP coverage is incomplete — many entries' PDFs get uploaded later, and CL's OCR runs asynchronously. The entry fingerprint includes each recap_doc's `is_available` / `is_sealed` / `plain_text`-presence flags, so a previously-missing PDF re-triggers processing automatically once it appears.
- **No-document detection.** Paperless orders and minute entries show up as `recap_documents` rows with `is_available: false`, no `filepath_local`, no `filepath_ia` — there is literally no document to fetch. `_is_fetchable` rejects these up-front so we never attempt the IA download. Entries that exist purely as docket-text (no document at all) are processed by the LLM from their description without any PDF stage at all.
- **No redundant `recap-documents` calls.** The `/docket-entries/` endpoint already returns each recap_document inline with `is_available`, `is_sealed`, `plain_text`, `filepath_local`, `filepath_ia`. We use that inline copy directly rather than refetching `/recap-documents/{id}/`, which would be one wasted call per document on first sync.
- **Court-local timezones, computed once.** Hearings are stored in UTC but ICS / Google events emit the courthouse's local time tagged with the court's IANA `TZID`. Calendar viewers display in their own tz while the event preserves "this is a 3 PM Pacific hearing" semantics across DST transitions and travel. We rely on the receiver's bundled IANA tz database for DST resolution and ship the Python `tzdata` package to keep our own `zoneinfo` lookups consistent across hosts. The court → tz table in `courts.py` is hand-maintained; CourtListener doesn't surface this.
- **Sticky timezone on hearing updates.** `timezone` is set on first insert and preserved across reschedules / detail updates. A hearing happens in one courthouse, even when sibling dockets in different timezones reference it — without this, a cand entry referencing a cadc oral argument would clobber the tz to PT and shift the displayed wall-clock time.
- **Multi-docket cases collapsed into one logical case.** Some cases have multiple docket numbers (e.g. district + appellate, or related defendants). The `dockets:` list in `config.yaml` aggregates them under one `case_id`, so hearings from any docket land in the same calendar.
- **ICS first, third-party calendars opt-in.** ICS works with anything that subscribes to a URL (Proton, Apple, etc.), needs no OAuth, and is always written. Google Calendar push requires `google_credentials_path` in the config and a staged token cache; Microsoft 365 / Outlook push requires an Entra app registration's `m365_client_id` (config or env) and a staged auth record. Both auto-enable in `emit_calendars` whenever the per-backend token cache file is present — no CLI flag — so `sync` / `serve` / `emit` push uniformly. Both can be on simultaneously; `emit_calendars` initializes each backend at most once per call and reuses it across calendars.
- **Webhook URL host is independent of `public_base_url`.** `public_base_url` is for the public ICS hosting (Caddy → `out/` directory); the webhook receiver typically lives on a different host (the shipped Caddyfile uses `calendars.example.com` for feeds and `webhook.example.com` for the receiver). `case-calendar webhook-url --host <host>` takes the webhook host as a flag rather than reading `public_base_url`, because a legitimate deployment is "run the webhook receiver privately, don't expose any public ICS feeds" — and that user has no `public_base_url` to read. Don't fold the two surfaces into a shared config key; either add a separate optional key with `--host` as a fallback, or leave the flag as the source of truth.
- **All input data is treated as untrusted.** Docket entries and PDF text can include arbitrary user-submitted content; the system prompt explicitly tells the model not to follow any instructions found inside.
- **Notifications use attendees, not email reminders.** Google Calendar's email reminder field only fires for the calendar owner; we add `notify_emails` as event attendees instead, so arbitrary addresses get the invitation + their own reminders. The `reminders` config emits per-event overrides (popup for the owner, plus an email VALARM rendered into the ICS for clients that support it).
- **Privacy-aware notification config.** `notify_emails` is exposed in the public ICS/gcal feed via `ATTENDEE` lines, so we document this trap in `config.example.yaml`: don't list addresses you don't want public. For public calendars, `reminders: [{method: popup, ...}]` works without leaking — popup fires in the subscriber's own calendar app from a `VALARM:DISPLAY` block.
- **Polling and webhook share one path.** Both `case-calendar sync` (polling) and `case-calendar serve` (webhook) call `CaseSyncer.process_entry`, so a hearing extracted via webhook is byte-identical to one extracted via polling. Webhook payloads include the docket entry with `recap_documents` inline (per the CL docs), so the only CL API call a webhook triggers is one `/dockets/{id}/` lookup the first time we see a new docket — to learn its `court_id` (timezone) and citation. After that, zero CL calls per webhook delivery; the LLM is the only outbound dependency.
- **Sync and serve are separate processes sharing one SQLite file, made safe with WAL + busy_timeout.** Polling (`case-calendar sync`, typically run via cron) and the webhook receiver (`case-calendar serve`, typically a systemd unit) usually run as two distinct OS processes pointed at the same `data/case-calendar.sqlite`. SQLite's default rollback-journal mode + `busy_timeout=0` would make the second writer raise `sqlite3.OperationalError: database is locked` *immediately* on any commit overlap — a webhook landing mid-sync would bubble up to a server-side 500 (CL retries via `Idempotency-Key`, so no data loss but transient errors in the log), and a sync that loses a race would abort the whole invocation. `Store.__init__` therefore opens every connection with `PRAGMA journal_mode=WAL` (lets multiple readers and one writer coexist; main writes go append-only to a `<db>-wal` sidecar and get checkpointed back into the main file later) and `PRAGMA busy_timeout=5000` (when a writer does hit a contending lock, block-and-retry for up to 5 seconds before raising — long enough to cover any of our actual transactions, all of which are single-row UPSERTs). The webhook server's in-process `_lock` (`WebhookServer.process_locked`) still serializes worker threads *within* the serve process; WAL + `busy_timeout` close the gap *between* processes. Hot backups must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main file, or run `PRAGMA wal_checkpoint(TRUNCATE)` first to consolidate everything back into the main file before copying.
- **Significance gate at render, not at extraction.** Every hearing is stored in the DB regardless of importance — that preserves the audit trail and lets the LLM use procedural events as context (`known_hearings` plus the new RELATED DOCKET ENTRIES block). Calendar visibility is filtered at render: the LLM tags each action with `significance='major' | 'minor'`, and the ICS/gcal renderers skip minor rows. Classification is by the proceeding's PURPOSE, not its EFFECT: final and CIPA pretrials and any pretrial/status conference whose agenda is a substantive motion are major; initial / Rule 16 / scheduling-only pretrials and phone calls whose only purpose is to rule on a Motion to Continue / Extend are minor — any rescheduling that comes out of those calls shows up on the trial / hearing row itself.
- **Past-date alone is not evidence a hearing occurred.** The old `_auto_mark_held_stale` sweep flipped any past 'scheduled' row to 'held' on the assumption that "date passed → it happened". Wrong for trials — they get continued or vacated by plea agreement without an explicit cancellation entry, the calendar date passes, and the row was falsely flipped to 'held' (see the us-v-moucka regression where the 4/13 trial was marked 'held' even though the docket had no minute entry, verdict form, transcript, or judgment-after-trial). Replaced by `_verify_scheduled_hearings` extended to past-dated rows: the verify-pass LLM only returns MARK_HELD when it can cite a minute entry / verdict / transcript / judgment-after as evidence. Past rows the docket doesn't confirm stay 'scheduled' — accurately communicating "outcome not confirmed" rather than asserting 'held' on guesswork. The summary LLM's existing trial-vs-plea invariant ("a trial DATE in a scheduling order is NOT proof a trial OCCURRED") matches this signal: status='scheduled' on a past trial row is the calendar's way of saying the same thing. Next sync re-verifies; once more entries land confirming the outcome, the row transitions to its terminal status.
- **Same-docket same-slot dedup sweep.** The per-entry extractor sees `known_hearings` and is told not to allocate a new `hearing_key` for a slot another hearing already occupies on the same docket — but a court order often describes a hearing using different vocabulary than the stipulation that proposed it ("Motion Hearing" in the signed order vs "Hearing on Motion for Summary Judgment" in the joint stipulation), and the LLM can fail to recognize the same event under the new label, allocating a fresh key with a `-N` suffix. The verify pass operates on one hearing at a time and has no view of sibling future hearings on the same docket, so it can't catch the duplicate either (it'll happily CONFIRM both rows independently). `_dedupe_concurrent_hearings` closes that gap: at end of `sync_case`, after `_verify_scheduled_hearings`, it queries `Store.find_concurrent_hearing_clusters` for any future `scheduled` rows sharing `(docket_id, starts_at_utc)` and sends each cluster to `llm.resolve_duplicate_hearings`. MERGE_INTO picks a target_key and the syncer cancels the other rows with a `[dedupe]` note pointing at the target, folding their `source_entry_ids` onto the target so the audit trail stays intact. KEEP_BOTH and UNCLEAR are no-ops. Past hearings are excluded — those are governed by the verify pass directly. Zero LLM cost on quiet syncs (no clusters → no LLM call); one extra call per duplicate cluster when one exists.
- **Per-hearing confidence pass (past + future, both states).** Every `scheduled` hearing — past OR future — gets audited by a focused LLM call (`verify_hearing` in `llm.py`), and every PAST `cancelled` hearing does too. The model receives just the candidate hearing + the last 15 hearing-relevant entries on its docket and returns one of: CONFIRM (no-op), RESCHEDULE, CANCEL, MARK_HELD, REINSTATE (cancelled-row only: revert to scheduled when the cancellation isn't docket-supported), DELETE_HALLUCINATION, or UNCLEAR (no-op, log only). The state coverage is asymmetric on purpose — future `cancelled` rows are NOT verified because a deliberately cancelled future hearing should stay cancelled until something actively un-cancels it. For past-dated rows the prompt is explicit on both sides: MARK_HELD requires a minute entry / verdict / transcript / judgment-after as evidence, not just an elapsed date (UNCLEAR on a past `scheduled` row leaves it as 'scheduled' — the correct "outcome not confirmed" communication); REINSTATE requires explicit positive signal that the cancellation is wrong (no vacatur entry AND docket activity contradicts cancellation), not just absence of evidence (UNCLEAR on a past `cancelled` row leaves it as 'cancelled' — conservative bias). The next sync re-verifies once more entries land. This pass catches missed reschedules / cancellations that the per-entry extractor didn't recognize, kills the hallucination class, prevents the trial-vacated-by-plea false-'held' that the date-only auto-held sweep used to produce, AND catches the inverse-McGonigal failure mode where a live trial got falsely marked cancelled and the case continued past the date. Cost is one LLM call per non-terminal hearing per sync; with the small/fast tier this is well under a cent per case per day even on dockets with many past rows.
- **CANCEL/MARK_HELD on unknown keys with a date insert directly.** When the LLM emits a CANCEL or MARK_HELD targeting a `hearing_key` that isn't in the store — common when a memo endorsement adjourns a hearing whose original scheduling entry was filtered out — and the action carries a `local_date`, the syncer creates a new row directly in the terminal status (`cancelled` or `held`). Without this fallback, those entries silently no-oped and the audit trail lost the proceeding. Without `local_date` we drop the action and log a warning so the operator can investigate. The LLM prompt now requires `local_date` on both CANCEL and MARK_HELD specifically to support this path.
- **MARK_HELD date-proximity validation.** When the LLM emits a MARK_HELD action carrying a `local_date`, the syncer compares it against the existing hearing's `starts_at_utc`. If they differ by more than 2 days the action is rejected and logged — the LLM probably matched the minute entry onto the wrong logical hearing (e.g. shoehorning a 3/6 CIPA hearing onto a 3/8 status conference). The real hearing's row gets caught by a properly-keyed MARK_HELD on a subsequent entry, or stays 'scheduled' if no such evidence ever materializes.
- **Cross-entry context for the LLM.** Each LLM call gets two bonus context streams alongside the entry being processed: explicit cross-references parsed from the entry text (`granting 65 Motion ...` → entry 65) and the last 5 hearing-relevant entries on the same docket (temporal proximity). This is how `[Telephonic Pretrial Conference]` orders that don't say "CIPA" get correctly titled `Telephonic Pretrial Conference (CIPA)` — the originating CIPA motion lives a few entries earlier and the LLM picks up the framing. To make this work without burning storage on irrelevant entries, the `entries.description` column is populated only for hearing-relevant entries; filter-failed entries (notices, attorney appearances, etc.) keep a fingerprint stub but no text body.
- **Per-entry known-events context is scoped to the current docket's court, not the case.** Aggregated multi-docket cases come in two shapes: (a) co-defendants on separate dockets in the *same* court — really one logical proceeding, contexts should aggregate; (b) parallel actions in *different* courts (e.g. district + circuit appeal + sibling district under a different statute) — separate proceedings, contexts must NOT bleed. Without this scope, a "stay appellate proceedings" order on docket B in court Y would surface court X's hearings and deadlines in the LLM context and trigger CANCEL actions against them. `Store.get_hearings_in_court` / `get_deadlines_in_court` do the LEFT JOIN against `dockets.court_id` so the per-entry extractor only sees same-court siblings. Rows with NULL docket_id or NULL court_id are included so a warm-up sync doesn't silently drop context. The end-of-sync verify pass is naturally docket-scoped (it reads `get_recent_relevant_entries(docket_id, …)`) and didn't need a fix.
- **Iterate oldest-first within a sync.** The CL API returns entries newest-first (which gives us a cheap `modified_after` early-stop), but cross-references resolve cleanly only when an entry's referenced motion is already in the local store. So `iter_entries` buffers a docket's worth of new entries and yields them sorted ascending by `date_modified`. Memory cost is bounded by `max_pages * page_size`; even a first-time sync of a busy docket is well under 10 MB.
- **No artificial cap on Retry-After.** CourtListener's free-tier daily bucket (300/day for authenticated, less for anonymous) can return Retry-After near 24h once exhausted. We honor it rather than abort, so a long-running sync resumes itself across the daily reset without manual restarts. We still log the URL, body, and rate-limit headers on every 429 so it's obvious which bucket fired (per-minute, per-hour, per-day) and whether requests are landing as anonymous (a CL-flagged misconfiguration symptom).
- **Filing deadlines tracked in parallel to hearings, in the same store.** The `deadlines` table mirrors `hearings` (stable kebab-case keys, `significance` major/minor, `gcal_event_id` for idempotent push, `source_entry_ids` for audit), and `cli._deadline_to_hearing` maps a deadline row into a hearing-shaped dict so ICS and gcal renderers stay deadline-agnostic. The hearing_key namespace is shared at the LLM layer (the addendum prompt explicitly tells the model to use a separate `deadline_key` field), and the renderers prefix the gcal/ICS UID with `deadline:` so a hearing and a deadline can never collide.
- **Deadlines without an explicit time fire end-of-business in court tz.** Most scheduling orders set a date but no clock time ("response due by 5/24"). The renderer fills in 5pm court-local time so the calendar pops a useful end-of-day reminder; the rendered duration is short (15 min). When the order DOES specify a time ("must be filed by 9 AM"), the LLM passes it through verbatim and the renderer uses it as-is.
- **Docket-aware auto-detect for deadline tracking.** `extract_deadlines` defaults to False on `CaseConfig`, but the syncer's `resolve_extract_deadlines(case, docket_id=None)` looks at the docket number when the static value is False: routine criminal dockets (`-cr-` / `-cm-` / `-po-` / `-mj-cr-`) stay off, civil / appellate / specialty default on. The static True is a force-on override for serious criminal cases where the briefing cadence IS what's being watched — pretrial motion practice (suppression / motions in limine / Daubert) and the run-up to sentencing (sentencing-memo deadlines, PSR objections). The end-of-case verify pass and auto-passed sweep gate on the same resolver, so a case that's "off" never accrues deadline-related work.
- **Auto-emit closes the loop.** Both `cmd_sync` (after each `sync_case`) and the webhook handler (after each `_handle_docket_alert`) call `cli.emit_calendars(cfg, store, only_calendars=...)` scoped to the calendars whose rows actually changed. ICS write is microseconds and idempotent — there's no separate flush step. gcal and M365 push auto-run inside `emit_calendars` whenever the per-backend token cache file exists; there's no CLI flag for it on `sync` / `serve` / `emit`, so the operator's workflow is just "run `setup gcal` / `setup m365` once, then everything pushes". The one-time `setup` subcommand is its own command precisely because its OAuth browser flow can't happen headless. Emit failures inside the webhook handler are logged but never fail the ack — the store is already updated, and a CL retry would just re-process.
- **Case summaries are documents-only — never inference.** The case-summary LLM (`llm.generate_docket_summary`) must state ONLY what the source documents say. Do not add prompt rules that ask the model to infer status from the *absence* of records — e.g., do not nudge it to label defendants "fugitives" merely because no appearance / arraignment is in the docket. Conservative refusal to assert facts the documents don't support is the correct behavior, not a defect. This applies to every future revision of `SUMMARY_SYSTEM_PROMPT`.
- **Case-summary prompt invariants — trial-vs-plea, sentence specifics, dated references, past-but-unconfirmed rows.** Four prompt rules in `SUMMARY_SYSTEM_PROMPT` are load-bearing and must survive future edits:
  - A trial DATE in a scheduling order (or the `starts_at_utc` of a "Jury Trial" row in the structured-events scaffold) is NOT proof a trial occurred. A guilty plea entered before the trial date moots the trial — Fed. R. Crim. P. 11 doesn't itself use "vacate" and courts vary on whether they enter a formal vacatur order or just take the date off-calendar. The model may say "tried before a jury" / "a jury trial was held" only when a verdict form or judgment-after-trial document is in the disposition set. Without that, it must state the plea/dismissal/etc. and stop — without claiming a trial occurred.
  - When a judgment / sentencing-judgment document is provided, the actual imposed sentence (term of imprisonment, supervised release, fine, restitution) is the most important fact about the case and MUST appear in the summary. Vague phrasing like "was sentenced" is wrong when the figures are in the document. When the judgment is unavailable and only a notice of sentencing or a plea agreement's recommendation exists, the model must say "was sentenced on [date]" without speculating about terms — the plea recommendation is not the imposed sentence.
  - If you mention a hearing or deadline by referring to its schedule, STATE THE DATE. Vague phrases like "a trial date set", "a hearing is scheduled", "a final pretrial conference scheduled", "with a hearing pending" are forbidden — they let stale or unverified scheduling slip past a subscriber unnoticed. The structured-events scaffold provides `starts_at_utc` / `due_at_utc` on every row; quote them. If the scaffold lacks a date (null), describe the trigger in the court's own language or omit the reference entirely — never substitute the absence of a date with vague wording.
  - Past-dated `status=scheduled` rows must be described as past-but-unconfirmed, NOT as upcoming. A row whose date has elapsed without the verify-pass LLM being able to MARK_HELD it (no minute entry / verdict / transcript / judgment-after on the docket) AND without an explicit vacatur entry means the public docket is silent on the outcome. The model must NOT speculate about the cause (a sealed order, a missed minute entry, an abandoned case) — the docket alone doesn't say. The honest framing states the original date AND the unconfirmed status without inventing a reason: "a trial was originally set for June 12, 2024; no public docket entry confirms either the proceeding or its vacatur, and the case has continued actively since." The McGonigal/Shestakov regression — summary said "a final pretrial conference scheduled and a trial date set" without dates after the trial-mcgonigal and FPC rows were correctly reverted from cancelled/held to scheduled, because the model hedged on stale-but-not-confirmed-wrong dates — is the canonical case.
- **Broad `is_disposition` for stale-flagging; narrow `_is_disposition_document` for the LLM doc set.** Two predicates in `summary.py` look alike but serve different jobs. `is_disposition` is intentionally broad: it matches a "Motion for Preliminary Injunction" because the motion's outcome is a disposition-class signal worth refreshing the case summary on. `_is_disposition_document` (used inside `find_operative_documents` only) is strict: it rejects motions, briefs, status reports, case-schedule orders (`ORDER RE 149 STIPULATION ... Motion for Summary Judgment due ...`), continuances, scheduling orders (`ORDER SETTING SENTENCING HEARING`), and minute entries of motion hearings (`Minute Entry ... Motion Hearing re: 6 Motion for Preliminary Injunction held ...`) — entries that mention disposition vocabulary in passing but are not themselves the ruling document. The split exists because feeding a 200-page PI brief or a procedural minute-entry to the summary LLM as "the disposition" produces summaries that describe the wrong posture: the original bug had `case_summaries` describing a granted preliminary injunction as "taken under submission" forever, because the only "disposition" the LLM saw was the motion text. Keep the two predicates separately tested — `is_disposition` should still flag motion practice; `_is_disposition_document` should only accept actual orders / judgments / minute-entries-of-decision.
- **Paperless dispositions use the docket text as the document text.** "Electronic Clerk's Notes for proceedings held … Court imposes sentence: 92 months imprisonment, 3 years Supervised Release; $200 Special Assessment" is a real sentencing record — but it's a minute entry with the figures inline in the docket text and no PDF attached. `_attach_text` takes `allow_description_fallback`, enabled for the disposition slot only, so the description body reaches the LLM when no PDF extracts. Operative pleadings deliberately do NOT get this fallback — an indictment minute-entry stub is not an acceptable substitute for the real charging document, and the asymmetry is locked in by tests.
- **Case summaries auto-refresh on docket changes.** `case_summaries.enabled: true` in the top-level config is the gate. The `case_summaries` table carries `stale` and `stale_since` columns per row; `sync.process_entry` flips `stale=1` and bumps `stale_since` to NOW whenever the entry it just saw matches `summary.is_operative_pleading` or `summary.is_disposition` (a superseding indictment, amended complaint, judgment, plea agreement, verdict, dismissal, dispositive memo). Two refresh paths:
  - `cmd_sync` (polling) calls `summary.refresh_stale` immediately at the end of the sync — polling is already batched into a single pass, so there's nothing to debounce, and the operator running `sync` wants results now.
  - `cmd_serve` (webhook) debounces: each delivery's `emit_fn` calls `_arm_debounce(only_calendars)`, which checks whether any docket on the touched calendars has `stale=1` and, if so, (re)starts a `threading.Timer` for `case_summaries.debounce_seconds` (default 300). Every subsequent delivery within the window cancels the existing timer and starts a new one, so a PACER batch upload that fires 5 webhooks across 90 seconds produces ONE Sonnet call, not 5. When the timer fires, it runs `refresh_stale` against the pending-calendar set and re-emits those calendars to update the index with the new prose. The immediate `emit_calendars` call still fires on each delivery for fast ICS / gcal / M365 push — only the expensive Sonnet step is debounced.
  Missing rows count as stale, so adding a case to the config seeds its summary on the next sync without a separate command. `case-calendar summarize --force` remains as an explicit per-case regeneration path; `case-calendar sync --force-summaries` is the equivalent for the polling flow — both bypass the stale check and regenerate every summary on the configured cases, useful after a prompt or model upgrade. Neither is required for the agentic flow.
- **Summaries use a higher model tier than the extractor.** Extraction is high-volume, short-context, classification-heavy — Haiku/Flash Lite/gpt-5.4-nano fit. Summaries are low-volume (one call per docket, rarely re-run), long-context (30-100K tokens of legal prose per docket), and synthesis-heavy with legal-terminology nuance — Sonnet/GPT-5.4/Gemini Pro fit. The two tracks are decoupled via separate env vars (`LLM_SUMMARY_PROVIDER` / `LLM_SUMMARY_MODEL`) and separate config keys (`case_summaries.provider` / `.model`) so neither track contaminates the other when defaults move.
- **Per-docket summaries, not per-case.** Multi-docket aggregated cases (Anthropic v. Dow) get one summary per constituent docket, rendered as a labeled paragraph block in the index. Operative pleadings live on individual dockets, so this is the natural unit; a judgment landing in one docket only dirties that docket's summary, not its siblings'.
- **Static legal disclaimers, never from the model.** "Case descriptions are generated by AI and may contain mistakes" and "Criminal defendants are presumed innocent unless and until convicted in a court of law" are rendered as a static `<footer>` block by the page template, NOT by the LLM. Keeping the legally-loaded text outside the synthesis step guarantees the language is stable regardless of model output or prompt revision. The prompt explicitly tells the model NOT to include these — they're added by the renderer.
- **`extra_documents` works around CourtListener data gaps without forging RECAP entries.** Some documents the public should be able to see don't surface through CL even when they should, and the gaps come from different upstream layers. The canonical case is the Zewei docket (S.D. Tex. 4:23-cr-00523), which has two unrelated examples on a single docket: (a) entries 1-4 — including the indictment — remain sealed in the public record at the courthouse even though the seal was lifted in connection with extradition; this is an upstream sealing decision on the court's side, NOT a CL data bug, and the workaround is `extra_documents` pointing at the DoJ press-release attachment that carries the indictment. (b) Entry 20's PDF (the agreed motion to designate the case as complex) sits in CL's storage bucket but the v4 API reports `is_available: false` because the file was uploaded under an older `pacer_case_id` (`1940635`) than the docket's current one (`1940634`) and CL's reconciler matches on the current id only ([CL bug #7345](https://github.com/freelawproject/courtlistener/issues/7345)); the workaround is `extra_documents` pointing at the `storage.courtlistener.com/recap/...` URL directly. Three options were considered: (1) upload a second copy to RECAP as a workaround — muddies the upstream bug report and writes under-seal stamps into the public RECAP record; (2) auto-search out-of-band sources (DoJ press releases etc.) — brittle discovery, burns LLM tokens on every summary regen, and "we found a PDF on the internet" is a much weaker provenance claim than "we read this from PACER"; (3) `extra_documents` — a per-case YAML list on `CaseConfig` where each entry has just three fields: `docket` (must be one of the case's tracked docket ids), `url`, and a required natural-language `note` describing what the document is and why it was added. There is deliberately no `role` field: real out-of-band documents don't always slot cleanly into "pleading" / "disposition", the operator's note already carries the meaning a role taxonomy would, and the note is what the summary LLM actually reads. `summary._fetch_extra_documents` pulls the bytes via `pdf.extract_text_from_url` (same pypdf → OCR fallback chain as CL recap_docs) and returns a single flat list that rides into the LLM prompt as its own `EXTRA DOCUMENTS PROVIDED BY OPERATOR` section, separate from the CL-walk's operative-pleading and disposition slots. Each entry's LLM block carries the `OPERATOR-PROVIDED DOCUMENT (sourced outside CourtListener)` header and a `NOTE FROM OPERATOR` line; the summary system prompt is explicit that the note is trusted metadata describing the document's identity ("treat this as the unsealed indictment despite the SEALED stamps"), while the document TEXT is still untrusted in the same way as CL/PACER text — matching the existing AGGREGATION NOTE convention. The `note:` field on each entry is *data fed to the summary LLM*, not operator scratchpad: keep it to one short sentence that identifies the document by name + case citation, and stop there. Bug numbers, `pacer_case_id` workaround details, "remove this entry once X ships" — all of that belongs in the YAML `#` comment block above the case in `config.yaml` (operator-facing) rather than in `note:`, because the LLM is summarizing a federal court case for public subscribers and any mention of CourtListener internals or tooling state in its output would be both off-topic and a leak of internal context. The content-gate that returns `None` when no operative-pleading text is available is extras-aware: extras alone satisfy it, so a docket whose only document is an operator-supplied URL still gets summarized (the canonical Zewei flow). No automatic dedup against CL: the operator is expected to remove the `extra_documents` entry once CL fixes the data gap. Misconfigured entries (missing field, docket id not on the case, note not a non-empty string) fail loud via `SystemExit` at config-load time so operator typos surface at boot rather than as silently-skipped documents.
- **M365 idempotency without server-side determinism.** Google lets us pick the event id; Graph doesn't. To stay idempotent across reschedules, DB restores, and manual calendar edits, M365 push uses two cooperating mechanisms: (1) cache the server-assigned id in `m365_event_id` so the common path is one PATCH per push; (2) on cache miss / 404, recover via `$filter` against a stable `CaseCalendarKey` single-value extended property keyed to `sha1(case_id::hearing_key)` that we set on every event we create. Plus `transactionId` on the create body so a retried POST from a webhook storm dedupes within Graph itself. The result is the same end-state as the gcal flow: one logical hearing → one calendar event, surviving reschedules, restores, and at-least-once delivery.

## One-shot maintenance scripts (`scripts/`)

- **`scripts/reprocess_entries.py <entry_id>...`** — re-run the LLM pipeline against specific stored entries (by CL `entry_id`, not docket position). Builds a synthetic entry from the local store, clears the entry's fingerprint to bypass dedup, and calls `CaseSyncer.process_entry`. Useful after prompt changes when you want to see the new behavior on existing rows. CAUTION: the LLM can allocate fresh `hearing_key`s if it interprets a previously-vague row as multiple specific events — duplicates may need manual cleanup.
- **`scripts/classify_significance.py`** — classifies hearing significance using a focused single-question LLM prompt (no full action extraction, so no risk of the duplicate-key issue above). Default is dry-run on NULL-significance rows; `--all` widens to every row, `--apply` writes verdicts back to the DB.

## Testing

```bash
uv sync --extra test
uv run pytest                                    # full suite (~25s, ~600 tests)

uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing
```

### Testing philosophy

- **Every change ships with the test that proves it.** Adding code → add the test that exercises the new branch. Fixing a bug → add the test that fails on the old code and passes on the new (so the regression can't sneak back). Changing behavior → update the tests that asserted the old behavior. Deleting code → delete the tests that covered it. The diff that lands a behavior change without a test change is the diff that breaks silently three months from now. CI runs the full suite with `--cov-branch` on every push to `main` and every pull request and fails the build under 90% project coverage (see [`.github/workflows/ci.yml`](.github/workflows/ci.yml)); 90% is a regression floor, not the bar to clear — the local invariant is stronger: no commit should reduce coverage at the module level. Run `uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing` before declaring a change done and confirm the touched modules either held or gained coverage.
- **Hermetic by construction.** No real HTTP, no real LLM, no real Google / Microsoft Graph, no real keyring. Every external dependency has a stub in `tests/conftest.py` or is monkey-patched per test. The `_no_real_token` autouse fixture strips any inherited `*_API_KEY` env vars so a developer with credentials in their shell can't accidentally hit a real provider.
- **Per-provider SDK calls are mocked at the SDK boundary, not the network boundary.** `tests/test_llm.py` injects fake `anthropic` / `openai` / `google.genai` modules into `sys.modules`; `tests/test_m365.py` bypasses `M365CalendarSync.__init__` via `__new__` and feeds a `MagicMock` Graph client with `AsyncMock`-backed `post` / `get` / `patch` / `delete`. This keeps tests fast and lets us assert on the call shape (model kwargs, JSON-mode toggles, extended-property writes) without needing real SDK fixtures.
- **Branch coverage, not just line coverage.** Run with `--cov-branch`. The suite holds at ≥96% branch coverage and every module is at ≥90%. New code should land with the test that drove it; the local invariant is "if a branch can fire, exercise it."
- **Cover error paths, not just happy paths.** For each external integration (HTTP 4xx/5xx, 404 on patch, malformed JSON, missing required fields, OAuth token absent), test that the code falls open or fails fast as documented in [Key Design Decisions](#key-design-decisions) rather than crashing. The 5xx-fail-open semantics, the `transaction_id` retry-dedup, and the silent webhook ack on emit failure are all enforced by tests.
- **Webhook integration tests boot a real server.** `tests/test_serve.py` starts an actual `ThreadingHTTPServer` on an ephemeral port and POSTs JSON to it. This catches routing / header / body-parse regressions that an HTTP-stubbed test would miss (411 on missing Content-Length, 413 on oversized payloads, 500 from inside the process-locked critical section).
- **Test files mirror the modules they cover** (`tests/test_store.py` ↔ `case_calendar/store.py`, etc.) so a coverage gap is one file away from its fix. Cross-module integration lives in `tests/test_sync_integration.py` and `tests/test_serve.py`.
- **Test the behavior, not the implementation.** Tests assert on what the renderer writes / what the store persists / what the LLM-shaped call kwargs are, not on which private helper was called. Refactors that preserve behavior shouldn't need test edits; refactors that change behavior should.
- **Unreachable defensive code is a test smell.** If a branch can't be exercised, either delete it or fix the upstream to make it reachable — don't paper over with `# pragma: no cover`. The `extractor._entry_text` filter-empty-blobs change and the `m365._to_local_dt` deletion both came from the coverage report surfacing dead code.
- **No real `sleep` in tests.** Where the production path uses `time.sleep` (CourtListener retry/backoff) or `threading.Timer` (webhook debounce), the tests monkey-patch the sleep / Timer so the test runs at memory speed. Burning wall-clock seconds during pytest is a defect.
- **Cross-check summaries against the calendar tables as an audit pass.** The unit + integration suite catches regressions within each layer, but the LLM-driven layers (extractor, verify pass, summary) can drift in ways no unit test foresees — the model can store a hearing on the wrong key, estimate a date the order didn't fix, or fail to mark a continuance — and each of those bugs is consistent with its layer's contracts in isolation. Periodically dump every case_summaries row alongside its case's hearings + deadlines and read the prose against the structured rows: dates mentioned in the summary should map to existing rows, status flags should agree with how the prose characterizes the proceeding, and every date in the prose should be one the COURT set (not an LLM estimate of an unknown future). This audit surfaced the duplicate same-slot hearing, the conditional deadline that was being stored as a guessed calendar date, and the trial row falsely flipped to 'held' by the date-only auto-held sweep — three different bugs in three different layers, all visible only when you put the prose and the calendar side-by-side. Run this any time the LLM stack is touched (prompt revisions, model upgrades, provider switch).

The suite is hermetic — no real network calls, no real LLM, no real Google API. `tests/conftest.py` provides:
