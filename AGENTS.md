# Agents

## Project Overview

case-calendar ŌĆö a CLI / webhook service that pulls federal court hearing dates AND filing deadlines from CourtListener / RECAP and emits them to ICS feeds (subscribable from Proton, Apple, etc.) and optionally pushes them to Google Calendar and/or Microsoft 365 / Outlook. Built for tracking cases where docket-watching by hand is too much: cybercrime prosecutions (e.g. DPRK IT-worker fraud), multi-docket tech litigation (e.g. Anthropic v. DOW), and similar. Designed to run agentically ŌĆö `case-calendar serve` accepts CL webhooks, processes each delivery, and re-renders affected calendars on the spot, so subscribers see updates within seconds of a filing without any manual command run.

## Architecture

- **case_calendar/cli.py** ŌĆö CLI entry point. Subcommands: `sync` (pull updates from CL into the store + auto-emit affected calendars), `serve` (run the webhook receiver + auto-emit affected calendars per delivery), `emit` (write ICS files and optionally push to Google Calendar ŌĆö used standalone for forced re-renders or first-run gcal OAuth), `show` (dump current hearings + deadlines). Loads `.env` via `python-dotenv` before any module reads env vars. Exposes `emit_calendars(cfg, store, *, only_calendars=None, push_gcal=False)` as the shared rendering primitive ŌĆö both `cmd_sync` and the webhook server's `emit_fn` callback go through it, so a single update flows from CL to ICS subscribers without a manual step. Also owns the deadline-to-hearing adapter (`_deadline_to_hearing` maps the deadline row into a hearing-shaped dict the renderers accept) and `_compose_title` (the title prefixer that adds `[HEARING]` / `[DEADLINE]` / `[time TBD]` / `[time unknown]` flags).
- **case_calendar/courtlistener.py** ŌĆö REST v4 client. Wraps `dockets/`, `docket-entries/`, `recap-documents/` endpoints with backoff for 429 (`Retry-After`) and 5xx responses. Honors any `Retry-After` value (the daily 300/day bucket can return a wait near 24h; we sleep through rather than abort) and logs URL + body + rate-limit headers on every 429 so it's obvious which bucket fired. Iterator-style pagination via `iter_entries`, with a `modified_after` cutoff for cheap incremental syncs; pages newest-first (preserves the cheap stop-condition) but buffers and yields oldest-first so cross-entry references resolve cleanly within a single sync.
- **case_calendar/courts.py** ŌĆö Court-id ŌåÆ IANA timezone mapping. CourtListener doesn't expose timezones in its API, so we maintain the table here. Coverage is comprehensive: all 13 federal circuits, all 94 district courts, the Supreme Court, specialty courts (Federal Claims, CIT, FISA, Tax, Armed Forces), and territories (PR, VI, Guam, NMI, American Samoa). Multi-tz states (Tennessee, Florida, Indiana, etc.) map to the principal-office tz of each district. Unknown court IDs log a warning and fall back to ET so the gap is visible rather than silent.
- **case_calendar/extractor.py** ŌĆö Cheap regex pre-filter. `is_hearing_relevant` matches hearing-vocabulary entries; `is_deadline_relevant` matches filing-deadline vocabulary (`due by`, briefing schedule, scheduling order, stipulation, etc.); `is_extractable(entry, *, want_deadlines)` is the combined gate the syncer calls ŌĆö hearing-relevant entries always pass, deadline-relevant entries pass only when the case opts into deadline tracking. Briefs, attorney appearances, and similar paperwork entries are skipped before any LLM tokens are spent. Intentionally over-inclusive ŌĆö false positives just cost an extra LLM call; false negatives lose hearings/deadlines.
- **case_calendar/llm.py** ŌĆö Provider-agnostic extractor for both hearings and filing deadlines. Selects between anthropic / openai / gemini via `LLM_PROVIDER` env (or auto-detects from whichever `*_API_KEY` is set). `LLM_MODEL` overrides the per-provider default. Defaults are the small/fast tier (Haiku, gpt-5.4-nano, Flash Lite) ŌĆö this is structured extraction, not deep reasoning. SDKs are imported lazily inside the per-provider call functions. The Anthropic path applies `cache_control: ephemeral` on the system block, so all four prompts (extract / verify_hearing / verify_deadline / extract+deadlines) get prompt-cache discounts within a sync. Four entry points:
  - `extract_actions` runs against ONE docket entry plus the case's known-hearings list (and known-deadlines list when `extract_deadlines=True`) and returns 0+ actions: hearing actions (ADD / RESCHEDULE / UPDATE_DETAILS / CANCEL / MARK_HELD / IGNORE) and, when deadlines are on, deadline actions (ADD_DEADLINE / RESCHEDULE_DEADLINE / CANCEL_DEADLINE / MARK_FILED). The system prompt is `SYSTEM_PROMPT` plus `DEADLINE_PROMPT_ADDENDUM` when deadlines are enabled ŌĆö keeping the addendum off-by-default keeps the simpler hearings-only prompt cheap on cases that don't need it.
  - `verify_hearing` runs against ONE candidate hearing plus the last 15 hearing-relevant entries on its docket and returns a single audit decision (CONFIRM / RESCHEDULE / CANCEL / MARK_HELD / DELETE_HALLUCINATION / UNCLEAR) ŌĆö used by the end-of-sync confidence pass to catch reschedules/cancellations the per-entry extractor missed and to reject rows the LLM hallucinated.
  - `verify_deadline` is the deadline analogue ŌĆö same shape, returns CONFIRM / RESCHEDULE / CANCEL / MARK_FILED / DELETE_HALLUCINATION / UNCLEAR for one pending deadline.

  Each ADD/RESCHEDULE/UPDATE_DETAILS action (hearing or deadline) carries a `significance` field (`major` / `minor`) that controls calendar visibility ŌĆö substantive proceedings (trial, sentencing, oral arg, any pretrial conference, evidentiary hearing, plea, etc.) and dispositive briefing deadlines (MTD/MSJ response/reply, sentencing memos, motion-in-limine, Daubert, suppression briefing) are major; procedural-only events (a phone call set just to rule on a Motion to Continue, routine joint status reports, proposed orders) are minor and filtered out at render time. The user message includes a "RELATED DOCKET ENTRIES" block (explicit cross-refs from the new entry's text + the last few hearing-relevant entries on the same docket) so the LLM can name a hearing correctly even when the entry that schedules it doesn't itself name the subject.
- **case_calendar/pdf.py** ŌĆö PDF text extraction with fallbacks. Order: CL `plain_text` ŌåÆ `pypdf` on the Internet Archive mirror ŌåÆ `pdftoppm` + `tesseract` OCR (optional system dependency). Sealed PDFs and not-yet-uploaded PDFs return `None` and are NOT cached ŌĆö the next sync re-fingerprints the entry and retries once the PDF arrives or gets OCR'd.
- **case_calendar/store.py** ŌĆö SQLite state. Tables: `dockets` (id + last `date_modified` + cached metadata for description rendering), `entries` (dedup of already-processed entries, keyed by `(docket_id, entry_id)` with content fingerprint, `entry_number` for cross-ref lookup, `description` / `short_description` / `recap_documents` JSON stored ONLY for entries that passed the regex pre-filter ŌĆö filter-failed entries get a fingerprint-only stub so dedup still works without storing dead-weight notice text. `recap_documents` keeps the compact projection (`id`, `document_number`, `attachment_number`, status flags, `filepath_ia`, `filepath_local`) that the description renderer turns into clickable per-doc URLs at emit time), `hearings` (per-case logical hearings keyed by `(case_id, hearing_key)` with `significance` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list), `deadlines` (per-case filing deadlines keyed by `(case_id, deadline_key)` with `due_at_utc` / `status` Ōłł {pending, met, passed, cancelled} / `significance` / `deadline_type` / `gcal_event_id` / `m365_event_id` / `docket_id` / source-entry list ŌĆö parallel structure to hearings so they share renderers), `courts` (CL court metadata, citation_string), `webhook_events` (idempotency-key dedup). The `m365_event_id` columns are populated by `set_m365_id_for_hearing` / `set_m365_id_for_deadline` after a Graph create; gcal ids are deterministic so `gcal_event_id` is currently unused at runtime. Connection is `check_same_thread=False` so the webhook server's worker threads can use the same Store instance under a server-wide lock.
- **case_calendar/sync.py** ŌĆö Per-case orchestration. For each docket: fetch entries newer than the store's high-water mark, run the regex filter, optionally pull PDFs, resolve cross-references (`granting 65 Motion ...` ŌåÆ look up entry 65 from the local store) plus the last 5 hearing-relevant entries on the docket as temporal context, hand entry + known-hearings list (+ known-deadlines list when deadlines are on) + related entries to the LLM, apply returned actions to the hearings/deadlines tables. `_needs_pdf` forces a PDF fetch on orders that grant scheduling-affecting motions (`Motion for Hearing`, `to Continue`, `for Continuance`, etc.) even when the order text inlines a date+time, since the brief description references the underlying motion only by docket position and the PDF often holds dates the description doesn't echo. Local time ŌåÆ UTC conversion happens here using the court's IANA timezone. End-of-case sweeps run in order: (1) `_verify_scheduled_hearings` ŌĆö per-hearing LLM confidence pass for every future scheduled hearing; (2) `_auto_mark_held_stale` ŌĆö flip past-dated 'scheduled' rows to 'held'; (3) when deadlines are on, `_verify_pending_deadlines` and `_auto_mark_passed_stale` do the same for pending deadlines. The `resolve_extract_deadlines(case, docket_id=None)` method decides per-call whether to extract deadlines: `case.extract_deadlines=True` is a force-on override; otherwise `_docket_implies_deadlines(docket_number)` classifies by federal docket-number prefix (`-cr-` / `-cm-` / `-po-` / `-mj-cr-` ŌåÆ off, civil/appellate/specialty ŌåÆ on, falls back to on when metadata is missing). Exposes `CaseSyncer.process_entry` so the webhook server can reuse the exact same per-entry pipeline.
- **case_calendar/serve.py** ŌĆö Webhook receiver for CourtListener `DOCKET_ALERT` events (real-time alternative to polling). Stdlib `ThreadingHTTPServer`; serializes processing with a server-wide lock so SQLite writes don't race. Auth model is the URL secret (CL has no signing mechanism), and `Idempotency-Key` headers are stored to make CL retries no-ops. The same store + entry-fingerprint dedup makes double-delivery safe even without the idempotency check. `WebhookServer` accepts an optional `emit_fn(only_calendars: set[str]) -> None` callback that the handler invokes after each successful docket-alert delivery, scoped to the calendars whose rows actually changed ŌĆö `cmd_serve` builds this callback to call `cli.emit_calendars` with the configured ICS paths and optional gcal push. Emit failures are logged but never fail the webhook ack: the store is already updated, and a CL retry would just re-process the same entries.
- **case_calendar/calendars/description.py** ŌĆö Shared event-body builder used by both ICS and gcal renderers. Composes notes ŌåÆ judge / panel ŌåÆ dial-in ŌåÆ case citation ŌåÆ docket URL ŌåÆ per-document URLs (`Documents:` block, one line per attached doc ŌĆö `65: <url>` for the main doc, `65-1: <url>` for attachment 1; `(sealed)` / `(not yet uploaded to RECAP)` replace the URL when no PDF is reachable) ŌåÆ PACER docket entry numbers (`Docket entries: 65, 82`) ŌåÆ CourtListener entry IDs. The docket-numbers line is what subscribers see in the CL UI; it sits directly above the opaque CL entry IDs so the audit trail reads docket-first. Both `Documents:` and `Docket entries:` are populated in `cli.emit_calendars` from `Store.get_entry_documents` / `Store.get_entry_numbers(source_entry_ids)`; source entries that lack a docket position (paperless minute orders) are silently dropped, and missing or empty doc lists produce no `Documents:` block at all. URL preference is IA mirror (`filepath_ia`) ŌåÆ CL storage (`https://storage.courtlistener.com/<filepath_local>`). New documents (e.g. an attachment uploaded later) land in the description automatically: the entry's fingerprint already includes per-doc availability flags, so a doc-add flips the fingerprint, `process_entry` re-runs and overwrites the stored `recap_documents` JSON, and the auto-emit (sync + webhook) re-renders the calendar. Also exports `no_time_title_prefix(starts_at_utc)` which returns `[time TBD]` for future date-only events (we expect the time to be set later) and `[time unknown]` for past ones (we never learned it). Renderers are deadline-agnostic: deadlines are mapped to hearing-shaped dicts in `cli._deadline_to_hearing` (with `hearing_key` namespaced as `deadline:<key>`) before they reach this layer.
- **case_calendar/calendars/ics.py** ŌĆö Hand-rolled RFC 5545 output. UIDs are derived from `case_id + hearing_key` so regenerations are stable across runs (and the `deadline:` prefix on mapped deadline keys keeps their UIDs in a separate namespace from real hearings). Rows with `significance='minor'` and rows with `status='cancelled'` are filtered out entirely ŌĆö the calendar tracks major case moments only, not procedural noise or events that are no longer happening. Each timed event is tagged with `DTSTART;TZID=<IANA tz>:<local time>`; receiving calendar apps resolve DST via their own bundled IANA databases (no VTIMEZONE shipped ŌĆö bare TZIDs work in Google, Apple, Proton, Outlook 2019+, Thunderbird). The title carries `[time TBD]` / `[time unknown]` only when the event is date-only; we don't add `[HELD]` (the date itself shows what's past).
- **case_calendar/calendars/gcal.py** ŌĆö Google Calendar push. Event IDs are deterministic (`sha1(case_id::hearing_key)`), so `patch` is idempotent and reschedules / detail updates land on the same event ŌĆö and the `deadline:` prefix on mapped deadline keys means a hearing and a deadline that happen to share a slug get distinct gcal events. Times are sent as `dateTime: "<local>", timeZone: "<IANA>"` so Google preserves the courthouse tz across DST and viewer-tz displays. First-run uses InstalledAppFlow OAuth; tokens cache to `~/.case-calendar/google-token.json` ŌĆö the daemon (`serve --push-gcal`) can't run InstalledAppFlow headless, so the operator must run `case-calendar emit --push-gcal` once interactively to stage the refresh token. Optional `notify_emails` and `reminders` decorate events with attendees + reminder overrides; emails are added as Google Calendar attendees (Google's email reminders only fire for the calendar owner ŌĆö attendees rely on their own client's reminder settings). `sendUpdates=externalOnly` so attendee invitations actually fire. Hearings flipped to `cancelled` get a `patch` with `{"status": "cancelled"}` against the existing remote event so subscribers stop seeing it ŌĆö but if the event was never created (404) we noop rather than insert+cancel. `significance='minor'` rows are skipped entirely.
- **case_calendar/calendars/m365.py** ŌĆö Microsoft 365 / Outlook push via Microsoft Graph. Uses the **official** `msgraph-sdk` + `azure-identity` libraries (not the third-party `O365` package). Auth is `InteractiveBrowserCredential` with `TokenCachePersistenceOptions` and a serialized `AuthenticationRecord`: first run opens a browser to grant `Calendars.ReadWrite`, the record JSON is cached at `~/.case-calendar/m365-token.json`, and the daemon thereafter constructs the credential with `disable_automatic_authentication=True` to refresh silently ŌĆö failing fast instead of trying to open a browser on a server. Graph generates event ids server-side (unlike Google's deterministic ids), so we persist the returned id on the hearings/deadlines row as `m365_event_id`; subsequent pushes patch by cached id, fall back to a `$filter` lookup on the `CaseCalendarKey` single-value extended property when the cache is stale (DB restore, manual delete) so we recover and re-attach to the existing event instead of orphaning it. `transactionId` is set on every create with the same key so retried POSTs from a webhook storm dedupe server-side. The SDK is async; `sync()` wraps `_sync_async` in `asyncio.run` so callers stay synchronous. Per-calendar config: `m365_calendar_id` routes to a specific Outlook calendar (`/me/calendars/{id}/events`); omit it (or set `m365_use_default_calendar: true`) to push to the user's default calendar (`/me/events`). Cancellation = `DELETE` (Graph already mails attendee cancellations on organizer DELETE, no separate verb). `significance='minor'` rows are skipped. Email reminders aren't representable in Graph's single `reminderMinutesBeforeStart`, so only popup reminders are honored ŌĆö the shortest popup minutes wins.

## Key Design Decisions

- **LLM-driven extraction over per-court regexes.** Courts describe hearings inconsistently (`Set/Reset Hearings` minute entries, `ELECTRONIC NOTICE OF RESCHEDULING`, embedded in PDF text, plain prose in clerk's notes). Maintaining regexes per court is a treadmill; the LLM sees the entry plus the case's known-hearing list and decides ADD vs RESCHEDULE vs UPDATE vs CANCEL in one call.
- **Stable hearing keys, not stable timestamps.** A "logical" hearing (e.g. "sentencing for Wang") may be rescheduled multiple times. We let the LLM assign a kebab-case `hearing_key` on first observation and re-use it on follow-ups, so the same calendar event is updated rather than duplicated. Google event IDs are derived from this key.
- **Regex pre-filter before LLM.** Only entries whose text matches hearing-related vocabulary are sent to the LLM. Briefs, attorney withdrawals, sealed-document placeholders, etc. are dropped for free.
- **Three-tier short-circuit** to keep API + LLM costs near zero on quiet days: (1) per-docket ŌĆö skip the entire docket if its `date_modified` hasn't advanced since the last sync (no entries API call, no LLM); (2) per-entry ŌĆö `iter_entries(modified_after=cutoff)` filters server-side to entries newer than our store's high-water mark; (3) per-fingerprint ŌĆö even if an entry comes back, dedup against `(docket_id, entry_id, content_fingerprint)` skips re-LLM-ing on entries whose substantive content didn't change. The docket high-water mark is only advanced on a clean run, so a mid-sync error retries the whole docket on the next run.
- **No separate PDF text cache.** Once a hearing is extracted from a PDF, the structured fields are in the `hearings` table and we never re-consult that PDF ŌĆö follow-up reschedules / dial-in updates arrive on new docket entries with their own PDFs. The 2 KB excerpt for description rendering lives on the `entries` row itself. The rare case of "same entry re-processed because its fingerprint changed" pays for one IA round-trip, which is cheap compared to the operational complexity of an immutable-blob cache.
- **PDF retries on availability flips.** RECAP coverage is incomplete ŌĆö many entries' PDFs get uploaded later, and CL's OCR runs asynchronously. The entry fingerprint includes each recap_doc's `is_available` / `is_sealed` / `plain_text`-presence flags, so a previously-missing PDF re-triggers processing automatically once it appears.
- **No-document detection.** Paperless orders and minute entries show up as `recap_documents` rows with `is_available: false`, no `filepath_local`, no `filepath_ia` ŌĆö there is literally no document to fetch. `_is_fetchable` rejects these up-front so we never attempt the IA download. Entries that exist purely as docket-text (no document at all) are processed by the LLM from their description without any PDF stage at all.
- **No redundant `recap-documents` calls.** The `/docket-entries/` endpoint already returns each recap_document inline with `is_available`, `is_sealed`, `plain_text`, `filepath_local`, `filepath_ia`. We use that inline copy directly rather than refetching `/recap-documents/{id}/`, which would be one wasted call per document on first sync.
- **Court-local timezones, computed once.** Hearings are stored in UTC but ICS / Google events emit the courthouse's local time tagged with the court's IANA `TZID`. Calendar viewers display in their own tz while the event preserves "this is a 3 PM Pacific hearing" semantics across DST transitions and travel. We rely on the receiver's bundled IANA tz database for DST resolution and ship the Python `tzdata` package to keep our own `zoneinfo` lookups consistent across hosts. The court ŌåÆ tz table in `courts.py` is hand-maintained; CourtListener doesn't surface this.
- **Sticky timezone on hearing updates.** `timezone` is set on first insert and preserved across reschedules / detail updates. A hearing happens in one courthouse, even when sibling dockets in different timezones reference it ŌĆö without this, a cand entry referencing a cadc oral argument would clobber the tz to PT and shift the displayed wall-clock time.
- **Multi-docket cases collapsed into one logical case.** Some cases have multiple docket numbers (e.g. district + appellate, or related defendants). The `dockets:` list in `config.yaml` aggregates them under one `case_id`, so hearings from any docket land in the same calendar.
- **ICS first, third-party calendars opt-in.** ICS works with anything that subscribes to a URL (Proton, Apple, etc.), needs no OAuth, and is always written. Google Calendar push requires `--push-gcal` and credentials in the config; Microsoft 365 / Outlook push requires `--push-m365` and an Entra app registration's client id. Both can be on simultaneously ŌĆö `emit_calendars` initializes each backend at most once per call and reuses it across calendars.
- **All input data is treated as untrusted.** Docket entries and PDF text can include arbitrary user-submitted content; the system prompt explicitly tells the model not to follow any instructions found inside.
- **Notifications use attendees, not email reminders.** Google Calendar's email reminder field only fires for the calendar owner; we add `notify_emails` as event attendees instead, so arbitrary addresses get the invitation + their own reminders. The `reminders` config emits per-event overrides (popup for the owner, plus an email VALARM rendered into the ICS for clients that support it).
- **Privacy-aware notification config.** `notify_emails` is exposed in the public ICS/gcal feed via `ATTENDEE` lines, so we document this trap in `config.example.yaml`: don't list addresses you don't want public. For public calendars, `reminders: [{method: popup, ...}]` works without leaking ŌĆö popup fires in the subscriber's own calendar app from a `VALARM:DISPLAY` block.
- **Polling and webhook share one path.** Both `case-calendar sync` (polling) and `case-calendar serve` (webhook) call `CaseSyncer.process_entry`, so a hearing extracted via webhook is byte-identical to one extracted via polling. Webhook payloads include the docket entry with `recap_documents` inline (per the CL docs), so the only CL API call a webhook triggers is one `/dockets/{id}/` lookup the first time we see a new docket ŌĆö to learn its `court_id` (timezone) and citation. After that, zero CL calls per webhook delivery; the LLM is the only outbound dependency.
- **Significance gate at render, not at extraction.** Every hearing is stored in the DB regardless of importance ŌĆö that preserves the audit trail and lets the LLM use procedural events as context (`known_hearings` plus the new RELATED DOCKET ENTRIES block). Calendar visibility is filtered at render: the LLM tags each action with `significance='major' | 'minor'`, and the ICS/gcal renderers skip minor rows. Pretrial conferences (any flavor) are always major; phone calls whose only purpose is to rule on a Motion to Continue / Extend are minor ŌĆö classified by the proceeding's PURPOSE, not its EFFECT, because any rescheduling that comes out of the call shows up on the trial / hearing row itself.
- **Self-healing status on past hearings.** At the end of every `sync_case`, any `scheduled` row whose `starts_at_utc` is in the past flips to `held`. `starts_at_utc` is already UTC and the server clock is UTC, so the comparison is timezone-free with no buffer needed: once the start is past, status='scheduled' is wrong by definition. If a late-arriving reschedule or cancellation entry lands later, it'll flip the row back to 'scheduled' or 'cancelled' on the next sync. Cancelled rows are never touched by this sweep.
- **Per-hearing confidence pass for the future.** Before the auto-held sweep, every future `scheduled` hearing gets audited by a focused LLM call (`verify_hearing` in `llm.py`). The model receives just the candidate hearing + the last 15 hearing-relevant entries on its docket and returns one of: CONFIRM (no-op), RESCHEDULE, CANCEL, MARK_HELD, DELETE_HALLUCINATION (cancel with note explaining no docket entry supports the hearing), or UNCLEAR (no-op, log only). This catches missed reschedules / cancellations that the per-entry extractor didn't recognize, and kills the hallucination class ŌĆö rows extracted from tangentially-related entries with no actual scheduling order behind them. The cost is one LLM call per future scheduled hearing per sync; with the small/fast tier this is well under a cent per case per day.
- **CANCEL/MARK_HELD on unknown keys with a date insert directly.** When the LLM emits a CANCEL or MARK_HELD targeting a `hearing_key` that isn't in the store ŌĆö common when a memo endorsement adjourns a hearing whose original scheduling entry was filtered out ŌĆö and the action carries a `local_date`, the syncer creates a new row directly in the terminal status (`cancelled` or `held`). Without this fallback, those entries silently no-oped and the audit trail lost the proceeding. Without `local_date` we drop the action and log a warning so the operator can investigate. The LLM prompt now requires `local_date` on both CANCEL and MARK_HELD specifically to support this path.
- **MARK_HELD date-proximity validation.** When the LLM emits a MARK_HELD action carrying a `local_date`, the syncer compares it against the existing hearing's `starts_at_utc`. If they differ by more than 2 days the action is rejected and logged ŌĆö the LLM probably matched the minute entry onto the wrong logical hearing (e.g. shoehorning a 3/6 CIPA hearing onto a 3/8 status conference). The real hearing's row gets caught later by the auto-held sweep if it's stale, or by a properly-keyed MARK_HELD on a subsequent entry.
- **Cross-entry context for the LLM.** Each LLM call gets two bonus context streams alongside the entry being processed: explicit cross-references parsed from the entry text (`granting 65 Motion ...` ŌåÆ entry 65) and the last 5 hearing-relevant entries on the same docket (temporal proximity). This is how `[Telephonic Pretrial Conference]` orders that don't say "CIPA" get correctly titled `Telephonic Pretrial Conference (CIPA)` ŌĆö the originating CIPA motion lives a few entries earlier and the LLM picks up the framing. To make this work without burning storage on irrelevant entries, the `entries.description` column is populated only for hearing-relevant entries; filter-failed entries (notices, attorney appearances, etc.) keep a fingerprint stub but no text body.
- **Iterate oldest-first within a sync.** The CL API returns entries newest-first (which gives us a cheap `modified_after` early-stop), but cross-references resolve cleanly only when an entry's referenced motion is already in the local store. So `iter_entries` buffers a docket's worth of new entries and yields them sorted ascending by `date_modified`. Memory cost is bounded by `max_pages * page_size`; even a first-time sync of a busy docket is well under 10 MB.
- **No artificial cap on Retry-After.** CourtListener's free-tier daily bucket (300/day for authenticated, less for anonymous) can return Retry-After near 24h once exhausted. We honor it rather than abort, so a long-running sync resumes itself across the daily reset without manual restarts. We still log the URL, body, and rate-limit headers on every 429 so it's obvious which bucket fired (per-minute, per-hour, per-day) and whether requests are landing as anonymous (a CL-flagged misconfiguration symptom).
- **Filing deadlines tracked in parallel to hearings, in the same store.** The `deadlines` table mirrors `hearings` (stable kebab-case keys, `significance` major/minor, `gcal_event_id` for idempotent push, `source_entry_ids` for audit), and `cli._deadline_to_hearing` maps a deadline row into a hearing-shaped dict so ICS and gcal renderers stay deadline-agnostic. The hearing_key namespace is shared at the LLM layer (the addendum prompt explicitly tells the model to use a separate `deadline_key` field), and the renderers prefix the gcal/ICS UID with `deadline:` so a hearing and a deadline can never collide.
- **Deadlines without an explicit time fire end-of-business in court tz.** Most scheduling orders set a date but no clock time ("response due by 5/24"). The renderer fills in 5pm court-local time so the calendar pops a useful end-of-day reminder; the rendered duration is short (15 min). When the order DOES specify a time ("must be filed by 9 AM"), the LLM passes it through verbatim and the renderer uses it as-is.
- **Docket-aware auto-detect for deadline tracking.** `extract_deadlines` defaults to False on `CaseConfig`, but the syncer's `resolve_extract_deadlines(case, docket_id=None)` looks at the docket number when the static value is False: routine criminal dockets (`-cr-` / `-cm-` / `-po-` / `-mj-cr-`) stay off, civil / appellate / specialty default on. The static True is a force-on override for serious criminal trials with substantive pretrial motion practice (suppression / motions in limine / Daubert / sentencing memos) where the briefing cadence IS what's being watched. The end-of-case verify pass and auto-passed sweep gate on the same resolver, so a case that's "off" never accrues deadline-related work.
- **Auto-emit closes the loop.** Both `cmd_sync` (after each `sync_case`) and the webhook handler (after each `_handle_docket_alert`) call `cli.emit_calendars(cfg, store, only_calendars=...)` scoped to the calendars whose rows actually changed. ICS write is microseconds and idempotent ŌĆö there's no separate flush step. gcal and M365 push are opt-in (`--push-gcal` / `--push-m365` on `sync` and `serve`) because each one's first-run OAuth flow needs a browser; the operator runs `emit --push-gcal` / `emit --push-m365` once to stage the token cache, after which the daemon refreshes non-interactively. Emit failures inside the webhook handler are logged but never fail the ack ŌĆö the store is already updated, and a CL retry would just re-process.
- **M365 idempotency without server-side determinism.** Google lets us pick the event id; Graph doesn't. To stay idempotent across reschedules, DB restores, and manual calendar edits, M365 push uses two cooperating mechanisms: (1) cache the server-assigned id in `m365_event_id` so the common path is one PATCH per push; (2) on cache miss / 404, recover via `$filter` against a stable `CaseCalendarKey` single-value extended property keyed to `sha1(case_id::hearing_key)` that we set on every event we create. Plus `transactionId` on the create body so a retried POST from a webhook storm dedupes within Graph itself. The result is the same end-state as the gcal flow: one logical hearing ŌåÆ one calendar event, surviving reschedules, restores, and at-least-once delivery.

## One-shot maintenance scripts (`scripts/`)

- **`scripts/reprocess_entries.py <entry_id>...`** ŌĆö re-run the LLM pipeline against specific stored entries (by CL `entry_id`, not docket position). Builds a synthetic entry from the local store, clears the entry's fingerprint to bypass dedup, and calls `CaseSyncer.process_entry`. Useful after prompt changes when you want to see the new behavior on existing rows. CAUTION: the LLM can allocate fresh `hearing_key`s if it interprets a previously-vague row as multiple specific events ŌĆö duplicates may need manual cleanup.
- **`scripts/classify_significance.py`** ŌĆö bulk-classifies every hearing whose `significance` column is NULL using a focused single-question LLM prompt (no full action extraction, so no risk of the duplicate-key issue above). Prints results to stdout; does NOT write to the DB. Pipe to a file for review and apply selectively.

- `tests/test_sync_integration.py` ŌĆö drives `CaseSyncer` end-to-end with `FakeCL` and a stubbed LLM extractor; covers hearing schedule ŌåÆ reschedule ŌåÆ cancel ŌåÆ mark-held flows, the parallel ADD_DEADLINE ŌåÆ RESCHEDULE_DEADLINE ŌåÆ MARK_FILED ŌåÆ auto-passed flow, the docket-level short-circuit, the entry-fingerprint dedup, docket/court metadata caching, and the docket-aware `extract_deadlines` auto-detect (criminal-off / civil-on / explicit-override).
