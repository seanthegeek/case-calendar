# Agents

## Project Overview

Case Calendar — a CLI / webhook service that pulls federal court hearing dates AND filing deadlines from CourtListener / RECAP and emits them to ICS feeds (subscribable from Proton, Apple, etc.) and optionally pushes them to Google Calendar and/or Microsoft 365 / Outlook. Built for tracking cases where docket-watching by hand is too much: cybercrime prosecutions (e.g. DPRK IT-worker fraud), multi-docket tech litigation (e.g. Anthropic v. DOW), and similar. Designed to run unattended — `case-calendar serve` accepts CourtListener webhooks, processes each delivery, and re-renders the affected ICS file on the spot. The end-to-end "when does a subscriber actually see this?" latency is bounded by signals outside our control on either side — see the [Limitations](docs/index.md#limitations) page; the project's own internal stage (webhook arrival → ICS file written) is the only step that runs in seconds.

## Conventions

These rules apply to anyone — human or agent — making changes to this repo. They are intentionally checked in (rather than living in any one agent's private scratch memory) so that every collaborator picks them up the same way.

- **Wait for explicit commit AND push permission on the default branch — these are separate grants.** Finish the implementation, run the tests, summarize the diff, then **stop and ask**. The author decides when a change is ready to land; auto-committing makes review noisier and harder to reverse. "Commit this" mid-session counts as permission for that one commit, not a standing grant — and crucially, permission to commit is NOT permission to push. Pushing publishes the change to the remote where collaborators / CI / production deploys can pick it up, and is much harder to walk back than a local commit. Wait for an explicit "push it" before `git push`. If the prior commit was itself unauthorized, do NOT push it to "tidy up" — surface the situation and let the author decide whether to keep, amend, or reset.
- **Self-test before every `git commit`:** has the author typed "commit" (or an unambiguous equivalent — "ok to commit", "commit this", "commit and push") in a present-tense imperative since your last commit? If no, **ask**. Conditional phrasings like "if everything works we can push" or "we could commit this" or "if it looks good ..." are NOT authorizations — they are plans you must confirm before acting on. Treat the literal text of the user's last message as the source of truth, not your own interpretation of where the conversation is going.
  - **Self-test before every `git push`:** has the author typed "push" since your last push? Same rule. Permission to commit is NEVER permission to push.
  - **Exception — branches you created in-session** When you have explicitly created a feature branch yourself (e.g. `git checkout -b feat/something`) in that session, commit and push to THAT branch freely without per-step permission. The entire branch is reviewed at PR-open, so the per-commit gate adds review noise without adding safety. The exception is scoped to branches Claude created in the current session; it does NOT extend to `main`, to other long-lived branches, or to branches the author created.
- **Back up the SQLite store before any schema or migration change.** Before running any schema-changing SQL (ALTER TABLE, CREATE/DROP, hand-rolled column rewrites, anything that mutates table shape) against a real store on disk, copy the DB file aside first (e.g. `cp case-calendar.sqlite case-calendar.sqlite.bak-YYYY-MM-DD-<reason>`). The store holds the full operational history (entries, hearings, deadlines, case summaries, webhook idempotency keys, source-entry provenance) — none of which is reconstructible without a multi-day CourtListener re-sync that burns the daily quota, costs LLM dollars, and still won't recover anything CourtListener has since rolled past. Migrations that look reversible on paper aren't always — a column rename + backfill can lose precision, an aborted migration mid-write can leave the DB in a shape no code path expects. The backup is the escape hatch. This applies in production AND in any dev environment whose DB the author cares about; ask if uncertain. (Tests use ephemeral tmp-path stores via the `store` fixture — those don't need backups.) **The store runs in WAL journaling mode** (see the concurrency design note below in Key Design Decisions), so a hot copy must include the `<db>-wal` and `<db>-shm` sidecar files alongside the main `.sqlite` file, OR run `PRAGMA wal_checkpoint(TRUNCATE)` first to fold the WAL contents back into the main file before copying just that one file. A bare `cp` of only the main file taken while writes are in flight will silently miss any committed-but-not-yet-checkpointed rows.
- **Project-specific rules belong in AGENTS.md, not in any agent's private memory store.** If you (Claude Code, Cursor, Codex, Aider, anything that has a "save this preference for next time" surface) catch yourself about to write down a rule that's actually about the codebase rather than about working with this particular user, write it here instead. Memory is fine for user-profile facts and tool-use preferences; project rules should be portable across agents.
- **Every behavior change ships with its test** — see [Testing philosophy](#testing-philosophy) for the full statement and the verification command.
- **Spell "CourtListener" in full everywhere.** No "CL" abbreviation, anywhere: not in user-facing documentation, not in AGENTS.md, not in source comments or docstrings, not in commit messages, not in PR descriptions, not in tests, not in identifier names. The full name reads only marginally longer than the abbreviation and removes a layer of jargon contributors and subscribers shouldn't have to translate. The only acceptable abbreviation is the lowercase `cl` parameter identifier for a `CourtListener` client object — an obvious local-binding name from context. Everything else is a defect: standalone "CourtListener" in prose, but also CamelCase suffixes like `FakeCL` (→ `FakeCourtListener`) or `_CLClient` (→ `_CourtListenerClient`). The CamelCase form slips through naive `\bCL\b` regex sweeps because lowercase→uppercase isn't a word boundary, so any bulk rename pass must also grep specifically for `CourtListener[^a-zA-Z0-9_]` and `CourtListener$` to catch identifier suffixes.
- **Do NOT claim CourtListener OCRs documents.** It doesn't. CourtListener's `plain_text` field on a `recap_document` is whatever extraction the Free Law Project's ingestion pipeline produced upstream — if that pipeline can't extract text (image-only PDFs without an embedded text layer), the field stays empty or garbled. Documents only become text-searchable when a downstream consumer (this project, via local poppler + tesseract, is one such consumer) runs its own OCR, OR when the originating PACER filing already shipped with embedded text. Saying "CourtListener hasn't OCR'd it yet" is wrong; the accurate framing is "CourtListener's `plain_text` was empty / unreliable, so we OCR the PDF ourselves." Applies in user docs, code comments, commit messages, and PR descriptions equally. (This also matters for the latency framing — a PDF that needs OCR is NOT a "wait for CourtListener to catch up" condition; it's a "wait for someone to run OCR on it, possibly us on next sync" condition.)
- **Plain language over jargon.** Comments, docstrings, AGENTS.md, commit messages, PR descriptions, and user-facing docs should describe what the code does in words a non-specialist would understand. Avoid terminology imported from neighboring fields that only loosely applies — e.g., "projection" from relational algebra to describe "the subset of recap_document fields we keep in the local store", or "compaction" / "denormalization" / similar when a plain description works. When a domain term IS the right word (because the code really is implementing that concept, or the reader needs to look it up to understand a library), use it AND a brief in-place gloss the first time it appears. When a term is borrowed loosely, replace it with the literal description. Past offenders that were swept out and should not return: "projection" of recap_documents (now "the subset of each recap_document we store locally" / "cached recap_documents"); "high-water mark" / "watermark" for the polling cutoff (now "last-modified cutoff"); "load-bearing" for rules-that-must-not-be-dropped (now "essential"); "happy path" for the success case (now "success path"); "self-heal" / "self-healing" for cache rebuild (now "automatically rebuild" / "rewrite from fresh data" / "repair"); "agentic" / "agentically" for unattended automation (now "automatic" / "unattended"); "first-class concern" for important matters (now "primary concern"); "DDL" for schema changes (now "schema-changing SQL"); "fan-out" for orphan child rows (now described literally); "column" for a per-model provider store in `model-comparison/SCORECARD.md` (an artifact of the build script's output layout — and the things are rows in the rendered tables, not columns — now "model", matching the Totals table's own axis label; the one surviving "column" there genuinely means a table column); "fixture" borrowed to mean the frozen benchmark-input snapshot (now "snapshot" / "benchmark snapshot", made by `model-comparison/snapshot_benchmark.py`) — but the pytest sense is EXEMPT: the `store` / `make_entry` test fixtures keep that name because it IS the testing library's own term (a reader needs it to follow the tests), the same "domain term that is the right word" carve-out above; only the loose borrow of "fixture" for data that is really just a snapshot is the defect. A specific sub-case to watch: **terms that are artifacts of internal tooling or a build script's output** ("column", a job name, an internal flag) read as self-evident to whoever wrote the tool and as undefined jargon to everyone else — name the real thing the reader cares about (the model, the case, the docket) instead of the tool's word for it. The test is whether a contributor coming into the codebase from a different background would have to stop and search to understand what a term refers to here; when in doubt, prefer the plainer rewrite even if it's a few extra words.
- **Don't make unsupportable claims in comments, docstrings, PR descriptions, or documentation pages (`docs/`, `model-comparison/`).** Base every documented fact on real data or code you actually queried/read in this session — never write an example or characterization first and verify later, and never generalize from memory of a build trace; query the store / read the source / name the specific row, THEN write. Phrasings like "the most-missed appellate deadline", "the foundational civil deadline", "the most common scheduling-order requirement", or generalizations from one observation ("appellate filings carry the party name between 'MOTION' and the verb" — true of the one 4th Cir docket we looked at, not necessarily true of appellate filings as a class) are claims about empirical frequency or community practice the author can't usually back. Replace them with verifiable substance: cite the rule / statute / case that establishes the fact ("FRAP 40", "FRCP 12(a)(1)"), name the specific docket or source where the pattern was observed ("Verbatim from 4th Cir docket 25-4607 (Gholinejad)"), or describe the invariant the code protects (the regex constraint a negative test pins). The reader gets a citation they can follow; nobody is asked to take an unattributable claim on faith. Past offenders that were swept out and should not return: "currently the most-missed appellate deadline" (now: "FRAP 40 — petition for panel rehearing / rehearing en banc"); "the foundational civil deadline" (now: "FRCP 12(a)(1)"); "the bare district-court phrasing" applied to court-neutral text; "Common scheduling-order requirement"; "`presentencing` also a common informal form"; "minute entries" applied to appellate docket entries (the term is district-court vocabulary); "entries like 'CIPA Section 5 notice' don't otherwise trip the regex" (often inaccurate after a sibling change in the same diff); the `model-comparison/SCORECARD.md` minor-deadline examples "briefing responses, status-report dates" (guessed by plausibility, never queried — the store's minor set is actually 61% transcript redaction-request, and dispositive briefing / recurring joint status reports are classed `major`, so the examples named the opposite of the truth); a SCORECARD trio example that named a "CLERK'S NOTICE CONTINUING" entry from memory of the build trace when the real entry was a `Set/Reset Hearing` notice for a different date (now: the actual three entries — stipulation, the order granting it, the `Set/Reset Hearing` notice — verified against the snapshot). **Priorities yes, empirical claims no — including inside LLM prompts.** A prompt can legitimately tell the model how to weight or order its output ("the imposed sentence is the most important fact about the case — include it"; "lead with the operative court and the most important charges or claims"). These are directives about emphasis applied to material the model already sees in the source documents — the model isn't being asked to take a worldly fact on faith, just to prioritize within what's in front of it. What a prompt CANNOT do is assert empirical claims the model couldn't verify from the source documents — don't write "FRAP 40 is the most-missed appellate deadline" or "this is the foundational civil deadline" or "most federal cybercrime cases involve X" into a prompt any more than into a comment, because the model has no way to fact-check the claim and will repeat it to subscribers as documented fact. The test: a *priority directive* ("lead with X if present", "X matters more than Y in the output", "if Z is mentioned, quote the date verbatim") shapes how the model orders / emphasizes what's already in front of it; an *empirical assertion* ("X is the most common", "in practice most courts...", "this is the foundational Y") adds a worldly claim the model can't verify and shouldn't be taught.
- **Fix underlying bugs, never just patch the data.** A manual SQL update or shell command that corrects ONE row of bad state in the local DB doesn't help other users running the same code, doesn't help future data hitting the same bug, and doesn't survive a fresh checkout or re-sync. Every observed bug must result in a code change that prevents the bad state from recurring, even when an immediate manual patch is also applied to unblock the operator. The manual patch is the bridge; the code fix is the destination — both happen, never just the bridge. Example: the us-v-moucka entry #1 stale cached-recap_documents bug surfaced as one row in the local DB whose stored `plain_text` was empty even though CourtListener had it. The bridge was an inline script that rebuilt that one row's stored recap_documents from a fresh CourtListener fetch; the destination was the new staleness detector + automatic cache rebuild in `summary.find_primary_documents` plus `Store.refresh_entry_recap_documents`, so the same bug repairs itself on any other affected docket without operator intervention.
- **Long-running operations (sync, store builds): emit one progress line per case as it starts.** When running `case-calendar sync` or `model-comparison/build_provider_stores.py` interactively for this user, wire up a per-case progress monitor that emits one line as each case begins processing. The shape: `[X/N] <case_slug> (<case_title>) — ETA HH:MM:SS to finish (avg Ns/case)`, where X is the position in the queue, N is the total case count (visible in the script's early-startup log lines — `clearing derived tables for N case(s)` for builds; the `config.yaml` case list for sync), and the rolling ETA = average interval between case starts × cases remaining (including the case just starting, so the projection covers when everything finishes). Use Monitor with `tail -F` of the script's stdout/log piped through grep --line-buffered or an awk that emits one line per match — don't poll. The first case starts the clock with no ETA yet ("first case starting, ETA pending"); subsequent cases each emit one notification with the rolling projection. The user wants to see this signal whenever a sync or store build is running; don't ask permission to set it up, just arm the monitor when you kick off the operation. Two failure modes this rule exists to prevent, both of which have recurred and are NOT acceptable: (1) **Stripping the format.** The `[X/N] … ETA HH:MM:SS (avg Ns/case)` shape is required on EVERY run, including scoped subsets — a `--case` filter, a trimmed/diagnostic config, an extraction-only `--skip-summaries` run — where N is simply the case-count of THIS run (cases × columns for a multi-column build), not the full caseload. Don't drop the counter or the ETA because a run is "small" or "just a diagnostic". (2) **Swallowing the signal.** Every status turn must RELAY the live line(s) with their real numbers; NEVER substitute a content-free acknowledgement ("holding for completion", "waiting", "still running", a bare restatement) for the actual per-case status — that defeats the entire point of arming the monitor. When you deliberately tighten the monitor's filter to cut per-line noise (e.g. excluding per-warning lines so the stream stays readable), you must still surface the accumulating tallies the run exists to collect — warning counts, error counts, cost — by reading them from the log periodically and reporting them, so the user always sees real numbers rather than vacuous holds or silence.
- **Before evaluating, benchmarking, or debugging any LLM, read its actual model card — not your training priors.** When a model's behavior on this project is in question (it's a benchmark candidate, it's misbehaving, you're choosing default sampling parameters, or you're writing a finding about why it scored the way it did), the FIRST step is to fetch the model's official model card / provider documentation and read its stated usage recommendations — sampling parameters, system-prompt guidance, known failure modes — and verify the exact text against the official source (Hugging Face card, provider docs page), not a search-result summary and never from memory. Training-data recall about a model is unreliable and often predates the release; the card frequently documents the exact cause of a problem you'd otherwise misattribute to "the model is just bad at this". The canonical instance: the `qwen3.5:9b` / `deepseek-r1:8b` reasoning-trace runaways were initially written up in `model-comparison/SCORECARD.md` as inherent model instability, when the Qwen3 model card states verbatim "DO NOT use greedy decoding, as it can lead to performance degradation and endless repetitions" and recommends `Temperature=0.6, TopP=0.95, TopK=20` for thinking mode, and the DeepSeek-R1 card recommends temperature 0.5-0.7 "to prevent endless repetitions" — i.e. our `temperature=0` (greedy, pinned for LLM-cache determinism) was the documented trigger, a cause readable in the card but invisible from training priors. A model-behavior claim in docs, a comment, or a commit message that wasn't checked against the current card is an unsupportable claim (see the matching Convention above) — cite the card.

## Python Code Style

- Formatter/linter: **Ruff**
  - All code must be linted and formatted
  - Ruff is version-pinned in `pyproject.toml` under the `lint` extra so CI and contributors share one format opinion. `uv sync --extra lint` installs the pinned version; `uv run ruff check` / `uv run ruff format --check` are what CI runs. When you want to roll forward, bump the pin in `pyproject.toml`, run `uv sync --extra lint`, run `uv run ruff format`, and commit the version bump + reformat in the same PR — never let CI and local drift
  - **ALWAYS run `uv sync --extra lint` before you format or lint, so you're on the project's PINNED ruff — not whatever version `uv run ruff` happens to resolve from your environment.** Ruff's formatter changes its line-wrapping between releases, so an older (or newer) local ruff will report "all formatted" while CI's pinned version reformats the very same file — a green local check that fails CI. Confirm `uv run ruff --version` matches the `lint = ["ruff==X.Y.Z"]` pin in `pyproject.toml` before trusting a local `ruff format`/`ruff check` result or committing formatted code. (This exact drift — local 0.14.8 vs pinned 0.15.13 — slipped an unformatted file past a local check and failed CI in 0.13.0.)
- Type annotations use `TypedDict` for structured results
- Supports all currently supported Python versions
- Modern type annotations across the entire project
  - Always use the latest version of pyright for static type checking. Run with `PYRIGHT_PYTHON_FORCE_VERSION=latest` to override the pyright-python wrapper's pinned release; CI sets this env var on the pyright step
- Testing framework: **pytest**
- Every bit of code should have a test
- Build backend: **hatchling**
- Module-level loggers: `logger = logging.getLogger(__name__)` — one logger per module, named for the module
- Project-defined errors subclass `RuntimeError`, not bare `Exception`, so callers can catch project failures specifically without sweeping in unrelated bugs

## Markdown Style

- All markdown must pass VSCode's default markdownlint config
  - VSCode projects must be configured with `"markdownlint.config": {"MD024": false, "MD004": {"style": "dash"}}` — `MD024` is disabled to allow for proper changelog headings, and `MD004` is pinned to `dash` because the project prefers `- list items` over `* list items` and the default `consistent` rule otherwise picks whichever style appears first in any given file, producing churn when a file's lists get reshuffled. Use `-` followed by a space for every unordered list across the codebase
- **Keep every inline code span on a single physical line, and never leave a `<placeholder>`-style angle-bracket token in body text outside backticks.** When a `` ` ``-delimited span opens on one line and its closing backtick lands on the next, not every Markdown renderer reassembles it — and when the span fails to form, the angle-bracket placeholders that were meant to be literal (`<created>`, `<failed>`, and the like) get parsed as raw HTML tags instead. An unclosed tag such as `<failed>` then mis-nests the DOM that follows it, so every paragraph after that point renders clipped and unstyled — the formatting silently disappears, and markdownlint does NOT catch it (the source is structurally valid). Wrap the surrounding prose rather than the code span, and reserve literal `<...>` for genuinely intended HTML (e.g. a real `<br>`). This bit [docs/webhooks.md](docs/webhooks.md): a line-wrapped `docket alerts: <created> created, <exists> already subscribed, <failed> failed` span broke the rendering of the entire rest of the page. The detector is a quick sweep for lines whose backtick count is odd (a code span that crosses a line boundary) plus a grep for `<word>` tokens sitting outside backticks.
- **Keep every Markdown link's `[text](target)` on a single physical line — never let a line-wrap fall *inside* the `[...]` text.** The docs are published by Jekyll on GitHub Pages, whose default `jekyll-relative-links` plugin is what rewrites in-repo `.md` links to `.html` on the live site. Its link-matching regex works within one line, so a link whose text straddles a newline — `[AI case` on one line, `summaries](case-summaries.md)` on the next — is never matched, and the plugin leaves the raw `.md` target in place, producing a dead link on the published page (the local file renders fine, and markdownlint does NOT flag it because the source is valid Markdown). When prose needs to wrap, put the break BEFORE the `[` or AFTER the `)`, not between the brackets. This bit four pages at once ([docs/local-llms.md](docs/local-llms.md) → case-summaries, [docs/index.md](docs/index.md) → calendars, [docs/subscribing.md](docs/subscribing.md) → public-page and calendars), each a wrapped sentence that split the link text across two lines. The detector is `grep -rPzo '\[[^\]]*\n[^\]]*\]\(' docs/` — any hit is a link whose text crosses a line boundary.
- **Escape a literal `~` (used for "approximately") as `\~` in prose — GitHub-Flavored Markdown treats `~` as a strikethrough delimiter.** GFM renders both `~~text~~` AND single `~text~` as strikethrough, so a line with two or more bare tildes (`~4× ... ~$60`) gets the text between a tilde pair silently struck through on GitHub.com — the rest of the line reads fine, only the spanned run is crossed out, and (like the inline-code case above) markdownlint does NOT flag it because the source is valid. Write `\~4×`, `\~$60`, `\~2000 tokens` etc.; the backslash renders as a literal `~` and can never pair. This bit [model-comparison/SCORECARD.md](model-comparison/SCORECARD.md), where a line of `~`-prefixed approximations struck through "4× what Gemini costs ... Gemini would save". Tildes INSIDE backtick code spans are safe and must stay unescaped. The detector is a grep for `~` outside code spans; any line with two or more is a live strikethrough.
- **No all-caps shouting in human-facing documentation (`docs/`, `model-comparison/`, the top-level README).** Reserve emphasis for `**bold**` or `*italic*`, never a capitalized word — write "the most detailed local extractor", not "the MOST detailed local extractor". Documentation is read by people, and all-caps emphasis reads as shouting. Two things legitimately stay capitalized and are not shouting: (1) genuine acronyms / identifiers (`JSON`, `GPU`, `VRAM`, `CourtListener`, a Modelfile `FROM` / `PARAMETER` directive); and (2) literal tokens — a toggle *value* (`ON` / `OFF`, `think-ON`) or a label the code actually prints (the `RUNAWAY` / `HUNG` / `WARNING` / `TOTAL` / per-entry `DECISION` log markers). AGENTS.md itself is the deliberate exception: it is written *to* an agent, so its emphatic `NOT` / `MUST` / `ONLY` stay — this rule governs the prose a subscriber or contributor reads, not these instructions. The detector is a grep for `\b[A-Z]{2,}\b` outside backticks with the acronym + label allow-list subtracted (the sweep used to clean [model-comparison/SCORECARD.md](model-comparison/SCORECARD.md)).

## One-shot maintenance scripts (`scripts/`)

- **`scripts/reprocess_entries.py <entry_id>...`** — re-run the LLM pipeline against specific stored entries (by CourtListener `entry_id`, not docket position). Builds a synthetic entry from the local store, clears the entry's fingerprint to bypass dedup, and calls `CaseSyncer.process_entry`. Useful after prompt changes when you want to see the new behavior on existing rows. CAUTION: the LLM can allocate fresh `hearing_key`s if it interprets a previously-vague row as multiple specific events — duplicates may need manual cleanup.
- **`scripts/classify_significance.py`** — classifies hearing significance using a focused single-question LLM prompt (no full action extraction, so no risk of the duplicate-key issue above). Default is dry-run on NULL-significance rows; `--all` widens to every row, `--apply` writes verdicts back to the DB.
- **`scripts/heal_proceeding_notes.py`** — deterministic backfill (no LLM, no CourtListener) for hearings whose `notes` regressed to a pre-hearing administrative notice (clerk's notice of Zoom access / courtroom change / scheduling) instead of the record of what happened. Calls `sync.heal_proceeding_notes`, which rebuilds each affected row's notes from its own record source entries (minute entry / transcript / clerk's notes of proceedings held), touching only rows whose notes are empty or an administrative notice. The sync-time MARK_HELD-supersede + dedupe-aware-notes rules only prevent NEW regressions; rows already collapsed in the store (their rich-notes sibling deleted by a dedupe merge) need this sweep because re-running sync can't recover them. Dry-run by default; `--apply` writes back. Only rewrites the `notes` column.
- **`scripts/heal_drifted_keys.py`** — deterministic canonicalization (no LLM, no CourtListener) for hearings AND deadlines left with a `base-<digits>` key-drift suffix (the "Sentencing Lytvynenko 2" class — the cross-record drift the dedupe sweeps collapse, where the survivor historically kept the `-N` key because it carried more `source_entry_ids`). Calls `sync.heal_drifted_keys` (hearings) and `sync.heal_drifted_deadline_keys` (deadlines), which repair ONLY provable-drift rows by two signals: (1) the survivor's `audit_notes` record it absorbed its own suffix-free base AND that base no longer exists → rename `base-N` → `base` (the new row starts with a NULL `m365_event_id` so the next emit re-creates the Graph event cleanly — the old key's gcal/Graph events orphan, the one-time cost of a rename; for hearings the recomputed key-derived title is name-aware per `_fallback_hearing_title`, for deadlines it's the plain humanization); (2) the suffix-free `base` still exists at the same slot (`starts_at_utc` for hearings, `due_at_utc` for deadlines) in the same logical PACER group → delete the `-N` row after folding its `source_entry_ids` onto `base`. A meaningful trailing number (sequential status conferences, trial days, dates, distinct same-day deadlines, `-docket-<id>` records) is NEVER touched — it has neither signal (its base, if any, sits at a different slot and carries no absorption note). The deadline half is NOT the forbidden deterministic same-slot deadline sweep — see the "Deadlines deliberately have NO same-slot dedupe sweep" carve-out in Key Design Decisions. The code fix in `llm.py`/`sync.py` (multi-record group canonicalization for the extractor + the dedupe-merge canonical-survivor preference) prevents NEW drift; this sweep repairs rows already collapsed in the store, which re-running sync can't re-cluster. Dry-run by default; `--apply` writes back. Renaming a key changes its ICS UID + gcal/M365 event id, so subscribers' clients re-create those events once.
- **`scripts/heal_key_derived_titles.py`** — deterministic cleanup (no LLM, no CourtListener) for hearings AND deadlines whose title is still a fallback humanization of the key: a hearing carrying a redundant defendant name on a single-defendant docket ("Sentencing Lytvynenko" → "Sentencing"), and either kind carrying a key abbreviation ("Status Conf" → "Status Conference", "Govt Status Report" → "Government Status Report"). Calls `sync.heal_key_derived_titles`, which rewrites ONLY rows whose CURRENT title is a fallback-derived form (`_title_is_key_derived`, which recognizes the plain humanization plus the name-aware/expanded fallback forms) — an explicit LLM title is never touched — judging single-vs-multi-defendant from each case's own hearing keys (`_case_defendant_names`) so genuine co-defendant dockets keep their names rendered `"<Type> - <Name>"`, and expanding abbreviations via `_KEY_ABBREVIATIONS`. The write-time fallbacks (`CaseSyncer._hearing_fallback_title`, `_fallback_deadline_title`) now produce these clean titles for NEW events; this script repairs the rows stored before that landed. Touches only the `title` column of either table (no key change, no schema change). Dry-run by default; `--apply` writes back. Deadlines get abbreviation expansion only — NOT name-stripping (their keys are freeform, not "type + defendant").

### Deployment scripts (`scripts/`, bash)

These are the operator wrappers for running the production deployment over ssh. They are checked in (no host-specific values are hardcoded) and read every deployment-identifying value from `.env` through a shared loader, so the same scripts work for any operator who fills in their own `.env`.

- **`scripts/_prod-env.sh`** — sourced (not executed) by every deployment script. Resolves the repo-root `.env` from its own location (so the scripts work from any CWD), reads ONLY the five `CC_PROD_*` keys it needs (rather than sourcing the whole secrets file, which would pull every key into the shell and could choke on a value bash can't evaluate), validates that all five are present (exits with a clear error naming the missing one otherwise), and exports `CC_PROD_SSH` = `<user>@<host>` for convenience. The five keys — `CC_PROD_HOST`, `CC_PROD_SSH_USER`, `CC_PROD_APP_DIR`, `CC_PROD_SERVICE` (systemd unit AND unix service account, assumed identical), `CC_PROD_STAGE_DIR` (a dir on prod writable by the ssh user, for staging scp drops) — are deployment-IDENTIFYING, so they live in the gitignored `.env` and are DELIBERATELY NOT mirrored into `.env.example` (the loader's header documents them instead, since `.env.example` is the fresh-checkout template and these are one operator's host details). This is the pattern to follow for any future host-specific value: externalize to `.env`, document in the script, leave out of `.env.example`.
- **`scripts/sync-prod`** — run `case-calendar sync` on prod over ssh, streaming output back to the terminal (`PYTHONUNBUFFERED=1` + `env` wrapper, since `sudo` strips env and python block-buffers a non-TTY stdout). Forwards extra args. Doesn't stop the service — sync and serve coexist under WAL journaling + `busy_timeout`.
- **`scripts/reconcile-prod`** — run `case-calendar reconcile` on prod over ssh, same shape as `sync-prod` (same `_prod-env.sh` loader, ssh-agent check, `PYTHONUNBUFFERED=1` + `sudo -u … env` wrapper). Runs the cheap placeholder re-check instead of a full sync — re-fetches only entries matching `sync.is_pending_enrichment` by id to pick up the upstream enrichment the webhook path can't see (see the placeholder-reconcile Key Design Decision and [CourtListener issue #7423](https://github.com/freelawproject/courtlistener/issues/7423)). Forwards extra args (`--case`, `--days`). Doesn't stop the service — reconcile and serve coexist under WAL + `busy_timeout`.
- **`scripts/sync-via-prod`** — push local `config.yaml` to prod, sync there (prod's CourtListener token does the work), restart the service so the running serve process picks up the new config, then pull the resulting store back to dev via SQLite's online-backup API. Config flows dev → prod, DB flows prod → dev (intentional). `--prune` runs `prune --apply` between code-pull and sync; all other args forward to `sync`. Backs up the local store before the swap.
- **`scripts/pull-prod-db`** — copy prod's `case-calendar.sqlite` down to dev (DB only, online-backup snapshot, no service downtime). Backs up the local store first.
- **`scripts/push-db-to-prod`** — the inverse: upload local store + `.env` + `config.yaml` to prod, pull code, re-emit, restart. Prompts for confirmation (it overwrites prod state). Stages in `CC_PROD_STAGE_DIR` then `cp`s into place (preserving destination ownership/perms a `mv` would clobber) and removes the staged `.env` afterward so secrets don't linger.
- **`scripts/upgrade-prod`** — apt upgrades, push local `.env`, pull code, `uv sync`, re-emit, restart, and flag if a reboot is required. Leaves the prod DB and `config.yaml` alone.

## Model comparison (`model-comparison/`)

- **`model-comparison/build_provider_stores.py`** — builds one complete store per LLM provider from the SAME cached CourtListener data, for a head-to-head comparison (cost AND output) before picking one to push to prod. Copies the warm store per provider into `data/provider-stores/<provider>/<extraction-model>/`, clears only the LLM-derived tables (hearings / deadlines / case_summaries — the `entries` rows are left untouched so the result is prod-faithful), then replays the real pipeline (`CaseSyncer._handle_entry` per cached entry + the verify/dedupe sweeps + `summary.refresh_stale`) pinned to that provider, and renders each one's ICS + `index.html` into `<provider>/<extraction-model>/out/` (push-ids stripped, so nothing reaches a real Google / M365 calendar). A shared, thread-safe response cache (CourtListener `_get`/`_post` + `pdf.extract_text` by document id) guarantees each CourtListener fetch happens at most once TOTAL across all providers — never per-provider — and the report prints the one-time CourtListener call count + peak per-minute/hour/day rate alongside per-provider/per-track token cost. Providers build in parallel (thread-local provider selection; `--no-parallel` to serialize), per-provider failures are isolated, and `--validate` diffs each store's row counts against the current prod store. `--fake` exercises the whole flow with synthetic tokens and no spend. Read-only on the live store (operates on copies). A SECOND, PERSISTENT cache — `_LLMCache`, a content-addressed SQLite sidecar at `data/llm-cache.sqlite` (`--llm-cache-path` to relocate, `--no-llm-cache` to disable, ignored under `--fake`) — wraps the real `providers._dispatch_llm_call` and serves identical requests across runs. Its key is the FULL resolved request (provider, model, system prompt, user message, `max_tokens`, `json_mode`, `temperature`); since every domain call pins `temperature=0`, a cached response is byte-identical to a fresh call, so there's no separate version tag — change the model or any prompt and the key changes and the call re-runs live, while every UNCHANGED call replays for free. **Ollama defaults to greedy like the hosted providers** (it forwards the domain `temperature=0`), so its default key needs nothing extra. The opt-in sampling knobs are the exception: for `provider=="ollama"` the key folds in `(_ollama_seed(), _ollama_temperature_override())` ONLY when one is set, so a seeded/overridden run re-keys (and an identical seed replays its stored response) while the common greedy run reuses its existing cache entry unchanged. Reproducible ollama benchmarking under the opt-in sampling wants `OLLAMA_SEED` set, but note even then the cache only stores a frozen response — a FRESH ollama call matches it byte-for-byte for schema-constrained extraction on a non-reasoning model, not necessarily for summaries or a reasoning model's output (the GPU-nondeterminism caveat in the Ollama-sampling-and-seed design note). The `model` slice of the key is the RESOLVED model, not the dispatch argument: `_LLMCache._effective_model` mirrors the per-provider `model or os.environ['LLM_MODEL'] or default` resolution exactly, so a `model=None` call keys on the model the request will actually carry. This is the cache's soundness pivot — keying on the unresolved `None` arg would let two `model=None` calls under different `LLM_MODEL` values collide on one response even though they build different requests. (In the harness `model` is always concrete at the cache boundary and `LLM_MODEL` is popped from the env, so the gap is closed by construction today — but the cache mirrors the resolution rather than silently depending on that invariant.) This is the developer-iteration cost lever: a single-track prompt tweak re-bills ONLY that track's calls (a summary-only edit drops from a full-caseload rebuild to just the \~34 summary calls), with no flag — the request hashes do the scoping automatically. A cache HIT skips `usage.record`, so the cost report's TOTAL already reflects only the live (miss) calls; `_LLMCache.log_summary` prints per-column + total hit/miss counts at the end. Errors are never cached (an exception propagates without a store, so a transient failure can't poison the cache); reads/writes are serialized under a lock and the model call on a miss runs outside it. It is distinct from the in-process CourtListener / PDF caches above, which dedupe FETCHES within one run; `_LLMCache` dedupes LLM CALLS across runs — the bigger lever, because the LLM calls are the entire dollar cost of a build. Delete the sidecar file to invalidate every cached response. For sub-track iteration (tighter than per-track), pair it with `config.dev.yaml` — a checked-in dev config covering only the \~18 cases that have driven a documented regression in one of the LLM-driven layers (each case annotated with the failure mode it pins), pointed at the prod store for warm replay — and reserve the full-caseload build (`-c config.yaml`) as the final pre-commit gate. `config.dev.yaml` works with any command (`case-calendar -c config.dev.yaml …` or `build_provider_stores.py --config config.dev.yaml`); it renders into `out/dev/` and sets `ensure_docket_alerts: false` so a dev run never touches the prod deployment's CourtListener alert subscriptions. The developer workflow that ties the LLM cache, the dev config, and the deployment scripts together is written up in [docs/development.md](docs/development.md). **Reproducible benchmarking:** `--source <store>` overrides `store_path` to replay from a specific store, and `--frozen` forbids ALL live data access — any CourtListener `_get`/`_post` or `pdf.fetch_pdf_bytes` cache miss raises `FrozenSnapshotError` instead of fetching — so a run provably uses only that store's data and can't silently drift. Pair them with a snapshot from `snapshot_benchmark.py` (next bullet). The per-entry extractor actions are captured to a CSV with `--entry-actions-csv model-comparison/model_actions.csv` (the committed comparison artifact `score_models.py` reads), recorded on every `extract_actions` call regardless of LLM-cache hit, so a warm-cache rebuild still produces a complete CSV.
- **While a build/sweep is running, run ad-hoc commands with `uv run --no-sync` — a competing `uv sync` thrashes the shared venv.** `build_provider_stores.py` launches a fresh `uv run` per provider/model, each of which syncs the project venv to the DEFAULT (core) dependency set. If you concurrently run something that targets a DIFFERENT extra-set — `uv sync --extra lint`, or `uv run pytest` (which pulls in the `test` extra) — the two invocations fight over the one venv, each adding/removing the differing extras, and a sync caught mid-operation can leave even a CORE dependency transiently unimportable: a spurious `ModuleNotFoundError` on a package `uv pip list` swears is installed. Observed exactly this way — `json_repair` and `pypdf` (both core deps) failing in a full-suite run during a live sweep, while `uv run --no-sync python -c "import json_repair, pypdf"` succeeded, proving the venv was intact and the failure was the race. So during a sweep: score and run one-off scripts with `uv run --no-sync` (the venv already holds the core deps the benchmark needs), and DEFER the full `uv sync --extra test --extra lint` + pytest/ruff gate until the sweep finishes and nothing else is calling `uv run`. Note that `uv sync --extra lint` (the pinned-ruff step under [Python Code Style](#python-code-style)) silently DROPS the `test` extra, so running it mid-sweep both starts the thrash and uninstalls pytest — another reason to leave the venv alone until the sweep is done.
- **`model-comparison/snapshot_benchmark.py`** — builds the benchmark INPUT snapshot: one immutable, dated, FULL-TEXT file so a model/prompt comparison stays reproducible while the live cases keep moving. Copies the current store (default: `store_path` from config) into a read-only `model-comparison/snapshots/benchmark-store.sqlite` via SQLite's online-backup API, CLEARS the model-output tables (hearings / deadlines / case_summaries — same set as `DERIVED_TABLES`) and VACUUMs so the shared snapshot is INPUT-ONLY (the benchmark rebuilds those; leaving a model's output in a shared file would let anyone peek, defeating the BLIND ground-truth scoring), then re-paginates each benchmark docket's full `docket-entries` feed from the v4 API (no `modified_after` cutoff) and overwrites EVERY entry's `description` / `short_description` / `recap_documents` with its COMPLETE text — no fingerprint-only stubs. The full text is what makes the snapshot serve as ground truth: the operational store stubs entries that fail the regex pre-filter, and a date hidden in a stubbed entry would be invisible to BOTH the models AND the human reading `build_scoring_page.py`, making the regex stage's own recall unmeasurable; with full text a regex-dropped date becomes a scoreable, provider-independent miss. (`--case <id>` does an in-place surgical refresh of one case in the existing snapshot — to pick up a new filing without re-pulling the whole benchmark.) Writes a sibling `benchmark-store.manifest.json` (snapshot date, source, caseload, per-table row counts, cleared tables, the snapshot's sha256). **The SQLite file is committed via Git LFS** (`.gitattributes` tracks `model-comparison/snapshots/*.sqlite`): contributors run `git lfs pull` once to fetch it; the committed manifest is a normal (non-LFS) file. Refuses to overwrite an existing snapshot unless `--force`. Used with `build_provider_stores.py --source … --frozen`: the snapshot freezes the data the models extract from, `ground_truth.csv` freezes the truth they're scored against, and only the model/prompt under test varies. The full workflow is in [model-comparison/README.md](model-comparison/README.md). (Formerly two scripts — a stubbed `snapshot_benchmark_store.py` and a `fetch_complete_benchmark.py` that added full text — merged into this one once the full-text snapshot became the only one used.)
- **`model-comparison/build_scoring_page.py`** — generates ONE self-contained offline HTML page (`ground_truth_scoring.html`) for BLIND human ground-truth scoring. Every benchmark docket entry is a card showing its COMPLETE text (description + extracted PDF text — exactly what the extractor saw) + clickable per-document links, beside the eight action-count boxes the extractor itself emits (hearings scheduled / rescheduled / held / cancelled, deadlines set / rescheduled / met-filed / cancelled). The maintainer reads top-to-bottom, types the counts, ticks "bad OCR" on unreadable entries, and downloads `ground_truth.csv`; counts autosave to localStorage keyed by `entry_id`. No model output is ever shown, so scoring stays blind. Content-less entries (empty clerk's notices whose only document isn't on RECAP — nothing for human or model to read) are hidden with a printed per-case count; entries sort chronologically by filing date so paperless minute entries land in place. Reads the `snapshot_benchmark.py` snapshot. A help block documents the scoring conventions (one slot is one hearing; a continuance is a reschedule; dark trial days are non-events; an amended minute entry supersedes the original; count both major and minor deadlines).
- **`model-comparison/score_models.py`** — the deterministic scorer (no model in the loop). Joins the human `ground_truth.csv` with `model_actions.csv` (per-entry extractor action counts per provider, captured by `build_provider_stores.py --entry-actions-csv`) on `entry_id` and reports, per provider: per-entry deviation (|model − human| summed over the 8 categories, split into OVER = duplicate keys / hallucination and UNDER = missed); per-docket-aggregate deviation (the same counts summed per logical docket first — robust to the model and human pinning an action to a slightly different entry); and regex-stage misses (entries the human counted > 0 but EVERY provider emitted 0 — `is_extractable` dropped them before any LLM ran, the provider-independent recall gap the over-inclusive-regex claim is measured against). `bad_ocr` entries are excluded; only `reviewed` rows score unless `--include-unreviewed`.
- **`model-comparison/SCORECARD.md`** — written analysis of the scoring run that backs the current default-provider choice (see the Key Design Decisions note on `LLM_PROVIDER` precedence). Re-read this when changing default-provider selection or moving a provider between the extractor and summary tracks.

## Testing

```bash
uv sync --extra test
uv run pytest                                    # full suite (\~35s, \~1760 tests)
uv run pytest tests/test_sync_integration.py    # one file
uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing
```

### Testing philosophy

- **Every change ships with the test that proves it.** Adding code → add the test that exercises the new branch. Fixing a bug → add the test that fails on the old code and passes on the new (so the regression can't sneak back). Changing behavior → update the tests that asserted the old behavior. Deleting code → delete the tests that covered it. The diff that lands a behavior change without a test change is the diff that breaks silently three months from now. CI runs the full suite with `--cov-branch` on every push to `main` and every pull request and fails the build under 90% project coverage (see [`.github/workflows/ci.yml`](.github/workflows/ci.yml)); 90% is a regression floor, not the bar to clear — the local invariant is stronger: no commit should reduce coverage at the module level. Run `uv run pytest --cov=case_calendar --cov-branch --cov-report=term-missing` before declaring a change done and confirm the touched modules either held or gained coverage.
- **Hermetic by construction.** No real HTTP, no real LLM, no real Google / Microsoft Graph, no real keyring. Every external dependency has a stub in `tests/conftest.py` or is monkey-patched per test. The `_no_real_token` autouse fixture strips any inherited `*_API_KEY` env vars so a developer with credentials in their shell can't accidentally hit a real provider.
- **Per-provider SDK calls are mocked at the SDK boundary, not the network boundary.** `tests/test_llm.py` injects fake `anthropic` / `openai` / `google.genai` modules into `sys.modules`; `tests/test_m365.py` bypasses `M365CalendarSync.__init__` via `__new__` and feeds a `MagicMock` Graph client with `AsyncMock`-backed `post` / `get` / `patch` / `delete`. This keeps tests fast and lets us assert on the call shape (model kwargs, JSON-mode toggles, extended-property writes) without needing real SDK fixtures.
- **Branch coverage, not just line coverage.** Run with `--cov-branch`. The suite holds at ≥96% branch coverage and every module is at ≥90%. New code should land with the test that drove it; the local invariant is "if a branch can fire, exercise it."
- **Cover error paths, not just success paths.** For each external integration (HTTP 4xx/5xx, 404 on patch, malformed JSON, missing required fields, OAuth token absent), test that the code falls open or fails fast as documented in [Key Design Decisions](agent-docs/DESIGN-DECISIONS.md) rather than crashing. The 5xx-fail-open semantics, the `transaction_id` retry-dedup, and the silent webhook ack on emit failure are all enforced by tests.
- **Webhook integration tests boot a real server.** `tests/test_serve.py` starts an actual `ThreadingHTTPServer` on an ephemeral port and POSTs JSON to it. This catches routing / header / body-parse regressions that an HTTP-stubbed test would miss (411 on missing Content-Length, 413 on oversized payloads, 500 from inside the process-locked critical section).
- **Test files mirror the modules they cover** (`tests/test_store.py` ↔ `case_calendar/store.py`, etc.) so a coverage gap is one file away from its fix. Cross-module integration lives in `tests/test_sync_integration.py` and `tests/test_serve.py`.
- **Test the behavior, not the implementation.** Tests assert on what the renderer writes / what the store persists / what the LLM-shaped call kwargs are, not on which private helper was called. Refactors that preserve behavior shouldn't need test edits; refactors that change behavior should.
- **Unreachable defensive code is a test smell.** If a branch can't be exercised, either delete it or fix the upstream to make it reachable — don't paper over with `# pragma: no cover`. The `extractor._entry_text` filter-empty-blobs change and the `m365._to_local_dt` deletion both came from the coverage report surfacing dead code.
- **No real `sleep` in tests.** Where the production path uses `time.sleep` (CourtListener retry/backoff) or `threading.Timer` (webhook debounce), the tests monkey-patch the sleep / Timer so the test runs at memory speed. Burning wall-clock seconds during pytest is a defect.
- **Cross-check summaries against the calendar tables as an audit pass.** The unit + integration suite catches regressions within each layer, but the LLM-driven layers (extractor, verify pass, summary) can drift in ways no unit test foresees — the model can store a hearing on the wrong key, estimate a date the order didn't fix, or fail to mark a continuance — and each of those bugs is consistent with its layer's contracts in isolation. Periodically dump every case_summaries row alongside its case's hearings + deadlines and read the prose against the structured rows: dates mentioned in the summary should map to existing rows, status flags should agree with how the prose characterizes the proceeding, and every date in the prose should be one the COURT set (not an LLM estimate of an unknown future). This audit surfaced the duplicate same-slot hearing, the conditional deadline that was being stored as a guessed calendar date, and the trial row falsely flipped to 'held' by the date-only auto-held sweep — three different bugs in three different layers, all visible only when you put the prose and the calendar side-by-side. Run this any time the LLM stack is touched (prompt revisions, model upgrades, provider switch).

The suite is hermetic — no real network calls, no real LLM, no real Google API. `tests/conftest.py` provides:

- `store` fixture — fresh tmp-path SQLite store per test
- `make_entry`, `make_recap_doc` factories — synthesize CourtListener-shaped dicts
- `FakeCourtListener` — drop-in for `CourtListener` that records calls and returns canned data
- `_no_real_token` autouse — strips real `*_API_KEY` env vars from inherited environments
