# Model comparison (`model-comparison/`)

Agent-facing reference for the comparison tooling's internals and operational
gotchas. For the reproduction method and step-by-step workflow, see
[README.md](README.md); the numbers behind the default-provider choice are in
[SCORECARD.md](SCORECARD.md).

- **`model-comparison/build_provider_stores.py`** — builds one complete store per LLM provider from the SAME cached CourtListener data, for a head-to-head comparison (cost AND output) before picking one to push to prod. Copies the warm store per provider into `data/provider-stores/<provider>/<extraction-model>/`, clears only the LLM-derived tables (hearings / deadlines / case_summaries — the `entries` rows are left untouched so the result is prod-faithful), then replays the real pipeline (`CaseSyncer._handle_entry` per cached entry + the verify/dedupe sweeps + `summary.refresh_stale`) pinned to that provider, and renders each one's ICS + `index.html` into `<provider>/<extraction-model>/out/` (push-ids stripped, so nothing reaches a real Google / M365 calendar). A shared, thread-safe response cache (CourtListener `_get`/`_post` + `pdf.extract_text` by document id) guarantees each CourtListener fetch happens at most once TOTAL across all providers — never per-provider — and the report prints the one-time CourtListener call count + peak per-minute/hour/day rate alongside per-provider/per-track token cost. Providers build in parallel (thread-local provider selection; `--no-parallel` to serialize), per-provider failures are isolated, and `--validate` diffs each store's row counts against the current prod store. `--fake` exercises the whole flow with synthetic tokens and no spend. Read-only on the live store (operates on copies). A SECOND, PERSISTENT cache — `_LLMCache`, a content-addressed SQLite sidecar at `data/llm-cache.sqlite` (`--llm-cache-path` to relocate, `--no-llm-cache` to disable, ignored under `--fake`) — wraps the real `providers._dispatch_llm_call` and serves identical requests across runs. Its key is the FULL resolved request (provider, model, system prompt, user message, `max_tokens`, `json_mode`, `temperature`); since every domain call pins `temperature=0`, a cached response is byte-identical to a fresh call, so there's no separate version tag — change the model or any prompt and the key changes and the call re-runs live, while every UNCHANGED call replays for free. **Ollama defaults to greedy like the hosted providers** (it forwards the domain `temperature=0`), so its default key needs nothing extra. The opt-in sampling knobs are the exception: for `provider=="ollama"` the key folds in `(_ollama_seed(), _ollama_temperature_override())` ONLY when one is set, so a seeded/overridden run re-keys (and an identical seed replays its stored response) while the common greedy run reuses its existing cache entry unchanged. Reproducible ollama benchmarking under the opt-in sampling wants `OLLAMA_SEED` set, but note even then the cache only stores a frozen response — a FRESH ollama call matches it byte-for-byte for schema-constrained extraction on a non-reasoning model, not necessarily for summaries or a reasoning model's output (the GPU-nondeterminism caveat in the [Ollama-sampling-and-seed design note](../agent-docs/DESIGN-DECISIONS.md#ollama-sampling-and-seed-greedy-temperature0-is-the-default-scorecard-valid-in-spec-sampling-is-an-opt-in-escape-hatch-not-the-default)). The `model` slice of the key is the RESOLVED model, not the dispatch argument: `_LLMCache._effective_model` mirrors the per-provider `model or os.environ['LLM_MODEL'] or default` resolution exactly, so a `model=None` call keys on the model the request will actually carry. This is the cache's soundness pivot — keying on the unresolved `None` arg would let two `model=None` calls under different `LLM_MODEL` values collide on one response even though they build different requests. (In the harness `model` is always concrete at the cache boundary and `LLM_MODEL` is popped from the env, so the gap is closed by construction today — but the cache mirrors the resolution rather than silently depending on that invariant.) This is the developer-iteration cost lever: a single-track prompt tweak re-bills ONLY that track's calls (a summary-only edit drops from a full-caseload rebuild to just the \~34 summary calls), with no flag — the request hashes do the scoping automatically. A cache HIT skips `usage.record`, so the cost report's TOTAL already reflects only the live (miss) calls; `_LLMCache.log_summary` prints per-column + total hit/miss counts at the end. Errors are never cached (an exception propagates without a store, so a transient failure can't poison the cache); reads/writes are serialized under a lock and the model call on a miss runs outside it. It is distinct from the in-process CourtListener / PDF caches above, which dedupe FETCHES within one run; `_LLMCache` dedupes LLM CALLS across runs — the bigger lever, because the LLM calls are the entire dollar cost of a build. Delete the sidecar file to invalidate every cached response. For sub-track iteration (tighter than per-track), pair it with `config.dev.yaml` — a checked-in dev config covering only the \~18 cases that have driven a documented regression in one of the LLM-driven layers (each case annotated with the failure mode it pins), pointed at the prod store for warm replay — and reserve the full-caseload build (`-c config.yaml`) as the final pre-commit gate. `config.dev.yaml` works with any command (`case-calendar -c config.dev.yaml …` or `build_provider_stores.py --config config.dev.yaml`); it renders into `out/dev/` and sets `ensure_docket_alerts: false` so a dev run never touches the prod deployment's CourtListener alert subscriptions. The developer workflow that ties the LLM cache, the dev config, and the deployment scripts together is written up in [docs/development.md](../docs/development.md). **Reproducible benchmarking:** `--source <store>` overrides `store_path` to replay from a specific store, and `--frozen` forbids ALL live data access — any CourtListener `_get`/`_post` or `pdf.fetch_pdf_bytes` cache miss raises `FrozenSnapshotError` instead of fetching — so a run provably uses only that store's data and can't silently drift. Pair them with a snapshot from `snapshot_benchmark.py` (next bullet). The per-entry extractor actions are captured to a CSV with `--entry-actions-csv model-comparison/model_actions.csv` (the committed comparison artifact `score_models.py` reads), recorded on every `extract_actions` call regardless of LLM-cache hit, so a warm-cache rebuild still produces a complete CSV.
- **While a build/sweep is running, run ad-hoc commands with `uv run --no-sync` — a competing `uv sync` thrashes the shared venv.** `build_provider_stores.py` launches a fresh `uv run` per provider/model, each of which syncs the project venv to the DEFAULT (core) dependency set. If you concurrently run something that targets a DIFFERENT extra-set — `uv sync --extra lint`, or `uv run pytest` (which pulls in the `test` extra) — the two invocations fight over the one venv, each adding/removing the differing extras, and a sync caught mid-operation can leave even a CORE dependency transiently unimportable: a spurious `ModuleNotFoundError` on a package `uv pip list` swears is installed. Observed exactly this way — `json_repair` and `pypdf` (both core deps) failing in a full-suite run during a live sweep, while `uv run --no-sync python -c "import json_repair, pypdf"` succeeded, proving the venv was intact and the failure was the race. So during a sweep: score and run one-off scripts with `uv run --no-sync` (the venv already holds the core deps the benchmark needs), and DEFER the full `uv sync --extra test --extra lint` + pytest/ruff gate until the sweep finishes and nothing else is calling `uv run`. Note that `uv sync --extra lint` (the pinned-ruff step under [Python Code Style](../AGENTS.md#python-code-style)) silently DROPS the `test` extra, so running it mid-sweep both starts the thrash and uninstalls pytest — another reason to leave the venv alone until the sweep is done.
- **`model-comparison/snapshot_benchmark.py`** — builds the benchmark INPUT snapshot: one immutable, dated, FULL-TEXT file so a model/prompt comparison stays reproducible while the live cases keep moving. Copies the current store (default: `store_path` from config) into a read-only `model-comparison/snapshots/benchmark-store.sqlite` via SQLite's online-backup API, CLEARS the model-output tables (hearings / deadlines / case_summaries — same set as `DERIVED_TABLES`) and VACUUMs so the shared snapshot is INPUT-ONLY (the benchmark rebuilds those; leaving a model's output in a shared file would let anyone peek, defeating the BLIND ground-truth scoring), then re-paginates each benchmark docket's full `docket-entries` feed from the v4 API (no `modified_after` cutoff) and overwrites EVERY entry's `description` / `short_description` / `recap_documents` with its COMPLETE text — no fingerprint-only stubs. The full text is what makes the snapshot serve as ground truth: the operational store stubs entries that fail the regex pre-filter, and a date hidden in a stubbed entry would be invisible to BOTH the models AND the human reading `build_scoring_page.py`, making the regex stage's own recall unmeasurable; with full text a regex-dropped date becomes a scoreable, provider-independent miss. (`--case <id>` does an in-place surgical refresh of one case in the existing snapshot — to pick up a new filing without re-pulling the whole benchmark.) Writes a sibling `benchmark-store.manifest.json` (snapshot date, source, caseload, per-table row counts, cleared tables, the snapshot's sha256). **The SQLite file is committed via Git LFS** (`.gitattributes` tracks `model-comparison/snapshots/*.sqlite`): contributors run `git lfs pull` once to fetch it; the committed manifest is a normal (non-LFS) file. Refuses to overwrite an existing snapshot unless `--force`. Used with `build_provider_stores.py --source … --frozen`: the snapshot freezes the data the models extract from, `ground_truth.csv` freezes the truth they're scored against, and only the model/prompt under test varies. The full workflow is in [README.md](README.md). (Formerly two scripts — a stubbed `snapshot_benchmark_store.py` and a `fetch_complete_benchmark.py` that added full text — merged into this one once the full-text snapshot became the only one used.)
- **`model-comparison/build_scoring_page.py`** — generates ONE self-contained offline HTML page (`ground_truth_scoring.html`) for BLIND human ground-truth scoring. Every benchmark docket entry is a card showing its COMPLETE text (description + extracted PDF text — exactly what the extractor saw) + clickable per-document links, beside the eight action-count boxes the extractor itself emits (hearings scheduled / rescheduled / held / cancelled, deadlines set / rescheduled / met-filed / cancelled). The maintainer reads top-to-bottom, types the counts, ticks "bad OCR" on unreadable entries, and downloads `ground_truth.csv`; counts autosave to localStorage keyed by `entry_id`. No model output is ever shown, so scoring stays blind. Content-less entries (empty clerk's notices whose only document isn't on RECAP — nothing for human or model to read) are hidden with a printed per-case count; entries sort chronologically by filing date so paperless minute entries land in place. Reads the `snapshot_benchmark.py` snapshot. A help block documents the scoring conventions (one slot is one hearing; a continuance is a reschedule; dark trial days are non-events; an amended minute entry supersedes the original; count both major and minor deadlines).
- **`model-comparison/score_models.py`** — the deterministic scorer (no model in the loop). Joins the human `ground_truth.csv` with `model_actions.csv` (per-entry extractor action counts per provider, captured by `build_provider_stores.py --entry-actions-csv`) on `entry_id` and reports, per provider: per-entry deviation (|model − human| summed over the 8 categories, split into OVER = duplicate keys / hallucination and UNDER = missed); per-docket-aggregate deviation (the same counts summed per logical docket first — robust to the model and human pinning an action to a slightly different entry); and regex-stage misses (entries the human counted > 0 but EVERY provider emitted 0 — `is_extractable` dropped them before any LLM ran, the provider-independent recall gap the over-inclusive-regex claim is measured against). `bad_ocr` entries are excluded; only `reviewed` rows score unless `--include-unreviewed`.
- **`model-comparison/SCORECARD.md`** — written analysis of the scoring run that backs the current default-provider choice (see the Key Design Decisions note on `LLM_PROVIDER` precedence). Re-read this when changing default-provider selection or moving a provider between the extractor and summary tracks.
